1. The Gateway Complication

I write this as I prepare to make a key feature request from the podman community BEFORE releasing the first version of the Recipe Bottle Workbench.

Prototypes of this vision have worked quite well, but ultimately only for bespoke Bottle Containers that I construct myself with custom networking (i.e. dhclient, IP address overrides, etc).

In short, I’m asking for podman to add something like a podman network connect feature flag perhaps called --as-gateway that causes podman to attach a Container expressly as the gateway for the network in question. This will let me run stock Bottle Containers without elevated privileges enabling after-boot redesignation of nameservers or late ip route additions. The dream is that such Containers do not even know they are operating in a security controlled regime: as long as they stay 'within their lane' and make only appropriate DNS requests that lead to permitted IP address use, their work can go forward.

After quite a lot of frustrating experiment, my favorite AIs eventually divulged that podman has been moving away from the CNI networking backend in favor of a 'netavark' implementation. As primarily an embedded guy, I have no sophistication in such things, but I do find it hopeful that the simpler 'netavark' approach might be easier to install the --as-gateway flag. Maybe the vision of podman is compatibile with my needs as a small developer.

More soon I hope!

2. The Recipe Bottle Workbench Vision

Figure 1. RBM System Architecture

Maintaining build and service environments is a headache for the small development organization. Containers are an amazing tool for controlling package version constellations. Necessary, but insufficient; I find that I need more redundancy, security, and control for my build setups.

The Recipe Bottle Workbench is my answer for filling in the gaps. The vision is simple: how can a User run only a few apps natively on a Workstation to coordinate and arrange Containers in sophisticated yet safe ways for me and my customer’s work?

Recipe Bottle Workbench uses only git, make, curl, yq and a Container Runtime (hopefully podman) natively to produce a safe and controlled space for development involving Containers heavily. Recipe Bottle Workbench itself is largely a set of Makefiles and bash scripts designed to be easily incorporated into arbitrary projects via git subtree graft. This is purely an open-source undertaking.

2.1. Part One: Image Management

The Recipe Bottle Workbench streamlines Container Image creation and maintenance through GitHub Actions and the Container Registry. With a configured personal access token, the Build Service:

  • Constructs Images in clean, isolated environments

  • Generates software bills of material for compliance

  • Maintains build transcripts and commit references in the user’s git repository

  • Stores validated Images in the User’s Container Registry

This approach ensures reproducible builds while minimizing dependencies on local resources and their configurations. The system uses only curl, jq, and makefiles to orchestrate remote Image construction, enabling small teams to maintain enterprise-grade Image management practices.

2.2. Part Two: Bottle Service Orchestration

For development services requiring internet and/ or IP connectivity at the Workstation, Recipe Bottle Workbench then orchestrates startup and configuration of Bottle Services, which are comprised of a Sentry Container and a Bottle Container operating in tandem.

Presuming a solution to the Gateway Complication, the Recipe Bottle Workbench allows any Image providing or using network services to function as a Bottle Container. Interposing the Sentry Container as the gateway means the Bottle Container won’t even know a security layer is in place.

The Sentry Container thus sets up a potentially sophisticated set of network security safeguards that prevent malicious or compromized Bottle Containers from exfiltrating the User’s assets. Through configuration of deeply mature tools iptables and dnsmasq, the Sentry Container prevents such illegal accesses.

These services support two key deployment patterns:

  • The Sessile Bottle Service: Persistent service Bottle Containers that provide stable, long-running development infrastructure

  • The Agile Bottle Service: Ephemeral tool Bottle Containers that execute specific operations and terminate

By providing these controlled yet accessible tools, the Recipe Bottle Workbench enables small development teams to maintain proper container hygiene throughout their workflow - from initial building through deployment and eventual cleanup. This empowers organizations to leverage the expanding ecosystem of containerized development tools without requiring specialized DevOps expertise.

3. Definitions

Bottle Service

A complete service instance consisting of one Bottle Container and one Sentry Container working together to provide secure functionality.

Sessile Bottle Service

A Bottle Service that maintains a persistent Bottle Container running continuously to provide long-term network services. These services maintain state and provide stable network ports while enforcing all Sentry Container security constraints.

Agile Bottle Service

A Bottle Service that creates temporary Bottle Containers for a single operation and then immediately removes them upon completion. This pattern suits toolchain operations that primarily operate through Volume Mounts with minimal network requirements.

User

An individual operating the Recipe Bottle Workbench to run containerized services securely on their Workstation.

Workstation

The local computing environment where the User runs the Recipe Bottle Workbench. This environment hosts all Bottle Services and their associated networks while providing isolation from other system resources.

Enclave Network

An isolated network connecting a Bottle Container to its Sentry Container, using the {rbn_enclave_base_ip}/{rbn_enclave_netmask} for addressing. This network enforces strict security policies and provides the only communication path for the Bottle Container.

Transit Network

The network interface that connects a Sentry Container to the Workstation’s network resources. This network enables controlled external communication while maintaining security boundaries.

Sentry Container

A privileged container that enforces network security policies for a Bottle Service. It connects to both Transit Network and Enclave Network, controlling all network traffic and providing DNS services.

Bottle Container

A container that runs the actual service functionality, connected only to the Enclave Network and configured with specific Volume Mounts. All external communication flows through its associated Sentry Container.

Recipe Bottle Workbench

The complete Recipe Bottle system, enabling secure deployment and management of containerized services through network isolation and controlled resource access.

Container

A lightweight, standalone executable package containing all necessary components to run a service.

Volume Mount

A configuration that allows Containers to access specified portions of the host filesystem.

Build Service

A system managing container creation through Build Workflows and the Container Registry, enabling Users to build containers with complete traceability.

Build History

A permanent record maintained by the Build Service for each container build, including build transcripts and commit references that verify container origins.

Container Registry

A managed repository where the Build Service stores validated containers, providing versioned images for use by Bottle Services.

Build Workflow

The automated process within the Build Service that standardizes how Users build and publish containers while maintaining security controls.

Container Runtime

Application capable of running Containers on the Workstation. Early experiments with docker led to disappointing results concerning interlacing host networking and internal networking. At the time of this writing, podman does not yet have the features for full Bottle Service provisions.
