==== Decision

Cloud Build configuration files for trigger-based builds use `cloudbuild.json` format,
generated programmatically via `jq`.
YAML is not used.

==== Rationale

JSON is the Cloud Build REST API's native wire format.
The `projects.builds.create` and `projects.triggers.run` endpoints accept JSON request bodies
modeled on the API's `Build` resource.
YAML is a file-authoring convenience that gets parsed into the same `Build` structure before submission.

Programmatic generation via `jq` produces correct JSON directly — no YAML indentation
fragility, no multiline string quoting hazards, no dependency on `yq` or custom emitters.
The stitch function (`zrbf_stitch_build_json`) already generates JSON;
converting to YAML would add complexity with zero functional benefit.

==== What Dies

* Any assumption that `cloudbuild.yaml` is required for triggers or SLSA provenance
* The linked term `rbtgr_build_yaml` — replaced by
{rbtgr_build_json}
* YAML-specific concerns: multiline string quoting, indentation sensitivity, `$$` escaping conventions

==== What Survives

* `zrbf_stitch_build_json()` — continues to emit JSON via `jq`, adapted for trigger structure
* `rbgjb/*.sh` step scripts — inlined into JSON `args` arrays at inscribe time
* All `_RBGY_*` substitution variables — baked into JSON `substitutions` object
* OCI Layout Bridge pattern — unchanged

==== Provenance Note

SLSA v1.0 provenance records `buildConfigSource.path` which will contain `cloudbuild.json`.
This is a cosmetic difference from `cloudbuild.yaml` — the structured provenance fields
(build inputs, parameters, outputs) are identical regardless of config file format.
