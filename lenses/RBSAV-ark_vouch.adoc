//axhob_operation
{rbtgo_ark_vouch}

NOTE: This operation vouches for an
{rbtga_ark}
in
{rbtgi_repository}
by running `slsa-verifier verify-image` against each per-platform image digest and pushing the verification results as a
{rbtga_ark_vouch}
artifact.
The
{rbtr_director}
submits a Cloud Build job via the RunBuild API (`builds.create`), which runs as the
{rbtr_mason}
service account on the default pool.

NOTE: Vouch uses `builds.create` (direct submission) rather than trigger dispatch because the
{rbtga_consecration}
is known only after conjure completes.
The default pool is acceptable: vouch steps only need internet access (binary download, GAR registry, Sigstore root of trust).

NOTE: {rbsk_no_gcloud} applies.
Registry authentication uses the GCE metadata server to obtain a service account token and writes a static Docker config.

The vouch operation:

//axhop_parameter_from_type
{rbtga_ark}
. *Validate Input Parameters*:
.. {rbbc_require}
vessel directory provided as first argument
.. {rbbc_require}
consecration provided as second argument

//axhos_step
. *Authenticate as
{rbtr_director}*:
.. Execute
{rbtoe_director_authenticate}
.. {rbbc_store}
`<<DIRECTOR_TOKEN>>` for subsequent API calls

//axhos_step
. *Gate on About Existence*:
.. {rbbc_call}
Docker Registry API: HEAD manifests for `<<CONSECRATION>>-about`
.. {rbbc_require}
HTTP 200 response -- conjure must complete before vouch
.. If `<<CONSECRATION>>-vouch` already exists:
{rbbc_warn}
re-vouch in progress (see idempotency below)

//axhos_step
. *Submit Vouch Cloud Build*:
.. Construct Build resource JSON with three step scripts, `serviceAccount` as {rbtr_mason}, substitution namespace `_RBGV_*`
.. *Step 1* ({rbrg_alpine_image_ref}): Download slsa-verifier binary from {rbrg_slsa_verifier_url} and verify checksum matches {rbrg_slsa_verifier_sha256}
.. *Step 2* ({rbrg_alpine_image_ref}): Authenticate to GAR via metadata server token, then run `slsa-verifier verify-image` for each per-platform image digest with `--source-uri` set to {rbtgr_rubric_repo} URL and `--print-provenance`, capturing per-platform verification output
.. *Step 3* ({rbrg_docker_image_ref}): Build and push a `FROM scratch` vouch container tagged `<<CONSECRATION>>-vouch`
.. {rbbc_call}
`POST <<GCB_BUILDS_URL>>` and poll until terminal state
.. {rbbc_fatal}
if any platform verification fails or build status is not SUCCESS -- no partial vouch artifacts are pushed

//axhoo_output_of_type
{rbtga_ark_vouch}
. *Vouch Artifact Content*:
.. `vouch_summary.json`: consecration, vessel, verifier identity (URL + SHA256), and per-platform verification status
.. Per-platform `verify-<<SUFFIX>>.json`: full provenance dump from `--print-provenance`
.. The vouch artifact is a single-platform container (verification results are architecture-independent metadata)
. *Display Results*:
.. {rbbc_show}
vouch complete with consecration and vouch artifact tag

//axhoc_completion
Successful execution creates a
{rbtga_ark_vouch}
artifact in the registry tagged as `<<VESSEL>>:<<CONSECRATION>>-vouch`.
+
*Source URI matching*: The `--source-uri` must match the source recorded in provenance.
For trigger-dispatched builds, this is the {rbtgr_rubric_repo} URL.
Format mismatch causes verification failure.
+
*Re-vouch idempotency*: Re-running vouch re-executes verification and pushes a new vouch artifact, overwriting the previous.
Content is presumed deterministic (same digests + same provenance + same verifier = identical output).
If this assumption fails in practice, narrow comparison scope to verification verdict, digests, and builder identity.
