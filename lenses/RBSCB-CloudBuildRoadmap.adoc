==== Multi-Architecture Strategy Caveat

All sections below assume Google Cloud Build (GCB) remains the sole builder.
GCB provides only x86 workers — default and private pools alike.
Multi-platform builds use QEMU emulation via the binfmt setup step (rbgjb04).

The vessel regime's `RBRV_CONJURE_BINFMT_POLICY` field ("allow" / "forbid") is the canary for this assumption.
As long as all vessels tolerate emulation, GCB-specific investment is sound.
If a vessel requires native ARM builds, the builder strategy must expand beyond GCB,
and further GCB-specific hardening should be re-evaluated.

If native ARM builds become necessary, builder-agnostic controls (cosign signing, portable SLSA provenance)
become more valuable than GCB-specific infrastructure (private pools, VPC Service Controls).
See cosign deferral rationale in Deferred Items.

==== Current Posture

Properties of the build pipeline as of 2026-02.
Authoritative procedures live in the operational specs (RBSRI, RBSTB, RBSDC, RBSDN);
this section states what holds and why it matters.

**Build definitions are fully static.**
Each vessel has a committed `cloudbuild.json` generated by the inscribe command
from step scripts (`rbgjb/*.sh`) and current regime pins — zero custom substitution overrides
at dispatch time.
The build is fully determined by the committed JSON at a specific rubric repo commit.
See RBSRI, RBSTB.

**SLSA v1.0 provenance is automatic.**
Google generates SLSA v1.0 provenance for all trigger-invoked builds stored in Artifact Registry.
Provenance chain: rubric repo commit → committed JSON → image digest.
No DIY provenance tooling.

**Security boundary: rubric repo.**
Cloud Build sees only the rubric repo via Developer Connect (one-time OAuth setup,
no ad-hoc credential management) — a single shared repo
with per-vessel directories (cloudbuild.json + Dockerfile + build context).
The main repo (tooling, specs, regime files) is never exposed to Google.
SLSA provenance records the rubric repo commit, which maps directly to the build material.
See RBSRI.

**Builder step images are digest-pinned.**
All `RBRR_GCB_*_IMAGE_REF` regime variables enforce `@sha256:<64hex>` digests,
validated by `buv_val_odref()`.
Inscribe verifies pin freshness (< 1 day) as a blocking gate;
stale pins require explicit refresh, commit, and re-inscribe.

**Private pool workers.**
Builds execute on a Private Pool peered to the depot VPC.
Expanded machine type options (64 vs default pool's 5), higher concurrent build limits.
Public egress is retained — no dependency mirroring cascade.
Pools are fully managed by Google (scale to zero, no VMs to maintain).
Foundation for a future egress lockdown if warranted.

**Artifact Registry storage.**
Built images are stored in Google Artifact Registry (GAR), not Docker Hub or other external registries.
Regime variables configure project, region, and repository.

**Dedicated Mason service account.**
Scoped to depot-project permissions, specified in the `serviceAccount` field of each build JSON.

**SBOM generation.**
Syft generates JSON and table-format SBOMs from OCI layout (rbgjb08),
packaged into the metadata container (rbgjb09).
Aligned with CISA 2025 minimum elements and EU CRA requirements.
The upgrade path is runtime-aware analysis, not removal.

**OCI Layout Bridge.**
Multi-platform builds export to OCI tarball, push via crane.
Clean separation: build phase (no credentials) from push phase (GAR auth).
See RBSOB trade study.

**Cloud Build logging.**
Build requests set `options.logging: CLOUD_LOGGING_ONLY`.
Log hygiene (no credential echo) was audited during trigger migration script inlining.

==== Potential Hardening: Egress Lockdown

Set `egressOption: NO_PUBLIC_EGRESS` on the Private Pool.
Workers lose all public internet access.
This would be the largest cascading change in the pipeline's history.

**Dependency mirroring required.**
Every external resource fetched during build must be pre-staged in GAR or GCS:

[cols="1,2,1",options="header"]
|===
|Build Step
|External Dependency
|Mitigation

|rbgjb04 (binfmt)
|`tonistiigi/binfmt` from Docker Hub
|Mirror to GAR

|rbgjb06 (build)
|Base images from Docker Hub (per vessel Dockerfile)
|Mirror to GAR; enforce GAR-only `FROM` refs

|rbgjb07 (crane)
|Crane tarball from GitHub Releases
|Host in GCS or bake into custom builder image

|rbgjb08 (SBOM)
|`anchore/syft` from Docker Hub
|Mirror to GAR
|===

**Ongoing maintenance cost.**
Mirror refresh becomes a recurring operation.
The existing pin-refresh workflow expands to include mirroring.
Every tool image security update requires a mirror-refresh-and-commit cycle.

**Additional infrastructure:**

* Private Cloud DNS zone for internal hostname resolution
* NAT/proxy if any exception egress paths are needed (treat as security event)
* All `RBRR_GCB_*_IMAGE_REF` pins must point to GAR mirrors, not upstream registries

**Note:** The rubric repo architecture (Current Posture) already narrows the build context to
per-vessel directories — Cloud Build never fetches from the main repo.
This limits the egress surface to builder images and base images only.

==== Potential Hardening: VPC Service Controls Perimeter

Wrap the depot project in a VPC Service Controls (VPC-SC) perimeter.
This restricts which GCP APIs can be called and from where, reducing data exfiltration risk.

**Required infrastructure:**

* VPC-SC perimeter around the depot project (organizational policy — may require org-level admin)
* Access levels for operator workstations (so builds can still be submitted from outside the perimeter)
* Ingress/egress rules for the perimeter (which services may cross the boundary)
* Testing: every GCP API call in the pipeline must be verified inside the perimeter — Cloud Logging writes, GCS uploads, Artifact Registry pushes, Cloud Build API calls can break subtly

**Prerequisite:** Egress Lockdown should be in place for the perimeter to be meaningful.

==== Deferred Items

Evaluated and consciously deferred; each has a specific revisit trigger.

**Dockerfile base image digest validation.**
Builder step images are digest-pinned (Current Posture), but user-authored Dockerfiles
in vessel build contexts are not validated.
A vessel with `FROM ubuntu:22.04` passes today — mutable tag, supply-chain risk.
This constrains vessel authors (policy decision with RBSRV implications),
orthogonal to pipeline infrastructure.
Separate future heat.

**BuildKit provenance flag.**
`--provenance=true` on `docker buildx build` records builder internals —
useful for deeper forensics.
Google's trigger-level SLSA v1.0 provenance covers the primary attestation use case.
Redundant for the current threat model.
Revisit if build reproducibility verification becomes a priority.

**Cosign image signing.**
Cryptographic signatures attached to images, verifiable by downstream consumers.
Two modes: keyless (Sigstore/Fulcio — requires public internet, conflicts with Egress Lockdown)
and keyed (Cloud KMS — no external dependency).
With Google SLSA provenance, cosign is redundant for the current threat model
(building and consuming within own GCP project).
Becomes valuable if: images are distributed externally, trust verification needed outside GCP,
or builder strategy expands beyond GCB (losing Google's provenance).
If adopted, prefer KMS-backed keyed signing (`cosign sign --key gcpkms://...`)
for Egress Lockdown compatibility.

**Binary Authorization.**
Deploy-time policy enforcement: only attested images can be deployed.
Relevant for GKE and Cloud Run.
Recipe Bottle deploys to Podman on workstations — no benefit until deployment targets change.

**Reproducibility verification.**
Periodic independent rebuilds compared by digest.
Requires reproducible toolchains — a vessel-author discipline, not a pipeline control.
Not actionable until vessel ecosystem matures.
