== Core Structure

A
{mcm_concept_model}
creates a cohesive gestalt - a unified constellation of ideas whose relationships and interactions transcend their individual definitions.
Through carefully curated terminology and precise linking patterns, each
{mcm_concept_model}
illuminates the deep conceptual relationships that give meaning to a domain.

The power of a
{mcm_concept_model}
lies in its ability to:

* Reveal the natural connections between ideas
* Preserve the integrity of concept relationships through evolution
* Enable clear visualization of how changes to one concept ripple through the constellation
* Guide understanding through carefully mapped idea spaces

Each
{mcm_concept_model}
manifests through a
{mcm_document}
that follows strict organizational patterns.
These patterns ensure that as the constellation evolves, their relationships remain clear and traceable.

The
{mcm_document}
contains two primary constrained element types:

* A
{mcm_mapping_section}
that defines all
{mcm_linked_term_s}
* One or more
{mcm_definition_section_s}
that provide precise meaning for each term

Through these elements, the
{mcm_document}
maintains the delicate balance between rigorous structure and organic concept evolution, allowing the idea constellation to grow while preserving its essential relationships.

== Text Organization

Text within a
{mcm_document}
follows specific structural rules to enable clear tracking of changes and relationships.

=== Line Break Fundamentals

The most fundamental organizing principle is the
{mcm_line_break}
.
Each
{mcm_line_break}
creates a clear boundary in the
{mcm_document}
while preserving fluid rendering in the output document.

==== Basic Line Break Rules

{mcm_line_break_s}
occur in these precise patterns:

* Between independent sentences
* Before and after each
{mcm_linked_term}
(including each term within a sequence)
* Between fragments that surround
{mcm_linked_term_s}

Basic example showing sentence and term breaks:

[source,asciidoc]
----
The system initializes its components.
The
{excm_processor}
begins operation.
Each component reports its status.
----

==== Term Integration Patterns

When incorporating
{mcm_linked_term_s}
the text follows these specific patterns:

Single term integration - breaks before and after the term:

[source,asciidoc]
----
The system processes
{excm_request}
according to defined rules.
----

Multiple
{mcm_linked_term_s}
in sequence - each term gets its own line, with commas attached:

[source,asciidoc]
----
The interface supports
{excm_drag},
{excm_drop},
and
{excm_scroll}
operations.
----

Complex
{mcm_linked_term}
relationships - each standalone term gets breaks:

[source,asciidoc]
----
The
{excm_validator}
applies each
{excm_rule}
to ensure
{excm_compliance}
with system requirements.
----

These patterns apply consistently across all document contexts, including:

* Regular paragraphs
* List items
* Definition entries
* Section introductions

=== Paragraph Organization

Paragraphs in the
{mcm_document}
follow these structural principles:

* {mcm_blank_line_s}
appear only between true paragraphs
* No
{mcm_blank_line_s}
between sentences within a paragraph
* No
{mcm_blank_line_s}
around source code blocks or list items
* Each paragraph focuses on a single concept or relationship

Example paragraph structure:

[source,asciidoc]
----
The system initializes its core components.
Each component performs self-diagnostics.
The
{excm_validator}
confirms system health.

The initialization process then moves to external systems.
The
{excm_connector}
establishes required links.
Each connection undergoes verification.
----

=== Special Cases

==== List Items

The
{mcm_document}
presents list items
with proper term integration
and conceptual breaks:

[source,asciidoc]
----
* The first item spans
{excm_component}
boundaries
* A second item demonstrates
{excm_process}
flow
* The final item shows
{excm_result}
formatting
----

==== Definition Entries

Definition entries follow standard sentence breaks while maintaining definition list structure:

The single space following `::` is necessary.

[source,asciidoc]
----
[[excm_term]]
{excm_example}:: 
The primary definition establishes core meaning.
Additional sentences expand understanding.
The
{excm_linked_term}
shows relationship handling.
----

==== Source Code Boundaries

Source code blocks establish strict boundaries for
{mcm_curation_action_s}.
These boundaries protect code content while maintaining proper term integration in surrounding text:

[source,asciidoc]
----
The
{excm_processor}
implements this interface:

[source,java]
-----
public interface Processor {
    void process();
    Status check();
}
-----

The implementation ensures
{excm_compliance}
with system requirements.
----

{mcm_curation_action_s}
respect these boundaries by:

* Maintaining term integration and line breaks in surrounding text
* Treating code block contents as opaque
* Preserving code formatting without modification
* Excluding code content from term matching and promotion analysis

This boundary pattern ensures code examples remain stable through document evolution while maintaining proper conceptual integration in their context.

=== Punctuation Rules

Punctuation follows standard English conventions regardless of line breaks:

* Sentence-ending punctuation stays with the sentence
* Commas remain with their preceding phrase
* Quotation marks enclose their content without breaks
* Parentheses maintain content unity

Example punctuation handling:

[source,asciidoc]
----
The system (version 2.0) activates its components.
The
{excm_processor}
begins its "primary sequence."
After initialization,
the system reports "ready."
----

=== Document Mechanics

=== {mcm_mapping_section} Constraints

The
{mcm_mapping_section}
enforces alignment rules to enhance readability and change tracking:

* The `<<` of each
{mcm_domain}
aligns to columns that are multiples of 10
* When entries require wider alignment, all lines adjust to the next larger multiple
* {mcm_attribute_reference_s}
maintain consistent indentation

[source,asciidoc]
----
// tag::mapping-example[]
:excm_doc_short:              <<excm_short,Short Term>>
:excm_doc_longer_term:        <<excm_longer,Longer Term>>
:excm_doc_much_longer_term:   <<excm_much_longer,Much Longer Term>>
// end::mapping-example[]
----

{mcm_normalization}
maintains these alignments during curation unless expansion is required.

=== Document Source Management

The
{mcm_document}
maintains compatibility with line-based tools
while preserving semantic structure.

This means:

* Favoring AsciiDoc's native list markers
over hardcoded sequence numbers

* Preserving processor directives
that enable clean rendering

* Ensuring changes remain visible
in both source differencing tools
and rendered comparisons

The goal is maintaining a document that:

* Shows ancestry clearly in source
* Renders cleanly through AsciiDoc
* Displays changes meaningfully in diffs
* Supports post-rendering analysis

=== Term Hierarchy Management

When
{mcm_linked_term_s}
appear to form hierarchical relationships, the
{mcm_document}
flattens this hierarchy through specific patterns:

* Each
{mcm_linked_term_s}
receives first-class presentation in the
{mcm_mapping_section}
* Related
{mcm_linked_term_s}
maintain separate definitions
* Relationships express through careful definition crafting

Example of hierarchical
{mcm_linked_term_s}
handling:

[source,asciidoc]
----
:excm_doc_validation:             <<excm_validation,Validation>>
:excm_doc_validation_rule:        <<excm_validation_rule,Validation Rule>>
:excm_doc_validation_set:         <<excm_validation_set,Validation Set>>
----

The definitions then express relationships:

[source,asciidoc]
----
[[excm_validation]]
{excm_doc_validation}:: 
The process of applying rules to ensure content correctness.

[[excm_validation_rule]]
{excm_doc_validation_rule}:: 
A specific constraint within the {excm_doc_validation} framework.

[[excm_validation_set]]
{excm_doc_validation_set}:: 
A collection of {excm_doc_validation_rule_s} that work together.
----

== Definition Patterns

=== Definition Form Patterns

Each
{mcm_linked_term}
in a
{mcm_document}
requires an anchor target and a definition.
The form determines how these elements are structured.

[[mcm_annotation]]
{mcm_annotation}::
A comment line providing type categorization or metadata about terms in a
{mcm_document}.
{mcm_annotation_s}
are ignored during first-order document reading.
When a referencing system is in context,
{mcm_annotation_s}
provide structured metadata interpreted by that system.
The content contains space-separated
{mcm_linked_term_s}
from the referencing system, which is typically out-of-context during first-order reading.
The grammar, interpretation, and placement rules for
{mcm_annotation_s}
are defined by the referencing system, not MCM.
+
{mcm_annotation_s}
use prefix-discriminated form: `//ax...` — no space after `//`, where the initial characters match a registered prefix identifying the referencing system.
The full content after `//` is interpreted by the referencing system as space-separated prefix terms.
+
The mechanical discriminator is the character immediately following `//`.
A space character indicates a regular comment.
A letter with no intervening space indicates an
{mcm_annotation}.
+
Registered prefixes:
+
* `ax` — Axial Lexicon

[[mcm_form_deflist]]
{mcm_form_deflist}::
The standard pattern using AsciiDoc definition list syntax.
The anchor target appears on its own line, followed by an optional
{mcm_annotation}
then the term reference with `::` and definition text.
Use for standalone terms with self-contained meaning.

[source,asciidoc]
----
[[term_anchor]]
//axk_motif
{term_reference}::
Definition text following sentence-per-line rules.
----

[[mcm_form_bullet]]
{mcm_form_bullet}::
A pattern for terms that belong to a parent structure or sequence.
Each term appears as a bullet item under its parent section.
The anchor target appears after the bullet marker, followed by an optional
{mcm_annotation}
then the term reference with definition prose.
Use for fields of a structure or steps in a procedure.

[source,asciidoc]
----
* [[field_anchor]]
//axr_member
{field_reference} defines the specific meaning within the parent structure.
----

[[mcm_form_section]]
{mcm_form_section}::
A pattern where the section heading itself is the term.
The anchor target appears before the section marker, followed by an optional
{mcm_annotation}
then the section heading containing the term reference.
The section content constitutes the definition.
Use for terms whose meaning requires extended explanation or contains nested child terms.

[source,asciidoc]
----
[[sequence_anchor]]
//axo_sequence
=== {sequence_reference}

The sequence executes these steps:

* [[step_one_anchor]]
//axo_step
First step description.
----

== Task Lens Support

{mcm_concept_model_s}
support focused work contexts through
{mcm_task_lens_es}
that emphasize relevant concepts while suppressing distractions.
See the
{mcm_task_lens_guide}
for compilation specifications and usage patterns.

=== Task Lens Structure

Each
{mcm_task_lens}
follows this pattern:

[source,asciidoc]
----
=== [Task Lens Name]

**Purpose**: [One sentence describing the work this lens enables]

**Gestalt**: [Metaphorical description guiding term interpretation]

**Core Activation**:

* {linked_term} - [Aspect emphasis if needed]
* {linked_term} - [Aspect emphasis if needed]

**Expanded Activation**:

* {linked_term} - [Reduced detail focus]
* {linked_term} - [Reduced detail focus]

**Suppressed Activation**:

* List of {linked_term_s} to remain dormant
* Additional suppression patterns

**Critical Relationships**:

* {term_a} ↔ {term_b}: [What aspect matters]
* {term_c} → {term_d}: [Specific mechanism focus]
----

Multiple
{mcm_task_lens_es}
may share terms but apply different
{mcm_focal_depth_s}
and aspect emphasis based on their purpose.

=== Blank Line Exception for Task Lenses

Despite the general document rule against
{mcm_blank_line_s}
around list items,
{mcm_task_lens}
sections require specific blank lines for valid asciidoc parsing:

* One blank line after each `**Section**:` header before its list
* One blank line between major sections
* No blank lines between list items within a section

This exception exists because asciidoc requires blank lines before bulleted lists to parse correctly, and the structured nature of
{mcm_task_lens_es}
makes the blank line placement predictable and consistent.

== Actor Definitions

[[mcm_editor]]
{mcm_editor}::
The human originator of
{mcm_concept_model_s}
and constraints.
The
{mcm_editor}
guides document evolution, establishes patterns, and directs transformations while maintaining conceptual integrity.

[[mcm_digital_mind]]
{mcm_digital_mind}::
An agent, typically an LLM, transforming
{mcm_document_s}
under
{mcm_editor}
direction and document constraints.
The
{mcm_digital_mind}
performs well-defined
{mcm_curation_action_s}
while preserving document patterns and relationships.

== {mcm_curation_action_s}

[[mcm_curation_action]]
{mcm_curation_action}::
A process executed by a
{mcm_digital_mind}
to adjust a
{mcm_document}
as directed by the
{mcm_editor}.
Each
{mcm_curation_action}
has a specific focus and constraints, modifying only those aspects of the
{mcm_document}
that fall within its defined scope.

[[mcm_normalization]]
{mcm_normalization}::
A
{mcm_curation_action}
that brings a
{mcm_document}
to canonical form through two phases.

*Phase 1: Text Normalization*

Applies consistent formatting throughout the
{mcm_document}
body:

* One sentence per line
* Each
{mcm_linked_term}
on its own line
* Commas and punctuation attached to preceding element
* {mcm_blank_line_s}
only between paragraphs and definition entries
* Preserve source code block contents without modification

*Phase 2: Mapping Section Normalization*

Maintains strict structural integrity of the
{mcm_mapping_section}:

* Preserve
{mcm_category}
comment headers as
{mcm_category_group}
delimiters
* Within each
{mcm_category_group},
align `<<` to the smallest multiple of 10 columns that accommodates the longest
{mcm_attribute_reference}
* Maintain alphabetical ordering by display text within each
{mcm_category_group}
* One
{mcm_attribute_reference}
per line

[[mcm_category_group]]
{mcm_category_group}::
A contiguous block of
{mcm_attribute_reference_s}
in the
{mcm_mapping_section}
preceded by a category comment header.
Each
{mcm_category_group}
maintains independent column alignment,
allowing terms to be added without affecting other groups.

[[mcm_demotion_analysis]]
{mcm_demotion_analysis}::
A
{mcm_curation_action}
that identifies
{mcm_linked_term_s}
whose formal status may no longer justify their maintenance overhead.
It suggests candidates for demotion to
{mcm_fundamental_text}
based on criteria such as:
* Rare usage outside their own definition
* Lack of meaningful variation requiring
{mcm_attribute_reference_s}
* Minimal role in expressing relationships between other terms
* Terms whose precise meaning has become self-evident in context

[[mcm_link_validation]]
{mcm_link_validation}::
A
{mcm_curation_action}
that verifies the mechanical integrity of the
{mcm_document_p}
linking system without modification:

* Reports
{mcm_attribute_reference_s}
that lack corresponding anchors
* Reports anchors that lack corresponding
{mcm_attribute_reference_s}
* Validates
{mcm_domain}
format compliance
* Identifies malformed link syntax

This action produces a diagnostic report for
{mcm_editor}
review.
Repairs are performed manually or through targeted
{mcm_curation_action_s}.

[[mcm_promotion_analysis]]
{mcm_promotion_analysis}:: 
A
{mcm_curation_action}
that identifies frequently occurring or conceptually significant elements in
{mcm_fundamental_text}
that warrant promotion to
{mcm_linked_term}
status.
It suggests candidates based on:
* Frequency in definitions of other
{mcm_linked_term_s}
* Use in expressing core conceptual relationships
* Occurrence in multiple distinct contexts
* Words whose precise meaning needs maintenance across document evolution

[[mcm_term_matching]]
{mcm_term_matching}::
A
{mcm_curation_action}
that identifies instances where concepts corresponding to existing
{mcm_linked_term_s}
appear in text without proper linking:
* Identifies important terms that warrant linking
* Proposes what should be linked

[[mcm_validation_summary]]
{mcm_validation_summary}::
An advisory report produced by
{mcm_link_validation}
that documents the state of the
{mcm_document}
without making modifications.
The
{mcm_validation_summary}
enables the
{mcm_editor}
to identify issues requiring attention.

== Lapidary

Vocabulary is the primary interface of every system the
{mcm_editor}
choreographs:
names are how intent survives across minds and sessions,
and the specification machinery around them is scaffolding in their service.
A name that fails its reader fails the system,
however sound the code beneath it.

Prefix discipline governs the left half of every persistent name;
this section governs the right half — the word itself.
The rules below bind the word part of every
{mcm_quoin},
the display text of every
{mcm_linked_term},
and any persistent name a system mints: verbs, nouns, file words, command words.
They constrain births, not the living —
existing names are swept only by deliberate remint tooling, never piecemeal.

* *Semantic uniqueness.*
One meaning per word across all persistent names, repo-wide.
A word already bound to a meaning anywhere in the system is unavailable for a second meaning, however distant the domain.
* *One word per concept.*
The dual of semantic uniqueness: no alias
{mcm_quoin_s} —
a concept already named is closed to a second word.
Near neighbors must be spaced:
adjacent definitions state their distinction, or one of the pair dies.
* *Vocabulary isolation.*
Words mint from source domains alien to software, cloud infrastructure, CI/CD, and corporate project management.
A word with heavy ambient presence in software prose is a
{mcm_trodden_word}
and is disqualified regardless of semantic fit.
Domain choice makes collision structurally impossible;
word-level checks merely verify.
* *The grep gate.*
A candidate word is verified by repo-wide grep before adoption;
a clean landing is a minting precondition, not a nicety.
* *Rare but real.*
Prefer rare, real words whose true meaning does semantic work.
Concrete, physical, sensory images lodge across sessions in ways abstract labels cannot.
Abbreviations spend recognition to save characters — the wrong trade.
* *Word husbandry.*
A word's commonness matches its operation's traffic:
daily surfaces earn short, plain, instantly-guessable words;
rare ceremonies must not spend them —
mint longer, rarer, more colorful words there,
conserving the small stock of plain English for the surfaces that need it.
Husbandry lowers the commonness ceiling, never the
{mcm_fair_faced}
floor: color may rise with rarity; opacity may not.
* *Sibling initials.*
Wherever a sibling set shares an abbreviation namespace —
colophon suffixes, quoin sub-letters, any single-character discriminator —
elect words whose first letters stay distinct,
so the abbreviation can remain the word's initial.
The verbs orbiting one critical noun are siblings from birth,
abbreviation namespace or not:
verb families acquire one eventually, and by then the words are minted.
An election-time check beside the grep gate;
the word-side twin of sub-letter monosemy.
* *Mint into an* {mcm_asterism}.
A new word joins a coherent metaphor family wherever one fits;
a lone orphan word is weaker than a family member.
* *Audible register.*
Sibling subsystems ride
{mcm_asterism_s}
whose registers are distinguishable by ear,
so a stray term's home system is audible without lookup.

[[mcm_asterism]]
{mcm_asterism}::
A named metaphor family whose source-domain relationships mirror the system relationships it names.
The name is the gestalt: a recognizable figure of stars that is not itself a constellation and may span several —
as a vocabulary family may span
{mcm_concept_model_s}.
Each term inherits relational meaning from the family
(the coronet sits below the firemark as the hoof marking sits below the brand),
so the metaphor performs structural work beyond retrieval,
and a new term in an established family arrives nearly pre-learned —
for the
{mcm_digital_mind}
and the
{mcm_editor}
alike.
Founding specimen: Job Jockey's equestrian vocabulary (JJS0, Equestrian Vocabulary), with the racing and dressage layers as sibling families sharing one register.
Model-side rationale: `Memos/retired/memo-20260610-quoin-minting-introspection.md`;
prefix-side sibling: `Memos/memo-20260110-acronym-selection-study.md`.

=== Exposure

Word-selection care scales with who must read the name.
Every persistent name sits on a ladder of increasing dressing:

* {mcm_hearting} — interior names, uncatalogued, governed by prefix discipline alone.
* A
{mcm_quoin}
— the catalogued default; the rules above bind.
* {mcm_ashlar} — a
{mcm_quoin}
that faces the operator; everything below binds, plus the rules at its definition.

The ladder is deliberate at both ends:
{mcm_hearting}
is permission to leave interior names plain,
granted so that care can concentrate where reading is costly.
Misclassification fails quiet in one direction —
a word that should be
{mcm_ashlar}
but is unmarked skips the stricter tests —
so when a word surfaces in operator output, ask whether it belongs on the
{mcm_broadside}.

The ladder sorts by exposure — who must read the name.
A second, finer mark cuts across it: *centrality*, how much of the user's model a name carries.
The few
{mcm_ashlar_s}
a user reasons *through* are
{mcm_bondstone_s};
the verbs that work them are
{mcm_toothing}.
Both draw a care beyond the ordinary
{mcm_ashlar}:
a model is held by its
{mcm_bondstone_s},
and a name that fails there fails the whole model, not a single reading.

[[mcm_ashlar]]
{mcm_ashlar}::
A
{mcm_quoin}
the operator must trigger, observe in output or errors, or reason about system state with.
The dressed facing stone — finished finely because it faces out.
Beyond the rules above, an
{mcm_ashlar}
must be
{mcm_fair_faced},
draws from the
{mcm_coffer},
and is not minted until it appears on the project's
{mcm_broadside}.
Scrutiny scales with contact frequency:
a word the operator meets daily outranks a founding-ceremony word,
and both outrank any spec-interior
{mcm_quoin}.
Verbs name what the operator does or sees, never the mechanism beneath.
Words appearing in failure output are
{mcm_ashlar_s}
even when the operator never invokes them.

[[mcm_bondstone]]
{mcm_bondstone}::
The few
{mcm_ashlar_s}
a user reasons *through* — the small set of operator-facing names that constitute the user's working model of the system, as against a name merely surfaced once in output.
The through-stone of the wall: dressed on the face like any
{mcm_ashlar},
yet reaching the wall's full thickness to bind the visible face to the unseen
{mcm_hearting} —
so a
{mcm_bondstone}
ties what the operator sees to what the system is.
Distinguished from a plain
{mcm_ashlar}
not by exposure — both face out — but by centrality: an
{mcm_ashlar}
in a rare error message faces the operator yet bears no model, where a
{mcm_bondstone}
is the name the user's whole model hangs on.
Every rule above binds hardest here.
It must above all be
{mcm_fair_faced}
and lodge unaided; it must dodge overloading in both general software prose and the system's own adjacent domain; and it earns first-letter separation from its siblings, so the set an operator holds at once stays distinct in the hand and the head.

[[mcm_toothing]]
{mcm_toothing}::
The verb-tier sibling of the
{mcm_bondstone}:
the operator-facing acts performed upon the model-constituting names, and the operations those names visibly perform upon one another so the system stays legible as it works.
The toothing-stones left projecting at a wall's end, the worked interlock by which a later course keys in — so a growing system's new names bond into the standing set rather than abutting it loose.
A
{mcm_toothing}
verb draws the same heightened care as the
{mcm_bondstone}
it acts on: it must be
{mcm_fair_faced}
and lodge unaided, dodge overloading in software prose and the adjacent domain, and hold first-letter separation across the set an operator wields at once.

[[mcm_hearting]]
{mcm_hearting}::
The wall-core fill — structural, unseen, taking no dressing.
Interior names governed by prefix discipline alone:
locals, private helpers, intermediate artifacts no operator and no spec must read.
{mcm_hearting}
is a deliberate grant of plainness, not neglect;
its existence is what lets
{mcm_ashlar}
care stay affordable.

[[mcm_fair_faced]]
{mcm_fair_faced}::
The property an
{mcm_ashlar}
must hold: finished so it needs no render coat.
An operator meeting the word for the first time, mid-ceremony,
must act correctly without a glossary lookup.
Rare enough to land clean in the grep gate,
common enough to guess warm —
the deliberate tightening of rare-but-real at the wall's face.
The
{mcm_cold_probe}
is its test.

[[mcm_coffer]]
{mcm_coffer}::
The bounded budget of
{mcm_ashlar_s}
an operator can be asked to hold.
Each mint spends from it;
spec-grain precision collapses into fewer operator words wherever the distinction does not change what the operator does.
The fine grain lives on in
{mcm_quoin_s};
the
{mcm_coffer}
governs only the wall's face.

[[mcm_broadside]]
{mcm_broadside}::
The single public sheet where a project posts its
{mcm_ashlar}
vocabulary.
Each project designates exactly one;
presence on the
{mcm_broadside}
is the registry contract —
an
{mcm_ashlar}
mint is incomplete until its entry exists.
A project whose operators work through digital minds may mirror the sheet into agent context files;
mirrors are derived, never canonical.

[[mcm_cold_probe]]
{mcm_cold_probe}::
A
{mcm_curation_action}
testing
{mcm_fair_faced}:
present each candidate to a fresh reader in one sentence of working context;
the reader states what they expect the word to do or mean.
The stated expectation is the verdict —
mismatch fails the candidate,
and no advocacy by the minter is admitted.

== Antipatterns

[[mcm_antipattern]]
{mcm_antipattern}::
A named, recurring defect formally catalogued so that
{mcm_curation_action_s}
and reviews can cite it precisely.
The catalog spans document defects — flaws in durable artifacts —
and delivery defects — failures in operator-facing presentation;
both recur, and both are dischargeable only once named.
Naming the defect is what makes it detectable and dischargeable;
an unnamed defect recurs because no rule can forbid it.

[[mcm_molehill]]
{mcm_molehill}::
An
{mcm_antipattern}:
a delivery defect — manufacturing a significant-looking decision-fork or challenge
out of something nearly inconsequential,
then building structure around it.
The name is the gestalt: a molehill presented at mountain scale,
spending the reader's deliberation on stakes that fit in a clause.

[[mcm_skidmark]]
{mcm_skidmark}::
An
{mcm_antipattern}:
version commentary embedded in a durable document — revision markers, dated change narration, "new in vX" annotations.
The name is the gestalt: the streak left by something that already passed, saying nothing about where it is now.
History-time knowledge belongs to the commit;
git log and blame carry every line's date and reason with a forcing function, while in-document
{mcm_skidmark_s}
drift and pollute present-tense reading.
A durable document — spec, guide, code, context file — speaks of itself only in the present tense.
A dated artifact — memo, scar, commit message — is itself a historical record and may carry revision metadata without fault.
Detection and drain moves are catalogued in ACG (ACGm_107).

[[mcm_stun]]
{mcm_stun}::
An
{mcm_antipattern}:
a delivery defect — a wall of dense, fully-structured prose with no on-ramp,
so the reader's comprehension seizes before it can enter.
The name is the gestalt: the delivery stuns before it informs.
The finished staircase, seen from outside, is a wall;
the defect is not completeness but delivering it whole,
with no sentence to stand on and no way to climb.

[[mcm_trodden_word]]
{mcm_trodden_word}::
An
{mcm_antipattern}:
a persistent name minted from vocabulary worn featureless by ambient software prose —
open, close, run, lock, task, manager, handler, grant.
The name is the gestalt: a path so heavily walked that no single footprint is findable on it;
grep returns a thousand strangers.
A
{mcm_trodden_word}
fails retrieval for the
{mcm_digital_mind}
and search for the
{mcm_editor},
and invites the plan-versus-product bleed that vocabulary isolation exists to prevent.
Semantic fit never rescues it:
the word is disqualified before fit is weighed.

[[mcm_word_cancer]]
{mcm_word_cancer}::
An
{mcm_antipattern}:
variations of one normative idea scattered across documents as paraphrases,
each restatement enlarging the drift surface,
none authoritative.
The name is deliberately ugly, and the ugliness is the diagnosis:
each paraphrase metastasizes —
a
{mcm_digital_mind}
completing "what should be said here" recreates a nearby idea in fresh words,
and every recreation seeds the next.
The cost is mechanical, not aesthetic:
near-paraphrases are nearly the same retrieval key,
and once they drift the
{mcm_digital_mind}
retrieves a confident blend of them — statement-level interference —
while the
{mcm_editor}
is left unable to know which wording is law.
Detection and collapse moves are catalogued in ACG (ACGm_105).

[[mcm_notional_snippet]]
{mcm_notional_snippet}::

{mcm_curation_action_s}.
While it may contain asciidoc noncompliance, the
{mcm_digital_mind}

{mcm_document}

{mcm_line_break}

{mcm_linked_term_s},

{mcm_linked_term_s}.

{mcm_blank_line}

== Term Management Elements

[[mcm_attribute_reference]]
{mcm_attribute_reference}::
A variable-style reference that connects a
{mcm_category}
prefixed attribute name to a specific
{mcm_domain}
and replacement text potentially with multiple
{mcm_variant_s}.
The
{mcm_attribute_reference}
forms the mechanical foundation for
{mcm_linked_term}
implementation in the
{mcm_document}.
Each
{mcm_attribute_reference}
follows strict formatting rules defined in the
{mcm_mapping_section}.

[[mcm_quoin]]
{mcm_quoin}::
A formal identifier with full cataloguing in the
{mcm_concept_model}:
{mcm_attribute_reference}
in the
{mcm_mapping_section},
anchor at the definition site, and definition text.
The cornerstone that locks the structure — addressable from anywhere, tracked across the constellation.
A
{mcm_quoin}
is the Diptych-era term for what
{mcm_linked_term}
describes in MCM's AsciiDoc expression.

[[mcm_inlay]]
{mcm_inlay}::
A formal identifier following the prefix naming discipline that exists only within a subdocument's scope.
No
{mcm_attribute_reference},
no anchor, no
{mcm_mapping_section}
entry.
An
{mcm_inlay}
is set permanently into a surface — recognized by its prefix pattern — but not catalogued in the parent document's lexicon.
Used by hierarchy markers that need to formally identify elements without elevating them to
{mcm_quoin}
status.

[[mcm_sprue]]
{mcm_sprue}::
A literal wire-level token in backtick delimiters representing a serialized name.
No
{mcm_attribute_reference},
no anchor, no
{mcm_mapping_section}
entry, no prefix naming discipline required.
A
{mcm_sprue}
is the exact character sequence as it appears in a wire format — a JSON property key, an API field name, a protocol identifier.
May originate from external schemas or foreign conventions.
A
{mcm_sprue}
that follows the prefix naming discipline is also an
{mcm_inlay};
one that does not is recognized solely by its backtick delimiters.
Used by arity rules when annotations read literal wire names from following text.

[[mcm_rivet]]
{mcm_rivet}::
A formal identifier for a *normative proposition* — an invariant, a deliberate
deviation, a foreign-behavior signature, or a Palisade membrane — that a
{mcm_concept_model}
defines once at a definition site and that source code and tests cite by its
identifier.
Unlike a
{mcm_quoin},
whose readable name carries its meaning, a
{mcm_rivet}
identifier is *opaque*: it leaks no semantics into released code, while one
repo-wide search on the identifier still returns the definition, every
defending code site, and every attacking test — a census with no registry to
drift.
The kind of proposition is declared at the definition site by an
{mcm_annotation}
voicing an axial motif (the `axvc_` family), never encoded in the identifier,
so re-categorizing a
{mcm_rivet}
never re-mints it.
A
{mcm_rivet}
is a *voiced subject*, as is the
{mcm_quoin};
the
{mcm_inlay}
and
{mcm_sprue}
are by contrast *lookahead arguments* that voicings read, never voiced
subjects.

[[mcm_category]]
{mcm_category}::
A prefix mechanism that classifies
{mcm_linked_term_s}
by their conceptual role.
Each
{mcm_category}
must be declared in the
{mcm_mapping_section}
header and follows consistent patterns for classifying terms by type (e.g., 'doc_' for document elements, 'term_' for terminology elements).

[[mcm_variant]]
{mcm_variant}::
A systematic modification of a base term that maintains its core meaning while expressing a different form.
Common
{mcm_variant_s}
include plurals (_s), possessives (_p), and verb conjugations (_ed, _ing), each following patterns defined in the
{mcm_mapping_section}.

[[mcm_domain]]
{mcm_domain}::
A unique identifier that anchors a term's definition within the document structure.
Each
{mcm_domain}
uses snake case formatting and maintains consistent patterns (e.g., 'mcm_<canonical_term>') to enable reliable linking and reference.

[[mcm_task_lens]]
{mcm_task_lens}::
A focused activation pattern within a
{mcm_concept_model}
that emphasizes specific terms and relationships for particular work contexts.
Each
{mcm_task_lens}
defines which
{mcm_linked_term_s}
activate at what
{mcm_focal_depth}
and guides interpretation through gestalt descriptions and critical relationship specifications.


[[mcm_task_lens_guide]]
{mcm_task_lens_guide}::
A separate guide document that specifies how to compile task-focused representations from a full
{mcm_concept_model}.
The
{mcm_task_lens_guide}
defines lens structure patterns, compilation mechanics, and
{mcm_focal_depth}
activation rules for creating work-optimized views of concept constellations.

[[mcm_lens_specification]]
{mcm_lens_specification}::
The structured definition within a
{mcm_task_lens}
that categorizes
{mcm_linked_term_s}
by their
{mcm_focal_depth}
and provides aspect focusing guidance.
The
{mcm_lens_specification}
enables mechanical compilation of task-optimized representations from the full
{mcm_concept_model}.

[[mcm_focal_depth]]
{mcm_focal_depth}::
The activation level of a
{mcm_linked_term}
within a
{mcm_task_lens},
determining how much detail and relationship information to preserve.
The three
{mcm_focal_depth_s}
are Core (full detail with relationships), Expanded (supporting context), and Suppressed (explicitly dormant).
