== Core Principles

The Axial Lexicon provides reusable, atomic terminology for technical specifications.
Each term maintains precise semantics while adapting to implementation context.
Specifications selectively import only needed axial terms, avoiding vocabulary bloat.

== Motif Vocabulary

AXL terms are
{axl_motif_s}
- abstract patterns that concrete specifications voice in their local context.

[[axl_motif]]
{axl_motif}::
An abstract pattern defined in the Axial Lexicon that can be voiced by concrete terms in specifications.
A
{axl_motif}
captures the essential shape of a concept while remaining independent of implementation context.
Specifications voice
{axl_motif_s}
through local terms that bind the abstract pattern to specific environments.

[[axl_voicing]]
{axl_voicing}::
A concrete term in a specification that gives local expression to A
{axl_motif}.
Each
{axl_voicing}
adapts the abstract pattern to a specific context while preserving the
{axl_motif_s}
essential character.

[[axl_voices]]
{axl_voices}::
The relationship where a concrete term expresses a
{axl_motif}
in a specific context.
When a term
{axl_voices}
a
{axl_motif},
it gives the abstract pattern local form and implementation-specific behavior.

[[axl_evokes]]
{axl_evokes}::
The relationship where A
{axl_motif}
calls forth its
{axl_voicing_s}
in the reader's understanding.
When encountering A
{axl_motif},
it
{axl_evokes}
the concrete realizations known from various specifications.

[[axl_annotation]]
{axl_annotation}::
A
{xref_MCM}
comment using Strachey brackets that declares the
{axl_motif_s}
a term
{axl_voices}.
The
{axl_annotation}
appears between anchor and definition as a space-separated list of AXL terms within Strachey brackets.
+
Format: `// ⟦relationship primary [dimension ...]⟧`
+
* First term: the relationship (typically `axl_voices`)
* Second term: the primary
{axl_motif}
being voiced
* Remaining terms: dimension terms (prefixed `axd_`) specifying characteristics
+
Examples:
+
* `// ⟦axl_voices axc_call⟧` - simple voicing
* `// ⟦axl_voices axr_member axd_required axd_nullable⟧` - voicing with dimensions
+
Category prefixes carry semantic meaning, eliminating need for punctuation.
Position determines role: relationship first, primary second, dimensions following.
This design prioritizes LLM interpretability over human visual parsing.
When ambiguity or friction arises during LLM processing, the format should evolve to resolve it.

== Motif Implications

{axl_motif_s} carry implied characteristics beyond their definitions.
These implications guide how
{axl_voicing_s}
should behave:

* Anchor placement expectations per
{xref_MCM}
form patterns
* Type/instance level - where the
{axl_motif}
sits in abstraction hierarchy
* Contextual requirements - parent structures or operations the
{axl_motif}
exists within
* Voicing expectation - whether the
{axl_motif}
typically requires local voicing

== Form Expectations

Each AXL category implies an expected
{xref_MCM}
definition form pattern for documents using that category.
During type-scheme linting, these expectations guide validation of definition structure.

* {axc_*} control terms expect
{xref_MCM}
`mcm_form_deflist`
* {axo_sequence} operations expect
{xref_MCM}
`mcm_form_section` with nested steps
* {axj_structure} definitions expect
{xref_MCM}
`mcm_form_section`
* {axj_field} definitions expect
{xref_MCM}
`mcm_form_bullet` under parent structure
* {axk_premise} definitions expect
{xref_MCM}
`mcm_form_deflist`
* {axs_*} section terms expect
{xref_MCM}
`mcm_form_section` for section headers
* Most other categories expect
{xref_MCM}
`mcm_form_deflist`

These expectations support the
{axl_annotation}
pattern where documents include `// ⟦axl_voices motif ...⟧` markers that are validated when AXL is in context.

== Axial Control Terms

[[axc_variable]]
{axc_variable}::
A
{axl_motif}
representing a temporary value holder denoted by distinctive delimiters that exists for the duration of an operational sequence.
Implementation varies by context: shell variables in bash, dictionary entries in Python, sessionStorage in browser JavaScript.

[[axc_fatal]]
{axc_fatal}::
A
{axl_motif}
representing immediate termination of the operational sequence with error status, preventing all subsequent steps from executing.
Requires operator intervention or error recovery before retry.

[[axc_warn]]
{axc_warn}::
A
{axl_motif}
representing display of a non-blocking warning message while allowing the operational sequence to continue execution.
Alerts operators to concerning conditions without halting progress.

[[axc_store]]
{axc_store}::
A
{axl_motif}
representing capture of a value from an operation result or computation into a
{axc_variable}
for later reference.
Context determines storage mechanism while preserving semantic intent.

[[axc_use]]
{axc_use}::
A
{axl_motif}
representing reference to a previously stored
{axc_variable}
value in the current operation.
Retrieves stored values from context-appropriate storage location.

[[axc_require]]
{axc_require}::
A
{axl_motif}
representing validation check that confirms an expected condition, triggering
{axc_fatal}
or alternate execution path if not met.
Enforces preconditions and invariants throughout execution.

[[axc_poll]]
{axc_poll}::
A
{axl_motif}
representing repeated execution of a status check at defined intervals until completion condition or timeout.
Implements retry logic with configurable intervals and limits.

[[axc_call]]
{axc_call}::
A
{axl_motif}
representing direct invocation of an external service, API endpoint, or command.
Captures both success and error responses for processing.

[[axc_submit]]
{axc_submit}::
A
{axl_motif}
representing initiation of an asynchronous operation that returns immediately with a handle for later completion tracking.
Distinct from
{axc_call}
which expects synchronous response.

[[axc_await]]
{axc_await}::
A
{axl_motif}
representing blocking wait on a submitted asynchronous operation until completion.
Polls operation status until done, then returns final result or error.

[[axc_show]]
{axc_show}::
A
{axl_motif}
representing presentation of operational facts or values to the operator.
Implementation varies from console output to UI display based on context.

[[axc_step]]
{axc_step}::
A
{axl_motif}
representing a normal progress message indicating successful completion of a processing stage.
Provides operational visibility without implying warnings or errors.

== Axial Environment Terms

{axe_environment_s} specify execution contexts for control operations.
{axl_voicing_s} of
{axc_call},
{axc_warn},
{axc_fatal}
and similar control motifs should include an appropriate
{axe_environment}
as a dimension to clarify how the control term manifests in that context.

[[axe_environment]]
{axe_environment}::
A
{axl_motif}
representing an execution context that determines how control terms manifest.
{axe_environment_s} appear as dimensions in
{axl_voicing_s}
to bind abstract control semantics to concrete operational behavior.

[[axe_bash_interactive]]
{axe_bash_interactive}::
A
{axl_motif}
representing a bash shell session with human operator present at an interactive terminal.
Feedback via stdout/stderr is immediate.
Operator can interrupt (Ctrl-C) or respond to prompts.
Warnings display as yellow text; fatals exit with non-zero status.

[[axe_bash_unattended]]
{axe_bash_unattended}::
A
{axl_motif}
representing a bash script executing without human presence.
Output captured to logs.
No interactive prompts possible.
Failures must be self-documenting for later analysis.

[[axe_bash_scripted]]
{axe_bash_scripted}::
A
{axl_motif}
representing a bash script deliberately invoked to run to completion.
No human interaction during execution.
Succeeds atomically or fails with error status.
Output may go to stdout/stderr for capture or display.
Distinct from
{axe_bash_unattended}
which implies background/scheduled execution with log capture.

[[axe_human_guide]]
{axe_human_guide}::
A
{axl_motif}
representing a human executing a specific rendered procedure document while interacting with an external system (web console, GUI application).
The procedure document presents numbered steps; the human performs each in the external system.
Feedback is visual confirmation in the external UI.
Human judges success or failure of each step.
Warnings are cautionary notes; fatals mean abort procedure.

[[axe_rest_api]]
{axe_rest_api}::
A
{axl_motif}

== Axial Operation Terms

=== Procedure Hierarchy

{axo_procedure}
is the superset
{axl_motif}
for procedural articulations.
{axo_command},
{axo_guide},
and
{axo_routine}
are specialized forms that voice
{axo_procedure}
with additional constraints.

{axo_sequence}
is a structural characteristic describing HOW behavior is organized, not a procedure type.
Most procedures ARE sequential.

The following
{axo_*}
terms are NOT procedures and remain outside the hierarchy:
{axo_entity},
{axo_role},
{axo_identity},
{axo_actor},
{axo_dependency}.

[[axo_procedure]]
{axo_procedure}::
A
{axl_motif}
representing an executable operation with defined inputs, behavior, outputs, and completion semantics.
{axo_procedure}
is the superset for all procedural articulations.
{axl_voicing_s}
specialize by executor (machine vs human), reusability (terminal vs composite), and interface (CLI, REST, console).
Documentation follows
{axs_*}
section patterns.

[[axo_command]]
{axo_command}::
A
{axl_motif}
representing an
{axo_procedure}
executed by machine without human intervention.
{axl_voices}
{axo_procedure}
with machine executor constraint.
Requires
{axe_environment}
dimension specifying execution context.

[[axo_guide]]
{axo_guide}::
A
{axl_motif}
representing an
{axo_procedure}
executed by a human operator following system-presented steps.
{axl_voices}
{axo_procedure}
with human executor constraint.
The system displays steps; the human performs them in external systems or consoles.
Behavior steps use
{axe_human_guide}
control voicings.

[[axo_routine]]
{axo_routine}::
A
{axl_motif}
representing a reusable
{axo_procedure}
invoked as a subsequence within other procedures.
{axl_voices}
{axo_procedure}
with composite reusability constraint.
Encapsulates common logic for consistent reuse across specifications.

[[axo_sequence]]
{axo_sequence}::
A
{axl_motif}
representing a structural characteristic: ordered processing steps executed in defined progression.
NOT a procedure type—describes HOW behavior is organized, not WHAT kind of procedure something is.
Most
{axo_procedure}
voicings ARE sequential.
Use within
{axs_behavior}
sections to document step ordering.

[[axo_role]]
{axo_role}::
A
{axl_motif}
representing an operational identity pattern that defines capabilities and permissions.
Determines available operations and access boundaries.

[[axo_identity]]
{axo_identity}::
A
{axl_motif}
representing the abstract concept of an operational identity.
An entity capable of authentication and holding authorities within a system.
{axo_identity} is the pattern;
{axo_actor_s}
are named instantiations with specific bindings.

[[axo_actor]]
{axo_actor}::
A
{axl_motif}
representing a specific named identity with defined responsibilities and exclusive authority over an operational domain.
{axo_actor_s} instantiate
{axo_identity}
with concrete bindings (human via OAuth, service via service account) and permissions.

[[axo_entity]]
{axo_entity}::
A
{axl_motif}
representing an object with both structure and behavior.
Unlike
{axr_record}
which captures pure data,
{axo_entity}
includes methods, builder patterns, or lifecycle operations.
{axl_voicing_s}
describe the entity's data members and its behavioral interface (construction, mutation, registration).
Typical manifestations include Rust structs with `impl` blocks, builder-pattern APIs, or objects that participate in registration protocols.

{axl_motif}
in annotation format: `// ⟦axl_voices primary dimension ...⟧`

=== Value Dimensions

These
{axd_dimension_s}
describe value characteristics and apply across most serialization formats.
They encompass both presence semantics (required, optional, defaulted) and structural semantics (repeated, list).

[[axd_required]]
{axd_required}::
Member or argument must be present.
Absence triggers validation error or
{axc_fatal}.

[[axd_optional]]
{axd_optional}::
Member or argument may be omitted.
Absence is valid and handled gracefully.

[[axd_nullable]]
{axd_nullable}::
Null, nil, or None is a valid value, distinct from absent.
Required for JSON where null is an explicit value type.
Some formats (protobuf) conflate nullable with optional.

[[axd_none]]
{axd_none}::
Section or field is explicitly declared as having no content.
Distinguishes intentional absence from omission.
Use in section annotations to mark compliance with required sections that have no applicable content.

[[axd_defaulted]]
{axd_defaulted}::
Has fallback value when absent.
Voicing should specify the default value in prose.

[[axd_repeated]]
{axd_repeated}::
May have multiple values (array, list, repeated field).
Absence typically means empty collection, not error.
Applies to usage sites (arguments, members) where multiple occurrences are permitted.

[[axd_list]]
{axd_list}::
Single value containing delimited multiple values.
Distinct from
{axd_repeated}
which indicates multiple separate occurrences.
Applies to type voicings where the type intrinsically represents a delimited collection.
Voicing should specify the delimiter in prose (space-delimited, comma-delimited, etc.).

=== Format-Specific Dimensions

These
{axd_dimension_s}
apply only to specific serialization formats.
Using them in forbidden contexts is a type-scheme error.

[[axd_field_number]]
{axd_field_number}::
Numeric field identifier for protobuf encoding.
Required for protobuf format.
Forbidden for JSON, XML, YAML, binary.

[[axd_wire_type]]
{axd_wire_type}::
Protobuf encoding type (varint, fixed32, fixed64, length-delimited).
Required for protobuf format.
Forbidden for JSON, XML, YAML, binary.

[[axd_endianness]]
{axd_endianness}::
Byte order for binary encoding (big-endian, little-endian).
Required for binary format.
Forbidden for JSON, XML, YAML, protobuf.

[[axd_xml_form]]
{axd_xml_form}::
Whether XML member is attribute or child element.
Required for XML format.
Forbidden for JSON, YAML, protobuf, binary.

=== Lifecycle Dimensions

These
{axd_dimension_s}
qualify
{axo_procedure}
voicings to specify execution lifecycle.
Exactly one lifecycle dimension is required for any
{axo_procedure}
voicing.
Lifecycle dimensions are mutually exclusive.

[[axd_transient]]
{axd_transient}::
{axo_procedure}
executes and completes.
The default lifecycle for most procedures.
{axs_completion}
documents exit status or return value.

[[axd_longrunning]]
{axd_longrunning}::
{axo_procedure}
runs continuously until explicitly stopped.
{axs_completion}
documents startup success criteria, not termination.
Requires
{axs_startup}
and
{axs_shutdown}
sections.
May include
{axs_health},
{axs_signals},
{axs_recovery}
sections.

[[axd_periodic]]
{axd_periodic}::
{axo_procedure}
executes on a schedule.
{axs_completion}
applies per-run.
Requires
{axs_schedule}
section.
May include
{axs_idempotency},
{axs_overlap}
sections.

=== Regime Dimensions

These
{axd_dimension_s}
qualify regime structure annotations.

[[axd_conditional]]
{axd_conditional}::
Group activation depends on one or more gate conditions.
A group annotated with
{axd_conditional}
requires corresponding hierarchy markers in the subdocument to include at least one
{axhrgc_gate}
specifying the activation criteria.

Per-format guidance for
{axd_dimension}
usage:

[cols="<,^,^,^,^,^",options="header"]
|===
|Dimension |JSON |Protobuf |XML |Binary |YAML

|{axd_required}     |recommend |recommend |recommend |recommend |recommend
|{axd_optional}     |recommend |recommend |recommend |recommend |recommend
|{axd_nullable}     |required  |implicit  |recommend |context   |recommend
|{axd_defaulted}    |optional  |required  |optional  |optional  |optional

== Axial Regime Annotation Terms

Regime annotations use the prefix-discriminated form (`//ax...` with no space after `//`) to provide structured metadata about regime definitions in AsciiDoc specifications.
Two complementary annotation families enable regime specification across parent documents and their subdocuments.

=== Definition-Site Voicing Annotations (axvr_*)

{axvr_*} annotations appear in parent documents between anchor and definition, declaring the regime role of the term being defined.
The surface syntax follows the prefix-discriminated pattern:

[source,asciidoc]
----
[[rbrv_sigil]]
//axvr_variable axd_required rbst_xname
{rbrv_sigil}::
Vessel identifier for build configurations.
----

The annotation line starts with `//axvr_` (no space after `//`), followed by the specific voicing term, then space-separated dimension and type terms.
The anchor immediately preceding the annotation must match the first
{mcm_attribute_reference}
in the definition text that follows.

[[axvr_regime]]
{axvr_regime}::
Voices a regime definition.
The annotated term defines a structured configuration system that binds related
{axrg_variable_s}
under a namespace prefix.
+
{axd_dimension_s}:
{axf_format}
(specifying how
{axrg_assignment_s}
are carried).

[[axvr_variable]]
{axvr_variable}::
Voices a regime variable definition.
The annotated term defines a named configuration element within a regime.
The definition text must include, as its second
{mcm_attribute_reference},
the parent regime (an anchor that voices
{axrg_regime}).
This ties each variable unequivocally to its regime.
+
{axd_dimension_s}:
{axd_required}
or
{axd_optional},
type
{axl_motif}
(base type or type voicing).

[[axvr_group]]
{axvr_group}::
Voices a regime group definition.
The annotated term defines a named grouping of related
{axrg_variable_s}
within a regime.
The definition text must include, as its second
{mcm_attribute_reference},
the parent regime (an anchor that voices
{axrg_regime}).
+
{axd_dimension_s}:
{axd_conditional}
(group activation depends on gate conditions in subdocument).

=== Standalone Hierarchy Markers (axhr*_)

Hierarchy markers appear in regime subdocuments as standalone annotations, NOT between anchor and definition.
Each marker uses the prefix-discriminated form and reads ahead a specified number of
{mcm_attribute_reference_s}
from the following text to establish hierarchical context.

The surface syntax:

[source,asciidoc]
----
//axhrb_regime
{rbrv_regime}

//axhrgv_variable
{rbrv_sigil}
----

The hierarchy marker line starts with `//axhr` (no space after `//`), followed by the sub-prefix identifying the marker type.
Following text contains the
{mcm_attribute_reference_s}
that the marker reads ahead.

==== Marker Terms and Lookahead Arities

[cols="<,^,<",options="header"]
|===
|Marker |Lookahead |Reads

|{axhrb_regime}     |1 {mcm_attribute_reference}  |The regime this subdoc describes
|{axhrv_variable}   |1 {mcm_attribute_reference}  |An ungrouped regime variable
|{axhrgb_group}     |1 {mcm_attribute_reference}  |A group (must match {axvr_group} in parent)
|{axhrgc_gate}      |2 {mcm_attribute_reference_s}|Enumerated variable + enum value
|{axhrgv_variable}  |1 {mcm_attribute_reference}  |A variable within current group
|===

[[axhrb_regime]]
{axhrb_regime}::
Marks the regime that a subdocument describes.
Lookahead: 1
{mcm_attribute_reference}
— the regime this subdoc describes.
The referenced attribute must correspond to an anchor annotated with
{axvr_regime}
in the parent document.

[[axhrv_variable]]
{axhrv_variable}::
Marks an ungrouped regime variable in a subdocument.
Lookahead: 1
{mcm_attribute_reference}
— the variable being detailed.

[[axhrgb_group]]
{axhrgb_group}::
Marks a group boundary within a subdocument.
Lookahead: 1
{mcm_attribute_reference}
— the group being detailed.
The referenced attribute must match an anchor annotated with
{axvr_group}
in the parent document.

[[axhrgc_gate]]
{axhrgc_gate}::
Marks a conditional activation gate within a group.
Lookahead: 2
{mcm_attribute_reference_s}
— the enumerated variable (voicing
{axt_enumeration})
and the specific
{axt_enum_value}
that activates this gate.
The gate variable may belong to a different regime (cross-regime gating is legal).

[[axhrgv_variable]]
{axhrgv_variable}::
Marks a variable within the current group in a subdocument.
Lookahead: 1
{mcm_attribute_reference}
— the variable being detailed within the current group scope.

==== Nesting Rules

Hierarchy markers establish implicit scope through document order:

* Higher-level markers implicitly close previous scope (no end markers needed)
* {axhrgb_group},
{axhrgc_gate},
and
{axhrgv_variable}
must appear within
{axhrb_regime}
scope
* {axhrgc_gate}
and
{axhrgv_variable}
must appear within
{axhrgb_group}
scope
* Multiple
{axhrgc_gate}
within one group have AND semantics

==== Cross-Document Validation

* {axhrb_regime}
attribute reference must match a regime defined with
{axvr_regime}
in parent
* {axhrgb_group}
attribute reference must match a group defined with
{axvr_group}
under that regime
* {axhrgc_gate}
first attribute reference must voice
{axt_enumeration};
second must be an
{axt_enum_value}
of that specific enumeration type

== Compliance Rules

This section specifies compositional requirements between motifs.
When a term voices a particular motif, these rules define what companion elements are required.

=== Procedure Documentation Completeness

Voicing as
{axo_procedure}
requires documentation sections for:

* {axs_inputs}
— what the procedure requires to execute

* {axs_behavior}
— what the procedure does

* {axs_outputs}
— what the procedure produces

* {axs_completion}
— how the procedure signals success or failure

Optional sections that MAY be included:

* {axs_preconditions}
— environmental state requirements before execution

* {axs_postconditions}
— environmental state guarantees after success

* {axs_errors}
— failure modes and recovery guidance

=== Command Documentation Completeness

Voicing as
{axo_command}
requires all
{axo_procedure}
sections plus:

* {axe_environment}

{axe_bash_interactive},
{axe_bash_unattended})

Voicing as
{axo_guide}

{axo_procedure}

* {axs_behavior}
steps use
{axe_human_guide}

* Steps describe actions the human performs in external systems

{axo_procedure}

* {axd_transient}

* {axd_longrunning}

* {axd_periodic}

Voicing with
{axd_periodic}

{axo_procedure}

* {axs_overlap}
