== Core Principles

The Axial Lexicon provides reusable, atomic terminology for technical specifications.
Each term maintains precise semantics while adapting to implementation context.
Specifications selectively import only needed axial terms, avoiding vocabulary bloat.

== Motif Vocabulary

AXL terms are
{axl_motif_s}
- abstract patterns that concrete specifications voice in their local context.

[[axl_motif]]
{axl_motif}::
An abstract pattern defined in the Axial Lexicon that can be voiced by concrete terms in specifications.
A
{axl_motif}
captures the essential shape of a concept while remaining independent of implementation context.
Specifications voice
{axl_motif_s}
through local terms that bind the abstract pattern to specific environments.

[[axl_voicing]]
{axl_voicing}::
A concrete term in a specification that gives local expression to A
{axl_motif}.
Each
{axl_voicing}
adapts the abstract pattern to a specific context while preserving the
{axl_motif_s}
essential character.

[[axl_voices]]
{axl_voices}::
The relationship where a concrete term expresses a
{axl_motif}
in a specific context.
When a term
{axl_voices}
a
{axl_motif},
it gives the abstract pattern local form and implementation-specific behavior.

[[axl_evokes]]
{axl_evokes}::
The relationship where A
{axl_motif}
calls forth its
{axl_voicing_s}
in the reader's understanding.
When encountering A
{axl_motif},
it
{axl_evokes}
the concrete realizations known from various specifications.

[[axl_annotation]]
{axl_annotation}::
A
{xref_MCM}
comment using Strachey brackets that declares the
{axl_motif_s}
a term
{axl_voices}.
The
{axl_annotation}
appears between anchor and definition as a space-separated list of AXL terms within Strachey brackets.
+
Format: `// ⟦relationship primary [dimension ...]⟧`
+
* First term: the relationship (typically `axl_voices`)
* Second term: the primary
{axl_motif}
being voiced
* Remaining terms: dimension terms (prefixed `axd_`) specifying characteristics
+
Examples:
+
* `// ⟦axl_voices axc_call⟧` - simple voicing
* `// ⟦axl_voices axr_member axd_required axd_nullable⟧` - voicing with dimensions
+
Category prefixes carry semantic meaning, eliminating need for punctuation.
Position determines role: relationship first, primary second, dimensions following.
This design prioritizes LLM interpretability over human visual parsing.
When ambiguity or friction arises during LLM processing, the format should evolve to resolve it.

== Motif Implications

{axl_motif_s}
carry implied characteristics beyond their definitions.
These implications guide how
{axl_voicing_s}
should behave:

* Anchor placement expectations per
{xref_MCM}
form patterns
* Type/instance level - where the
{axl_motif}
sits in abstraction hierarchy
* Contextual requirements - parent structures or operations the
{axl_motif}
exists within
* Voicing expectation - whether the
{axl_motif}
typically requires local voicing

== Form Expectations

Each AXL category implies an expected
{xref_MCM}
definition form pattern for documents using that category.
During type-scheme linting, these expectations guide validation of definition structure.

* {axc_*} control terms expect
{xref_MCM}
`mcm_form_deflist`
* {axj_structure} definitions expect
{xref_MCM}
`mcm_form_section`
* {axj_field} definitions expect
{xref_MCM}
`mcm_form_bullet` under parent structure
* {axk_premise} definitions expect
{xref_MCM}
`mcm_form_deflist`
* Most other categories expect
{xref_MCM}
`mcm_form_deflist`

These expectations support the
{axl_annotation}
pattern where documents include `// ⟦axl_voices motif ...⟧` markers that are validated when AXL is in context.

== Axial Control Terms

[[axc_variable]]
{axc_variable}::
A
{axl_motif}
representing a temporary value holder denoted by distinctive delimiters that exists for the duration of an operational sequence.
Implementation varies by context: shell variables in bash, dictionary entries in Python, sessionStorage in browser JavaScript.

[[axc_fatal]]
{axc_fatal}::
A
{axl_motif}
representing immediate termination of the operational sequence with error status, preventing all subsequent steps from executing.
Requires operator intervention or error recovery before retry.

[[axc_warn]]
{axc_warn}::
A
{axl_motif}
representing display of a non-blocking warning message while allowing the operational sequence to continue execution.
Alerts operators to concerning conditions without halting progress.

[[axc_store]]
{axc_store}::
A
{axl_motif}
representing capture of a value from an operation result or computation into a
{axc_variable}
for later reference.
Context determines storage mechanism while preserving semantic intent.

[[axc_use]]
{axc_use}::
A
{axl_motif}
representing reference to a previously stored
{axc_variable}
value in the current operation.
Retrieves stored values from context-appropriate storage location.

[[axc_require]]
{axc_require}::
A
{axl_motif}
representing validation check that confirms an expected condition, triggering
{axc_fatal}
or alternate execution path if not met.
Enforces preconditions and invariants throughout execution.

[[axc_poll]]
{axc_poll}::
A
{axl_motif}
representing repeated execution of a status check at defined intervals until completion condition or timeout.
Implements retry logic with configurable intervals and limits.

[[axc_call]]
{axc_call}::
A
{axl_motif}
representing direct invocation of an external service, API endpoint, or command.
Captures both success and error responses for processing.

[[axc_submit]]
{axc_submit}::
A
{axl_motif}
representing initiation of an asynchronous operation that returns immediately with a handle for later completion tracking.
Distinct from
{axc_call}
which expects synchronous response.

[[axc_await]]
{axc_await}::
A
{axl_motif}
representing blocking wait on a submitted asynchronous operation until completion.
Polls operation status until done, then returns final result or error.

[[axc_show]]
{axc_show}::
A
{axl_motif}
representing presentation of operational facts or values to the operator.
Implementation varies from console output to UI display based on context.

[[axc_step]]
{axc_step}::
A
{axl_motif}
representing a normal progress message indicating successful completion of a processing stage.

== Axial Dimension Terms

{axd_dimension_s}
are modifiers that appear in
{axl_annotation_s}
to specify characteristics of members and arguments.
Unlike primary
{axl_motif_s},
{axd_dimension_s}
do not stand alone but qualify how a term voices its primary motif.
Applicability of
{axd_dimension_s}
depends on serialization format context.

[[axd_dimension]]
{axd_dimension}::
A characteristic modifier that qualifies a member or argument in an
{axl_annotation}.
{axd_dimension_s}
appear after the primary
{axl_motif}
in annotation format: `// ⟦axl_voices primary dimension ...⟧`

=== Value Dimensions

These
{axd_dimension_s}
describe value characteristics and apply across most serialization formats.
They encompass both presence semantics (required, optional, defaulted) and structural semantics (repeated, list).

[[axd_required]]
{axd_required}::
Member or argument must be present.
Absence triggers validation error or
{axc_fatal}.

[[axd_optional]]
{axd_optional}::
Member or argument may be omitted.
Absence is valid and handled gracefully.

[[axd_nullable]]
{axd_nullable}::
Null, nil, or None is a valid value, distinct from absent.
Required for JSON where null is an explicit value type.
Some formats (protobuf) conflate nullable with optional.

[[axd_none]]
{axd_none}::
Section or field is explicitly declared as having no content.
Distinguishes intentional absence from omission.
Use in section annotations to mark compliance with required sections that have no applicable content.

[[axd_defaulted]]
{axd_defaulted}::
Has fallback value when absent.
Voicing should specify the default value in prose.

[[axd_repeated]]
{axd_repeated}::
May have multiple values (array, list, repeated field).
Absence typically means empty collection, not error.
Applies to usage sites (arguments, members) where multiple occurrences are permitted.

[[axd_list]]
{axd_list}::
Single value containing delimited multiple values.
Distinct from
{axd_repeated}
which indicates multiple separate occurrences.
Applies to type voicings where the type intrinsically represents a delimited collection.
Voicing should specify the delimiter in prose (space-delimited, comma-delimited, etc.).

=== Format-Specific Dimensions

These
{axd_dimension_s}
apply only to specific serialization formats.
Using them in forbidden contexts is a type-scheme error.

[[axd_field_number]]
{axd_field_number}::
Numeric field identifier for protobuf encoding.
Required for protobuf format.
Forbidden for JSON, XML, YAML, binary.

[[axd_wire_type]]
{axd_wire_type}::
Protobuf encoding type (varint, fixed32, fixed64, length-delimited).
Required for protobuf format.
Forbidden for JSON, XML, YAML, binary.

[[axd_endianness]]
{axd_endianness}::
Byte order for binary encoding (big-endian, little-endian).
Required for binary format.
Forbidden for JSON, XML, YAML, protobuf.

[[axd_xml_form]]
{axd_xml_form}::
Whether XML member is attribute or child element.
Required for XML format.
Forbidden for JSON, YAML, protobuf, binary.

=== Lifecycle Dimensions

These
{axd_dimension_s}
qualify
{axo_procedure}
voicings to specify execution lifecycle.
Exactly one lifecycle dimension is required for any
{axo_procedure}
voicing.
Lifecycle dimensions are mutually exclusive.

[[axd_transient]]
{axd_transient}::
{axo_procedure}
executes and completes.
The default lifecycle for most procedures.
{axhoc_completion}
documents exit status or return value.

[[axd_longrunning]]
{axd_longrunning}::
{axo_procedure}
runs continuously until explicitly stopped.
{axhoc_completion}
documents startup success criteria, not termination.
Requires
{axhos_step} (startup)
and
{axhos_step} (shutdown)
sections.
May include
{axhos_step} (health, signals, recovery)
sections.

[[axd_periodic]]
{axd_periodic}::
{axo_procedure}
executes on a schedule.
{axhoc_completion}
applies per-run.
Requires
{axhos_step} (schedule)
section.
May include
{axhos_step} (idempotency, overlap)
sections.

=== Regime Dimensions

These
{axd_dimension_s}
qualify regime structure annotations.

[[axd_conditional]]
{axd_conditional}::
Group activation depends on one or more gate conditions.
A group annotated with
{axd_conditional}
requires corresponding hierarchy markers in the subdocument to include at least one
{axhrgc_gate}
specifying the activation criteria.

[[axrd_singleton]]
{axrd_singleton}::
Regime has exactly one assignment source.
No enumeration or cross-instance operations apply.

[[axrd_manifold]]
{axrd_manifold}::
Regime has multiple assignment sources (one per instance).
Enumeration and cross-instance operations are meaningful.

A
{axl_motif}
representing a structured configuration system.

{axrg_regime}

{axrg_variable_s}

{axrg_variable}::
A
{axl_motif}

{axrg_regime}

A
{axl_motif}

are carried in files of a particular
{axf_format}

[[axrd_file_sourced]]

== Axial Regime Annotation Terms

Regime annotations use the prefix-discriminated form (`//ax...` with no space after `//`) to provide structured metadata about regime definitions in AsciiDoc specifications.
Two complementary annotation families enable regime specification across parent documents and their subdocuments.

=== Definition-Site Voicing Annotations (axvr_*)

{axvr_*} annotations appear in parent documents between anchor and definition, declaring the regime role of the term being defined.
The surface syntax follows the prefix-discriminated pattern:

[source,asciidoc]
----
[[rbrv_sigil]]
//axvr_variable axd_required rbst_xname
{rbrv_sigil}::
Vessel identifier for build configurations.
----

The annotation line starts with `//axvr_` (no space after `//`), followed by the specific voicing term, then space-separated dimension and type terms.
The anchor immediately preceding the annotation must match the first
{mcm_attribute_reference}
in the definition text that follows.

[[axvr_regime]]
{axvr_regime}::
Voices a regime definition.
The annotated term defines a structured configuration system that binds related
{axrg_variable_s}
under a namespace prefix.
+
{axd_dimension_s}:
{axf_format}
(specifying how
{axrg_assignment_s}
are carried).

[[axvr_variable]]
{axvr_variable}::
Voices a regime variable definition.
The annotated term defines a named configuration element within a regime.
The definition text must include, as its second
{mcm_attribute_reference},
the parent regime (an anchor that voices
{axrg_regime}).
This ties each variable unequivocally to its regime.
+
{axd_dimension_s}:
{axd_required}
or
{axd_optional},
type
{axl_motif}
(base type or type voicing).

[[axvr_group]]
{axvr_group}::
Voices a regime group definition.

{axrg_variable_s}
within a regime.
The definition text must include, as its second
{mcm_attribute_reference},
the parent regime (an anchor that voices
{axrg_regime}).
+
{axd_dimension_s}:
{axd_conditional}
(group activation depends on gate conditions in subdocument).

[[axvr_kindle]]
{axvr_kindle}::
Voices a regime kindle operation definition.
The annotated term defines the silent initialization phase for a
{axrg_regime}:
setting defaults, detecting unexpected variables, and building derived structures.

[[axvr_validate]]
{axvr_validate}::

[[axvr_render]]

[[axvr_list]]
{axvr_list}::
Voices a regime list operation definition.
The annotated term defines an enumeration operation for an
{axrd_manifold}
{axrg_regime}:
listing available instances.

[[axvr_survey]]
{axvr_survey}::

{axrd_manifold}
{axrg_regime}.

[[axvr_audit]]
{axvr_audit}::
Voices a regime audit operation definition.

{axrd_manifold}
{axrg_regime}.

== Axial Hierarchy Operation Markers

{axho*} markers appear at the
{axl_detail_site}

{axhr*}

|===
|Marker |Lookahead |Reads

{axhob_operation}::
Opens the operation hierarchy block at the
{axl_detail_site}.

{mcm_attribute_reference}
— the operation this detail site describes.

{axvo_procedure}
or
{axvo_method}
at the

{axhoc_completion}::
Marks the signaling contract section in an operation
{axl_detail_site}.
0-arity bare marker — no lookahead required; content follows directly.
Documents how the operation communicates success or failure: exit codes for CLI, HTTP status for REST, return values for functions.
Distinct from
{axhog_guarantee}
which describes state promises, and
{axhoo_output}

[[axhop_parameter]]
{axhop_parameter}::
Abstract parent for parameter markers in an operation
{axl_detail_site}.
Do not use directly — use one of the concrete variants:
{axhop_parameter_from_type}
or
{axhop_parameter_from_arg}.

{axhop_parameter}

{mcm_attribute_reference}

[[axhop_parameter_from_arg]]
{axhop_parameter_from_arg}::
Variant of
{axhop_parameter}
whose lookahead reuses a shared CLI argument term.

Used when the parameter is a well-known CLI argument that appears across multiple operations.
Lookahead: 1
{mcm_attribute_reference}
— must resolve to an existing shared CLI argument anchor; small vocabulary per spec.
Individual markers per parameter; no section wrapper needed.

[[axhoo_output]]
{axhoo_output}::
Abstract parent for output markers in an operation
{axl_detail_site}.
Do not use directly — use
{axhoo_output_of_type}.

{axhoo_output_of_type}::

{axhoo_output}

{axhop_parameter_from_type}

{mcm_attribute_reference}

{axhoq_precondition}::
Marks a required precondition in an operation

[[axhog_guarantee]]
{axhog_guarantee}::
Marks a postcondition guarantee in an operation
{axl_detail_site}.
No lookahead required — bare marker; content follows directly.

[[axhos_step]]
{axhos_step}::
Marks a named behavioral step in an operation
{axl_detail_site}.
No lookahead required — bare marker; content follows directly.

{axhos_waymark}::

== Compliance Rules

This section specifies compositional requirements between motifs.
When a term voices a particular motif, these rules define what companion elements are required.

=== Operation Detail-Site Completeness

A term voiced
{axvo_procedure}
or
{axvo_method}
with a detail-site subdocument requires:

* {axhob_operation}
— opens the hierarchy block, with lookahead to the operation's linked term

* At least one
{axhos_step}
— what the operation does

* {axhoc_completion}
— how the operation signals success or failure

Optional markers that MAY be included:

* {axhop_parameter}
— input parameters (with lookahead to existing domain type or CLI argument term)

* {axhoo_output}
— produced artifacts (with lookahead to existing domain type term)

* {axhoq_precondition}
— environmental state requirements before execution

* {axhog_guarantee}
— environmental state guarantees after success

=== CLI Subcommand Completeness

Voicing as
{axi_cli_subcommand}
requires:

* Exit Status section voiced as
{axa_exit_uniform}
or
{axa_exit_enumerated}
(which voice
{axhoc_completion}
in CLI context)

* Arguments section voiced as
{axa_argument_list}
(which voices
{axhop_parameter}
in CLI context; may list zero arguments if operation takes none)

=== CLI Option Completeness

Voicing as
{axa_cli_option}
requires the definition text to specify:

* Long form (e.g., `--file`)

* Short form or explicit absence (e.g., `-f` or "none")

* Default value, if the argument is used with
{axd_defaulted}

=== Argument List Item Qualification

Within an
{axa_argument_list},
each item requires a preceding
{axd_dimension}
annotation specifying at minimum:

* {axd_required}
or
{axd_optional}

* {axd_defaulted}
if a default value applies

=== Type Voicing Qualification

Voicing a type (subspecializing
{axt_*}
or
{axtu_*}
motifs) constrains dimension usage.

Valid dimensions:

* {axd_list}
— type intrinsically represents delimited collection

Invalid dimensions:

* {axd_required},
{axd_optional},
{axd_defaulted}
— presence applies to usage sites, not type definitions

* {axd_repeated}
— repetition applies to usage sites, not type definitions

=== Regime Specification Completeness

Voicing as
{axrg_regime}
requires:

* Format dimension (
{axf_bash},
{axf_makefile},
{axf_json},
etc.) specifying how assignments are carried

* Provenance dimension (
{axrd_file_sourced},
{axrd_env_inherited},
or
{axrd_constructed})
declaring how
{axrg_assignment_s}
arrive

=== Regime Variable Qualification

Voicing as
{axrg_variable}
requires:

* Type in annotation (base type motif or type voicing)

* At minimum
{axd_required}
or
{axd_optional}

* {axd_defaulted}
if a default value applies

Valid dimensions:

* {axd_required}
— variable must have assignment

* {axd_optional}
— variable may be absent

* {axd_defaulted}
— fallback value when absent

* {axd_list}
— inherited from type voicing; variable's type is a delimited collection

Invalid dimensions:

* {axd_repeated}
— regime files use unique keys; multiple assignments of same variable not permitted

* {axd_nullable}
— format-dependent;
{axf_bash}
has no null distinct from empty string

=== Regime Voicing Annotation Completeness

{axvr_regime}
requires:

* {axf_format}
dimension specifying how
{axrg_assignment_s}
are carried

{axvr_variable}
requires:

* Second
{mcm_attribute_reference}
in definition text must be the parent regime

* {axd_required}
or
{axd_optional}

* Type
{axl_motif}
(base type or type voicing)

{axvr_group}
requires:

* Second
{mcm_attribute_reference}
in definition text must be the parent regime

=== Regime Hierarchy Consistency

* {axvr_group}
with
{axd_conditional}
requires corresponding
{axhrgb_group}
in subdocument to have at least one
{axhrgc_gate}

* {axvr_group}
without
{axd_conditional}:
corresponding
{axhrgb_group}
must have no
{axhrgc_gate}

* Every
{axvr_variable}
under a regime should appear as either
{axhrv_variable}
or
{axhrgv_variable}
in the subdocument

=== Operation Voicing Annotation Completeness

{axvo_procedure}
requires:

* Exactly one lifecycle dimension:
{axd_transient},
{axd_longrunning},
or
{axd_periodic}

* If
{axd_grouped}:
second
{mcm_attribute_reference}
in definition text must be an
{axo_group}

{axvo_method}
requires:

* Second
{mcm_attribute_reference}
in definition text must be the affiliated
{axo_entity}

* Exactly one lifecycle dimension:
{axd_transient},
{axd_longrunning},
or
{axd_periodic}

* If
{axd_grouped}:
third
{mcm_attribute_reference}
in definition text must be an
{axo_group}

{axvo_group}
requires:

* Second
{mcm_attribute_reference}
in definition text must be the affiliated
{axo_entity}

* Optional exposure dimension:
{axd_tabtarget},
{axd_slash_command},
or other interface mechanism specifying the default exposure for external members

=== Operation Hierarchy Consistency

* {axhob_operation}
attribute reference must match a term voiced
{axvo_procedure}
or
{axvo_method}
at the
{axl_definition_site}

* Every
{axhoc_completion},
{axhop_parameter},
{axhoo_output},
{axhoq_precondition},
{axhog_guarantee},
and
{axhos_step}
must appear within
{axhob_operation}
scope
