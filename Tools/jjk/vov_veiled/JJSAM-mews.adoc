== What the mews is

The *mews* is the operator's fleet: the machines available for Job Jockey's remote work — stood
up, tracked, and allocated.
It is the maturation of foray: today foray dispatches to an explicit `(alias, reldir)`; the mews
lets it pick from a registered, reserved pool.
Falconry register (soaking): *mews* = the operator-scoped home where provisioned environments are
kept between flights; *jess* = the key tether binding a provisioned user.

The fleet lifecycle is one domain: *provision → register → reserve → dispatch*.

== Why it lives in Job Jockey

The mews is operator-private operational knowledge — the testing apparatus, the machines.
JJK is not released the way RBK is, so housing the fleet here keeps it out of any public surface
by construction, not by vigilance.
The dependency concern resolves the same way: RBK test suites are *dispatched through foray*,
never direct consumers of the fleet registry, so no RBK→JJK coupling arises.

== Distinct state repo (sibling to the gallops repo)

The mews *code and vocabulary* live in JJK; its *runtime state* — the fleet inventory plus live
reservations — lives in its own git repository, logically distinct from the gallops/planning
record.
The two separate on scope (operator-global fleet vs. project planning), cadence (slow operational
vs. fast intentional), and blast radius.
One vvx engine drives both; two repos, two locks.
The fleet store is the anticipated *second instance* of the locked-store entity
(*blotter*, candidate-minted at the revision-control canon workup, 260705):
one engine drives both stores through the same entity machinery — the sibling rhyme resolved
into a designed convergence, while two repos, two locks stands.

== The fleet, in three parts

* *Registry* — which nodes exist, of what type, how reached.
  This is the BURN node-profile set matured: BURN is proto-mews.
* *Reservation* — soft, token-granted node ownership, so two masters can work in parallel from
  different places without collision; an operator can administratively reclaim a stuck
  reservation. No distributed-consensus machinery (sole-operator fit).
* *Provisioning* — standing a node up: admin host posture and workload-account creation (today's
  caparison / invigilate / garrison over BURN/BURP).
  The privileged transport stays bash under BCG+WSG — the Windows transport Palisade is not
  rewritten for purity — so the mews is Rust core plus owned bash transport.

=== Invariants (re-gestate intact)

* *Two records, one identity.* Node shape (host, OS family — per-node, git-transported,
  changes rarely) separates from credentials (privileged + workload key material —
  per-station-user); they join 1:1 on a shared node identity (₣A-'s *investiture*), and a
  node's host is stated only in the node record, reached by reference from the credential side.
  BURN/BURP was the proto-mews carrier of this split; the split outlives any carrier.
* *Three verbs, one division of labor.* Establish admin posture (nuclear — purge and
  re-establish, never skip-if-present); verify posture (read-only, no standalone entry point —
  invoked as establish's post-check and provision's precondition, the single source of truth
  for "correctly configured", each fact carrying a read command, expected value, and a failure
  pointer to whichever side owns it); provision the workload account (workload-only,
  destructive).
* *Destructive ownership.* Every provision wipes prior workload state; node runtime state
  belongs to whoever last provisioned; one workload user per node, ever.
  The account-lifecycle verb zoo (conscript / discharge / inventory / withdraw) was weighed
  and rejected — provisioning subsumes it.
* *Keys are operator-owned.* The system never generates or modifies SSH key material; pubkeys
  derive from privkeys at use-time.
  Corollaries that held: passphrase-protected privkeys unsupported (derivation would prompt);
  out-of-band pubkey registration (e.g. GitHub) is a precondition the system states but cannot
  validate.
* *The handbook boundary.* Security-sensitive host edits stay operator-manual by design
  (registry edits, VPN/IdP first-auth, first Remote-Login enablement); automation owns only
  idempotent admin-shell operations; the verify verb's failure pointers encode which side owns
  each fact.
* *Self-contained nodes.* Standing up a fresh node needs no other node: author its records,
  run the operator-manual handbook at the keyboard, then establish and provision over the
  tailnet the node self-authed to.
  (₣A-'s converge-against-a-known-host-first was a sequencing choice, not architecture.)
* *Transport posture.* Bash conducts; the platform shell is a syscall layer taking one atomic
  command per call — the BCG+WSG discipline the provisioning bullet above already pins.

=== Windows Palisade facts (surveyed; do not re-derive)

* *Two-phase SSH trust is structural.* Hardening sshd restarts it, killing the session that
  did the hardening; reconnect-and-verify is a phase boundary, not a retry.
  First contact may fall back to a password typed on /dev/tty (never seen by the tool); every
  later session is key-only.
* *The workload owns its WSL distribution.* Per-user distro registration lands in the hive of
  the logon session that performs the import, so provisioning drives the import from an
  SSH-as-workload session; the admin side only produces a seed image on disk and never
  registers the distro itself (the admin-no-WSL shape).
  The identity boundary is the SSH session, not a scheduler hop.
* *Reboot is the canonical Windows state reset.* In-session destructive cleanup (hive unload,
  profile teardown) is unreliable; the chosen primitive is an unconditional reboot, with the
  rejected alternatives recorded in
  `Memos/memo-20260511-windows-hive-cleanup-reboot-decision.md`.
* *Headless accounts break relative-path key resolution.* Windows OpenSSH resolves a relative
  `AuthorizedKeysFile` via GetUserProfileDirectoryW, which fails for never-logged-on accounts;
  an absolute-path Match block is structurally required.
  `Memos/memo-20260516-windows-headless-account-anatomy.md` is the account-internals bible.
* *One container daemon per Windows host.* WSL2 distros share one network namespace; two
  daemons fight over kernel iptables/NAT rules.
  ₣A- cinched Docker Desktop only (per-distro WSL integration); a native in-distro daemon is
  deferred to operator-managed exclusive lifecycle.
* *Docker Desktop requires a live desktop session.* The engine itself boots headless — service
  start plus a backend launch from an admin SSH session brings up dockerd and its API — but the
  GUI process cannot survive without an interactive desktop, and the backend tears down with it
  (surveyed on rocket, 260712, under heat ₣Bs).
  A disconnected RDP session satisfies the requirement; a Home-edition host has no RDP server,
  so a headless Home box structurally cannot run the Desktop engine and — under the one-daemon
  fact above — can serve only the deferred native-dockerd shape.
  Fleet consequence: the release-1 Windows runtime, and the off-path-RST behavior only it
  exhibits (the `rbsq_wdd_offpath_reply` survey), are provable only on a node holding a
  persistent logon session.
* *Known parked defects in the veiled code*, recorded so re-gestation does not rediscover
  them: workload stdout arrives UTF-16LE-mangled because the forced-command directive does not
  set `$env:WSL_UTF8=1`; obliterate may need wildcard cleanup of orphan `C:\Users\<workload>.*`
  profile dirs, which otherwise mint `.machinename`-suffixed profiles on the next cycle.

  the mews or re-gestates RBK-side is undecided.
* *Still deferred, still named:* PowerShell as a workload shell-letter; localhost provisioning
  variants; per-purpose workload→GitHub keypair separation.
