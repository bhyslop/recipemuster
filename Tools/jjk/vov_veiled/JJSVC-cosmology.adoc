= Revision-Control Cosmology

The world model of the revision-control substrate: the shared gestalt the sibling contract
sheaves — store, driver, dispatch, tackle — assume and none restates.

== The world model

Job Jockey's own record — the {jjdgr_gallops}, and later the captured chat histories — lives
in a git repository *separate from the code being jockeyed and global across stations*: the
{jjdb_studbook}, whose entity contract is its own sheaf.
Work happens in vanilla clones of consumer projects ({jjdw_hippodrome_s}) and their transient
per-pace worktrees ({jjdd_billet_s}), all peers in one parent directory (the {jjdw_infield}).
A polymorphic revision-control driver (the {jjdf_farrier}) fits the same JJ intent to
whatever ground each repo runs on.

JJ launched anywhere derives everything it needs from where it stands:
origin URL → sire (the registered body of work) → {jjdb_pedigree} → {jjdf_farrier} kind,
trunk posture, {jjdl_tackle} bindings.
One derivation chain, no per-repo pointers, no clone-local state.
The trunk of a {jjdw_hippodrome} belongs to the {jjdw_operator} alone — the {jjdw_engine}
works in billets and records in the {jjdb_studbook}, and never handles the
{jjdw_operator_p} serializing branch.

== The two parties

The world model has exactly two acting parties; everything else is ground they act on.

[[jjdw_operator]]
{jjdw_operator}::
The human party: the person at the shell who directs the work and dispatches every session.
The {jjdw_operator} alone touches the trunk of a {jjdw_hippodrome} — the serializing branch
is never the {jjdw_engine_p} to advance.
The word is the corpus's standing name for this party, catalogued here;
the sole-operator premise (`jjdk_sole_operator`) bounds how many there are.

[[jjdw_engine]]
{jjdw_engine}::
JJ's acting machinery as one party: the {jjdx_mcp} and the {jjdf_farrier} and
{jjdb_blotter} apparatus beneath it — the single mechanical actor that drives every
{jjdb_blotter} and carries its own stores' coordinates.
Sibling sheaves' *engine-driven* and *engine-known* speak against this definition.
Word tolerance: foreign proper nouns elsewhere in the corpus (Docker Engine, Podman's
engine) are Palisade names, never this party; the collision is deliberate and tolerated —
the living interior word was blessed over a fresh mint.

== Why decouple

Four recorded pressures, each dissolved structurally rather than patched:

* *Contamination.* Co-residence makes JJ bookkeeping commits part of the work repo's history
  and advances its HEAD — the recorded symptom is the pensum-seed eviction (JJ commits
  advanced HEAD past origin and tripped foray's curia-readiness guard).
  A separate repo dissolves the whole contamination class, not one instance.
* *Merge relief.* Across synced clones, every {jjdgr_gallops} change is a 3-way merge of
  canonical JSON; a single global record edited under a lock has no cross-clone merge at all.
* *Identity collision.* Distributed next-free {jjdt_coronet} allocation across divergent
  clones deterministically double-mints (the ₢BfAA3 incident,
  `Memos/memo-20260702-jjk-gallops-coronet-collision-incident.md`).
  One record behind one lock means one seed — the collision class dies at the root.
* *Vanilla hippodromes.* State out of the work repo is what lets a work repo carry no JJ
  inserts at all ({jjdw_hippodrome}).

== Infield

[[jjdw_infield]]
{jjdw_infield}::
The peer ring: one parent directory holding every JJ-managed tree as a sibling.
Residents: the kit repo and the {jjdb_studbook} (both {jjdw_yard} dirnames), consumer
{jjdw_hippodrome_s} and their transient {jjdd_billet_s}, and the BUK station family
(`../station-files/burs.env`, `../logs-buk`, `../temp-buk`, `../output-buk`), which already
resolves to this level — every peer tree shares station geography with zero setup.
Infield-peer placement is therefore load-bearing: nesting a tree deeper breaks the station
geography.
+
The {jjdw_infield} is a plain directory, not a repository.
Because JJ's own residents land here rather than inside any repo, no repo needs a JJ
gitignore entry — the infield tracks nothing, and nothing JJ owns ever lands in a
{jjdw_hippodrome}.

[[jjdw_yard]]
{jjdw_yard}::
The dirname brand (mnemonic: yard) marking JJ-owned residents of the {jjdw_infield},
carried by two signets split by resident kind:
`jjq_` for the directory residents — `jjqa_app` the kit repo, `jjqs_studbook` the record
repo, and each `jjqb_{coronet}` pace billet — and `jjy_` for the launcher scripts
(dispatch sheaf).
Charter: *every* JJ-owned infield resident carries a yard brand — `jjq_` if a directory,
`jjy_` if a script.
The split is yard-side vs. wire-side legibility: the yard signets name what the operator
touches on disk; `jjx_` names the MCP wire.
{jjdw_hippodrome_s} stay consumer-named — the brand marks JJ's own residents only.

== Hippodrome

[[jjdw_hippodrome]]
{jjdw_hippodrome}::
A work clone JJ operates from — a vanilla clone of a consumer project, carrying *no JJ
inserts*: no committed dossier, no `.claude/jjm` state, no JJK includes.
A vanilla external project is a working {jjdw_hippodrome} the moment its {jjdb_pedigree}
exists.
Identity is the directory name — there is no clone-local state at all (v1 ruling) — and the
clone's origin URL derives the upstream key that resolves its sire.
Several clones of one sire are several {jjdw_hippodrome_s}: the operator's standing
concurrent-parallel-clone work model is preserved, not replaced.
