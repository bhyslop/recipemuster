== Overview

The
{jjdgr_gallops}
is the persistent data store for Job Jockey.
It tracks
{jjdhr_heat_s}
(bounded initiatives) and their constituent
{jjdpr_pace_s}
(discrete actions).

== Design Principles

=== Exclusive JSON Ownership

All access to the
{jjdgr_gallops}
goes through {jjdx_handler} functions within {jjdx_vvx}.
No jq or direct JSON manipulation ever touches this data.
This is an absolute architectural constraint, not a preference.

The rationale: Rust {jjdx_handler} functions replace fragile jq invocations with compiler-enforced correctness.
The compiler is tireless; humans are not.

=== Git as Journal

The
{jjdgr_gallops}
remains JSON specifically so git can serve as the journalling layer.
Every mutation creates a diffable commit.
Git owns the complete history; we need not implement our own audit trail.

Git has the full history of every mutation.
Plan evolution is visible through commit diffs without additional in-JSON bookkeeping.

=== Diff-Friendly Mutations

The
{jjdhm_order}
array contains full
{jjdt_coronet_s}.
Each
{jjdt_coronet}
embeds its parent
{jjdt_firemark}
(redundant within a single
{jjdhr_heat}).
This redundancy is cheap and enables grep, copy-paste, and self-documenting JSON.

=== Crash-Safe Architecture

Every operation follows the same invariant regardless of transport:
lock → load → transform → save → unlock.
+
The {jjdx_handler} acquires a git-ref lock (`git update-ref`),
loads the {jjdgr_gallops} JSON,
performs the transformation,
writes atomically (temp file then rename),
and releases the lock.
If the {jjdx_handler} crashes mid-transform, no partial writes occur.
+
No in-memory {jjdgr_gallops} state persists between operations.
Each operation reads from disk and writes back.
This is critical for the {jjdx_mcp}, which is a long-lived process:
caching state between calls would produce stale reads when multiple sessions write concurrently.

=== MCP Transport

[[jjdx_mcp]]
//axvo_procedure axd_longrunning
{jjdx_mcp}::
The `vvx mcp` subcommand starts a long-lived stdio server implementing the Model Context Protocol (MCP) specification version 2025-11-25.
The server communicates via JSON-RPC 2.0 over stdin/stdout.
Implemented using the `rmcp` crate (official Rust MCP SDK), version pinned exactly.
+
The {jjdx_mcp} process is stateless infrastructure — it does not manage {jjdxo_officium} lifecycle.
{jjdxo_officium_s} are created by explicit agent action ({jjdo_open}), not by server process startup.
See <<jjdxo_officium>> for lifecycle details.

[[jjdx_tool]]
//axl_voices axo_entity
{jjdx_tool}::
A JSON-RPC tool call within an {jjdx_mcp} session.
Each operation is registered as an {jjdx_tool} with typed parameter schemas and descriptions.
{jjdx_tool_s} are the sole invocation path for Claude Code.
The {jjdx_cli} CLI subcommand path is removed;
{jjdx_tool_s} are NOT also available as CLI subcommands.
+
Infrastructure commands ({jjdx_vvx} emplace, unlock) remain CLI-only.

[[jjdx_handler]]
//axl_voices axo_entity
{jjdx_handler}::
The transport-agnostic Rust function that implements an operation.
Both {jjdx_tool_s} and (historical) CLI subcommands invoke the same {jjdx_handler}.
{jjdx_handler} functions accept typed parameters and return structured results;
the transport layer (MCP or CLI) handles serialization.

Each {jjdx_tool} call is stateless: the {jjdx_handler} performs the full crash-safe invariant (lock → load → transform → save → unlock) per invocation.
The {jjdx_mcp} holds no {jjdgr_gallops} state between calls.
+
Operations that previously read text from stdin (
{jjdo_enroll},
{jjsoprd_revise_docket},
{jjdo_transfer},
{jjdo_close},
{jjdo_landing})
accept structured string parameters instead.
The {jjdx_mcp} resolves the {jjdgr_gallops} path internally;
{jjda_file} is not exposed as a tool parameter.

=== Officium Lifecycle

[[jjdxo_officium]]
//axve_entity
{jjdxo_officium}::
A bounded agent session with a unique identity and a dedicated filesystem directory.
Each chat session has exactly one {jjdxo_officium}, created by explicit agent action ({jjdo_open}).
+
The {jjdxo_officium} is decoupled from {jjdx_mcp} process lifetime.
The {jjdx_mcp} is stateless disposable infrastructure — it may restart at any time without affecting the {jjdxo_officium}.
The agent (chat session) is the stable identity anchor; the {jjdxo_officium} ID is a routing parameter passed on every {jjdx_tool} call.
+
The term derives from the Latin _Officium Divinum_ (Divine Office) — a structured, bounded period of intentional work.
+
Multiple {jjdxo_officium_s} may coexist concurrently (multiple Claude Code windows).
Per {jjdk_sole_operator}, all belong to a single operator.
The commit lock serializes their mutations; the {jjdxo_exchange} isolates their gazette file I/O.

==== Identity

Each {jjdxo_officium} is assigned a unique identifier by {jjdo_open}.
Format: `YYMMDD-NNNN` where `YYMMDD` is the local datestamp and `NNNN` is an autonumber (enumerate existing directories for the date, pick next unused starting at 1000).
No persistent counter file — the directory listing is the source of truth.
+
Unicode verification prefix: ☉ (U+2609 SUN, evoking the Divine Office's daily cycle).
The ☉ appears in parameters and display; it is stripped for the directory name.
This parallels the ₣/₢ convention for {jjdt_firemark_s} and {jjdt_coronet_s}.
+
The identifier appears in the {jjdxo_exchange} path and in the {jjdxo_invitatory} commit.

==== Directory Layout

Each {jjdxo_officium} creates a directory via {jjdo_open}:
+
----
.claude/jjm/officia/<id>/gazette_in.md   (ephemeral: agent → server)
.claude/jjm/officia/<id>/gazette_out.md  (ephemeral: server → agent)
.claude/jjm/officia/<id>/heartbeat
----
+
The `officia/` directory is the parent for all active {jjdxo_officium_s}.
Gazette files (`gazette_in.md`, `gazette_out.md`) have single-MCP-call lifetime — they are parameters and return values, not persistent state.
The {jjdxo_heartbeat} file provides liveness detection.
+
The entire `.claude/jjm/officia/` tree is gitignored — it is ephemeral exchange infrastructure, not tracked state.
This eliminates git noise from officium operations and makes incorrect {jjdxo_exsanguination} benign (no tracked files lost).
+
Directories that survive session termination are stale and are reaped by {jjdxo_exsanguination} at the next {jjdo_open}.

[[jjdxo_exchange]]
//axve_entity
{jjdxo_exchange}::
The per-{jjdxo_officium} directory (`.claude/jjm/officia/<id>/`) used for {jjsz_gazette} file exchange between {jjdx_tool_s} and Claude.
+
Only one {jjsz_gazette} file exists at a time: {jjdxo_gazette}.
Operations that produce gazette output write to this file; Claude reads it.
Claude writes gazette input to this file; operations that accept gazette input read it.
+
The sequential nature of tool calls (one at a time per session) means the single file never has concurrent readers and writers within one {jjdxo_officium}.

[[jjdxo_gazette]]
//axve_entity
{jjdxo_gazette}::
Two directional gazette files within an {jjdxo_exchange}: `gazette_in.md` (agent to server) and `gazette_out.md` (server to agent).
Contain {jjsz_gazette} wire-format markdown.
Both have single-MCP-call lifetime — every {jjdx_tool_s} call deletes both files on entry before dispatch.
+
`gazette_in.md` is written by Claude before setter operations ({jjdo_enroll}, {jjsoprd_revise_docket}, {jjdo_paddock} setter).
`gazette_out.md` is written by the server after getter operations ({jjdo_orient}, {jjdo_show}, {jjdo_paddock} getter).

[[jjdxo_invitatory]]
//axvo_procedure axd_internal
{jjdxo_invitatory}::
The officium creation procedure.
Triggered by the agent calling {jjdo_open} — not by {jjdx_mcp} process startup.
The agent calls {jjdo_open} once per chat session.
+
{jjds_behavior}
+
. Run {jjdxo_exsanguination} (reap stale officia before creating new one)
. Generate {jjdxo_officium} identity (datestamp + autonumber)
. Create {jjdxo_exchange} directory with {jjdxo_gazette} and {jjdxo_heartbeat} files (both empty)
. Collect model inventory via `vvcp_probe()` if daily probe has not yet run (check `.claude/jjm/officia/.probe_date` datestamp file)
. Create invitatory commit (action code `i`, `jjb`-branded) with model inventory in body (or empty body if probe skipped)
. Return {jjdxo_officium} identity string (e.g. `☉260327-1000`)
+
The agent receives the identity and passes it as a parameter on all subsequent {jjdx_tool} calls.

[[jjdxo_compline]]
//axve_entity axd_retired
{jjdxo_compline}::
Retired.
The original design called for a clean shutdown procedure at {jjdx_mcp} process termination.
This proved unworkable: {jjdx_mcp} processes restart at unpredictable frequency, decoupled from chat session boundaries.
+
Staleness is now detected by {jjdxo_heartbeat} mtime absence.
Cleanup is performed by {jjdxo_exsanguination} at the next {jjdo_open}.
Sessions simply stop — no compline runs.
+
The action code `o` is reserved-unused.

[[jjdxo_heartbeat]]
//axve_entity
{jjdxo_heartbeat}::
A sentinel file within the {jjdxo_exchange} directory: `heartbeat`.
Every {jjdx_tool} dispatch touches this file (updates mtime).
+
The {jjdxo_heartbeat} provides liveness detection without coupling to process trees or Claude Code internals.
A stale {jjdxo_heartbeat} (mtime exceeding the staleness threshold) indicates the owning session has ended.
{jjdxo_exsanguination} uses {jjdxo_heartbeat} mtime to identify directories safe to reap.

[[jjdxo_exsanguination]]
//axvo_procedure axd_internal
{jjdxo_exsanguination}::
The stale officium reaping procedure.
Runs at the beginning of each {jjdxo_invitatory}, before creating the new {jjdxo_officium} directory.
+
{jjds_behavior}
+
. Scan `.claude/jjm/officia/` for directories
. For each directory, check {jjdxo_heartbeat} mtime
. Remove directories whose {jjdxo_heartbeat} exceeds the staleness threshold (7 days)
. Skip directories with no {jjdxo_heartbeat} file (legacy or corrupt — log warning, leave for manual cleanup)
+
The week-long staleness threshold ensures all same-day directories survive long enough for the autonumber scan to see them, preventing {jjdxo_officium} identity collisions within a day.
Because the `officia/` tree is gitignored, incorrect reaping is benign — no tracked files are lost.

==== Steeplechase Commit Patterns

{jjdxo_invitatory} commit (action code `i`):
+
----
jjb:BRAND::i: OFFICIUM <id>

haiku: <model-id>
sonnet: <model-id>
opus: <model-id>
host: <hostname>
platform: <platform>
----
+
The body contains model inventory when `vvcp_probe()` runs (first {jjdxo_invitatory} of the day).
On subsequent invitatory commits the same day, the body is empty.
+
The invitatory commit uses empty identity (no firemark or coronet) — the officium is infrastructure, not affiliated with a heat.
+
The {jjdxo_compline} action code `o` is reserved-unused.
No compline commits are created — sessions end without ceremony.

=== Deterministic Serialization

Implementations MUST serialize JSON object keys in deterministic order to minimize diff churn.
For `heats` and `paces` objects, use sorted key order (e.g., Rust `BTreeMap` instead of `HashMap`).
This ensures that reordering paces via
{jjdo_reorder}
produces minimal diffs — only the `order` array changes, not the entire `paces` object.
+
The `heats` object uses BTreeMap (alphabetical key order).
{jjdgm_order}
is the authoritative source for heat display and selection order.
Operations that need heat order (
{jjdo_list},
default heat selection in
{jjdo_orient})
iterate
{jjdgm_order},
not the `heats` object directly.

== Premises

[[jjdk_sole_operator]]
//axk_premise
{jjdk_sole_operator}::
All concurrent {jjdxo_officium_s} belong to a single operator.
The commit lock serializes operations across parallel {jjdxo_officium_s};
cross-user concurrency is out of scope.
+
Consequently, every operation acquires the lock unconditionally — there is no
read-only fast path, and the {jjsodp_command_lifecycle} dispatcher need not
distinguish mutating from non-mutating operations.
+
{jjdxo_officium_s} need only identification (which session), not authentication (who).

[[jjdk_reldir_safety]]
//axk_premise
{jjdk_reldir_safety}::
Remote dispatch involves destructive operations on the
{jjdxr_fundus}
(workspace cleanup via
{jjdo_plant}).
A malformed
{jjdt_reldir}
could resolve to `/` or `~`, destroying the filesystem or the user's home.
Three layers defend against this; no single layer is sufficient alone.
+
*Layer 1 — Rust constant validation (compile-time obvious):*
Must not be empty, must not start with `/` (absolute), must not start with `.` (traversal), must contain at least one `/` (forces subdirectory depth).
+
*Layer 2 — Fundus-side resolved-path check:*
The remote command validates the resolved path before any destructive operation: must be strictly under `$HOME`, must have minimum depth (at least 2 components under `$HOME`).
+
*Layer 3 — Bind-time probe:*
{jjdo_bind}
performs the same resolved-path check at session creation.
Fail fast before any
{jjdo_relay}
is ever attempted.

== Equestrian Vocabulary

=== Vocabulary Isolation

Job Jockey's terminology is drawn entirely from equestrianism.
This is not decoration — it is a namespace discipline applied to human cognition and language model disambiguation.

Planning systems face a persistent hazard: when the vocabulary used to _organize_ work overlaps with the vocabulary used to _do_ work, the boundary between plan and product dissolves.
"Task" could be a planning unit or a runtime construct.
"Phase" could be a workflow stage or a build step.
"Diff" could be a review artifact or a git object.
Every ambiguous term forces a context switch — the human pauses to resolve which layer a word belongs to, and the language model may silently resolve it wrong.

In early incarnations, Job Jockey used generic planning terms that bled directly into source code.
Planning concepts became load-bearing in two systems simultaneously: changing one risked breaking the other.
The fix was not better naming within the same domain — it was choosing a source domain so alien to software engineering that collision becomes structurally impossible.

Equestrianism shares zero terminology with software, cloud infrastructure, CI/CD pipelines, or corporate project management.
No codebase has a `heat` object.
No ticketing system uses `firemark`.
No CI step is called `corral`.
When a human says "mount," there is no ambiguity about which system they are addressing.
When a language model encounters "school," it does not reach for a compiler.

The vocabulary also exploits a property of human memory: concrete, physical, sensory metaphors lodge more durably than abstract labels.
A jockey wearing silks, a horse being schooled in a dressage ring, a volte circling back to the same point with refined intent — these images persist across sessions in ways that "human-claimed-pending-interactive-work" cannot.
The cognitive cost of re-establishing context drops when the terms are vivid and the metaphor is coherent.

=== Racing Vocabulary

The original layer, established in V3.
Terms drawn from horse racing — the competitive, forward-driving aspect of initiative management.

* {jjdhr_heat} — a bounded initiative. A race: defined start, defined finish, sustained effort.
* {jjdpr_pace} — a discrete action within a heat. The rhythm of hooves — each stride advances the race.
* {jjdt_firemark} — heat identity, `₣`-prefixed. The hot-iron brand that marks ownership on a living animal — permanent, unmistakable.
* {jjdt_coronet} — pace identity, `₢`-prefixed. The marking at the hoof line — fine-grained, below the firemark, identifying the individual step.
* {jjdt_silks} — display name. The jockey's colors — vivid, human-recognizable, worn for the crowd, not the horse.
* {jjdgr_gallops} — the persistent data store. Training grounds where real work happens before race day — not the race itself, but where readiness is built.
* *paddock* — heat-wide context. The gathering area where horses are assessed before entering the course.
* *muster* — list heats. Assembling the field before the race.
* *parade* — show heat or pace detail. The formal presentation of horses before the crowd.
* *furlough* — alter heat status. A horse temporarily withdrawn from racing — not retired, not scratched, resting.
* *retire* — archive a completed heat. The horse leaves the racing life — honored, not discarded.
* *rail* — reorder paces. The rail governs position on the track — who runs where.

=== Dressage Vocabulary

The V4 layer, introduced for the school-breeze-corral execution pipeline.
Terms drawn from dressage and classical horsemanship — the disciplined, precise, iterative aspect of training and evaluation.

* *school* — per-pace planning phase. Schooling a horse: patient, attentive work that develops capability before asking for performance. The rider (human) is fully present.
* *longe* — heat-level readiness assessment. Working a horse on a long line to assess soundness before mounting — observation without commitment.
* *breeze* — autonomous execution phase. A breeze work: the horse runs at controlled speed, demonstrating trained capability without the rider directing each stride.
* *corral* — review and integration phase. Gathering horses after work — assessing condition, accepting or turning back for more training.
* *volte* — an execution attempt covering one or more paces. A dressage figure: a precise, controlled circle back to the same point with refined intent. Each volte attempts the same work with accumulated understanding.
* *caracole* — volte identity, `₡`-prefixed. A cavalry half-turn — circling back for another pass at the same objective.
* *gait* — reusable recipe of beats. The trained movement patterns a horse carries — walk, trot, canter, each with distinct rhythm and purpose.
* *beat* — atomic execution unit. The indivisible footfall within a gait — one hoof striking ground.
* *quirt* — gait identity, `Ꝗ`-prefixed. A short riding whip — the thing that sets a gait in motion.
* *reined* — under human authority. The rider holds the reins; the pipeline does not touch this horse.

=== Presentation Aliases

The display and output layer uses distinct terms for certain schema fields, following the same vocabulary isolation principle:

* *docket* — the presentation term for the `text` field. What the {jjdpr_pace} should accomplish. Used in CLI output labels and slash command descriptions.

These are display/CLI terms, not schema changes. Field renames are deferred to a future heat.

== Upper API

Job Jockey has two distinct API layers.
The *lower layer* is the {jjdx_tool} set: boring, heterogeneous verb names implemented in Rust ({jjdo_enroll}, {jjdo_orient}, {jjdo_close}, etc.).
The *upper layer* is the equestrian vocabulary that humans use to direct a language model ({jjsuv_slate}, {jjsuv_mount}, {jjsuv_wrap}, etc.).

The separation is deliberate and follows the vocabulary isolation principle described in Equestrian Vocabulary.
Shared vocabulary between layers caused the LLM to conflate them — guessing tool parameters by analogy, reaching for raw tools instead of following workflows.
Boring lower-layer names resist pattern matching.
Vivid upper-layer names stay memorable for humans and unambiguous for language models.

The {jjsuc_insert} is the primary artifact that translates between layers — most verbs are delivered via the CLAUDE.md verb table.
V4 reserves slash commands for protocol-heavy orchestration (school, corral).
The human speaks equestrian vocabulary; the LLM invokes {jjdx_tool_s} via MCP.

== Serialization

[[jjdz_charset]]
{jjdz_charset}::
The 64-character alphabet for identity encoding.
Uses URL-safe base64 (RFC 4648 §5):
`ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_`
Position in charset determines numeric value (A=0, B=1, ..., `-`=62, `_`=63).
This alphabet is safe for JSON keys, URLs, shell, git refs, and filenames.
Matches Rust `base64::engine::general_purpose::URL_SAFE_NO_PAD`.

[[jjdz_encoding]]
{jjdz_encoding}::
The algorithm for encoding
{jjdt_firemark}
and
{jjdt_coronet}
values.
+
*{jjdt_firemark} (Heat identity):*
+
. Format: `₣` + 2 base64 characters
. Given integer H (0-4095): encode as `₣` + `charset[H/64]` + `charset[H%64]`
. To decode `₣XY`: H = pos(X) * 64 + pos(Y)
+
*{jjdt_coronet} (Pace identity):*
+
. Format: `₢` + 5 base64 characters
. First 2 characters encode the parent Heat (same as {jjdt_firemark} without prefix)
. Last 3 characters encode the pace index P (0-262143)
. Given Heat H and pace index P: encode as `₢` + `charset[H/64]` + `charset[H%64]` + `charset[P/4096]` + `charset[(P/64)%64]` + `charset[P%64]`
. To decode `₢XYZAB`: H from XY, P = pos(Z) * 4096 + pos(A) * 64 + pos(B)
+
*Input flexibility:*
+
CLI commands accept identities with or without prefix.
Length determines type: 2 base64 chars = {jjdt_firemark}, 5 base64 chars = {jjdt_coronet}.
Output always includes the prefix (`₣` or `₢`).

== Operation Taxonomy

This section defines the structural vocabulary for how operations decompose into
spec-governed layers.
Each MCP command ({jjsoprd_revise_docket}, etc.) is a *procedure* that delegates to
a *composed method* on {jjdgr_gallops}, which itself composes *shared primitive methods*.
The spec names every layer so implementation cannot drift from design.

=== Dispatch Protocol

[[jjsodp_command_lifecycle]]
{jjsodp_command_lifecycle}::
The uniform frame that the MCP dispatcher applies to every {jjdx_tool} call.
Two layers: an {jjdxo_officium} envelope that wraps all commands, and a {jjdgr_gallops} lifecycle for commands that touch persistent state.
+
*{jjdxo_officium} envelope* (all commands except {jjdo_open}):
+
. Validate {jjdxo_officium} parameter — {jjdxo_exchange} directory must exist
. Touch {jjdxo_heartbeat} file in the {jjdxo_exchange} directory
+
{jjdo_open} is exempt: it creates the {jjdxo_officium} rather than validating one.
If the {jjdxo_exchange} directory is missing (stale session, incorrect ID), the dispatcher returns an error prompting the agent to call {jjdo_open}.
+
*{jjdgr_gallops} lifecycle* (gallops-touching commands):
+
Per {jjdk_sole_operator}, every gallops operation acquires the lock unconditionally.
+
{jjds_behavior}
+
. Acquire commit lock
. {jjdr_load} — deserialize {jjdgr_gallops} from JSON
. Call the procedure handler with `&mut` {jjdgr_gallops} and deserialized parameters
. {jjdr_persist} — save {jjdgr_gallops} and commit with affiliated firemark
. Return the handler's output text as MCP response
+
The lifecycle is implemented once in the MCP dispatcher.
Procedures never acquire locks, load {jjdgr_gallops}, or call {jjdr_persist} directly.
The handler returns output text on success or an error — it has no knowledge of
locks, persistence, or commit messages.
+
The dispatcher determines the affiliated firemark from the command's parameters.
Heat-affiliated commands supply a {jjdt_firemark} directly.
Pace-affiliated commands supply a {jjdt_coronet}; the dispatcher derives the
parent {jjdt_firemark}.
+
Officium operations ({jjdo_open}, {jjdo_chapter}, {jjdo_absolve}) bypass the {jjdgr_gallops} lifecycle entirely — they are infrastructure that does not touch persistent state.
+
*{jjsz_gazette} I/O wiring* (output and input operations):
+
Output operations ({jjdo_orient}, {jjdo_show}, {jjdo_paddock} getter):

. Dispatcher creates an empty {jjsz_gazette} with appropriate vocabulary
. Dispatcher passes `&mut` gazette to the handler
. Handler calls `jjrz_add` on the gazette to populate notices
. After the handler returns, dispatcher emits the gazette to {jjdxo_gazette} (skips write if gazette is empty)
+
Input operations ({jjdo_enroll}, {jjsoprd_revise_docket}, {jjdo_paddock} setter):

. Dispatcher reads {jjdxo_gazette} at the start of dispatch
. If gazette file is non-empty, dispatcher immediately overwrites it with empty content (consume)
. Non-empty content is parsed via the appropriate gazette parse function; parsed values replace inline params
. If gazette file is empty, dispatcher falls back to inline params from the MCP call
. Both paths converge before calling the handler — the handler receives the resolved values regardless of source

=== Officium Operations

Operations for inspecting and maintaining {jjdxo_officium} state.
These are infrastructure operations — they do not touch {jjdgr_gallops}.

[[jjdo_open]]
//axvo_procedure axd_transient
==== {jjdo_open}

Create a new {jjdxo_officium} for the calling agent session.
Takes no parameters; returns the {jjdxo_officium} identity string.
+
{jjds_behavior}
+
. Execute {jjdxo_invitatory} procedure ({jjdxo_exsanguination} → identity → directory → probe → commit)
. Return identity string (e.g. `☉260327-1000`)
+
The agent calls {jjdo_open} once per chat session, at the beginning.
The returned identity becomes a required parameter on all subsequent {jjdx_tool} calls.
+
{jjds_stdout}
+
The {jjdxo_officium} identity string, prefixed with ☉.

[[jjdo_chapter]]
//axvo_procedure axd_transient
==== {jjdo_chapter}

List active {jjdxo_officium_s}.
Scans `.claude/jjm/officia/` for directories.
Bypasses the {jjdgr_gallops} lifecycle — infrastructure operation.
+
{jjds_arguments}
+
* `officium` (string, required) — identity of the calling {jjdxo_officium}.
  Used to mark the caller's own entry distinctly in output.
+
{jjds_behavior}
+
. Scan `.claude/jjm/officia/` for directories
. For each directory, report:
  * {jjdxo_officium} identity: ☉-prefixed (e.g. `☉260327-1000`)
  * Creation time: directory ctime (fall back to mtime if ctime unavailable)
  * {jjdxo_heartbeat} mtime: timestamp of last dispatch that touched the heartbeat file
  * Age: human-readable elapsed time since heartbeat mtime (e.g. `42s`, `5m`, `1h23m`)
  * Staleness status: `active` (within threshold), `stale` (exceeds threshold), or `unknown` (heartbeat metadata unavailable)
. Mark the caller's own {jjdxo_officium} entry distinctly (e.g. `← you`)
+
{jjds_stdout}
+
Tabular listing of {jjdxo_officium_s}.
Columns: Officium, Created, Heartbeat, Age, Status.
Rows sorted by identity.
Caller's own row marked with a distinguishing indicator.
If no officia directories exist, reports "No officia found."

[[jjdo_absolve]]
//axvo_procedure axd_transient
==== {jjdo_absolve}

Reap stale {jjdxo_officium} directories.
Removes {jjdxo_exchange} directories whose {jjdxo_heartbeat} exceeds the staleness threshold.
Bypasses the {jjdgr_gallops} lifecycle — infrastructure operation.
+
{jjds_arguments}
+
* `officium` (string, required) — identity of the calling {jjdxo_officium}.
  The caller's own directory is never removed, regardless of staleness.
+
{jjds_behavior}
+
. Scan `.claude/jjm/officia/` for directories
. Skip the calling {jjdxo_officium_p} own directory (identified by `officium` param)
. For each remaining directory:
  * If no {jjdxo_heartbeat} file exists, treat as stale and remove
  * Otherwise check {jjdxo_heartbeat} mtime; remove if it exceeds the staleness threshold (same threshold as {jjdxo_exsanguination})
. Report each removed directory by ☉-prefixed identity
. Report each removal error separately
. If no directories were removed and no errors occurred, report "Nothing to absolve."
+
{jjds_stdout}
+
List of removed {jjdxo_officium} identities (count + per-identity lines), followed by any removal errors.
When nothing was removed and no errors occurred, reports "Nothing to absolve."
+
NOTE: {jjdo_absolve} vs. {jjdxo_exsanguination}: these are complementary, not redundant.
{jjdxo_exsanguination} is *automatic and silent* — it runs inside every {jjdo_open} invocation and produces no output.
{jjdo_absolve} is *explicit and diagnostic* — it is user-invoked, reports what was removed, and can be called at any time without opening a new {jjdxo_officium}.
Use {jjdo_absolve} when stale directories accumulate between {jjdo_open} calls, or to inspect cleanup results directly.

[[jjdo_bind]]
//axvo_procedure axd_transient
==== {jjdo_bind}

Create a
{jjdt_legatio}
session handle by probing the
{jjdxr_fundus}
via SSH and validating
{jjdt_reldir}
per
{jjdk_reldir_safety}.

{jjds_arguments}

//axd_required
* {jjda_host}
— hostname or IP of the
{jjdxr_fundus}

//axd_required
* {jjda_user}
— SSH user on the
{jjdxr_fundus}

//axd_required
* {jjda_reldir}
— relative path from user's home to the project directory on the
{jjdxr_fundus}

{jjds_stdout}
{jjdt_legatio}
token and probe results

{jjds_exit_uniform} 0 success, non-zero error.

{jjds_behavior}

. Validate
{jjdt_reldir}
(Layer 1 —
{jjdk_reldir_safety}):
must not be empty, must not start with `/` or `.`, must contain at least one `/`
. SSH probe (Layer 3 —
{jjdk_reldir_safety}):
execute remote shell script on
{jjdxr_fundus}
that resolves `$HOME/{reldir}` to an absolute path, verifies it is strictly under `$HOME`, and verifies minimum depth (at least 2 components under `$HOME`).
Expect `PROBE_OK` sentinel on success.
. Read `$HOME/{reldir}/.buk/burc.env` from
{jjdxr_fundus}
via SSH and extract `BURC_OUTPUT_ROOT_DIR`
. Mint
{jjdt_legatio}
token: scan existing legatio files in
{jjdxo_officium}
directory, assign next sequential `L{N}` token
. Persist legatio state (host, user, reldir, output_root_dir) as JSON in
{jjdxo_officium}
directory
. Output
{jjdt_legatio}
token and binding summary

//axvo_procedure axd_transient
==== {jjdo_send}

Synchronous remote execution via
{jjdt_legatio}.
Single command string interpreted by `bash -c` on the
{jjdxr_fundus}
after `cd RELDIR`.

{jjds_arguments}

//axd_required
* {jjda_legatio}
—
{jjdt_legatio}
token from
{jjdo_bind}

//axd_required
* {jjda_command}
— shell command string to execute on the
{jjdxr_fundus}

{jjds_stdout} Remote stdout, stderr, and exit code

{jjds_exit_uniform} 0 on SSH transport success, non-zero on SSH transport failure.

{jjds_behavior}

. Load
{jjdt_legatio}
state from
{jjdxo_officium}
directory
. Construct remote command: `cd "$HOME/{reldir}" && bash -c {quoted_command}`
(POSIX single-quote wrapping with `'\''` for embedded quotes)
. Execute via SSH with `BatchMode=yes` and `ConnectTimeout=10`
. Capture remote stdout, stderr, and exit code
. Output remote stdout, stderr, and `Exit: {code}`

[[jjdo_relay]]
//axvo_procedure axd_transient
==== {jjdo_relay}

Asynchronous dispatch via nohup on the
{jjdxr_fundus}.
Mints a
{jjdt_pensum}
token (incrementing
{jjdhm_pensum_seed}
under
{jjdgr_gallops}
lock), sets `BURE_LABEL`, launches via SSH, captures `burx.env` start fields via a second SSH connection.

{jjds_arguments}

//axd_required
* {jjda_legatio}
—
{jjdt_legatio}
token from
{jjdo_bind}

//axd_required
* {jjda_tabtarget}
— bare filename of the tabtarget to dispatch (e.g., `rbw-tf.QualifyFast.sh`).
Must end with `.sh`, no path components.

//axd_required
* {jjda_timeout}
— maximum execution time in seconds.
A watchdog on the
{jjdxr_fundus}
kills the tabtarget after this duration.

//axd_required
* {jjdt_firemark}
— parent
{jjdhr_heat}
identity (used for
{jjdt_pensum}
token minting under
{jjdgr_gallops})

{jjds_stdout}
{jjdt_pensum}
token and remote temp directory path

{jjds_exit_uniform} 0 success, non-zero error.

{jjds_behavior}

. Validate tabtarget: must not be empty, must not contain `/` or `..`, must end with `.sh`
. Load
{jjdt_legatio}
state from
{jjdxo_officium}
directory
. Mint
{jjdt_pensum}
token:
.. Acquire commit lock
.. Load
{jjdgr_gallops}
.. Increment
{jjdhm_pensum_seed}
.. Persist
{jjdgr_gallops}
with commit
.. Release lock
. SSH #1 — launch nohup wrapper on
{jjdxr_fundus}:
.. Set `BURE_LABEL` to
{jjdt_pensum}
token in environment
.. Background the tabtarget via nohup with timeout watchdog
.. Expect `RELAY_OK` sentinel on successful launch
. SSH #2 — poll for initial `burx.env`:
.. Read `{output_root_dir}/current/burx.env` from
{jjdxr_fundus}
(up to 10 retries, 1 second apart)
.. Validate `BURX_LABEL` matches
{jjdt_pensum}
token
.. Extract `BURX_TEMP_DIR` and `BURX_PID`
. Persist
{jjdxp_pensum}
state (legatio, firemark, temp_dir, pid, tabtarget, timeout, began_at) as JSON in
{jjdxo_officium}
directory
. Output
{jjdt_pensum}
token and remote temp directory

=== Steeplechase Operations

The steeplechase is Job Jockey's session history, stored in git commits rather than the
{jjdgr_gallops}
JSON.
Commit messages follow patterns that
{jjdo_log}
can parse to reconstruct session history.

==== Commit Message Patterns

All JJ commits use the format: `jjb:BRAND:IDENTITY:ACTION: message`

* `jjb` — JJ brand prefix (literal)
* `BRAND` — Version identifier. Format: NNNN (installed) or NNNN-xxxxxxx (dev). Source: `.vvk/vvbf_brand.json` (installed) or max brand from `Tools/vok/vov_veiled/vovr_registry.json` + git HEAD (Kit Forge)
* `IDENTITY` — ₢CORONET for pace-level, ₣FIREMARK for heat-level
* `ACTION` — single-letter code (required for all commits)

*Standard notch* (created by {jjdo_record}):
----
jjb:BRAND:₢XXXXX:n: commit message

Co-Authored-By: ...
----

Standard notch commits use `n` to denote routine progress tracking.

*Landing marker* (created when spawned agent completes):
----
jjb:BRAND:₢XXXXX:L: {agent} {landed|crashed}

{agent completion report}

Co-Authored-By: ...
----

Subject format: `{agent} {landed|crashed}` where agent is the executor tier (haiku/sonnet/opus) and landed/crashed indicates success/failure. Body contains the agent's raw completion report.

*{jjdxo_invitatory} marker* (created by {jjdo_open}):
----
jjb:BRAND::i: OFFICIUM <id>

haiku: claude-3-5-haiku-20241022
sonnet: claude-sonnet-4-20250514
opus: claude-opus-4-5-20251101
host: macbook-pro.local
platform: darwin-arm64
----
+
The body contains model inventory when `vvcp_probe()` runs (first {jjdxo_invitatory} of the day).
On subsequent invitatory commits the same day, the body is empty.

{jjdxo_invitatory} markers record when an {jjdxo_officium} opens.
Created by explicit agent action ({jjdo_open}) — not by {jjdx_mcp} process startup.
Model IDs are collected via `vvcp_probe()`, which spawns parallel claude invocations via tokio async subprocess spawning.
+
The {jjdxo_compline} action code `o` is reserved-unused. No compline commits are created.

Pace-level action codes:

* `n` — Notch: standard commit
* `L` — LANDING: records agent return after autonomous execution
* `W` — WRAP: pace completion summary
* `d` — discussion: significant decision (lowercase, can also be heat-level)

*Heat-level operations*:
----
jjb:BRAND:₣XX:N: heat-silks
jjb:BRAND:₣XX:S: pace-silks
jjb:BRAND:₣XX:s: YYMMDD-HHMM session  (superseded — see below)
jjb:BRAND:₣XX:r: reordered
jjb:BRAND:₣XX:D: old-coronet → new-coronet
jjb:BRAND:₣XX:R: heat-silks
----

Heat-level action codes:

* `N` — Nominate: create new heat
* `S` — Slate: add new pace
* `r` — rail: reorder paces (lowercase)
* `D` — Draft: move pace between heats (uppercase, rare)
* `R` — Retire: archive heat (uppercase)

NOTE: {jjdxo_invitatory} commits use action code `i`.
The {jjdxo_compline} action code `o` is reserved-unused — sessions end without ceremony.
See <<jjdxo_invitatory>> and <<jjdxo_compline>> for design rationale.

== V3 Legacy Schema Reference

WARNING: This section is a frozen reference for V3→V4 migration.
It documents the V3 schema as implemented in `jjrt_v3_types.rs`.
Do NOT modify — will be removed by ₣An (jjk-v4-release-and-legacy-removal).

=== V3 Pace Record

In V3, a {jjdpr_pace} contains a `tacks` array of Tack records.
Each Tack captures a snapshot of the pace's state at a point in time.

==== V3 Tack Record

* `ts` (string, required) — timestamp
* `state` (enum, required) — pace state (see V3 Pace State Enum)
* `text` (string, required) — docket text at this snapshot
* `silks` (string, required) — pace silks at this snapshot
* `commit` (string, required) — basis commit SHA (Rust field: `basis`, serde rename: `commit`)
* `direction` (string, optional) — plan/direction text

==== V3 Pace State Enum

* `rough` — needs clarification
* `bridled` — human-claimed for interactive work (serde alias: `primed`)
* `complete` — finished
* `abandoned` — stopped without completion

=== V3 Heat Record

Same as V4 (no structural changes to Heat in V4 migration).

=== V3 Gallops Record

Same as V4 except:

* `schema_version` absent (added in V4)
* `heat_order` absent (added in V4; populated from sorted BTreeMap keys on migration)

=== V3→V4 Migration Path

Migration is handled by {jjdr_load} in `jjri_io.rs`.
Current behavior:

. Deserialize JSON as current types
. Detect migration mode: `schema_version` absent or `heat_order` empty
. Recompute `paddock_file` from firemark for every heat
. Set `schema_version` to 4
. Populate `heat_order` from sorted BTreeMap keys
. Next {jjdr_save} writes V4 format

The frozen V3 types in `jjrt_v3_types.rs` capture the V3 schema for reference when V4 type changes are implemented.
