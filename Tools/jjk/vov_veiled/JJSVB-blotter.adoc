= Blotter

The locked-store entity: the kind of git repository Job Jockey itself owns and writes.
Its methods — the shared write bracket and the stale-lock break — are homed in the sibling
routine sheaf (`JJSVJ-journal.adoc`, {jjdb_journal} and {jjdb_break});
this sheaf summarizes and cites, never restates.

[NOTE]
====
*Forward references.* The mews fleet store is a census-minted word whose binding sheaf
lands after this one;
where it appears below it reads as a forward reference to a forthcoming contract home, not a
citation.
====

== The entity

[[jjdb_blotter]]
//axo_entity
{jjdb_blotter}::
A git repository JJ itself owns and drives:
*linear* — it never branches;
*single-writer-under-lock*;
and *engine-driven* — the human never hand-manages its git.
Git is its source of truth and its journal at once:
additive history, commit-as-intent, free immutability.
Linearity plus single-writer make a monotonic ordinal per {jjdb_blotter} clean and
authoritative.
+
A {jjdb_blotter} is the physical vessel, not the tenant:
what the repository holds is each instance's own contract.
+
Every write passes through the {jjdb_journal} ({jjdk_sole_door});
a stale lock is cleared only by the {jjdb_break}.
No writer ever forces:
history is additive, and the break is lease-guarded, never blind.

== Instances

Two instances are designed; one engine drives both through the same entity machinery:

* the {jjdb_studbook} — the regional record repo carrying the {jjdgr_gallops};
* the *mews fleet store* — the anticipated second instance
  (sibling nucleation, `JJSAM-mews.adoc`).

Two repos, two locks: instances share machinery, never state.
The one genuine seam between them is the `(work repo, SHA)` cross-reference both journal —
in the {jjdb_studbook}, the {jjdb_counterfoil}.

== Revision ordinals

Every {jjdb_blotter} commit carries a decimal revision ordinal — entirely emission-layer:
a denormalized order label (AXLA `axd_ordinal`) allocated under the store's lock at journal
time and baked into the commit message.
The SHA stays the identity;
single-writer makes the bake race-free, and the linear history makes it verifiable —
no side table exists.
The genesis commit takes the store's founding value: no zeroth special case.
A founding value leads with a digit that pins width without zero-padding,
so lexicographic and numeric order agree until the width horizon;
a future store founds at a fresh leading-digit × width pair.
Bare decimal is charset-valid, so an ordinal never circulates glyphless:
machine contexts carry the SHA,
and any future ingestion convenience must require the glyph
(the minted-mark carriage law, AXLA Minted-Mark Dimensions).

== Lock ref and guidon

Each {jjdb_blotter} carries exactly one well-known lock ref in its own git namespace:
`refs/jjv/guidon`.
The store discriminator is the repository itself — a {jjdb_blotter_p} lock lives in the
{jjdb_blotter} it locks — so the namespace interior stays flat:
one ref, no per-store subtree.
`refs/jjv/*` is reserved to JJ entire;
any future resident beyond the guidon ref must be allocated in this sheaf.

The lock is the ref's existence, taken and released by compare-and-swap ref pushes —
the {jjdf_farrier} {jjdf_lock} ops `stake`/`pluck`/`sight`, sequence-internal, never
operator-typed.

[[jjdb_guidon]]
{jjdb_guidon}::
The lock-holder mark flown on the lock ref:
officium, station, acquire time, and operation.
The {jjdb_guidon} is what makes break-on-error safe —
the {jjdb_break} acts only against an observed {jjdb_guidon}, never blind.
+
*Acquire time is the load-bearing field*: it is what distinguishes a crashed holder from a
writer three seconds in, and the {jjdd_cashier} can warn on nothing else.
The four fields are the mark's *composition*, not four catalogued terms — they are cited
nowhere but here and at the door that renders them.
+
*The composed form.* The engine composes a {jjdb_guidon} as one line of `key=value` fields,
space-separated, in the fixed order officium, station, acquire time (RFC 3339, UTC),
operation.
Values carry no spaces, so the line reads back field-wise;
the composer is the single home of the format, and no other site spells it.
+
*The mechanics never parse it.* `stake`, `pluck`, and `consign`'s lease compare {jjdb_guidon_s}
by *blob-hash equality alone* — the text is opaque to every lock primitive.
Two consequences bind:
a malformed {jjdb_guidon} — one this engine did not compose, one an older engine composed
differently, one a hand-staked probe left behind — is *still a lock*, and must still be
sightable and breakable;
and a parse of the fields is therefore permitted *for display alone*, never for a decision.
The {jjdd_cashier} sights the mark verbatim, breaks against the verbatim value, and parses
tolerantly only to render it —
an unparseable field renders as itself, and the lock stays as breakable as any other.

== Read posture

[[jjdk_lockless_reads]]
//axk_premise
{jjdk_lockless_reads}::
*Reporting* reads take no lock and tolerate slight staleness.
No read-lock machinery exists to cite; the bound is the design itself.
Consumers reach a {jjdb_blotter_p} content through the engine's read path, never raw JSON.
+
The premise's scope is the read that merely reports — display, listing, orientation.
A read that *arms an act* (its answer decides a mutation of the world) is not a reporting
read and does not ride this premise:
it goes through the journal sheaf's read bracket — lock, advance, read, release — and pays
the lock's costs as the price of deciding from the store's truth.
One rule decides which side a consumer is on:
if a stale answer could only ever mis-*report*, the read is lock-free;
if it could mis-*act*, it takes the bracket.

*Layer independence* — the {jjdb_blotter_p} correctness argument never reaches into its
tenants:
what the stored records semantically can or cannot do (a state that never reverses, an
ordering an application maintains) is tenant knowledge,
and a lock or bracket is never removed on the strength of it.
The store's guarantees come from the store's own mechanics — the lock, the advance, the
lease — so they survive every change of tenant semantics unexamined.
A proposed optimization that begins "the data can never..." is the tell that this rule is
about to be broken.

== Local clone

A {jjdb_blotter_p} local clone is a *cache of the remote*, never a work tree.
The remote is the store;
the clone is one station's copy of it, engine-driven and sole-door ({jjdk_sole_door}),
holding nothing a human authored.
The {jjdb_journal} carries it to the remote tip under the lock before every write, and its
branch adopts only positions the remote has accepted (`JJr_b52`), so *tracked* content there
is always the store's own history — the store, not the clone, is the truth.
Nothing tracked in the clone is worth preserving:
a station may discard and re-clone it whenever it likes, and lose no record.

The cache reading governs *tracked content alone*, and the distinction is load-bearing.
A clone's untracked space is a reserved tenancy, not residue:
the {jjdb_studbook} co-locates its per-officium {jjdxo_exchange} and un-graduated record half
there (studbook sheaf, "Officium exchange and record").
`advance` neither touches that space nor guards it —
it is the tenant's, held to the tenant's own contract, and a station discarding its clone
discards it too.
So *re-clone loses no record, and is not therefore free*:
what the store holds always returns; what a tenant was still holding untracked does not.

== Bootstrap config

A {jjdb_blotter_p} own configuration — where each instance lives, remote URL and local clone —
is *engine-known, never pedigree-resolved*:
a store cannot look itself up in itself.
The pedigree resolves consumer repos; the engine carries its own stores' coordinates.
