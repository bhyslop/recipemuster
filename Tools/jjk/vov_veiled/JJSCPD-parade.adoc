//axvd_sheaf axd_normative
This is a normative portion of the {jjs0_imprimatur}: binding — code, tests, and decisions may rely on it.
Display
{jjdhr_heat}
or
{jjdpr_pace}
information in human-readable text format.
Output varies by target type.

*Invocation:*
----
vvx jjx_show [--detail] [--remaining] [--hark <rev>]
----
Targets are not parameters — they are supplied via {jjezs_halter} notices in `gazette_in.md`.

{jjds_arguments}

//axd_required
* Targets — one or more {jjezs_halter} notices in `gazette_in.md`, each lede a
{jjdt_firemark}
(heat view) or
{jjdt_coronet}
(pace view), self-typed by length (2 base64 chars = heat, 5 base64 chars = pace).
At least one notice is required; there is no auto-select — an absent gazette is a loud error, never a silent first-racing-heat default.
A `targets` or `firemark` parameter is rejected, not honored — the forced gazette write is the point.

//axd_optional
* `--detail` — Show paddock and full pace dockets (heat mode only)

//axd_optional
* `--remaining` — Show only remaining paces (exclude
{jjdpe_complete}
and
{jjdpe_abandoned})

//axd_optional
* {jjda_hark} `<rev>` — Hark back: render the heat or pace as of a prior git revision (read-only,
via {jjdr_hark}), output stamped `AS OF <rev>`.
Presence flips show into hark mode; absent, show reads the working tree as usual.
{jjdo_orient} (mount) never accepts it.


{jjds_stdout} Human-readable text (not JSON).

*Example output — heat list (default):*
----
 No  State       Pace                    Coronet
-------------------------------------------------
  1  abandoned   vvw-workbench-tabtarget AAAAC
  2  complete    install-arch-decision   AAAAI
  3  rough       vvx-push-rename         AAAAJ
----

*Example output — heat list with `--remaining`:*
----
Heat: my-heat (₣AA) [racing]
Progress: 1 complete | 1 abandoned | 1 remaining (1 rough)

 No  State     Pace                Coronet
-------------------------------------------
  1  rough     vvx-push-rename     AAAAJ

Next: vvx-push-rename (₢AAAAJ) [rough]
----

*Example output — heat list with file-touch census (always shown):*
----
File touches (file: the paces whose commits touched it):
  (2 of 4 files; those touched by a single pace are omitted)

  B  ₢AAAAB  vos-add-routines-section
  D  ₢AAAAD  jjsa-session-probe-update

  jjrg_gallops.rs  B D
  jjrt_types.rs    B D
----

*Example output — commit swim lanes (always shown after file-touch census):*
----
Commit swim lanes (x = commit affiliated with pace):

  1 B vos-add-routines-section
  2 D jjsa-session-probe-update
  3 * heat-level

123456789ab
··xx·xx····  D  4c
xx··x······  B  3c
··········x  *  1c
----

Commit swim lanes show paces × commits: each column is a commit (chronological, dense-encoded 1-9 then a-z then A-Z for 61 max), each row is a pace ordered by first chronological appearance.
`*` row captures heat-level commits.
Row label (terminal char) and commit count suffix appear right of the bitmap.
Default truncation: last 35 commits; if truncated, prefixed with `(showing last 35 of N commits)`.

The file-touch census names, per file, the paces whose commits touched it: a legend line per pace with commits (its mark, coronet, and silks), then one row per file listing the marks, busiest file first.
It is sparse by construction, never a pace × file grid.
A wide heat's touch matrix runs a few percent dense, so a grid spends nearly all its width drawing absence, and the reader — an agent, which decodes a position in a dotted row unreliably — pays for a decode it cannot trust.
Naming the paces present costs less than drawing the ones that are not.
A file touched by a single pace is omitted, and the omitted count is stated: a lone-pace file poses no collision question, which is the planning signal the census exists to carry.
The floor is the reader's, not the data's — the trophy of
{jjdo_archive}
renders the same census with every touched file, because a retired heat's record is the last account of what it moved.
Administrative files (`.claude/jjm/*`) are excluded from the census and the swim lanes alike.

In `--detail` mode, tacks are displayed in chronological order (oldest first).
Each tack is numbered with a display index starting from `[0]` (oldest) through `[N]` (most recent).
Each tack shows its display index, state, commit hash, silks (which may change over time), and the full docket text.
Note: display indices are chronological position, not storage position.
Commit hash displays "(no commit)" for placeholder value `0000000`.

{jjds_exit_uniform} 0 success, non-zero error.

{jjds_behavior}

. Read
{jjdgr_gallops}
. Resolve the target set from the
{jjezs_halter}
notices in `gazette_in.md` — at least one required, each lede a firemark or coronet; error on an absent gazette or a rejected `targets`/`firemark` parameter
. For each resolved target, detect type by length (strip prefix if present):
.. 2 chars → heat view
.. 5 chars → pace view
.. Otherwise: error "target must be Firemark (2 chars) or Coronet (5 chars)"
. For heat view:
.. Parse as
{jjdt_firemark},
verify heat exists
.. If `--detail` flag:
... Read paddock file content
... Print heat header (silks, firemark, status, created)
... Print paddock section
... For each pace in
{jjdhm_order}
(respecting `--remaining`): print pace header and full tack text
.. Otherwise (default list):
... If `--remaining`: print plain text table with heat header, progress stats, remaining paces in aligned columns, and next up indicator
... If not `--remaining`: for each pace in
{jjdhm_order}:
print numbered line with state, silks, coronet
.. After pace listing, always append the file-touch census: query steeplechase entries for heat, run `git diff-tree` per commit to identify touched work files (excluding `.claude/jjm/*`), invert to file → the paces that touched it, drop files below the cross-pace floor of two and state how many were dropped, order rows by pace count descending then filename; emit a legend line per pace with commits (mark, coronet, silks) above the rows
.. After the file-touch census, always append commit swim lanes: query steeplechase entries for heat (newest-first from git log), take last 35 (most recent), reverse for chronological column order; rows are paces ordered by first chronological appearance plus `*` row for heat-level commits; columns dense-encoded 1-9 then a-z then A-Z; row label and commit count suffix after bitmap; truncation indicator if >35 commits
. For pace view:
.. Parse as
{jjdt_coronet},
extract parent
{jjdt_firemark}
.. Verify heat and pace exist
.. Query steeplechase entries for parent heat, filter by coronet, run `git diff-tree` per commit, collect unique work file names (excluding `.claude/jjm/*`), display as "Work files:" line (bare filenames, sorted)
.. Print pace header (silks, coronet, state, heat)
.. Print docket text from the latest tack (storage `tacks[0]`, i.e. most recent) as plain text, without display index prefix

*Hark mode ({jjda_hark} present) — wraps the behavior above:*

. Source the {jjdgr_gallops} and each targeted heat's paddock from the {jjda_hark} revision via
{jjdr_hark} (both fetched at the same revision), in place of the working-tree {jjdgr_gallops} read
and the per-heat paddock-file read.
. Render exactly as above, with every view stamped `AS OF <rev>`.
. Omit the file-touch census and commit swim lanes: both derive from live git history (current
HEAD), not the revision's, so they carry no coherent meaning in a snapshot.
. Set no {jjdxw_emblem} identity and write nothing — a hark mutates nothing live.
. On an unresolvable or too-old revision, surface {jjdr_hark}'s error verbatim and render nothing.
. {jjdo_orient} never enters hark mode — a mount is always the latest state.
