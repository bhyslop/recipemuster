== Overview

The
{jjdgr_gallops}
is the persistent data store for Job Jockey.
It tracks
{jjdhr_heat_s}
(bounded initiatives) and their constituent
{jjdpr_pace_s}
(discrete actions).

== Design Principles

=== Exclusive JSON Ownership

All access to the
{jjdgr_gallops}
goes through {jjdx_handler} functions within {jjdx_vvx}.
No jq or direct JSON manipulation ever touches this data.
This is an absolute architectural constraint, not a preference.

The rationale: Rust {jjdx_handler} functions replace fragile jq invocations with compiler-enforced correctness.
The compiler is tireless; humans are not.

=== Git as Journal

The
{jjdgr_gallops}
remains JSON specifically so git can serve as the journalling layer.
Every mutation creates a diffable commit.
Git owns the complete history; we need not implement our own audit trail.

Git has the full history of every mutation.
Plan evolution is visible through commit diffs without additional in-JSON bookkeeping.

=== Diff-Friendly Mutations

The
{jjdhm_order}
array contains full
{jjdt_coronet_s}.
Each
{jjdt_coronet}
embeds its parent
{jjdt_firemark}
(redundant within a single
{jjdhr_heat}).
This redundancy is cheap and enables grep, copy-paste, and self-documenting JSON.

=== Crash-Safe Architecture

Every operation follows the same invariant regardless of transport:
lock → load → transform → save → unlock.
+
The {jjdx_handler} acquires a git-ref lock (`git update-ref`),
loads the {jjdgr_gallops} JSON,
performs the transformation,
writes atomically (temp file then rename),
and releases the lock.
If the {jjdx_handler} crashes mid-transform, no partial writes occur.
+
No in-memory {jjdgr_gallops} state persists between operations.
Each operation reads from disk and writes back.
This is critical for the {jjdx_mcp}, which is a long-lived process:
caching state between calls would produce stale reads when multiple sessions write concurrently.

=== MCP Transport

[[jjdx_mcp]]
//axl_voices axi_cli_subcommand
{jjdx_mcp}::
The `vvx mcp` subcommand starts a long-lived stdio server implementing the Model Context Protocol (MCP) specification version 2025-11-25.
The server communicates via JSON-RPC 2.0 over stdin/stdout.
Implemented using the `rmcp` crate (official Rust MCP SDK), version pinned exactly.

[[jjdx_tool]]
//axl_voices axo_entity
{jjdx_tool}::
A JSON-RPC tool call within an {jjdx_mcp} session.
Each operation is registered as an {jjdx_tool} with typed parameter schemas and descriptions.
{jjdx_tool_s} are the sole invocation path for Claude Code.
The {jjdx_cli} CLI subcommand path is removed;
{jjdx_tool_s} are NOT also available as CLI subcommands.
+
Infrastructure commands ({jjdx_vvx} emplace, unlock) remain CLI-only.

[[jjdx_handler]]
//axl_voices axo_entity
{jjdx_handler}::
The transport-agnostic Rust function that implements an operation.
Both {jjdx_tool_s} and (historical) CLI subcommands invoke the same {jjdx_handler}.
{jjdx_handler} functions accept typed parameters and return structured results;
the transport layer (MCP or CLI) handles serialization.

Each {jjdx_tool} call is stateless: the {jjdx_handler} performs the full crash-safe invariant (lock → load → transform → save → unlock) per invocation.
The {jjdx_mcp} holds no {jjdgr_gallops} state between calls.
+
Operations that previously read text from stdin (
{jjdo_enroll},
{jjdo_revise_docket},
{jjdo_transfer},
{jjdo_close},
{jjdo_landing})
accept structured string parameters instead.
The {jjdx_mcp} resolves the {jjdgr_gallops} path internally;
{jjda_file} is not exposed as a tool parameter.

=== Deterministic Serialization

Implementations MUST serialize JSON object keys in deterministic order to minimize diff churn.
For `heats` and `paces` objects, use sorted key order (e.g., Rust `BTreeMap` instead of `HashMap`).
This ensures that reordering paces via
{jjdo_reorder}
produces minimal diffs — only the `order` array changes, not the entire `paces` object.
+
The `heats` object uses BTreeMap (alphabetical key order).
{jjdgm_order}
is the authoritative source for heat display and selection order.
Operations that need heat order (
{jjdo_list},
default heat selection in
{jjdo_orient})
iterate
{jjdgm_order},
not the `heats` object directly.

=== Identity Encoding

{jjdt_firemark}
(`₣` prefix) and
{jjdt_coronet}
(`₢` prefix) are the canonical identities used in JSON keys, CLI arguments, and display.
The base64 encoding provides compact, URL-safe, shell-safe strings.
Internally, Rust may decode to integers for arithmetic, but the string form is authoritative.

== Equestrian Vocabulary

=== Vocabulary Isolation

Job Jockey's terminology is drawn entirely from equestrianism.
This is not decoration — it is a namespace discipline applied to human cognition and language model disambiguation.

Planning systems face a persistent hazard: when the vocabulary used to _organize_ work overlaps with the vocabulary used to _do_ work, the boundary between plan and product dissolves.
"Task" could be a planning unit or a runtime construct.
"Phase" could be a workflow stage or a build step.
"Diff" could be a review artifact or a git object.
Every ambiguous term forces a context switch — the human pauses to resolve which layer a word belongs to, and the language model may silently resolve it wrong.

In early incarnations, Job Jockey used generic planning terms that bled directly into source code.
Planning concepts became load-bearing in two systems simultaneously: changing one risked breaking the other.
The fix was not better naming within the same domain — it was choosing a source domain so alien to software engineering that collision becomes structurally impossible.

Equestrianism shares zero terminology with software, cloud infrastructure, CI/CD pipelines, or corporate project management.
No codebase has a `heat` object.
No ticketing system uses `firemark`.
No CI step is called `corral`.
When a human says "mount," there is no ambiguity about which system they are addressing.
When a language model encounters "school," it does not reach for a compiler.

The vocabulary also exploits a property of human memory: concrete, physical, sensory metaphors lodge more durably than abstract labels.
A jockey wearing silks, a horse being schooled in a dressage ring, a volte circling back to the same point with refined intent — these images persist across sessions in ways that "human-claimed-pending-interactive-work" cannot.
The cognitive cost of re-establishing context drops when the terms are vivid and the metaphor is coherent.

=== Racing Vocabulary

The original layer, established in V3.
Terms drawn from horse racing — the competitive, forward-driving aspect of initiative management.

* {jjdhr_heat} — a bounded initiative. A race: defined start, defined finish, sustained effort.
* {jjdpr_pace} — a discrete action within a heat. The rhythm of hooves — each stride advances the race.
* {jjdt_firemark} — heat identity, `₣`-prefixed. The hot-iron brand that marks ownership on a living animal — permanent, unmistakable.
* {jjdt_coronet} — pace identity, `₢`-prefixed. The marking at the hoof line — fine-grained, below the firemark, identifying the individual step.
* {jjdt_silks} — display name. The jockey's colors — vivid, human-recognizable, worn for the crowd, not the horse.
* {jjdgr_gallops} — the persistent data store. Training grounds where real work happens before race day — not the race itself, but where readiness is built.
* *paddock* — heat-wide context. The gathering area where horses are assessed before entering the course.
* *muster* — list heats. Assembling the field before the race.
* *parade* — show heat or pace detail. The formal presentation of horses before the crowd.
* *furlough* — alter heat status. A horse temporarily withdrawn from racing — not retired, not scratched, resting.
* *retire* — archive a completed heat. The horse leaves the racing life — honored, not discarded.
* *rail* — reorder paces. The rail governs position on the track — who runs where.

=== Dressage Vocabulary

The V4 layer, introduced for the school-breeze-corral execution pipeline.
Terms drawn from dressage and classical horsemanship — the disciplined, precise, iterative aspect of training and evaluation.

* *school* — per-pace planning phase. Schooling a horse: patient, attentive work that develops capability before asking for performance. The rider (human) is fully present.
* *longe* — heat-level readiness assessment. Working a horse on a long line to assess soundness before mounting — observation without commitment.
* *breeze* — autonomous execution phase. A breeze work: the horse runs at controlled speed, demonstrating trained capability without the rider directing each stride.
* *corral* — review and integration phase. Gathering horses after work — assessing condition, accepting or turning back for more training.
* *volte* — an execution attempt covering one or more paces. A dressage figure: a precise, controlled circle back to the same point with refined intent. Each volte attempts the same work with accumulated understanding.
* *caracole* — volte identity, `₡`-prefixed. A cavalry half-turn — circling back for another pass at the same objective.
* *gait* — reusable recipe of beats. The trained movement patterns a horse carries — walk, trot, canter, each with distinct rhythm and purpose.
* *beat* — atomic execution unit. The indivisible footfall within a gait — one hoof striking ground.
* *quirt* — gait identity, `Ꝗ`-prefixed. A short riding whip — the thing that sets a gait in motion.
* *reined* — under human authority. The rider holds the reins; the pipeline does not touch this horse.

=== Presentation Aliases

The display and output layer uses distinct terms for certain schema fields, following the same vocabulary isolation principle:

* *docket* — the presentation term for the `text` field. What the {jjdpr_pace} should accomplish. Used in CLI output labels and slash command descriptions.

These are display/CLI terms, not schema changes. Field renames are deferred to a future heat.

== Upper API

Job Jockey has two distinct API layers.
The *lower layer* is the {jjdx_tool} set: boring, heterogeneous verb names implemented in Rust ({jjdo_enroll}, {jjdo_orient}, {jjdo_close}, etc.).
The *upper layer* is the equestrian vocabulary that humans use to direct a language model ({jjsuv_slate}, {jjsuv_mount}, {jjsuv_wrap}, etc.).

The separation is deliberate and follows the vocabulary isolation principle described in Equestrian Vocabulary.
Shared vocabulary between layers caused the LLM to conflate them — guessing tool parameters by analogy, reaching for raw tools instead of following workflows.
Boring lower-layer names resist pattern matching.
Vivid upper-layer names stay memorable for humans and unambiguous for language models.

The {jjsuc_insert} is the primary artifact that translates between layers — most verbs are delivered via the CLAUDE.md verb table.
V4 reserves slash commands for protocol-heavy orchestration (school, corral).
The human speaks equestrian vocabulary; the LLM invokes {jjdx_tool_s} via MCP.

== Types

[[jjdt_firemark]]
//axl_voices axt_string
{jjdt_firemark}::
The identity of a
{jjdhr_heat}.
Format: `₣` + 2 base64 characters (e.g., `₣AB`).
The 2 base64 characters encode an integer 0-4095 (2^12).
Used as JSON keys and in CLI arguments.
Input may omit the `₣` prefix; output always includes it.

[[jjdt_coronet]]
//axl_voices axt_string
{jjdt_coronet}::
The identity of a
{jjdpr_pace}.
Format: `₢` + 5 base64 characters (e.g., `₢ABCDE`).
Globally unique: the first 2 base64 characters are the parent
{jjdhr_heat_p}
encoded identity; the remaining 3 encode the pace index (0-262143, 2^18).
Used as JSON keys and in CLI arguments.
Input may omit the `₢` prefix; output always includes it.
+
*Reassignment:* When a
{jjdpr_pace}
is moved between
{jjdhr_heat_s}
via
{jjdo_relocate},
the old
{jjdt_coronet}
becomes invalid and a new one is assigned using the destination
{jjdhr_heat_p}
{jjdhm_seed}.
The new
{jjdt_coronet}
embeds the destination
{jjdt_firemark}.
Pace data is preserved, with a note recording the original
{jjdt_coronet}.

{jjdt_silks}::
A case-insensitive kebab display name.
Used for human-readable identification of
{jjdhr_heat_s}

== Serialization

[[jjdz_charset]]
{jjdz_charset}::
The 64-character alphabet for identity encoding.
Uses URL-safe base64 (RFC 4648 §5):
`ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_`
Position in charset determines numeric value (A=0, B=1, ..., `-`=62, `_`=63).
This alphabet is safe for JSON keys, URLs, shell, git refs, and filenames.
Matches Rust `base64::engine::general_purpose::URL_SAFE_NO_PAD`.

[[jjdz_encoding]]
{jjdz_encoding}::
The algorithm for encoding
{jjdt_firemark}
and
{jjdt_coronet}
values.
+
*{jjdt_firemark} (Heat identity):*
+
. Format: `₣` + 2 base64 characters
. Given integer H (0-4095): encode as `₣` + `charset[H/64]` + `charset[H%64]`
. To decode `₣XY`: H = pos(X) * 64 + pos(Y)
+
*{jjdt_coronet} (Pace identity):*
+
. Format: `₢` + 5 base64 characters
. First 2 characters encode the parent Heat (same as {jjdt_firemark} without prefix)
. Last 3 characters encode the pace index P (0-262143)
. Given Heat H and pace index P: encode as `₢` + `charset[H/64]` + `charset[H%64]` + `charset[P/4096]` + `charset[(P/64)%64]` + `charset[P%64]`
. To decode `₢XYZAB`: H from XY, P = pos(Z) * 4096 + pos(A) * 64 + pos(B)
+
*Input flexibility:*
+
CLI commands accept identities with or without prefix.
Length determines type: 2 base64 chars = {jjdt_firemark}, 5 base64 chars = {jjdt_coronet}.
Output always includes the prefix (`₣` or `₢`).

=== Steeplechase Operations

The steeplechase is Job Jockey's session history, stored in git commits rather than the
{jjdgr_gallops}
JSON.
Commit messages follow patterns that
{jjdo_log}
can parse to reconstruct session history.

==== Commit Message Patterns

All JJ commits use the format: `jjb:HALLMARK:IDENTITY:ACTION: message`

* `jjb` — JJ brand prefix (literal)
* `HALLMARK` — Version identifier. Format: NNNN (installed) or NNNN-xxxxxxx (dev). Source: `.vvk/vvbf_brand.json` (installed) or max hallmark from `Tools/vok/vov_veiled/vovr_registry.json` + git HEAD (Kit Forge)
* `IDENTITY` — ₢CORONET for pace-level, ₣FIREMARK for heat-level
* `ACTION` — single-letter code (required for all commits)

*Standard notch* (created by {jjdo_record}):
----
jjb:HALLMARK:₢XXXXX:n: commit message

Co-Authored-By: ...
----

Standard notch commits use `n` to denote routine progress tracking.

*Landing marker* (created when spawned agent completes):
----
jjb:HALLMARK:₢XXXXX:L: {agent} {landed|crashed}

{agent completion report}

Co-Authored-By: ...
----

Subject format: `{agent} {landed|crashed}` where agent is the executor tier (haiku/sonnet/opus) and landed/crashed indicates success/failure. Body contains the agent's raw completion report.

*Session marker* (created automatically by {jjdo_orient}):
----
jjb:HALLMARK:₣XX:s: YYMMDD-HHMM session

haiku: claude-3-5-haiku-20241022
sonnet: claude-sonnet-4-20250514
opus: claude-opus-4-5-20251101
host: macbook-pro.local
platform: darwin-arm64

Co-Authored-By: ...
----

Session markers record when a new work session begins. {jjdo_orient} creates these automatically when:

* The most recent commit in the steeplechase is more than 1 hour old, OR
* No commits exist in the steeplechase

The subject contains the session start timestamp in `YYMMDD-HHMM` format. The body contains:

* Model IDs for haiku, sonnet, and opus tiers (full model ID strings)
* Host identifier (machine name)
* Platform string (e.g., `darwin-arm64`, `linux-x86_64`)

Model IDs are collected via {vosr_probe} routine, which spawns parallel claude invocations via tokio async subprocess spawning. This approach is token-cheap (~10 token prompt, ~15 token response per probe) and provides accurate version tracking for multi-device workflows.

Pace-level action codes:

* `n` — Notch: standard commit
* `L` — LANDING: records agent return after autonomous execution
* `W` — WRAP: pace completion summary
* `d` — discussion: significant decision (lowercase, can also be heat-level)

*Heat-level operations*:
----
jjb:HALLMARK:₣XX:N: heat-silks
jjb:HALLMARK:₣XX:S: pace-silks
jjb:HALLMARK:₣XX:s: YYMMDD-HHMM session
jjb:HALLMARK:₣XX:r: reordered
jjb:HALLMARK:₣XX:D: old-coronet → new-coronet
jjb:HALLMARK:₣XX:R: heat-silks
----

Heat-level action codes:

* `N` — Nominate: create new heat
* `S` — Slate: add new pace
* `r` — rail: reorder paces (lowercase)
* `D` — Draft: move pace between heats (uppercase, rare)
* `R` — Retire: archive heat (uppercase)

NOTE: Officium markers migrated to VVC INVITATORY (vvb:...:i:), see VOS0-VoxObscuraSpec.adoc
