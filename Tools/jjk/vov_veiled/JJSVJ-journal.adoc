= Journal Ceremony and Break Sequence

The shared write bracket of the {jjdb_blotter}, and the break sequence that clears a stale
lock.
This sheaf is the ONE home of both sequences:
every operation that writes a {jjdb_blotter} cites this bracket and restates none of it,
so the operations cannot drift apart in the shared stretch.
The op names composing the sequences — `lodge`, `glean`, `advance`, `proffer`, `consign`,
`sync_state`, and the lock trio `stake`/`pluck`/`sight` — are {jjdf_farrier} primitives
({jjdf_core} and {jjdf_lock}); their method contracts live in the farrier sheaf's op census.

== The journal ceremony

[[jjdb_journal]]
{jjdb_journal}::
The bracket every {jjdb_blotter} write passes through, ordered *durable-first*:
any work artifact the journal will reference commits — and reaches the remote — before the
journal entry that names it.

The bracket:

. *Work half* (lock-free; present only when the write journals issued work — a notch or a
  wrap):
  `lodge` the explicit file list on the work repo's current or billet branch,
  take the {jjdb_counterfoil},
  and on a billet branch `consign` the branch —
  branches are the cross-station WIP carrier, and an unpushed branch is no carrier.
  Fast-forward and additive, so safe by construction; the trunk is never pushed by JJ.
  Offline, the ceremony proceeds with the {jjdb_counterfoil} flagged unpushed and
  `sync_state` nags at next open.
  The commit the journal will name now exists.
. *Take the lock*:
  `glean` (sync lock-ref state) →
  `stake` (the compare-and-swap: atomically push the {jjdb_guidon}-bearing lock ref, or be
  rejected) →
  `sight` to confirm the held {jjdb_guidon} is ours.
. *Advance*:
  carry the local {jjdb_blotter} forward to the remote tip — *fast-forward-only*,
  a no-op when already there.
  A clone holding commits the tip does not have is an impossible state under this ceremony
  (the branch adopts only accepted positions, step 5),
  so advance refuses it (diverged) and surfaces it for the attended session and operator —
  never auto-destroys.
. *Mutate*:
  apply the record change against the tip's own state — post-lock, post-advance,
  so a stale pre-lock read can never be carried across the lock —
  plus the {jjdb_counterfoil} naming the work repo and its commit,
  returning an explicit file list.
. *Proffer content*:
  compose ONE commit whose sole parent is the tip, carrying exactly the named files,
  *without moving the local branch*;
  push it `--atomic`, binding the content update to a lease on our own lock ref —
  if the lock was broken under us, the whole push fails;
  and only on acceptance advance the local branch to the accepted position.
  No corruption, no service but git.
  A refused push lands nothing on the remote and leaves the local branch and its record
  untouched — the refused commit never rides the branch,
  so no residue exists for any later ceremony to scrub.
. *Release*:
  `pluck`, best-effort;
  a die here leaves a stale lock still flying its {jjdb_guidon} for the {jjdb_break}.

*Compose-then-push* — a rivet, cited in shipped code by the opaque token `JJr_b52`:
when `mutate` runs, the local {jjdb_blotter} holds *nothing of its own* —
it stands at the remote tip as of this ceremony's `glean`,
so the one commit `proffer` composes takes a tip the remote itself authored as its sole
parent;
and the local branch moves *only to positions the remote has accepted*,
so content the lease refused never existed on the branch and is never re-consigned.
The citation sites are the `advance` and `proffer` implementations and the ceremony's own
steps;
one `grep JJr_b52` returns this definition and every site it guards.

*As gleaned* is the rivet's exact reach, and the limit is deliberate.
`glean` precedes `stake` and is opportunistic (step 2), so a station that lands a full
ceremony between our `glean` and our `stake` leaves us standing at a tip that is no longer
the remote's —
whereupon `mutate` builds on a stale image and `proffer` is refused (diverged) under a lock we
legitimately hold.
That is a wasted ceremony, never a corruption:
the refusal is the push's ordinary fast-forward protection, the next ceremony gleans afresh
and self-heals,
and the refusal leaves nothing behind —
the composed commit was never on the branch, so the stale image costs one retry and no
residue.
Closing the window (a re-`glean` after `stake`) is a separate act, weighed on its own.
WEIGHED, and left open deliberately:
the window costs at most one wasted, self-healing ceremony in a glean-to-stake race measured
in seconds,
while closing it puts a second network fetch inside every lock hold —
lengthening the very window in which a contender is refused LockHeld and in which a crash
strands a stale lock.
The cure would spread the disease it treats; the window stays open until the trade reverses.

The rivet is what committing on the branch cannot give, however the branch is later repaired.
A ceremony that commits on its branch *before* the remote accepts has, on refusal, a commit
the remote never authorized sitting where the next lodge would stack on it and the next push
would carry it —
so content one lock refused arrives on the shared store under a later, unrelated lock,
with nothing recording that it was ever refused;
or, had another station written in between, the same commit wedges the station instead,
with a divergence it has no verb to recover from.
The retired construction answered by *equalize-and-retrench*:
advance made the clone equal to the tip, destroying the refused commit under the next lock.
That worked, but it kept a destructive verb (`git reset --hard`) in the write path to scrub a
state the ceremony itself had created.
Compose-then-push removes the state instead of the evidence:
the branch never holds an unaccepted commit, so there is nothing to carry, nothing to diverge
from, and nothing to destroy —
the fast-forward that equalize was once too weak for is now sufficient,
because the invariant it could not deliver is delivered at the proffer instead.

The working tree beneath the ceremony is *writer-only scratch* (entity sheaf, Local clone;
readers reach the store by ref-read, never the tree):
a die between `mutate` and `proffer` leaves stray file content but no commit,
the next ceremony's advance passes unrelated scratch untouched,
and stray content cannot ride onto the store —
`proffer` composes from the tip's own tree plus an explicit file list, never a stage-all.
A refused commit is unreachable rather than destroyed
(git's object store keeps it forensically recoverable until pruned),
and the never-force rivet (`JJr_d81`, farrier sheaf) — which governs the remote's history —
is untouched throughout.

Durable-first is the bracket's law, not a preference:
an orphaned work commit (committed, never journaled) is recoverable;
a journal entry naming a nonexistent SHA is the worse failure.
The ordering classifies as a rivet;
its opaque tail mints at implementation, citing this section.

== The read bracket

The ceremony's read-only variant, for a consumer whose read *arms an act* —
a decision that mutates the world (the type specimen is {jjdd_muck}, whose snapshot decides
which billets are destroyed) —
as opposed to a read that merely reports, which stays on the entity sheaf's lock-free read
path ({jjdk_lockless_reads}).

The bracket is the {jjdb_journal} with its write steps removed and nothing else changed:
*Take the lock* (`glean` → `stake` → `sight`) →
*Advance* (fast-forward-only, `JJr_b52`) →
read →
*Release* (`pluck`, best-effort).
No mutate, no proffer.
What the acting consumer reads under this bracket is the store's truth at that moment —
no other station can be mid-write, and the advance step is what makes the local clone that
truth rather than a stale copy.

An acting read pays the lock's full costs, and they are accepted here once for every consumer:
it can be refused LockHeld while a stranger is mid-write,
and a crash inside the bracket strands a stale lock for the {jjdb_break}.
Uniformity is what buys the acceptance —
the store's correctness comes from its own bracket, never from an argument about what the
tenant data can or cannot do
(the entity sheaf's layer-independence rule),
so an acting consumer cites this bracket rather than reasoning its way out of it.

Every {jjdb_blotter} write passes through the {jjdb_journal}.
A no-other-door-exists claim has no runtime guard, only structure —
the bound is the design itself, and operations rely on it by citing this sheaf.

== The break sequence

[[jjdb_break]]
{jjdb_break}::
Clearing a stale lock is *never a method on the holder's guard* — it is its own sequence,
acting on someone else's lock:
`sight` the lock ref and report the observed {jjdb_guidon} (officium, station, acquire time,
operation),
then lease-guarded `pluck` against exactly that observed value — never blind, never forced.
There is no personified break entity.
+
The operator-deliberate break-all door — `sight` every JJ lock, report the {jjdb_guidon_s},
lease-guarded `pluck` per lock — is the {jjdd_cashier} (dispatch sheaf;
precedent `/vvc-BREAK-LOCK`);
the door cites this sequence per lock, adding nothing to it.

*The break itself takes no bracket*, and this is not a {jjdk_lockless_reads} violation.
The break's `sight` does arm an act — its own `pluck` — but the arming is never blind:
the `pluck` is lease-guarded against exactly the value `sight` observed, so a guidon that
changed in between refuses `LockBroken` rather than clearing a lock the operator never saw.
The mechanism *is* the compare-and-swap the bracket would otherwise buy.
A break that first took the lock it exists to clear could never run — stated here so the
absence reads as design rather than oversight.

=== What the lease protects, and what it does not

*The lease protects a live writer* — a rivet, cited in shipped code by the opaque token
`JJr_c4e`:
a writer whose lock is broken and restaked under it fails its own atomic `proffer` and lands
*nothing* on the remote (journal step 5).
The whole safety case of the {jjdd_cashier} rests here.
Breaking a lock cannot corrupt the store, and cannot half-apply a write, *however wrong the
operator's judgment of staleness was* —
the worst outcome against a live writer is a lost ceremony, which the writer's station reports
as a `LockBroken` refusal and can simply run again.
The citation sites are the `proffer` lease contract (farrier sheaf) and the door's own
consequence line;
one `grep JJr_c4e` returns this definition and every site it guards.

*The lease does not protect a live acting reader*, and this is an ACCEPTED EXPOSURE, ruled
here in writing.
The read bracket has no `proffer`, so nothing detects a lock broken under a reader:
the reader completes and *acts* on what it read, believing the exclusivity the bracket
promises — that no station can be mid-write — when in fact a writer may have staked and
proceeded in the meantime.
Nothing corrupts the store (a reader writes nothing), and nothing reports it either.
The exposure is accepted on the same footing as the `JJr_b52` re-glean window above:
closing it would mean giving the read bracket a proffer-shaped epilogue — machinery whose
only purpose is to detect an operator breaking a lock they were warned not to break —
and the loss it would prevent is one acting reader's decision made on a stale image, in a
race the operator has to lose deliberately to enter.
+
The exposure's operator face is the {jjdd_cashier_p} consequence line, which must
*differentiate*: a broken writer loses its ceremony and nothing else; a broken *reader* may
act on a stale image with nothing to catch it.
That differentiation is the whole reason the door names the held {jjdb_guidon_p} operation
rather than merely reporting that a lock is held.
+
The layer-independence rule (entity sheaf, Read posture) forbids arguing this exposure away
from tenant semantics — that a reaped billet is closed anyway, that the sweep is monotonic,
that no harm follows *for today's tenants*.
That argument was raised and OVERRULED (260714).
The exposure is accepted at the *store's* layer, on the store's own terms, so it stays true
whatever a tenant later stores.
