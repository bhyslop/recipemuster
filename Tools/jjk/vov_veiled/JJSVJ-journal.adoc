= Journal Ceremony and Break Sequence

The shared write bracket of the {jjdb_blotter}, and the break sequence that clears a stale
lock.
This sheaf is the ONE home of both sequences:
every operation that writes a {jjdb_blotter} cites this bracket and restates none of it,
so the operations cannot drift apart in the shared stretch.
The op names composing the sequences — `lodge`, `glean`, `advance`, `consign`, `sync_state`,
and the lock trio `stake`/`pluck`/`sight` — are {jjdf_farrier} primitives ({jjdf_core} and
{jjdf_lock}); their method contracts live in the farrier sheaf's op census.

== The journal ceremony

[[jjdb_journal]]
{jjdb_journal}::
The bracket every {jjdb_blotter} write passes through, ordered *durable-first*:
any work artifact the journal will reference commits — and reaches the remote — before the
journal entry that names it.

The bracket:

. *Work half* (lock-free; present only when the write journals issued work — a notch or a
  wrap):
  `lodge` the explicit file list on the work repo's current or billet branch,
  take the {jjdb_counterfoil},
  and on a billet branch `consign` the branch —
  branches are the cross-station WIP carrier, and an unpushed branch is no carrier.
  Fast-forward and additive, so safe by construction; the trunk is never pushed by JJ.
  Offline, the ceremony proceeds with the {jjdb_counterfoil} flagged unpushed and
  `sync_state` nags at next open.
  The commit the journal will name now exists.
. *Take the lock*:
  `glean` (sync lock-ref state) →
  `stake` (the compare-and-swap: atomically push the {jjdb_guidon}-bearing lock ref, or be
  rejected) →
  `sight` to confirm the held {jjdb_guidon} is ours.
. *Advance*:
  *equalize* the local {jjdb_blotter} with the remote tip — made equal to it, never merely
  moved toward it.
  Total against every position the clone can be in;
  a clone ahead of the tip is retrenched back to it.
. *Mutate and lodge*:
  apply the record change, plus the {jjdb_counterfoil} naming the work repo and its commit;
  `lodge` with explicit files.
. *Consign content*:
  `--atomic`, binding the content update to a lease on our own lock ref —
  if the lock was broken under us, the whole push fails.
  No corruption, no service but git.
  A refused push lands nothing on the remote and leaves its commit in the local clone alone,
  unpushed and unauthorized;
  the next ceremony's *Advance* retrenches it away — that is its disposition, and the ceremony
  owes it no other.
. *Release*:
  `pluck`, best-effort;
  a die here leaves a stale lock still flying its {jjdb_guidon} for the {jjdb_break}.

*Equalized under the lock* — a rivet, cited in shipped code by the opaque token `JJr_b52`:
when `mutate` runs, the local {jjdb_blotter} holds *nothing of its own* —
it equals the remote tip as of this ceremony's `glean`,
so every commit `lodge` writes takes a tip the remote itself authored as its sole parent,
and content the lease refused is *retrenched*, never re-consigned.
The citation sites are the `advance` implementation and the ceremony's own advance step;
one `grep JJr_b52` returns this definition and every site it guards.

*As gleaned* is the rivet's exact reach, and the limit is deliberate.
`glean` precedes `stake` and is opportunistic (step 2), so a station that lands a full
ceremony between our `glean` and our `stake` leaves us equalized to a tip that is no longer
the remote's —
whereupon `mutate` builds on a stale image and `consign` is refused (diverged) under a lock we
legitimately hold.
That is a wasted ceremony, never a corruption:
the refusal is `consign`'s ordinary fast-forward protection, the next ceremony gleans afresh
and self-heals,
and the rivet's *retrench* half is untouched by the window —
a refused commit is local-only, so it is never in the tracking ref this step equalizes to,
whatever that ref's vintage.
Closing the window (a re-`glean` after `stake`) is a separate act, weighed on its own.

The rivet is what a fast-forward cannot give.
A refused ceremony leaves a commit the remote never accepted sitting in the clone (step 5),
and a fast-forward leaves a clone that is merely *ahead* exactly where it stands:
`lodge` then stacks on the refused commit and `consign` carries both,
so content one lock refused arrives on the shared store under a later, unrelated lock,
with nothing recording that it was ever refused —
and had another station written in between, the same commit wedges the station instead,
with a divergence it has no verb to recover from.
Which of the two befalls a station is pure timing.
Equalize forecloses both:
the clone is made equal to the tip, so there is nothing to carry and nothing to diverge from.

Retrenching is safe because a {jjdb_blotter_p} clone is a cache, never a work tree
(entity sheaf, Local clone):
it is engine-driven and sole-door, so it holds nothing a human authored.
Discarding a local-only commit there is a deliberate ruling, stated here — never a silent
papering-over — and git's reflog keeps the commit forensically recoverable regardless.
Equalizing moves the local cache and pushes nothing,
so the never-force rivet (`JJr_d81`, farrier sheaf) — which governs the remote's history — is
untouched.
Dirt is equalized away with everything else local,
which heals a second wedge: a die between `mutate` and `lodge` leaves the clone dirty with no
one to clean it,
and an advance that refused dirt would refuse every ceremony thereafter.
Untracked residue alone survives the equalize, and cannot ride onto the store:
`lodge` deposits an explicit file list, never a stage-all.

Durable-first is the bracket's law, not a preference:
an orphaned work commit (committed, never journaled) is recoverable;
a journal entry naming a nonexistent SHA is the worse failure.
The ordering classifies as a rivet;
its opaque tail mints at implementation, citing this section.

[[jjdk_sole_door]]
//axk_premise
{jjdk_sole_door}::
Every {jjdb_blotter} write passes through the {jjdb_journal}.
A no-other-door-exists claim has no runtime guard, only structure —
the bound is the design itself, and operations rely on it by citing this sheaf.

== The break sequence

[[jjdb_break]]
{jjdb_break}::
Clearing a stale lock is *never a method on the holder's guard* — it is its own sequence,
acting on someone else's lock:
`sight` the lock ref and report the observed {jjdb_guidon} (officium, station, acquire time,
operation),
then lease-guarded `pluck` against exactly that observed value — never blind, never forced.
There is no personified break entity.
+
The operator-deliberate break-all door — `sight` every JJ lock, report the {jjdb_guidon_s},
lease-guarded `pluck` per lock — is the {jjdd_break_door} (dispatch sheaf;
precedent `/vvc-BREAK-LOCK`);
the door cites this sequence per lock, adding nothing to it.
