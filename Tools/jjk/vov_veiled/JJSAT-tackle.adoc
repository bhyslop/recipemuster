== What tackle is

Tackle is a strong way to identify and curate sets of files, and to declare how rulesets
interrelating those filesets apply — the two-part charter this sheaf hangs from:

* *Filesets* — the identified, curated sets:
  "these globs are BCG bash, those are RCG rust, that subtree is theurge, that one is ifrit."
  Identification is the claim rules; curation is the coverage check that keeps every claim
  deliberate and every gap loud (Scope semantics below).
* *Interrelating rulesets* — the declared relations between filesets, and the disciplines that
  govern the relations themselves:
  a spec family balances an implementation, a test domain exercises a subject domain, a guide
  governs a fileset (The association graph below).

One codebase holds several discipline domains at once (ACG's own observation; the worked
census below), so the binding is file-grain, not repo-grain — and it is the binding that
spares the operator from writing "write BCG-compliant code for all the bash you touch" into
every chat.

*Tackle already exists in the home repo, denormalized.*
The CLAUDE.md `@`-include tree, the test-suite table ("regime changes → reveille +
regime-poison"), the hand-maintained acronym maps, the matricula's file-selection allowlist
(VOSMM Tier 0), and the retired SlickEdit workspace files (`Tools/vslf-rbw/`) are each a
hand-maintained *projection* of a scope→discipline→association structure that has no single
home.
Tackle is the normalization of that standing structure, not new machinery;
every consumer becomes a projection that re-derives instead of drifting.

Two structural discoveries the census forces:

* *The operator thinks domains-first.*

== A tackle is countable

*A tackle is one domain binding* — a fileset plus the governance it carries;
the *tackle set* is an upstream's table of them.
The breeze sheaf's usage ("blaze — owned by *a* tackle"; tackles among the silks-identified
gallops entities) agrees, and V4's "silks-identified resource bundle" maps onto the same
node.
The aggregate needs no minted name of its own:
plain prose — "the tackle set", "the upstream's tackles" — does the work, the pedigree is
the aggregate's home, and a mint waits until the aggregate itself acquires behavior worth
naming.

== Scope semantics

The allowlist/denylist question dissolves once *how a set is defined* (polarity) is
separated from *what happens to a file nothing claims* (the default).
The position, argued in full in the memo:

* *Claims are positive, at domain grain.*
  A claim rule is an anchored subtree × shape conjunction — wildcards retained as the
  intension mechanism (`**` subtree recursion, `*.rs` shape globs): what is banned is the
  *unanchored* floating pattern, and what is audited is maximal width (below).
  New files and new subdirectories inside a claimed domain auto-fall-in — negative
  inclusion's auto-coverage exactly where it is correct, because the domain owner chose the
  criterion once.
  A *new domain* is not automatic, and that is right: it deserves a founding act, the
  pedigree's own deliberate-ceremony posture one level down.
  With domain-grain claims, *placing a file is affiliating it* — the location decision that
  already happens is the discipline decision.
* *There is no default.*
  No catch-all row; totality is reached because humble disciplines (generated, inert asset,
  provenance-only) are first-class rows, never because a bucket swallowed the remainder.
* *Overlap means co-membership.*
  A file may sit in two domains at once — the standing example: workstation bash that with
  slight modification also runs in the cloud, BCG and CBG both applying.
  Consumers take the union of trappings; no precedence machinery exists; coverage reads
  *at-least-one* claim.
  Tackle wins by being simple rather than over-fitted.
  Accidental overlap is guarded by claims written at honest grain (the census's own rows are
  naturally disjoint) and by the audit report showing every overlap — never hand-mirrored
  exclusions (the SlickEdit disease, memo §autopsy).
  A supplanting relation ("this specific claim excludes that general one"), if ever truly
  needed, is a future deliberate addition.
* *The failure asymmetry is the decisive argument.*
  Under-coverage (positive's failure) is mechanically detectable: universe minus claims is a
  set subtraction any validator runs, no judgment needed.
  Mis-affiliation (negative's failure) is undetectable by construction: the table *is* the
  definition of "right", so the wrong discipline surfaces only when it misbehaves — for
  security-posture bindings, discovery by incident.
  Given teeth, choose the failure mode that can be automated away.
  (Prior art agrees: firewall default-deny, gitattributes, CODEOWNERS — see memo.)
* *Recursion is preserved in the claim; the hazard is anchor width.*
  Root-anchored recursive claims have maximal swallow radius (the SlickEdit
  launcher-swallow, memo §autopsy); deep-anchored ones are tame.
  Claims are therefore always path-anchored, and a root-anchored recursive claim or a
  bare-`*` shape is an *audit flag*, not a prohibition.
* *The table is intensional — rules, never lists; every fileset is derived.*
  The two query directions scale differently and the design leans on that:
  path→discipline is O(rules) per path with no tree walk (the saddle-time direction — how
  gitattributes serves million-file repos);
  the census direction (fileset materialization, coverage audit) walks the ambit only, is
  batch by nature, and is where caching lands, keyed (table version, tree state).
  A materialized roster is derived, per-clone, disposable state: it lives in station-local
  scratch, *never the studbook* — the same law as worktree paths (regional record,
  station-local fact).
  Caching, if the trees go huge, is the deliberate supersession VOSMM's transience invariant
  (VOr_k3p) already anticipates — never drift.
* *Coverage is owed within the ambit; three verdicts.*
  Per upstream, the pedigree declares the ambit (for the home repo: the whole git-tracked
  tree; for a large vanilla upstream: the subtrees JJ works).
  Any path then has one of three mechanical verdicts:
  *claimed* (some tackle governs it);
  *unclaimed inside the ambit* (the defect — loud, surfaced by the coverage subtraction);
  *beyond the ambit* (a known-unknown, not a defect, but a session touching one is told so).
  The same boundary bounds census cost: the coverage walk prunes beyond-ambit subtrees
  entirely — governance honesty and traversal cost turn out to be one boundary.
* *Dead rows are flagged.*
  A claim rule matching zero files is rot (the specimen's `RBM-recipes` row); the census
  reports it (lean: notice, not error).

== The association graph

The scope containment of claim rules is a tree;
the declared interrelations make it a graph.
A hame is a typed edge between two tackles — *specifies*, *tests*, *crosscuts* are the
census-observed types; the taxonomy stays open — and an edge can carry its own governing
discipline:
ACG is the type specimen, governing the code↔spec citation relation (reference-the-home,
rivet residue) rather than either side alone.

*Declared, never deduced.*
Every edge is authored — the pedigree-ceremony posture again — and traversal is mechanical:
no similarity inference, no embedding neighborhood, no linguistic guess (the VOSMM
mechanical-never-linguistic ethos, VOr_m7w, applied to context provisioning).
This is what makes the graph trustworthy enough to *govern* rather than merely suggest.

The consumer horizon (vision, not this sheaf's design):
a frontier-tier planning session judges "this implementation needs balancing against this
spec" *once*, captures the judgment as a declared edge, and every later provisioning replays
it mechanically — school walks the graph to queue exactly the context a weaker session needs
to execute cheaply.
A weak model cannot discover missing context; the declared graph is the precondition for
handing work down-tier at all.
The same freeze pattern as the grimoire: authored once with full wisdom, read forever by
exact match.
Execution-side design (gait affinities, blaze resolution, exclusivity) remains joint work
with the breeze sheaf (JJSAB, Reconciliation section) and is deferred below.

*Boxes before arrows* (sequencing comfort): the concept model holds nodes and edges, but the
build sequence need not — an MVP of nodes alone (remuda → trappings) is complete and useful
on its own, with hames arriving when their first consumer (school, or an ACG-citation lint)
actually lands.
The grammar stays tiny either way: a node is a row, an edge is a tagged pair of rows;
sophistication lives in what consumers do with the tags, app-side, per the substrate split.

== Consumers (all projections)

* *Session provisioning* (farrier sheaf, launch-time): the dispatcher reads the pedigree's
  tackle set and provisions context at file grain, replacing repo-grain `@`-includes.
  Honing is prediction and starvation fails silent, so the farrier sheaf's standing repairs
  ride along: an invariant conduct core ships unhoned, and a pull door turns starvation into
  one extra round-trip.
* *The matricula* (VOSMM Tier 0): its file-selection allowlist is tackle's first working
  consumer — shaped schema-compatible, homed VOK-side until representation delivery,
  migrating to the pedigree home at studbook founding.
  Its file-role layer (declaration-bearing / reference-only / index-of-record / generated) is
  the trappings' facet at scan grain.
* *Dynamic context*: the hand-maintained CLAUDE.md / `claude-*.md` acronym tables become
  generated projections, presented only when tackle says they are relevant — the matricula
  MVP's acronym-index regeneration is the first step already in motion.
  Doctrinal ancestor noted: MCM's task-lens machinery (focused activation, focal depths —
  Core / Expanded / Suppressed) is this same selective-context idea at concept grain;
  tackle-driven provisioning is its file-grain sibling, compiling a work-scoped view instead
  of shipping the whole constellation.
* *School / breeze* (horizon): gait selection and beat-map context assembly read the graph;
  design deferred to the joint seam.
  Type-specimen use case: the recurring failure where a model writes bash from ambient
  training habits unless reminded to read BCG.
  The tackle binding is what lets school *mechanically* place a
  read-the-governing-guide/review-against-guide beat in the beat map for any beat whose file
  scope lands in a BCG domain — retiring the operator's memory as the enforcement mechanism.
  Tackle's part is only the binding and the integrity declarations; the beat-shape is gait
  design, breeze-side.

Each consumer reads facets of the same rows; none owns the table.

== The substrate split (direction — not yet an API commitment)

Where tackle services divide between the shared substrate crate (the vvx-land library both
JJ and the matricula read) and the applications above it.
The dividing line is the matricula's own mechanical/judgment boundary (VOr_m7w), applied to
tackle:

* *The crate owns the mechanical half* — everything that is set operations over declared
  rules, no judgment: schema and load, claim-rule evaluation (path → verdict), roster
  derivation (rule × tree state → files), coverage subtraction (ambit minus claims),
  declared-edge traversal.
  Four query surfaces, matching the query directions in Scope semantics: resolve one path,
  derive one roster, audit coverage, walk edges.
* *The applications own the judgment half* — what to *do* with verdicts: JJ dispatch decides
  provisioning policy (honing, conduct core, pull door, warn-vs-refuse at the loudness
  seats); the matricula maps trappings facets to vestures and scan roles; school composes
  beats.
  Policy never sinks into the crate; rules never float into the apps.
* *Not yet claimable*: signatures and cache mechanics; the serialized form is settled (the
  schema section); the sequencing ruling holds.
  The shape above is a lean the matricula's interim allowlist reader should stay compatible
  with, not an interface to build against.
* *Distribution*: the crate ships (it is the reader the direction already places in the
  shipped substrate library); the matricula consumes it and never flows into it (VOr_q4f);
  the tackle *data* is regional, pedigree-side, while any materialized cache stays
  station-local (Scope semantics).
* *Names in data, shape in code*: the crate and apps know the *schema* — what a domain row
  is, what a tagged edge is — and never any particular name: no `BCG`, no `theurge`, no
  edge-type enum baked into vvx.
  Guides are referenced by path or id in the table; edge labels are opaque tags the apps
  match by string; the BURE tweak-name gate is the house precedent (enforce the shape, never
  the census).
  A tag graduates into code behavior only by a later deliberate act, never by drift.
  This is also what keeps domain fluidity cheap: a development-time domain change is a table
  edit (journaled, studbook-side), never a rebuild.

== The table — settling schema

One file per upstream's tackle set, studbook-side, engine-written under the journal
ceremony.
Serialization is gallops-grade: deterministic key order; optional members skip-when-absent
so an untouched store stays byte-canonical; operator-mutable values held raw and validated
at read — bad config never makes the store illegitimate.
A tackle serializes as one contiguous, self-contained, diff-clean block, so crossbreeding —
copying a tackle between projects' tables — is a block copy; identity re-mints at import
(derivation is generative).

Wire keys carry the `jjot` sprue.
Standing reservation: `jjo` is the JJ JSON-sprue container — a non-terminal naming nothing
itself; each JJ-authored JSON wire format takes one fourth letter beneath it (`jjot` =
tackle); gallops `jjg` predates the rule and stands unchanged.
(The `jj` third-letter space is nearly full, and reservations live partly in the JJK README
prefix registry, which code grep cannot see — the registry is part of any future gate.)
Tail grammar per gallops: one structure letter (`r` root, `t` tackle row, `e` hame), then
kind `n` (field key) or `e` (enum token).
The sprue carries uniqueness, so tail words stay plain English unless the concept is
operator-spoken; the spoken five are *silks*, *remuda*, *trappings*, *hames*, *ambit*.

----

      "jjottn_silks": "theurge",
      "jjottn_remuda": ["Tools/rbk/rbtd/**/*.rs"],

Members:

* `jjotrn_next_tackle_seed` — the 3-character base64url mint counter, advanced only under
  the store lock.
* `jjotrn_ambit` — the governed boundary: positive globs within which coverage is owed;
  a beyond-ambit path is a known-unknown, never a defect (Scope semantics).
* `jjotrn_tackles` — the tackle rows, keyed by tackle identity: the type-sentinel glyph
  (`‹g›`, pending eyeball election) plus a 3-character base64url body.
  The glyph rides the wire as type safety — a wrong-type identity fails at the parse
  boundary; the identity is immutable, seed-minted, never re-minted in place.
  References never ride names, so a relabel strands nothing (the gallops law,
  `jjrt_types.rs`).
* `jjottn_silks` — display name; freely relabelable; never a reference target.
* `jjottn_remuda` — the fileset definition: anchored subtree × shape globs assembling the
  flight of files this tackle governs.
  Wildcards are the mechanism; unanchored patterns are banned; root-anchored recursion and
  bare-`*` shapes are audit flags.
* `jjottn_trappings` — governance attachments: references to the guides, spec homes, and
  conduct documents the tackle carries.
  A reference names the guide's home — the illustrated bare paths are the home repo's
  self-governing special case — with the form pending the guide-home fork (Open forks).
  Names live in data, never in code; consumers read facets of the same attachments.
* `jjotrn_hames` — declared interrelations: `from`/`to` are tackle identities, never silks;
  `tag` is an opaque label apps match by string; `rule` optionally names the document
  governing the relation itself (ACG the type specimen).
  Skip-when-absent: a boxes-only table shows no hames key at all.
* Reserved, unbuilt: an integrity-checks row member; enum tokens (`jjot«s»e_`) — none yet.

Identity nature: algebraic — fixed count 3, base64url body, seed-minted under lock, no
hierarchical embedding (a tackle hangs on its pedigree by origin URL, never by prefix).
The glyph and the identity-kind word (the "firemark" analog) pend: glyph by eyeball
election, word when speech demands one.

== Where it lives

Studbook-side, beside the pedigree (one tackle set per upstream) — never in the hippodrome,
which carries no JJ inserts (canon: the revision-control cosmology sheaf,
vanilla-hippodrome premise).
Tackle and farrier config are sibling entries on the pedigree answering different questions
with different change cadences:
tackle binds *files to rules* (changes when code domains shift);
farrier binds *the repo to its mechanics* (changes when repo topology shifts).
The ambit plausibly seats on the pedigree beside the tackle set (open fork).
Letting tackle swallow VCS config would regrow the maximal tackle the MVP deliberately
defers.
The vanilla-hippodrome premise also forecloses the gitignore/gitattributes scaling trick of
in-tree distributed rule files: tackle must be centralized-external, which is what makes the
intensional-table + live-evaluator design mandatory rather than optional.

The trappings' targets obey the same exclusion, harder: the guides are proprietary — as
closed as the record itself — so a governed repo never carries them, and JJ operates on a
foreign upstream without disclosing that they exist, or disclosing only deliberately
whittled variants.
The home repo's in-tree guides are an ownership coincidence — the one upstream that owns
its own guides — never the pattern.
A trappings reference therefore names the guide's home, never a bare governed-repo-relative
path; which private home holds the guides, and with it the reference form, is an open fork
(Open forks).

== How it reaches a session

At saddle time (farrier sheaf, launch-time provisioning): the dispatcher reads the pedigree's
tackle set and provisions the session's context accordingly.
This replaces repo-resident `@`-include context at repo grain with runtime binding at file
grain, and is what lets vanilla external projects receive discipline context without carrying
any of it.

== Deferred (explicitly not MVP) — the banked maximal layer

Banked from the retired V4 workup so the ideas survive their source heat; each is a future
design decision, not a commitment:

* *₿ blazes* — symbolic file/resource references defined by a tackle; dockets and gaits
  reference them, school resolves them to concrete paths when writing warrants — breeze never
  sees ₿ symbols.
  Uniqueness scope is unsettled: the workup recorded both per-heat uniqueness and
  scoped-to-tackle identity (never global in any recording).
  Validation sketch: no duplicate definitions, no orphan references.
* *Actions* — named commands with exclusivity scopes (candidate enum: worktree / workstation /
  project), the orchestrator holding mutexes during concurrent dispatch when an action's
  scope demands it.
* *Gait affinities and read-also entries* — which gaits a tackle suits, and standing context
  attachments.
* *Integrity declarations* — build / test / lint checks a tackle makes available, informing
  chukker-boundary decisions in the breeze pipeline.
  Note: this is relocation, not speculation — the CLAUDE.md after-code-changes suite table is
  this content, hand-maintained today; promotion is cheap when its consumer lands.
* *Tackle–gait interface* — gaits declare required blazes; compatibility validates at school
  time.
* *Registry shape* — tackles as silks-identified, mutable resource bundles in a top-level
  registry beside the gait registry.
* *Per-heat tackle mutability* — MVP tackle is per-upstream standing config.
  A provisional reconciliation (unreviewed supposition — design review required):
  the per-upstream tackle set is the standing base; a heat may later overlay heat-scoped
  additions (blazes, actions) that reference and extend the base; "tackle" names the layered
  whole.
  This bullet is that fork's one home.
