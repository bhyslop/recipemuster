== What MVP tackle is

A *tackle* is the standing binding from file scope to governing discipline:
"these globs are BCG bash, those are RCG rust, that subtree is theurge, that one is ifrit."
Nearly a table — glob patterns on one side, discipline/guide references on the other — and it is
the binding that spares the operator from writing "write BCG-compliant code for all the bash you
touch" into every chat.

One codebase holds *several* discipline domains at once (ACG's own observation: multiple bash
domains; discrete rust domains in theurge, ifrit, and JJ itself), so the binding is file-grain,
not repo-grain.

== Where it lives

Studbook-side, beside the pedigree (one tackle set per upstream type) — never in the hippodrome,
which carries no JJ inserts (canon: the revision-control cosmology sheaf, vanilla-hippodrome premise).
Tackle and farrier config are sibling entries on the pedigree answering different questions with
different change cadences: tackle binds *files to rules* (changes when code domains shift);
farrier binds *the repo to its mechanics* (changes when repo topology shifts).
Letting tackle swallow VCS config would regrow the maximal tackle the MVP deliberately defers.

== How it reaches a session

At saddle time (farrier sheaf, launch-time provisioning): the dispatcher reads the pedigree's
tackle bindings and provisions the session's context accordingly.
This replaces repo-resident `@`-include context at repo grain with runtime binding at file grain,
and is what lets vanilla external projects receive discipline context without carrying any of it.

== One word, one concept (the V4 resolution)

The retired V4 execution workup defined a maximal tackle: a silks-identified resource bundle
carrying ₿-blaze file scopes, actions with exclusivity, gait affinities, read-also entries —
school-time execution machinery for the breeze pipeline.
Resolution: *this MVP definition becomes the tackle; the maximal version extends it later.*
One definition re-homed and grown — never two tackles side by side.
The maximal layer is banked below (Deferred) as this sheaf's own content; its execution context
(school, gaits, chukker boundaries) lives in the breeze aspirant sheaf (JJSAB), and the future
tackle design pace inherits this sheaf as its floor: blazes, actions, exclusivity, and gait
affinities are additions to the scope→discipline core.

== Scope layering (provisional — agent supposition, requires design review)

A possible reconciliation of the two scopes, recorded 260704 and *not to be acted on without
design review*: the per-upstream tackle is the standing base (discipline bindings, rarely
changing); a heat may later overlay heat-scoped additions (blazes, actions) that reference and
extend the base; "tackle" names the layered whole.
Operator ruling (260704): tackle architecture waits until the operator has worked inside the
new system long enough to find footing — it is sequenced *after* the studbook/farrier MVP
ships, never alongside it.

== Deferred (explicitly not MVP) — the banked maximal layer

Banked here from the retired V4 workup so the ideas survive their source heat; each is a future
design decision, not a commitment:

* *₿ blazes* — symbolic file/resource references defined by a tackle; dockets and gaits
  reference them, school resolves them to concrete paths when writing warrants — breeze never
  sees ₿ symbols.
  Uniqueness scope is unsettled: the workup recorded both per-heat uniqueness and
  scoped-to-tackle identity (never global in any recording).
  Validation sketch: no duplicate definitions, no orphan references.
* *Actions* — named commands with exclusivity scopes (candidate enum: worktree / workstation /
  project), the orchestrator holding mutexes during concurrent dispatch when an action's scope
  demands it.
* *Gait affinities and read-also entries* — which gaits a tackle suits, and standing context
  attachments.
* *Integrity declarations* — build / test / lint checks a tackle makes available, informing
  chukker-boundary decisions in the breeze pipeline: whether mechanical merge with
  tackle-declared checks suffices or an explicit alignment beat is needed.
* *Tackle–gait interface* — gaits declare required blazes; compatibility validates at school
  time.
* *Registry shape* — tackles as silks-identified, mutable resource bundles in a top-level
  registry beside the gait registry.
* Any per-heat tackle mutability story — MVP tackle is per-upstream standing config.
