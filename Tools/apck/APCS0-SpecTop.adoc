== Overview

This document defines the formal vocabulary for the detection pipeline in Ann's PHI Clipbuddy.
It specifies the entities the algorithm operates on, the resources it consults, the procedures it performs, and the execution sites at which each discerner runs.

The application specification (APCAS) owns the user experience.
The prototype specification (APCPS) owns implementation details, tech stack, data sources, wire-format JSON schemas, and the container image contents.
This document owns the detection pipeline's conceptual structure — the terms that must be precise for the pipeline to be reasoned about correctly.

The pipeline has three named procedures with ten parallel discerners between the first and second.
{apcs_normalization}
produces the canonical input.
Ten discerners each run as a pure function of
{apcs_normalized_text},
producing their own intimately-named finding type;
seven of them run inside the Rust application and three of them run inside
{apcs_container}.
{apcs_combining}
fuses the ten finding streams into
{apcs_unified_finding_s}
covering regions of
{apcs_normalized_text}.
{apcs_anonymization}
produces
{apcs_anonymized_text}.

Byte offsets, character arithmetic, and tokenizer mechanics live in code.
This vocabulary talks about regions of
{apcs_normalized_text} —
the addressable units of detection — and leaves their concrete representation as an implementation concern.


== Input

[[apcs_clipboard_content]]
{apcs_clipboard_content}::
Raw content read from the system clipboard on window focus.
May be HTML or plain text.
Consumed once, then the system clipboard is cleared.

[[apcs_epic_html]]
{apcs_epic_html}::
HTML produced by Epic's "Copy All" function.
Semi-structured: bold labels for demographics, section headers for clinical categories, narrative freetext for clinical prose, signature blocks for provider attribution.
The structural formatting is the primary advantage over plain text — labels anchor PHI detection with high precision.

[[apcs_clinical_gate]]
{apcs_clinical_gate}::
A heuristic that decides whether
{apcs_clipboard_content}
is clinical.
Scans for known label patterns (Patient:, DOB:, MRN:, etc.).
Acceptance threshold: two or more recognized labels present.
Content that fails the
{apcs_clinical_gate}
is not processed; the triage view clears.


== Parsed Structure

The parsed structure captures the semantic regions Epic's HTML encodes — labeled demographic fields, section headers, narrative prose.
Parsing is an implementation-level step of
{apcs_normalization};
this vocabulary exists because the narrative discussion of individual discerners refers to structural categories when explaining how they decide.
Region membership belongs to the concept model; byte offsets and string arithmetic do not.

[[apcs_document]]
{apcs_document}::
The result of parsing
{apcs_epic_html}.
A sequence of typed
{apcs_span_s}
whose concatenated text contributions form
{apcs_normalized_text}.

[[apcs_span]]
{apcs_span}::
A typed region within the
{apcs_document}.
Each
{apcs_span}
has a kind, an optional label, and text content.
Three kinds:
{apcs_labeled_field},
{apcs_section_header},
{apcs_narrative}.

[[apcs_labeled_field]]
{apcs_labeled_field}::
A
{apcs_span}
produced by a bold label followed by a value (e.g., `Patient: Margaret J. Thornton`).
The label is the bold text before the colon.
The value is the text after the colon, extending to the next structural boundary (another label, line break, or HTML element boundary).

[[apcs_section_header]]
{apcs_section_header}::
A
{apcs_span}
produced by a bold label matching a known clinical section name (Chief Complaint, History of Present Illness, Assessment/Plan, etc.).
Contributes the header text to
{apcs_normalized_text}
but carries no value — the section body follows as subsequent
{apcs_span_s}.

[[apcs_narrative]]
{apcs_narrative}::
A
{apcs_span}
containing freetext not attached to a label.
Clinical prose, family history details, plan items.
PHI in
{apcs_narrative}
{apcs_span_s}
lacks positional anchoring and must be detected by the discerners that do not depend on Epic labels.


== Detection Resources

=== Registries

[[apcs_label_registry]]
{apcs_label_registry}::
The enumerated set of
{apcs_labeled_field}
labels that trigger per-word PHI extraction in the
{apcs_label_scan}.
Each entry specifies the label text (case-insensitive match) and the
{apcs_phi_category}
assigned to extracted words.
+
[cols="2,1"]
|===
| Label | Category

| Patient:           | NAME
| Attending:         | PROVIDER
| Provider:          | PROVIDER
| Referring:         | PROVIDER
| Facility:          | FACILITY
| Electronically signed by: | PROVIDER
| Emergency Contact  | NAME
| Primary Care Physician | PROVIDER
|===

[[apcs_regex_table]]
{apcs_regex_table}::
The enumerated set of regular expression patterns that detect structurally identifiable PHI in the
{apcs_regex_scan}.
Each entry specifies a pattern, a
{apcs_phi_category},
and a capture group index.
See APCPS for the full pattern table.

=== Dictionaries

[[apcs_surname_dictionary]]
{apcs_surname_dictionary}::
US Census 2010 surnames, top 1,000 by frequency.
Source: public domain.
Consumed by the
{apcs_surname_scan}.

[[apcs_firstname_dictionary]]
{apcs_firstname_dictionary}::
SSA baby names, top 1,000 by aggregate frequency.
Source: public domain.
Consumed by the
{apcs_firstname_scan}.

[[apcs_city_dictionary]]
{apcs_city_dictionary}::
US state capitals plus top 100 cities by population.
Source: public domain.
Consumed by the
{apcs_city_scan}.

[[apcs_english_whitelist]]
{apcs_english_whitelist}::
Top 5,000 common English words (Norvig/Google n-grams).
Source: public domain.
Consumed by the
{apcs_english_scan}.

[[apcs_medical_whitelist]]
{apcs_medical_whitelist}::
Hand-curated medications, lab tests, diagnoses, anatomy, abbreviations.
Source: original curation.
Consumed by the
{apcs_medical_scan}.

NOTE: There is no umbrella "blacklist" or "whitelist" concept in this vocabulary.
Earlier drafts grouped the three name/location dictionaries under a `blacklist` concept and the two common-term dictionaries under a `whitelist` concept, then parameterized a single dictionary scan over both.
The unified framing was load-bearing only for the unified scan; with each dictionary now paired to its own intimately-named scan, the umbrella adds grouping without carrying weight.


=== PHI Categories

[[apcs_phi_category]]
{apcs_phi_category}::
The HIPAA Safe Harbor category attached to findings that identify PHI.
Determines the
{apcs_placeholder}
text used in
{apcs_anonymized_text}.
Enumerated: NAME, PROVIDER, FACILITY, DATE, DOB, PHONE, EMAIL, SSN, MRN, ACCOUNT, ENCOUNTER_ID, ADDRESS, HEALTH_PLAN_ID, DEVICE_ID, NPI, DEA, URL, IP_ADDRESS.
+
Container discerners carry their own native taxonomies (Stanford's 8-label flat set; spaCy's POS tags, dependency relations, and scispaCy biomedical entity labels; Stanza's Universal Dependencies POS tags, morphological features, dependency relations, and OntoNotes NER labels).
Mapping native taxonomies to the
{apcs_phi_category}
enumeration is a
{apcs_combining}
concern and is deferred along with the rest of combining.


== Normalization

[[apcs_normalization]]
{apcs_normalization}::
The one-time light-cleanup procedure that produces the canonical input for every discerner.
Consumes
{apcs_clipboard_content} —
parsed as
{apcs_epic_html}
into a
{apcs_document}
when the content is HTML, or taken as-is when plain text — and produces
{apcs_normalized_text}.
One of the three named procedures of the pipeline, together with
{apcs_combining}
and
{apcs_anonymization}.
Runs entirely inside the Rust application.
Internal mechanics (HTML stripping, whitespace collapse, line-break handling, encoding normalization) are implementation detail; the vocabulary concerns only the procedure's inputs and output.

[[apcs_normalized_text]]
{apcs_normalized_text}::
The canonical string every discerner consumes.
The output of
{apcs_normalization}.
Regions within
{apcs_normalized_text}
are the addressable units of detection; character positions are counted into this string, not into
{apcs_clipboard_content}
or
{apcs_epic_html}.
All ten discerners are pure functions of
{apcs_normalized_text}.


== Placement and the Container

[[apcs_placement]]
{apcs_placement}::
The execution site at which a discerner runs.
Enumerated: `rust_application` (the discerner runs in-process inside the Tauri-hosted Rust application, emitting findings via direct function calls) and `container` (the discerner runs inside
{apcs_container},
emitting findings via a JSON file written to the bind-mounted directory).
Every discerner declares its
{apcs_placement}
as part of its definition.
{apcs_combining}
reads both streams into the same
{apcs_evidence}
pool;
from combining's perspective the
{apcs_placement}
of a finding's originating discerner is a provenance field, not a structural dimension.

[[apcs_container]]
{apcs_container}::
A long-running Linux container that hosts the three sophisticated discerners —
{apcs_stanford_scan},
{apcs_spacy_scan},
and
{apcs_stanza_scan} —
in their native Python runtime.
Parameterized by the set of discerner models it bundles (see APCPS for the concrete images, versions, and Dockerfile).
Reads inputs from and writes outputs to a single bind-mounted host directory; exposes no network interface (`--network=none`), runs as a non-root user, and drops all Linux capabilities.
Lifecycle is managed by tabtargets, not by the Rust application — start, stop, and image build are human-directed actions, distinct from Clipbuddy startup.
Communication with the Rust application is file I/O only:
Clipbuddy writes
{apcs_normalized_text}
as `{N}-in.txt` into the shared directory;
the container reads the highest-indexed
`{N}-in.txt`
it finds, runs its three discerners, and publishes their consolidated findings atomically as `{N}.json`
(write to `{N}.json.tmp`, then `rename(2)` to `{N}.json`).
The container emits a running log to `container-log.txt` in the same directory; the start-container tabtarget truncates the log at each container restart.
No sockets, no pipes, no HTTP — POSIX file I/O is the entire wire format.
See APCPS for the directory path, JSON schema, Dockerfile, and tabtarget set.


== Discerners

Each discerner is an independent algorithm that consumes
{apcs_normalized_text}
and emits an intimately-named finding type.
Ten discerners run in parallel on the same input; no discerner refines or filters another's output.
{apcs_combining}
fuses their finding streams.

There is deliberately no generic supertype for discerners and no generic finding type.
Each discerner is named individually, parameterized individually, and reasoned about individually;
{apcs_combining}
rules reference discerners and findings by name, not by a shared abstract shape.
This is load-bearing: earlier drafts that introduced `apcs_algorithm` and `apcs_reliability_profile` collapsed distinct concerns (what a discerner knows, how confident it is, how its output should be displayed) into non-load-bearing abstractions that each had to be re-specialized at every rule site.

[[apcs_finding]]
{apcs_finding}::
A single detection emitted by any discerner, covering a region of
{apcs_normalized_text}.
Prose noun — there is no generic schema.
Each discerner's
{apcs_finding}
subtype has its own intimate schema carrying whatever information that discerner's algorithm produces.

[[apcs_evidence]]
{apcs_evidence}::
The pool of every
{apcs_finding}
emitted by every discerner for a single input.
{apcs_combining}
is a function of
{apcs_evidence}.
The pool preserves provenance — a finding's originating discerner is never lost — and allows multiple findings to cover the same region without any one being discarded or subordinated before combining.

=== Regex Scan

[[apcs_regex_scan]]
{apcs_regex_scan}::
The discerner that detects structurally identifiable PHI by regular-expression pattern matching.
Parameterized by the
{apcs_regex_table}.
Scans
{apcs_normalized_text}
against every pattern.
Each match emits one
{apcs_regex_finding}
with the
{apcs_phi_category}
specified by the pattern entry.
{apcs_placement}: `rust_application`.

[[apcs_regex_finding]]
{apcs_regex_finding}::
The output unit of
{apcs_regex_scan}.
Carries: the matched text, the region of
{apcs_normalized_text}
it covers, and the
{apcs_phi_category}
specified by the pattern that fired.

=== Label Scan

[[apcs_label_scan]]
{apcs_label_scan}::
The discerner that extracts PHI positionally from recognized Epic labels.
Parameterized by the
{apcs_label_registry}.
For each registry entry whose label text appears in
{apcs_normalized_text},
segment the value following the label into words and emit one
{apcs_label_finding}
per word with the
{apcs_phi_category}
specified by the registry entry.
{apcs_placement}: `rust_application`.

[[apcs_label_finding]]
{apcs_label_finding}::
The output unit of
{apcs_label_scan}.
Carries: the word text, the region of
{apcs_normalized_text}
it covers, the
{apcs_phi_category}
specified by the registry entry, and the originating label (retained for audit and for combining rules that reference specific labels).

[[apcs_surname_scan]]
{apcs_surname_scan}::

or
{apcs_medical_scan}

{apcs_combining}

[[apcs_surname_finding]]
{apcs_surname_finding}::
The output unit of
{apcs_surname_scan}.
Carries: the matched text and the region of
{apcs_normalized_text}
it covers.
Implied
{apcs_phi_category}: NAME.

=== English Scan

[[apcs_english_scan]]
{apcs_english_scan}::
The discerner that detects common-English-word dictionary membership.
Parameterized by the
{apcs_english_whitelist}

Scans

{apcs_english_finding}.

[[apcs_english_finding]]
{apcs_english_finding}::

{apcs_english_scan}.
Carries: the matched text and the region of
{apcs_normalized_text}
it covers.
No implied
{apcs_phi_category} —
this finding asserts common-English membership, not a PHI classification.

=== Stanford Scan

[[apcs_stanford_scan]]
{apcs_stanford_scan}::
The discerner that detects PHI by neural named-entity recognition using `StanfordAIMI/stanford-deidentifier-base`.
Parameterized by the model's flat 8-label taxonomy:
`O, VENDOR, DATE, HCW, HOSPITAL, ID, PATIENT, PHONE`
(labels are flat — no BIO prefix — so runs of identical labels form a single entity).
Observed strengths (recorded from spike evaluation on the APCK synthetic fixtures): multi-token person names merged as a single span
(`Margaret J. Thornton` → one `PATIENT` span),
facility names merged
(`Maine Medical Center` → one `HOSPITAL` span),
narrative-scope catches where positional anchors are absent.
Observed weaknesses: structured identifiers (SSN, URL, address, device serial) fragment across multiple labels and are less reliable than
{apcs_regex_scan}
for the same categories.
The two are complementary, and
{apcs_combining}
is where the complementarity is cashed in.
Runs once per input inside the
{apcs_container};
emits a
{apcs_stanford_finding}
per contiguous labeled span.
{apcs_placement}: `container`.

[[apcs_stanford_finding]]
{apcs_stanford_finding}::
The output unit of
{apcs_stanford_scan}.
Carries: the matched text, the region of
{apcs_normalized_text}
it covers, the native Stanford entity label (one of the seven non-`O` labels above), and the model's confidence score for that span.
The native label is preserved verbatim; mapping to
{apcs_phi_category}
happens in
{apcs_combining}.

=== spaCy Scan

[[apcs_spacy_scan]]
{apcs_spacy_scan}::
The discerner that produces syntactic and biomedical-entity analysis via spaCy parameterized by a clinical-domain model (scispaCy's `en_core_sci_md`; see APCPS for the concrete pipeline component list).
Emits per-token structural signals (POS tag, morphological features, lemma, dependency head index, dependency relation) and any biomedical named-entity spans the pipeline produces.
The syntactic signals are the load-bearing output for this discerner's role in combining — the "may/May/may" disambiguation relies on POS tags separating modal verbs (AUX) from proper nouns (PROPN).
Runs once per input inside the
{apcs_container};
emits a
{apcs_spacy_finding}
per token and per entity span.
{apcs_placement}: `container`.

[[apcs_spacy_finding]]
{apcs_spacy_finding}::
The output unit of
{apcs_spacy_scan}.
Two variants share this name:
per-token findings carry text, region, POS tag, morphological features, lemma, and dependency information;
per-entity findings carry text, region, and the biomedical entity label.
Concrete JSON schema for both variants lives in APCPS.

=== Stanza Scan

[[apcs_stanza_scan]]
{apcs_stanza_scan}::
The discerner that produces syntactic and general-domain named-entity analysis via Stanza, parameterized by the English Universal Dependencies pipeline (see APCPS for concrete version).
Emits the full set of per-token signals Stanza's pipeline produces (Universal POS tag, language-specific POS tag, morphological features, lemma, dependency head, dependency relation, BIO-encoded NER tag) and entity-level spans derived from the BIO-tagged tokens.
Like
{apcs_spacy_scan},
chosen for its POS and dependency signals; its general-domain NER (OntoNotes-style PERSON/ORG/GPE labels) supplements
{apcs_stanford_scan}
with a second NER signal trained on different data.
Runs once per input inside the
{apcs_container};
emits
{apcs_stanza_finding_s}
at both per-token and per-entity granularities.
{apcs_placement}: `container`.

[[apcs_stanza_finding]]
{apcs_stanza_finding}::
The output unit of
{apcs_stanza_scan}.
As with
{apcs_spacy_finding},
per-token and per-entity variants share the name;
the concrete JSON schema for both lives in APCPS.


== Combining

NOTE: **Deferred section.**
The combining algorithm — the rules that fuse the ten discerner finding streams into authoritative per-region classifications — is deferred.
The decision to defer is deliberate: combining rules should be written with empirical
{apcs_evidence}
in hand from real clinical text processed through all ten discerners, not invented before the signals are visible.
The quoins below are placeholders so downstream procedures have referenceable inputs and so the document's link graph remains complete; their authoritative shape will be written in a later pace once representative inputs have flowed through the pipeline.

Specifically deferred:

* The noun for what
{apcs_combining}
produces per region (candidates under consideration include verdict, decision, call, outcome, determination, or remaining as
{apcs_unified_finding}).
* Per-discerner-pair precedence rules and the complete combining rule table.
* Granularity handling when findings of different sizes overlap (phrase-scoped Stanford findings covering word-scoped label findings, etc.).
* The concrete mapping from
{apcs_stanford_scan}'s
native labels onto the
{apcs_phi_category}
enumeration, and similar mappings for spaCy and Stanza native taxonomies.
* Clinician-toggle composition — whether clinician decisions enter combining as an additional finding stream, operate as a post-combining mutator on
{apcs_unified_finding_s},
or take some third shape.
* Second-order reprocessing — whether phrase-scoped findings derive additional word-scoped findings for uniform word-grained combining, while preserving the original phrase-scoped finding as evidence.

[[apcs_combining]]
{apcs_combining}::
The procedure that consumes
{apcs_evidence}
and produces
{apcs_unified_finding_s}
covering regions of
{apcs_normalized_text}.
One of the three named procedures of the pipeline, together with
{apcs_normalization}
and
{apcs_anonymization}.
Rules are deferred (see above).
The procedure is named
{apcs_combining}
rather than the earlier draft name `reconciliation` because the latter implied smoothing competing verdicts into one, whereas the intent is to read all evidence in parallel and produce a classification per region without discarding any finding's contribution to the record.

[[apcs_unified_finding]]
{apcs_unified_finding}::
The output of
{apcs_combining}
for a single region of
{apcs_normalized_text}:
a single authoritative classification consumed by
{apcs_anonymization}.
The concrete shape of a
{apcs_unified_finding}
(the fields carried, the classification vocabulary, the clinician-toggle composition) is deferred.


== Anonymization

[[apcs_anonymization]]
{apcs_anonymization}::
The procedure that consumes
{apcs_unified_finding_s}
and
{apcs_normalized_text}
and produces
{apcs_anonymized_text}.
Regions classified as PHI are replaced by their category-specific
{apcs_placeholder_s};
regions classified as retained text pass through verbatim.
Runs entirely inside the Rust application.
One of the three named procedures of the pipeline, named for symmetry with
{apcs_normalization}
and
{apcs_combining}.

[[apcs_anonymized_text]]
{apcs_anonymized_text}::
{apcs_normalized_text}
with every region marked for elision in the set of
{apcs_unified_finding_s}
replaced by its
{apcs_placeholder}.
The output format for pasting into Open Evidence.

[[apcs_placeholder]]
{apcs_placeholder}::
A category-specific substitution string emitted by
{apcs_anonymization}.
Format: `[CATEGORY]` (e.g., `[NAME]`, `[PHONE]`, `[ADDRESS]`).
Exception: DOB emits `Age: NN` (computed from date of birth and current date; ages over 89 emit `Age: 90+` per HIPAA Safe Harbor).


== Diagnostic Projections

The assay projections render the pipeline's internal state for review and debugging.
They are not user-facing — the user-facing surface is
{apcs_anonymized_text}
together with the triage view (owned by APCAS).
Two sibling projections exist because the pipeline now has two meaningful stages to inspect: the raw per-discerner
{apcs_evidence}
before
{apcs_combining},
and the fused view after.

[[apcs_discerner_assay]]
{apcs_discerner_assay}::
{apcs_normalized_text}
annotated inline with markers drawn from
{apcs_evidence}:
every
{apcs_finding}
from every discerner is rendered, with its originating discerner and any native tag it carries distinguishable in the marker.
Shows each discerner's raw output — overlaps, disagreements, and granularity differences are all visible side-by-side, which is the point.
The exact marker content per discerner varies with what each discerner's
{apcs_finding}
carries (see
{apcs_elision_marker});
the format is a debugging aid, not a stable interface.

[[apcs_unified_assay]]
{apcs_unified_assay}::
{apcs_normalized_text}
with every region of every
{apcs_unified_finding}
annotated inline using
{apcs_elision_marker_s}.
Shows the post-combining view — what
{apcs_anonymization}
actually sees.

[[apcs_elision_marker]]
{apcs_elision_marker}::
The annotation unit in an
{apcs_discerner_assay}
or an
{apcs_unified_assay}.
In an
{apcs_discerner_assay}
the format is `‹DISCERNER:TAG text›` (e.g., `‹REGEX:SSN 123-45-6789›`, `‹SURNAME:NAME Thornton›`, `‹STANFORD:PATIENT Margaret Thornton›`, `‹SPACY:AUX may›`) so each discerner's contribution is distinguishable;
the TAG slot carries the discerner's native tag (PHI category, native entity label, POS tag, etc.) or is omitted for discerners whose findings have no meaningful tag (e.g., the
{apcs_english_scan}
and
{apcs_medical_scan},
where membership itself is the finding and rendered as `‹ENGLISH may›` or `‹MEDICAL aspirin›`).
In an
{apcs_unified_assay}
the format is `‹CATEGORY text›` — the discerner that originated the region is absorbed into the combining decision and no longer shown.
Guillemets delimit the marker to avoid collision with clinical prose punctuation.
