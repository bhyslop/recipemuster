== Overview

The Bash Utilities Specification defines the tabtarget dispatch system — the infrastructure for tab-completion-friendly launchers that route to kit formularies.

BUK invented tabtargets; BUK defines them.

For universal naming vocabulary (Cipher, Signet, Epithet, Inscription, Vesture), see VLS-VoxLiturgicalSpec.adoc.

== Tabtarget Vesture

[[bust_tabtarget]]
//axl_voices axo_entity
{bust_tabtarget}::
Vesture for tabtarget launcher identifiers.
Defines how launcher scripts in `tt/` are named.
+
* signet_case: lowercase with hyphen (the colophon)
* separator: `.`
* epithet_case: PascalCase (the frontispiece)
* envelope: `.sh`
+
Pattern: `{colophon}.{frontispiece}[.{imprint}].sh`
+
Examples:
+
* `rbw-cr.Rack.sh` — RBW workbench, Rack operation
* `rbw-cr.Rack.tadmor.sh` — same, with imprint specifying target
* `jjw-.HeatMount.sh` — JJK workbench, HeatMount operation
* `buw-tt.LauncherList.sh` — BUK tabtarget testbench, LauncherList operation

== Dispatch

The dispatch vocabulary defines the tabtarget launcher system.
A
{busdl_launcher}
validates arguments and delegates to a
{busdm_formulary}.
The
{busdc_colophon}
identifies the routing path;
the
{busdf_frontispiece}
provides human-readable description;
the
{busdi_imprint}
embeds target parameters.

[[busdc_colophon]]
//axl_voices axt_string
{busdc_colophon}::
The routing identifier in a tabtarget inscription.
A
{busdc_colophon}
determines which
{busdm_formulary}
handles the dispatch.
Includes the hyphen as part of its identity (e.g., `rbw-`, `buw-tt-`).
+
The
{busdc_colophon}
is the first component of the tabtarget pattern, before the first dot separator.
+
Examples:
+
* `rbw-` routes to RBW workbench
* `jjw-` routes to JJK workbench
* `buw-tt-` routes to BUK tabtarget testbench

[[busdf_frontispiece]]
//axl_voices axt_string
{busdf_frontispiece}::
The human-readable description in a tabtarget inscription.
A
{busdf_frontispiece}
uses PascalCase and describes the operation.
+
Appears after the
{busdc_colophon}
and dot separator.
+
Examples: `Build`, `Rack`, `RunVVX`, `HeatMount`.

[[busdi_imprint]]
//axl_voices axt_string
{busdi_imprint}::
An optional embedded parameter in a tabtarget inscription.
An
{busdi_imprint}
specifies a target, instance, or configuration variant.
+
Appears after a second dot separator following the
{busdf_frontispiece}.
+
Examples:
+
* `tadmor` in `rbw-cr.Rack.tadmor.sh`
* `srjcl` in `rbw-cr.Rack.srjcl.sh`

[[busdm_formulary]]
//axl_voices axo_entity
{busdm_formulary}::
A component that routes
{busdc_colophon_s}
to implementations.
Each
{busdm_formulary}
registers handlers for a set of
{busdc_colophon_s}
and dispatches to the appropriate function.
+
Two types exist: the
{busdm_workbench}
routes commands for a kit, and the
{busdm_testbench}
routes tests.
+
A
{busdm_formulary}
is sourced by the
{busdl_launcher}
and provides the dispatch function.

[[busdm_workbench]]
//axo_entity
{busdm_workbench}::
A
{busdm_formulary}
that routes commands for a kit.
Named `{prefix}w_workbench.sh`.
Provides a `{prefix}w_dispatch` function that routes by
{busdf_frontispiece}.

[[busdm_testbench]]
//axo_entity
{busdm_testbench}::
A
{busdm_formulary}
that routes tests for a kit.
Named `{prefix}t_testbench.sh`.
Provides a `{prefix}t_dispatch` function that routes test cases.

[[busdl_launcher]]
//axl_voices axo_entity
{busdl_launcher}::
A bootstrap script in the `tt/` directory that validates arguments and delegates to a
{busdm_formulary}.
A
{busdl_launcher}
is the entry point for tabtarget invocation.
+
Naming follows
{bust_tabtarget}:
`{colophon}.{frontispiece}[.{imprint}].sh`
+
Responsibilities:
+
* Parse
{busdc_colophon}
from own filename
* Validate required arguments
* Source the target
{busdm_formulary}
* Invoke the dispatched function

== Routing Behavior

=== Dispatch Flow

. User invokes launcher: `./tt/rbw-cr.Rack.tadmor.sh`
. Launcher parses its filename to extract:
* {busdc_colophon}:
`rbw-cr`
* {busdf_frontispiece}:
`Rack`
* {busdi_imprint}:
`tadmor`
. Launcher sources the appropriate
{busdm_formulary}
(`rbw_workbench.sh`)
. Formulary's dispatch function routes based on frontispiece
. Target function executes with imprint as parameter

=== Workbench Pattern

A
{busdm_workbench}
dispatches with a `case` over the
{busdf_frontispiece}:

[source,bash]
----
# Tools/{kit}/{prefix}w_workbench.sh

{prefix}w_dispatch() {
    local frontispiece="$1"
    local imprint="$2"
    shift 2

    case "$frontispiece" in
        Build)          {prefix}w_build "$imprint" "$@" ;;
        Rack)           {prefix}w_rack "$imprint" "$@" ;;
        *)              buc_fatal "Unknown frontispiece: $frontispiece" ;;
    esac
}
----

=== Testbench Pattern

{busdm_testbench_s}
follow the same pattern but route to test functions:

[source,bash]
----
# Tools/{kit}/{prefix}t_testbench.sh

{prefix}t_dispatch() {
    local frontispiece="$1"
    shift

    case "$frontispiece" in
        Unit)        {prefix}t_unit "$@" ;;
        Integration) {prefix}t_integration "$@" ;;
        *)           buc_fatal "Unknown test: $frontispiece" ;;
    esac
}
----

=== Test Registry Model

The test registry (`butr_registry.sh`) uses two orthogonal enrollment dimensions: the
{busr_fixture}
and the
{busr_suite}.

[[busr_fixture]]
//axo_entity
{busr_fixture}::
An init/setup execution context.
Each test case belongs to exactly one
{busr_fixture}.
{busr_fixture_s}
provide isolation via `_tsuite` subshell boundaries.
Registered with `butr_fixture_enroll(name, init_fn, setup_fn)`.

[[busr_suite]]
//axo_entity
{busr_suite}::
A cross-cutting selection group (N:M with cases).
A case can belong to multiple
{busr_suite_s}.
{busr_suite_s}
classify by dependency tier (what environment must exist), not by test methodology.
A
{busr_suite}
is a membership tag.
It enables suite dispatch: run a cross-cutting subset of cases grouped by their owning
{busr_fixture_s}.
Suite context is set with `butr_suite_enroll(SUITE1 [SUITE2...])` before case enrollments.
Queried by `butd_run_suite(suite_name)`.

Standard suite constants (defined as `readonly` in testbench kindle):

* `BUTR_SUITE_FAST` — no external dependencies, CI-safe
* `BUTR_SUITE_SERVICE` — needs credentials, no container runtime
* `BUTR_SUITE_COMPLETE` — full test matrix including container runtime

Dispatch entry points:

* `butd_run_all()` — all fixtures, all cases
* `butd_run_fixture(name [case...])` — single fixture, optional case filter
* `butd_run_one(fn)` — single case with its fixture setup
* `butd_run_suite(suite)` — cross-cutting suite, grouped by fixture

=== Colophon Conventions

{busdc_colophon_s}
follow a structure that encodes both kit identity and formulary type:

* `{cipher}-` — default workbench (e.g., `rbw-`, `jjw-`)
* `{cipher}-t-` — testbench (e.g., `buw-t-`, `jjw-t-`)
* `{cipher}-{variant}-` — specialized workbench (e.g., `rbw-cr-` for crucible rack operations)

The trailing hyphen is always part of the colophon — it separates the routing prefix from the dot-delimited tabtarget components.

== Test Invocation

The test invocation vocabulary defines how test cases exercise code under controlled capture.
All invocation tiers share a common primitive,
`zbuto_invoke`,
which captures stdout, stderr, exit status, and optional BURV output into
{busig_stdout},
{busig_stderr},
{busig_status},

=== Fact-File Pattern

[[busff_fact_file]]
//axl_voices axo_entity
{busff_fact_file}::
Inter-step communication pattern using named output files.
+
A producer writes a value to `${BURD_OUTPUT_DIR}/${CONSTANT}` during command execution.
A consumer reads it from `${ZBUTO_BURV_OUTPUT}/current/${CONSTANT}` after invocation.
The kindle constant name is the contract between producer and consumer.
+
Example:
+
[source,bash]
----
# Producer (in command implementation):
echo "${z_hallmark}" > "${BURD_OUTPUT_DIR}/${RBF_FACT_HALLMARK}"

# Consumer (in test case, after buto_tt_expect_ok):
z_hallmark=$(<"${ZBUTO_BURV_OUTPUT}/current/${RBF_FACT_HALLMARK}")
----
+
The `current/` path segment maps
{burd_output_dir}
inside the BURV isolation tree.
Fact-file constants are defined as `readonly` in kindle (e.g., `RBF_FACT_HALLMARK`).
+
*Cross-tabtarget chaining.* Beyond the same-dispatch producer/consumer above, a fact written by one tabtarget is read by a *later* tabtarget through the depth-1 chain: at dispatch start {burd_output_dir} (`current/`) is promoted to {burd_previous_dir} (`previous/`), so the prior tabtarget's facts are readable via {buf_read_fact} from {burd_previous_dir}. A consuming tabtarget calls {buf_relay} first to forward the baton (so the chain survives more than one hop), then {buf_read_fact} for the fact it needs.

[[busff_multi_fact]]
//axl_voices axo_entity
{busff_multi_fact}::
Inter-step communication pattern for emitting multiple facts of a uniform shape during a single dispatch.
Each fact lands as a file named `<root>.<extension>` under
{burd_output_dir}
(and durably under
{burd_temp_dir}),
where the extension classifies the fact type and the root identifies the per-row instance.
+
The pattern composes three rules:
+
* *Type-as-extension naming.* The extension token carries the fact's classification; the root carries the row identity.
A producer emitting per-alias observation facts writes `<alias>.buf_ext_alias` rather than encoding the type into the filename stem.
* *Closed registry.* Each allowed extension is a named constant of the shape `BUF_EXT_<NAME>` declared in `buf_fact.sh`.
Constant values match constant names downcased (e.g., `BUF_EXT_ALIAS="buf_ext_alias"`), so any fact file is recognizable on disk by extension alone.
Adding an extension is a deliberate edit to the registry; opaque ad-hoc extensions are not the shape.
* *BUK/RB boundary discipline.* The BUK registry holds only BUK-domain extensions.
RB-domain extensions register in `rbgc_constants.sh` under the `RBGP_FACT_EXT_*` family, and neither universe enumerates the other.
Boundary is by ownership of the producer module, not by where the fact lands on disk.
+
Emission cardinality: one write per `<root, extension>` key per dispatch.
A second write of the same key during one dispatch is a bug — the implementing primitive
{buf_write_fact_multi}
fails fatally on preexistence to surface it.
+
Semantically distinct from
{busff_fact_file}:
single-form is one named-content slot per dispatch (the filename is the contract); multi-form is many same-shaped emissions per dispatch (the extension is the contract, the root is per-row).

[[buf_write_fact_single]]
//axl_voices axo_entity
{buf_write_fact_single}::
Dual-write fact-file primitive — single-form.
Writes a named file to both
{burd_output_dir}
and
{burd_temp_dir}.
Dies if either copy already exists — a preexisting file indicates a double-write bug.
+
{burd_output_dir}
is the latest-command convenience copy, cleared on next dispatch.
{burd_temp_dir}
is the durable copy, addressable by anyone who captured the path.
+
All single-form fact-file producers must use
{buf_write_fact_single}
to ensure durable copies exist; multi-row producers use
{buf_write_fact_multi}.
+
[source,bash]
----
# Single-value fact:
buf_write_fact_single "${RBF_FACT_HALLMARK}" "${z_hallmark}"

# Single-slot fact whose body has structured content (e.g., burx.env):
buf_write_fact_single "${BUF_burx_env}" "${z_content}"
----

[[buf_write_fact_multi]]
//axl_voices axo_entity
{buf_write_fact_multi}::
Dual-write fact-file primitive — multi-form, implementing the
{busff_multi_fact}
pattern.
Writes a value to `<file_root>.<ext>` in both
{burd_output_dir}
and
{burd_temp_dir}.
Dies if either copy already exists.
+
Signature: `buf_write_fact_multi <file_root> <ext> <value>`.
The extension is an opaque pass-through — the function performs no registry validation; the registered `BUF_EXT_*` constant at the call site is the contract, and out-of-registry usage is a producer-side bug to be caught by code review, not by this primitive.
+
[source,bash]
----
# Per-row emission inside a producer loop:
for z_alias in "${z_aliases[@]}"; do
  buf_write_fact_multi "${z_alias}" "${BUF_EXT_ALIAS}" ""
done
----
+
Mirror of
{buf_write_fact_single}
for the multi-row shape; both use the same dual-write + fail-on-preexist semantics.

[[buf_relay]]
//axl_voices axo_entity
{buf_relay}::
Consume-side primitive — forwards the prior dispatch's facts from {burd_previous_dir} into this dispatch's {burd_output_dir} (the baton forward), so a multi-hop tabtarget chain survives past one hop.
No-op when there is no prior dispatch.
Files already present in {burd_output_dir} are preserved, never clobbered — so this dispatch's own reserved files (e.g. {buf_burx_env}) and any fact it already wrote are kept.
Install ordering invariant: a consuming tabtarget calls {buf_relay} FIRST, before any {buf_read_fact} or producer write.

[[buf_read_fact]]
//axl_voices axo_entity
{buf_read_fact}::
Consume-side primitive — reads a single named fact from {burd_previous_dir} and emits its bare value (trailing newline stripped) on stdout.
Fails hard if the fact is absent — a missing upstream fact is a broken chain, not a default-to-empty.
Single-form only: the value is an opaque singular string, never parsed.
Signature: `buf_read_fact <filename>`.

[[buf_burx_env]]
//axpof_fact
{buf_burx_env}::
Tinder constant naming the
{burx_regime}
fact-file (`burx.env`).
Written by bud_dispatch.sh via
{buf_write_fact_single}
at dispatch start, with terminal fields appended at completion.

[[buf_ext_alias]]
//axl_voices axt_enum_value axd_string
{buf_ext_alias}::
Registered BUK-domain extension constant for BURN alias observation facts (`BUF_EXT_ALIAS="buf_ext_alias"`).
Written by `burn_list_capture` (in `burn_regime.sh`) once per discovered alias profile, with empty content — presence-of-file is the fact, not file content.
Serialized as `buf_ext_alias` in fact-file extension token.
+
First and currently sole member of the
{busff_multi_fact}
registry; serves as the proof-of-shape for the convention.

=== Yelp Pattern

[[busy_yelp]]
//axl_voices axk_premise
{busy_yelp}::
A formatted string fragment for interpolation into `buh_line` calls.
Yelp content **must not contain literal `%` characters** — this premise allows safe interpolation into `printf` format strings via bash variable expansion.
+
Produced by
{busy_yawp_s}
via pure assignment of inert
{busy_diastema}
markers, resolved at display time by
{busy_format_yawp}.
+
Yelps are the vocabulary-first primitive for handbook output.
A kit defines vocabulary constants at kindle time by capturing yelps into readonly globals (e.g., `RBYC_DEPOT`, `RBYC_HALLMARK`).
Call sites compose lines by interpolating these constants into `buh_line` strings.

[[busy_yawp]]
//axl_voices axo_entity
{busy_yawp}::
A `buyy_*_yawp` function in `buym_yelp.sh` that produces a
{busy_yelp}
via pure bash assignment into the group return variable `z_buym_yelp`.
No subshells, no stdout, no process spawning — cannot fail.
+
Yawp functions stamp inert
{busy_diastema}
markers into strings.
All terminal capability decisions are deferred to
{busy_format_yawp}
at display time.
Eight yawp functions exist: `buyy_cmd_yawp` (command), `buyy_ui_yawp` (UI element), `buyy_href_yawp` (web hyperlink), `buyy_link_yawp` (vocabulary link), `buyy_tt_yawp` (resolved tabtarget), `buyy_pass_yawp` (pass/green), `buyy_warn_yawp` (warn/orange), `buyy_fail_yawp` (fail/red).

[[busy_diastema]]
//axl_voices axo_entity
{busy_diastema}::
A non-printing byte sequence (`\x02\x1N`) embedded in a
{busy_yelp}
by a
{busy_yawp}
to delimit styled regions.
Eight markers form the diastema family: CMD, UI, HREF_URL, HREF_TEXT, LINK_URL, LINK_TEXT, TT, and END.
Openers identify the region type; END returns to the ambient color.
HREF and LINK have separate markers because vocabulary links (teaching terms) and web links (external references) serve different pedagogical purposes and carry independent palette slots.

[[busy_format_yawp]]
//axl_voices axo_entity
{busy_format_yawp}::
The single intelligence point for yelp display: `buyf_format_yawp(color, string)`.
Takes a `BUYC_*` ambient color constant and a
{busy_diastema}-marked
string, resolves all markers into ANSI color sequences and OSC-8 hyperlinks, and sets `z_buym_format`.
+
Resolution order: structured markers (HREF, LINK) via BASH_REMATCH extraction, then simple markers (CMD, UI, TT) via string replacement, then END markers replaced with the ambient color.

[[busy_configurator]]
//axl_voices axo_entity
{busy_configurator}::
A `buyc_*` function called before kindle to set the formatting mode flag.
Three configurators exist: `buyc_dispatch` (probe TERM/NO_COLOR/BURD_*), `buyc_unconditional` (always color), `buyc_plain` (never color).
Kindle reads the mode flag and defines the readonly `BUYC_*` public color palette and hyperlink mode.

=== Regime Enum Sprue Convention

{axrg_regime_s} across BUK and the kits layered on it (Recipe Bottle's
RBRV/RBRN families among them) declare enumerated {axrg_variable_s} through
`buv_enum_enroll`.
This convention governs how the *values* of those enumerations are named, so
that each value is discoverable from its enum family by a single grep.

[[bus_enum_sprue]]
//axl_voices axt_enum_value
{bus_enum_sprue}::
A regime enum value carried in prefix-disciplined form rather than as a bare
English word.
The value is a sprue in the MCM sense (`mcm_sprue`): a literal wire-level
token appearing verbatim in regime assignment files, Cloud Build
substitutions, build-info JSON, and shell/Python comparisons.
Bare, the token is invisible to its enum family; a recognizable prefix makes
it also an MCM inlay (`mcm_inlay`) — recognized by pattern, so a `grep` on
the family prefix returns the whole cluster.
+
Pattern: `<proj><n><regime><e>_<value>`
+
* `<proj>` — project cipher (`rb` Recipe Bottle, `bu` BUK)
* `<n>` — regime-namespace signal (`n`; `r` is taken by the regime-variable
  family `rbr*`/`bur*`)
* `<regime>` — regime family letter (`v` vessel, `n` node, `c` config,
  `s` station, `e` environment)
* `<e>` — enum-value signal
* `_<value>` — the canonical value name
+
Family-sharing is deliberate, not accidental: vessel-mode and egress-mode
both carry `rbnve_` (both vessel-regime); the three node-mode enums share
`rbnne_`, so `rbnne_disabled` is voiced alike by ENTRY, DNS, and ACCESS.
Uniqueness lives in the full value plus its owning {axrg_variable}, not in
the shared prefix — one grep per family returns the complete cluster.

==== Target Enums

Vessel regime (`rbnve_`):

* `RBRV_VESSEL_MODE` — bind/conjure/graft → `rbnve_bind` / `rbnve_conjure` / `rbnve_graft`
* `RBRV_EGRESS_MODE` — tether/airgap → `rbnve_tether` / `rbnve_airgap`

Node regime (`rbnne_`):

* `RBRN_ENTRY_MODE` — disabled/enabled → `rbnne_disabled` / `rbnne_enabled`
* `RBRN_UPLINK_DNS_MODE` — disabled/global/allowlist → `rbnne_disabled` / `rbnne_global` / `rbnne_allowlist`
* `RBRN_UPLINK_ACCESS_MODE` — disabled/global/allowlist → same `rbnne_*` set

BURN node regime (`bunne_`):

* `BURN_PLATFORM` — linux/mac/windows → `bunne_linux` / `bunne_mac` / `bunne_windows`

Auth regime (`rbnae_`):

* `RBRA_ROLE` — governor/retriever/director → `rbnae_governor` / `rbnae_retriever` / `rbnae_director`

The auth role is the one enum whose value axis uniquely splits in two. The
minted family `RBCC_role_*` (`rbnae_*`, domain = governor/retriever/director)
is the `RBRA_ROLE` value — written into credential files, validated, carried as
the auth folio, swizzle-checked. A separate bare family `RBCC_account_*` (domain
also includes payor/assay/mason) composes GCP service-account account-ids/emails
and local secret-directory names. The bare side is justified by class-4 below:
those derived strings live in cloud/filesystem resource-name space and stay
bare even though they share the role words. The folio→file `case` in the RBRA
resolver is the translation point — the minted folio maps to a bare-named
secret directory.

==== Use-Site Classification

A bare-word grep on a value yields four classes; only two of them migrate.

. *Phase / concept references* — "the conjure phase", "the airgap build":
  name the workflow, not the enum value. *Leave bare.*
. *Enum-value references* — equality tests, case arms, JSON literals, and
  default-value defaults (`${z_mode:-conjure}`): the actual variable value.
  *Migrate.*
. *Identifier embeds* — `RBSAC-ark_conjure.adoc`, sigils like
  `rbev-bottle-ifrit-airgap`, test-step labels like `ordain-conjure`: the
  word is part of a larger identifier. *Leave bare.*
. *Coupled derived strings* — the value drives a derived token living in
  resource-name space that must itself stay bare. *Leave the derived string
  bare; migrate only the dispatch that selects it.* Two confirmed witnesses:
  egress → worker-pool suffixes (`RBGC_POOL_SUFFIX_*`, the `-tether` /
  `-airgap` fragments in pool paths), and vessel-mode → hallmark prefixes
  (`RBGC_HALLMARK_PREFIX_*`).
  Because of class 4, even a class-2 case arm is not blind find/replace: the
  arm becomes `rbnve_tether)` while the pool string it maps to stays
  `-tether`.

==== RBRN_RUNTIME Exclusion

`RBRN_RUNTIME` (`docker` / `podman`) is deliberately *excluded* — its value
stays bare.
It is consumed as the literal executable name
(`command -v "${RBRN_RUNTIME}"`, `${RBRN_RUNTIME} image inspect`).
Adopting the convention would demand a permanent value→binary indirection
layer at every dispatch site — a runtime cost paid for grep-discoverability
of two values that `RBRN_RUNTIME` itself already marks.
This is the one regime enum where the convention would cost more than it
returns; the exclusion is a decision, not an oversight.

==== Tweak Mechanism

The tweak mechanism provides a single caller-supplied name/value override that flows through invocation without BUK interpreting its semantics.
BUK enrolls, validates, and propagates the pair; consuming kits (e.g., RBK) check the name and act on the value.

Doctrine — what a tweak is for:
a tweak forces one hard-to-produce condition so a test can observe the infrastructure handle it correctly —
a colliding identity stamp (`buorb_ensconce_stamp`), a poisoned configuration value (`buost_regime_poison`), a denied credential path (`buorb_credless_guard`) —
ideally asserting the rejecting gate's named exit code, never bare nonzero (the precision exit-code band; see BCG "Precision Exit-Code Band").
Those three are the live behavioral census — exactly what `grep buo` finds, beside `buost_example`, BUK's own stub that exercises this channel's name/value plumbing in the BUK self-test rather than forcing any condition in a consumer.
An injection that enables a positive path, carries a parameter, or short-circuits work for convenience is not a tweak;
it wants a CLI parameter, a regime slot, or its own command.

Doctrine — one tweak at a time, by design:
any one test, fixture, or suite run carries at most one tweak.
This is sufficiency, not a limitation —
one forced condition per observation is what keeps the seam non-intrusive,
and suites composed of single-condition observations cover the bases without massive test machinery.
A test that appears to need two simultaneous tweaks is mis-tiered or mis-designed;
a dual need that survives scrutiny mints a second mechanism deliberately, never by widening this one.

Doctrine — a suite may reserve the slot:
a test tier may dedicate the tweak slot to one standing guard for its whole run;
cases in that tier then carry no tweaks of their own — the reservation is part of the tier's definition.
The fast tier is the live instance: it reserves the slot for `buorb_credless_guard`, which the token-mint membranes reject with the credless band code, so a passing fast run can never spend a credential.

The caller sets at most one
{bure_tweak_name}/{bure_tweak_value}
pair in the environment before invoking a tabtarget.
The pair is ambient — it propagates through `zbuto_invoke` like all BURE variables, without requiring re-kindle in the invoked command.

[[bure_tweak_name]]
//axl_voices axrg_variable axtu_string axd_optional
{bure_tweak_name}::
Consumer-interpreted tweak identifier.
The name selects which behavioral override the consuming kit applies.
BUK validates length (0–64 characters) and the `buo` tweak sprue — a `buo<segment>_` prefix — so an unregistered or mistyped name fails loud at enforce rather than silently no-op'ing at the consumer; BUK validates the shape only and assigns no meaning beyond it.
The sprue doubles as a virtual registry: `grep buo` enumerates every tweak across the codebase, with no central list to maintain.
+
Example: `buorb_ensconce_stamp` — tells RBK to pin a capture's identity stamp, forcing the stamp-collision condition the cloud-side guard exists to reject.

[[bure_tweak_value]]
//axl_voices axrg_variable axtu_string axd_optional
{bure_tweak_value}::
Consumer-interpreted tweak payload.
The value is opaque to BUK; its format and interpretation are defined by whatever kit checks
{bure_tweak_name}.
BUK validates length (0–256 characters) but assigns no meaning to the value.
+
Either variable may be set independently — BUK does not enforce pairing.
Consumer kits that require both must enforce that themselves.

BUK reaches remote nodes under the
{bus_jurisdiction}

{bus_jurisdiction}::
The BUK feature-area for reaching remote nodes deterministically.
Three verbs cover the privileged-side ceremonies:

precondition for {busn_garrison} and as a post-completion check for
{busn_caparison}; {busn_garrison} provisions and refreshes a workload
account from an existing admin SSH foothold. The workload-side

{bujb_jurisdiction}.

{bujb_jurisdiction}::

{busn_caparison},
{busn_invigilate},

{busn_garrison}

{bus_jurisdiction}
contracts not expressed in regime data:
the three shell-letter→`command=` directive mappings
(`b` = native bash for Linux/Mac;

=== Privileges

A
{burn_regime}
instance lives under `.buk/rbmn_nodes/<{burn_investiture}>/` and is shared
across station users (git-transported); it carries node shape only and
no key material.
A
{burp_regime}
instance lives under `.buk/rbmu_users/<{burs_user}>/<{burp_investiture}>/`;
the
{burs_user}
owns its credential subtree exclusively, and operations write only
under the current station user's tree.

{busn_garrison}
authenticates as
{burp_privileged_user}
using
{burp_privileged_key_file}
by key alone; admin trust and host posture are established
beforehand by
{busn_caparison}
per platform (Windows uses a one-time password fallback on its own
phase-1 admin SSH session for first-run trust; Linux/Mac wrap
ssh-copy-id orchestration).
Workload account is unprivileged, ssh-only, and named by
{bujb_workload_user}
(project-wide convention, single source of truth).

=== Garrison-Destructive Model

Every
{busn_garrison}
invocation wipes prior workload state on the node regardless of which
{burp_investiture}
or station user runs it.
There is exactly one workload user per node, ever; node runtime state
belongs to whoever last garrisoned it.
Garrison subsumes the workload-account lifecycle: there is no separate
verb to materialize, remove, or enumerate workload accounts — every
garrison rebuilds from a clean slate.
+
Admin-side host posture is out of garrison's scope.
{busn_caparison}
owns it per platform: Windows OpenSSH `sshd_config` hardening, WSL
distribution staging, sleep/hibernate disable, and Tailscale service
auto-start on Windows; Remote Login + pmset + Tailscale on macOS;
ssh-copy-id orchestration + sleep target masking + tailscaled on
Linux.
{busn_invigilate}
verifies the posture as garrison's precondition.

=== Verb Definitions

[[busn_caparison]]
//axl_voices axt_command
{busn_caparison}::
Establish admin host posture per platform — SSH key trust plus the
host-side state (sshd_config hardening on Windows OpenSSH, WSL
distribution staging, power/sleep configuration, Tailscale service
auto-start where applicable) — leaving the host ready for
{busn_garrison} and {busn_invigilate} verification.
Per-platform tabtargets
(`buw-jpCW`, `buw-jpCM`, `buw-jpCL`);
each implements its platform's posture establishment from a single
operator-facing verb.
Nuclear: every invocation purges and re-establishes the
caparison-managed posture from scratch — no skip-if-present logic.
Operator-managed scope (security-sensitive registry edits, Tailscale
interactive auth, etc.) is documented per-platform in
{bust_handbook_windows},
{bust_handbook_macos},
and
{bust_handbook_linux};
caparison cannot or does not automate that scope.
+
On Windows the SSH-trust phases retain a structural two-phase shape
forced by `Restart-Service sshd` terminating its own ssh session:
phase 1 (over the password-or-key admin SSH session) places the
admin pubkey, applies icacls lockdown, writes hardened sshd_config
directives, validates via `sshd -t`, and restarts sshd; phase 2
(key-auth-only reconnect) verifies the post-restart admin session
authenticates by key alone. Between phase 2 and the post-trust
admin posture, caparison-windows runs a preflight gate that asserts
{busn_invigilate}'s operator-precondition fact subset (the registry
keys + Tailscale-service-presence facts whose pointer is
{bust_handbook_windows}); the gate fails fast on missing operator
posture rather than after the multi-minute WSL-distribution
download. Post-trust admin posture (WSL stage, powercfg, Tailscale
service) runs after the preflight passes. The bash orchestrator
treats the disconnect immediately after `Restart-Service sshd` as
the expected phase boundary, not a failure.

[[busn_invigilate]]
//axl_voices axt_command
{busn_invigilate}::
Verify admin host posture per platform, read-only — assert each
posture fact (admin SSH key trust, service states, registry values,
power configuration, host-availability resources) matches its
expected value, with no mutations and no remediation.
Per-platform tabtargets
(`buw-jpIW`, `buw-jpIM`, `buw-jpIL`);
each owns its platform's posture fact list.
Failure pointers discriminate operator-managed posture
(→ the per-platform handbook) from caparison-managed posture
(→ the per-platform caparison).
+
Used three times in the caparison/garrison flow: as
{busn_caparison}'s preflight gate (caparison runs invigilate's
operator-precondition fact subset between phase 2 and post-trust
posture, so missing operator scope fails fast rather than after
expensive admin-posture work), as {busn_caparison}'s post-completion
check (caparison runs invigilate's caparison-deliverable fact subset
after applying its operations to verify they took — operator facts
already verified at preflight do not re-run within a single
caparison run), and as {busn_garrison}'s precondition check
(garrison runs the full invigilate before any destructive workload
work). Standalone invocation runs the full fact list.

[[busn_garrison]]
//axl_voices axt_command
{busn_garrison}::
Provision or refresh the workload account on a node from an existing
admin SSH foothold.
A single tabtarget per shell-letter
(`buw-jpGb`, `buw-jpGc`, `buw-jpGw`)
runs the same destructive ceremony with the shell-letter selecting
remote shell mappings via
{bujb_jurisdiction}.
Garrison-`b` is the native-bash workload-shell flavor on both Linux
and macOS; step bodies branch on {burn_platform}.
The elevation matrix differs across letters:
garrison-`b` runs destructive ops via `sudo -n` (Linux + Mac);
garrison-`c` rides on {busn_caparison}'s Windows-admin key trust;
garrison-`w` runs inner-WSL ops as `wsl.exe --user root` without
sudo.
Garrison never reads or writes any `sshd_config` and never alters
admin authorized_keys; step 1 is a uniform preflight gate provided
by {bujp_preflight}.
Idempotent on re-run; destructive of any prior workload state.
+
Ceremony steps:
+
. Preflight gate via {bujp_preflight}: probe admin SSH reachability
under key-only auth; layer letter-specific probes (sudo NOPASSWD
on garrison-`b`/linux; admin-group + sudo on garrison-`b`/mac;
WSL distribution presence on garrison-`w`; nothing additional for
garrison-`c`).
Across platforms this presupposes
{busn_caparison}
has run for this {burp_investiture};
{busn_invigilate}
runs as the posture precondition before preflight to surface
caparison-managed posture drift with specific guidance.
. Obliterate any existing workload account named
{bujb_workload_user}
via `zbujb_obliterate_workload` — the letter-agnostic, platform-aware
destroy helper. Scope spans every namespace a prior garrison of any
letter could have written to (on Windows: SAM entry, Windows profile
directory, Cygwin home, WSL-side Linux user inside `rbtww-main`; on
Linux/Mac: native account + home). Idempotent on absent state.
. Create fresh workload account named
{bujb_workload_user}:
unprivileged, no sudo, no admin group, ssh-only.
. Place workload pubkey
(derived from
{burp_workload_key_file}
via `ssh-keygen -y`)
in workload's authorized_keys with the shell-letter's `command=`
directive.
. Copy
{burp_workload_key_file}
to the remote workload account at the path hardcoded in
{bujb_jurisdiction}
per shell-letter,
so the workload can authenticate outbound (e.g., to GitHub).
. Validate round-trip: SSH as workload, exec a no-op, verify exit 0.
