//axhob_operation
{bust_garrison_wsl}

NOTE: {busn_garrison} ceremony with WSL bash as the workload shell, for
Windows nodes. Workload identity inhabits *three namespaces*: a Windows
local user (auth boundary discovered by Windows OpenSSH), the workload's
*own* registered WSL distribution `rbtww-main` (populated in workload's
`HKCU\Lxss` via `wsl --import` from a seed tarball), and a Linux user
inside that workload-owned distribution (execution context entered by
the locked-down `command=` directive's `wsl.exe --user`). The third
namespace is forced by Microsoft's per-user WSL design -- distributions
register only in the running user's `HKCU\Lxss`; admin's distributions
are invisible to workload sessions.

NOTE: Mechanism is *SSH-as-workload conducted by the privileged
orchestrator* -- bare workload authorized_keys → single workload SSH
session imports `rbtww-main` into workload's `HKCU\Lxss`, mints inner
Linux user, plants privkey, rewrites authorized_keys with the
locked-down `command=` form as its final file operation, then runs
`wsl.exe --shutdown` to release the `NTUSER.DAT` hive cleanly before
session exit. The single-session discipline closes the Windows User
Profile Service demotion path -- WSL helpers hold the `NTUSER.DAT`
hive while a workload's WSL VM is running, and a second workload
logon during the holding window would resolve to a numbered fallback
profile directory (`bujuw_user.<host>.NNN`) where the just-written
authorized_keys is invisible to sshd. Round-trip validation under the
locked-down directive is deferred to {bust_invigilate_windows}. The
brief bare-authorized_keys window exposes only the operator's own
running garrison -- workload pubkey derived from operator-owned
{burp_workload_key_file}.

NOTE: Destruction is letter-agnostic via `zbujb_obliterate_workload` in
{bujb_jurisdiction} -- single greppable destroy verb running the
five-phase nuclear cleanup: hive release (`wsl --shutdown` +
helper-process teardown) → `Win32_UserProfile` WMI delete (atomic
`ProfileList` SID, `NTUSER.DAT`, and folder removal including
numbered fallback dirs) → SAM scrub (`Remove-LocalUser`) → filesystem
sweep (any leftover `C:\Users\<user>*` dirs and the
`C:\cygwin64\home\<user>\` Cygwin home) → WSL vestige cleanup inside
admin's `rbtww-main` (vestige from prior garrison shapes; the
workload's own distribution lives in the workload's `HKCU\Lxss`).
Idempotent on absent state and tolerant of partial prior destruction.
Best-effort: Windows User Profile Service performs asynchronous
internal bookkeeping (`.bak` SID lifecycle, mounted-state markers)
after profile removal, so a 100% pristine state is not always
achievable on a single pass; the discipline is to sweep thoroughly
enough that `New-LocalUser` lands on a clean SID with the canonical
profile path. Admin trust is {busn_caparison}'s scope; sshd hardening
inside the WSL distribution remains operator-manual. Documentation
sources for the cleanup pattern: Microsoft Learn ("Profile loading
fails", "Scripts to clean profile folder and prevent TEMP user profile
creation"); helgeklein.com (Delprof2 -- documented authority for safe
profile teardown sequencing).

The {busn_garrison} operation on Windows via WSL:

{burp_investiture}

//axhoq_precondition
A {burp_regime} profile exists at the given {burp_investiture} and its
{burn_regime} targets a Windows OpenSSH node, reachable as
{burp_privileged_user} under key-only auth using
{burp_privileged_key_file}.

//axhoq_precondition
{busn_caparison} has run for this {burp_investiture} -- admin SSH trust
established, `rbtww-main` staged under the admin user (canonical:
`BUJB_wsl_distribution` in {bujb_jurisdiction}), operator-managed
registry posture / Tailscale / powercfg all in the shape
{bust_invigilate_windows} expects.

//axhoq_precondition
{burp_workload_key_file} is unencrypted and exists on the operator's
station.

//axhos_step
. *Assert admin posture*:
.. {busc_call}
{bust_invigilate_windows} for this {burp_investiture}
.. {busc_store}
`«INVIGILATE_RESULT»`
.. {busc_require}
`«INVIGILATE_RESULT»` is zero
.. {busc_fatal}
on non-zero -- {bust_handbook_windows} scope (operator-managed) or
{bust_caparison_windows} scope (caparison-managed); diagnostic surfaced
verbatim

//axhos_step
. *Open admin SSH session*:
.. {busc_call}
ssh as {burp_privileged_user} with {burp_privileged_key_file}; key-auth only
.. {busc_store}
`«ADMIN_SESSION»`
.. {busc_fatal}
on connection failure -- {bust_caparison_windows} scope

//axhos_step
. *Export admin's rbtww-main as workload-readable seed*:
.. {busc_call}
PowerShell `wsl.exe --export rbtww-main <seed-path>` over `«ADMIN_SESSION»`
to a path inside the workload's Windows profile -- admin can write there
before the workload first logs in, workload session can read it
.. {busc_store}
`«SEED_TARBALL»` (the chosen path)
.. {busc_fatal}
on export failure -- {bust_caparison_windows} scope (rbtww-main missing
or admin HKCU\\Lxss corrupted)

//axhos_step
. *Obliterate prior workload*:
.. {busc_call}
`zbujb_obliterate_workload` over `«ADMIN_SESSION»` -- five-phase
nuclear cleanup running each phase as one or more atomic admin-context
PowerShell calls; all phases idempotent on absent state and tolerant
of partial prior destruction
.. *Phase 1 -- Release hives*: `wsl.exe --shutdown` host-wide;
supplement with `Stop-Process -Name wslhost,wslservice -Force
-ErrorAction SilentlyContinue` for any orphan helpers that survive the
shutdown. Releases `NTUSER.DAT` handles held by WSL helpers from any
previous garrison-w run. Without this phase, Windows User Profile
Service can demote a subsequent workload logon to a numbered fallback
profile directory (`bujuw_user.<host>.NNN`), rewriting
`ProfileImagePath` in `HKLM\...\ProfileList\<SID>` -- a permanent
corruption that survives reboot
.. *Phase 2 -- Profile destruction via `Win32_UserProfile`*:
`Get-CimInstance Win32_UserProfile` filtered on
`LocalPath -like 'C:\Users\${BUJB_workload_user}*'` (wildcard catches
canonical AND numbered fallback dirs from prior demotions), piped to
`Remove-CimInstance`. This is the documented Microsoft path: User
Profile Service handles `ProfileList` SID key removal (canonical AND
`.bak` variants), `NTUSER.DAT` hive unload, and folder deletion
atomically
.. *Phase 3 -- SAM scrub*: `Get-LocalUser` probe →
`Remove-LocalUser` if present. Fresh local user is created in step 5
claiming a new SID; removing the old SAM entry severs lingering
references and guarantees the next `New-LocalUser` is not a no-op
.. *Phase 4 -- Filesystem sweep*: `Remove-Item -Recurse -Force` over
any `C:\Users\${BUJB_workload_user}*` paths still present after
Phase 2 (belt-and-braces when the previous `ProfileList` pointer no
longer matched the on-disk path), and over
`C:\cygwin64\home\${BUJB_workload_user}\` Cygwin home
.. *Phase 5 -- WSL vestige*: workload's Linux user inside admin's
`rbtww-main` distribution -- vestige from prior garrison shapes; the
workload's own distribution lives in its own `HKCU\Lxss`, populated
in step 7. `wsl.exe --distribution rbtww-main --user root userdel` +
`rm -rf /home/${BUJB_workload_user}` for any orphan home. Run inside
admin's distribution which {busn_caparison} provisioned
.. NOTE: Cleanup is best-effort. Windows User Profile Service
performs asynchronous bookkeeping (`.bak` SID lifecycle,
mounted-state markers) so a 100% pristine state is not always
achievable on a single pass. The discipline is to sweep thoroughly
enough that step 5's `New-LocalUser` lands on a clean SID with the
canonical profile path; incremental residue does not affect
operational behavior
.. References for the nuclear cleanup pattern: Microsoft Learn
("Profile loading fails", "Scripts to clean profile folder and
prevent TEMP user profile creation"); helgeklein.com (Delprof2)
.. {busc_fatal}
on PowerShell session failure -- {bust_handbook_windows} scope

//axhos_step
. *Create Windows workload account*:
.. provision {bujb_workload_user} as a Windows local user via
`net.exe user /add /passwordreq:no /active:yes` over `«ADMIN_SESSION»`
.. register the resulting SID under
`HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList` so
Windows OpenSSH builds a complete logon session on inbound SSH --
Win32-OpenSSH #1383
.. no Linux user is created inside admin's `rbtww-main` -- the
workload's Linux user is minted inside the workload's *own* imported
distribution in step 7
.. {busc_fatal}
on Windows account-management failure -- prior obliterate left state

//axhos_step
. *Place bare workload trust*:
.. {busc_call}
`ssh-keygen -y` on {burp_workload_key_file}
.. {busc_store}
`«WORKLOAD_PUBKEY»` from stdout
.. over `«ADMIN_SESSION»` create
`/mnt/c/Users/${BUJB_workload_user}/.ssh/` (mkdir -p, chmod 700) and
write `«WORKLOAD_PUBKEY»` to `authorized_keys` (chmod 600) as a bare
entry -- locked-down `command=` would route through workload's
`HKCU\Lxss` before the import lands
.. NTFS ACLs inherited from the parent profile (admin-owned, non-admin
read-only) satisfy StrictModes -- POSIX `chown` meaningless to Windows
OpenSSH which inspects NTFS ACLs not POSIX modes
.. {busc_fatal}
on key-write failure -- workload account left without trust

//axhos_step
. *Provision workload's WSL and lock down trust (single SSH-as-workload session)*:
.. {busc_call}
ssh as {bujb_workload_user} with {burp_workload_key_file} through the
just-placed bare entry -- session is a real Windows logon for the
workload, HKCU mounted. This is the *only* SSH-as-workload session
inside garrison-w; round-trip validation under the locked-down
directive is deferred to {bust_invigilate_windows}
.. {busc_store}
`«WORKLOAD_SESSION»`
.. over `«WORKLOAD_SESSION»` run `wsl.exe --import rbtww-main
<workload-wsl-root> «SEED_TARBALL»` -- registers the workload's own
`rbtww-main` in workload's `HKCU\Lxss`; `<workload-wsl-root>` is a
workload-writable directory inside the workload's Windows profile
.. then `wsl.exe --distribution rbtww-main --user root useradd
--create-home --shell /bin/bash ${BUJB_workload_user}` -- mints the
inner Linux workload user
.. then `passwd --lock ${BUJB_workload_user}` (root inside the
distribution) -- enforces SSH-key-only on the inner Linux account
.. then plant {burp_workload_key_file}'s contents at
`/home/${BUJB_workload_user}/.ssh/id_ed25519` (chmod 600) inside the
distribution -- workload's outbound auth, e.g., GitHub
.. then overwrite the workload's authorized_keys at
`/mnt/c/Users/${BUJB_workload_user}/.ssh/authorized_keys` with the
locked-down `command=` form, replacing the bare entry placed in
step 6 -- workload-owned file, workload-driven write avoids
ownership-transfer gymnastics. SSH server reads authorized_keys at
connection time only, so the running session is unaffected by the
rewrite
.. canonical directive content: `BUJB_command_w` in {bujb_jurisdiction}
with `${BUJB_workload_user}` substituted at directive-emit time:
+
[listing]
----
command="C:/Windows/System32/wsl.exe --distribution rbtww-main --user ${BUJB_workload_user} --exec /bin/bash -lc \"$SSH_ORIGINAL_COMMAND\"",no-port-forwarding,no-X11-forwarding,no-agent-forwarding
----
.. *Final act before session exit*: `wsl.exe --shutdown` --
terminates the workload's WSL VM and helper processes (`wslhost.exe`,
`wslservice`), releasing the `NTUSER.DAT` hive handle BEFORE Windows
unloads the logon session. Without this release, any subsequent
workload logon during the helper-holding window resolves to a
numbered fallback profile directory and the just-written
authorized_keys becomes invisible to sshd (see {busn_garrison}
mechanism NOTE for the demotion-bug rationale)
.. on session exit, delete `«SEED_TARBALL»` over `«ADMIN_SESSION»` --
seed has served its purpose
.. {busc_fatal}
on workload session failure, `wsl --import` failure, inner `useradd` /
`passwd` failure, privkey plant failure, authorized_keys lockdown
rewrite failure, `wsl --shutdown` failure, or seed delete failure --
workload state partially provisioned across namespaces; revalidate
from a clean garrison

//axhog_guarantee
On success:

* Exactly one workload identity named {bujb_workload_user} exists,
  mirrored across all three namespaces.
* Windows local user with SID registered under `HKLM\...\ProfileList`
  (auth boundary).
* Workload-owned `rbtww-main` registered in workload's `HKCU\Lxss`
  (distribution namespace).
* WSL Linux user inside that distribution (execution context entered
  by the locked-down `command=` directive's `wsl.exe --user`).
* Workload authorized_keys carries `«WORKLOAD_PUBKEY»` with the
  shell-letter `w` `command=` directive.
* {burp_workload_key_file} planted at `/home/${BUJB_workload_user}/.ssh/id_ed25519`
  inside the workload's distribution.
* `wsl.exe --shutdown` executed inside the workload session before
  exit; no orphan `NTUSER.DAT` hive handles outlive garrison-w.
* Admin posture is unchanged -- no `sshd_config` writes, no admin
  authorized_keys writes.

Garrison-w does *not* self-validate the locked-down round-trip;
{bust_invigilate_windows} asserts the round-trip as a post-completion
fact. Operator runs invigilate after garrison-w to confirm the
locked-down directive routes end-to-end through workload's WSL.

//axhoc_completion
Exit 0 when the authorized_keys lockdown rewrite, `wsl.exe --shutdown`,
and seed cleanup all complete. Non-zero on invigilate-windows
precondition failure, admin SSH failure, seed export failure,
obliterate failure, Windows account-management failure, key-write
failure, workload SSH session failure, `wsl --import` failure, inner
account-management failure, privkey plant failure, lockdown rewrite
failure, hive-release failure, or seed cleanup failure.
