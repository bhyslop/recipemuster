//axhob_operation
{bust_garrison_wsl}

NOTE: {busn_garrison} ceremony with WSL bash as the workload shell, for
Windows nodes. Workload identity inhabits *three namespaces*: a Windows
local user (auth boundary discovered by Windows OpenSSH), the workload's
*own* registered WSL distribution `rbtww-main` (populated in workload's
`HKCU\Lxss` via `wsl --import` from a seed tarball), and a Linux user
inside that workload-owned distribution (execution context entered by
the locked-down `command=` directive's `wsl.exe --user`). The third
namespace is forced by Microsoft's per-user WSL design -- distributions
register only in the running user's `HKCU\Lxss`; admin's distributions
are invisible to workload sessions.

NOTE: Mechanism is *SSH-as-workload conducted by the privileged
orchestrator* -- bare workload authorized_keys → workload SSH session
imports `rbtww-main` into workload's `HKCU\Lxss` + mints inner Linux
user + plants privkey → locked-down `command=` rewrites the bare entry
from a second workload session. Two workload sessions are structurally
distinct -- first lands a real Windows logon (HKCU mounted); second
overwrites the workload-owned authorized_keys without ownership
gymnastics. The brief bare-authorized_keys window exposes only the
operator's own running garrison -- workload pubkey derived from
operator-owned {burp_workload_key_file}.

NOTE: Destruction is letter-agnostic via `zbujb_obliterate_workload` in
{bujb_jurisdiction} -- single greppable destroy verb covering the
Windows SAM entry, `C:\Users\<user>\` profile (which by convention
contains the workload's WSL filesystem root, so purging the profile
transitively destroys the workload's WSL distribution registration
on-disk and `HKCU\Lxss` hive), `C:\cygwin64\home\<user>\` Cygwin home,
and the WSL-side Linux user inside admin's `rbtww-main` (vestige from
prior garrison shapes; idempotent on absent state). Admin trust is
{busn_caparison}'s scope; sshd hardening inside the WSL distribution
remains operator-manual.

The {busn_garrison} operation on Windows via WSL:

//axhopt_typed_parameter
{burp_investiture}

//axhoq_precondition
A {burp_regime} profile exists at the given {burp_investiture} and its
{burn_regime} targets a Windows OpenSSH node, reachable as
{burp_privileged_user} under key-only auth using
{burp_privileged_key_file}.

//axhoq_precondition
{busn_caparison} has run for this {burp_investiture} -- admin SSH trust
established, `rbtww-main` staged under the admin user (canonical:
`BUJB_wsl_distribution` in {bujb_jurisdiction}), operator-managed
registry posture / Tailscale / powercfg all in the shape
{bust_invigilate_windows} expects.

//axhoq_precondition
{burp_workload_key_file} is unencrypted and exists on the operator's
station.

//axhos_step
. *Assert admin posture*:
.. {busc_call}
{bust_invigilate_windows} for this {burp_investiture}
.. {busc_store}
`«INVIGILATE_RESULT»`
.. {busc_require}
`«INVIGILATE_RESULT»` is zero
.. {busc_fatal}
on non-zero -- {bust_handbook_windows} scope (operator-managed) or
{bust_caparison_windows} scope (caparison-managed); diagnostic surfaced
verbatim

//axhos_step
. *Open admin SSH session*:
.. {busc_call}
ssh as {burp_privileged_user} with {burp_privileged_key_file}; key-auth only
.. {busc_store}
`«ADMIN_SESSION»`
.. {busc_fatal}
on connection failure -- {bust_caparison_windows} scope

//axhos_step
. *Export admin's rbtww-main as workload-readable seed*:
.. {busc_call}
PowerShell `wsl.exe --export rbtww-main <seed-path>` over `«ADMIN_SESSION»`
to a path inside the workload's Windows profile -- admin can write there
before the workload first logs in, workload session can read it
.. {busc_store}
`«SEED_TARBALL»` (the chosen path)
.. {busc_fatal}
on export failure -- {bust_caparison_windows} scope (rbtww-main missing
or admin HKCU\\Lxss corrupted)

//axhos_step
. *Obliterate prior workload*:
.. {busc_call}
`zbujb_obliterate_workload` over `«ADMIN_SESSION»` -- single PowerShell
session purges Windows SAM entry, profile directory (transitively the
workload's WSL filesystem and `HKCU\Lxss` hive), Cygwin home, and the
WSL-side Linux user inside admin's `rbtww-main`
.. nuclear -- idempotent on absent state
.. {busc_fatal}
on PowerShell session failure -- {bust_handbook_windows} scope

//axhos_step
. *Create Windows workload account*:
.. provision {bujb_workload_user} as a Windows local user via
`net.exe user /add /passwordreq:no /active:yes` over `«ADMIN_SESSION»`
.. register the resulting SID under
`HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList` so
Windows OpenSSH builds a complete logon session on inbound SSH --
Win32-OpenSSH #1383
.. no Linux user is created inside admin's `rbtww-main` -- the
workload's Linux user is minted inside the workload's *own* imported
distribution in step 7
.. {busc_fatal}
on Windows account-management failure -- prior obliterate left state

//axhos_step
. *Place bare workload trust*:
.. {busc_call}
`ssh-keygen -y` on {burp_workload_key_file}
.. {busc_store}
`«WORKLOAD_PUBKEY»` from stdout
.. over `«ADMIN_SESSION»` create
`/mnt/c/Users/${BUJB_workload_user}/.ssh/` (mkdir -p, chmod 700) and
write `«WORKLOAD_PUBKEY»` to `authorized_keys` (chmod 600) as a bare
entry -- locked-down `command=` would route through workload's
`HKCU\Lxss` before the import lands
.. NTFS ACLs inherited from the parent profile (admin-owned, non-admin
read-only) satisfy StrictModes -- POSIX `chown` meaningless to Windows
OpenSSH which inspects NTFS ACLs not POSIX modes
.. {busc_fatal}
on key-write failure -- workload account left without trust

//axhos_step
. *Initialize workload's WSL via SSH-as-workload*:
.. {busc_call}
ssh as {bujb_workload_user} with {burp_workload_key_file} through the
just-placed bare entry -- session is a real Windows logon for the
workload, HKCU mounted
.. {busc_store}
`«WORKLOAD_SESSION_INIT»`
.. over `«WORKLOAD_SESSION_INIT»` run `wsl.exe --import rbtww-main
<workload-wsl-root> «SEED_TARBALL»` -- registers the workload's own
`rbtww-main` in workload's `HKCU\Lxss`; `<workload-wsl-root>` is a
workload-writable directory inside the workload's Windows profile
.. then `wsl.exe --distribution rbtww-main --user root useradd
--create-home --shell /bin/bash ${BUJB_workload_user}` -- mints the
inner Linux workload user
.. then `passwd --lock ${BUJB_workload_user}` (root inside the
distribution) -- enforces SSH-key-only on the inner Linux account
.. then plant {burp_workload_key_file}'s contents at
`/home/${BUJB_workload_user}/.ssh/id_ed25519` (chmod 600) inside the
distribution -- workload's outbound auth, e.g., GitHub
.. on session exit, delete `«SEED_TARBALL»` over `«ADMIN_SESSION»` --
seed has served its purpose
.. {busc_fatal}
on workload session failure, `wsl --import` failure, inner `useradd` /
`passwd` failure, privkey plant failure, or seed delete failure --
workload state partially provisioned across namespaces

//axhos_step
. *Lock down workload trust*:
.. {busc_call}
ssh as {bujb_workload_user} with {burp_workload_key_file} through the
still-bare entry
.. {busc_store}
`«WORKLOAD_SESSION_LOCKDOWN»`
.. over `«WORKLOAD_SESSION_LOCKDOWN»` overwrite the workload's
authorized_keys with the locked-down `command=` form, replacing the
bare entry -- workload-owned file, workload-driven write avoids
ownership-transfer gymnastics
.. canonical directive content: `BUJB_command_w` in {bujb_jurisdiction}
with `${BUJB_workload_user}` substituted at directive-emit time:
+
[listing]
----
command="C:/Windows/System32/wsl.exe --distribution rbtww-main --user ${BUJB_workload_user} --exec /bin/bash -lc \"$SSH_ORIGINAL_COMMAND\"",no-port-forwarding,no-X11-forwarding,no-agent-forwarding
----
.. {busc_fatal}
on workload session failure or rewrite failure -- bare trust still
in place; revalidate from a clean garrison

//axhos_step
. *Validate round-trip*:
.. {busc_call}
ssh as {bujb_workload_user} with {burp_workload_key_file}; run a no-op
command -- session routes through locked-down `command=`, dispatches
to workload's own imported `rbtww-main`, runs the inner Linux user's
shell
.. {busc_store}
`«WORKLOAD_RT_RESULT»`
.. {busc_require}
`«WORKLOAD_RT_RESULT»` is zero
.. {busc_fatal}
on non-zero -- locked-down form broken; workload reachable bare but
not under the operational directive

//axhog_guarantee
On success:

* Exactly one workload identity named {bujb_workload_user} exists,
  mirrored across all three namespaces.
* Windows local user with SID registered under `HKLM\...\ProfileList`
  (auth boundary).
* Workload-owned `rbtww-main` registered in workload's `HKCU\Lxss`
  (distribution namespace).
* WSL Linux user inside that distribution (execution context entered
  by the locked-down `command=` directive's `wsl.exe --user`).
* Workload authorized_keys carries `«WORKLOAD_PUBKEY»` with the
  shell-letter `w` `command=` directive.
* {burp_workload_key_file} planted at `/home/${BUJB_workload_user}/.ssh/id_ed25519`
  inside the workload's distribution authenticates the round-trip.
* Admin posture is unchanged -- no `sshd_config` writes, no admin
  authorized_keys writes.

//axhoc_completion
Exit 0 when the locked-down round-trip validation succeeds. Non-zero on
invigilate-windows assertion failure, admin SSH failure, seed export
failure, obliterate failure, Windows account-management failure,
key-write failure, workload SSH session failure, `wsl --import` failure,
inner account-management failure, privkey plant failure, seed cleanup
failure, lockdown rewrite failure, or workload round-trip failure.
