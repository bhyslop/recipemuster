NOTE: Workload identity inhabits *three namespaces*: a Windows local
user (auth boundary discovered by Windows OpenSSH), the workload's
*own* registered WSL distribution `rbtww-main` (populated in
workload's `HKCU\Lxss` via `wsl --import` from a seed tarball), and a
Linux user inside that workload-owned distribution (execution context
entered by the locked-down `command=` directive's `wsl.exe --user`).
The third namespace is forced by Microsoft's per-user WSL design --
distributions register only in the running user's `HKCU\Lxss`;
admin's distributions are invisible to workload sessions.

NOTE: Auth is decoupled from workload-profile state via an
admin-owned absolute-path authorized_keys.
{busn_caparison}
installs `Match User ${BUJB_workload_user}` in `sshd_config` routing
`AuthorizedKeysFile` to
`__PROGRAMDATA__/ssh/users/${BUJB_workload_user}/authorized_keys`.
Sshd's trust-file lookup never resolves the workload's `$HOME`.

NOTE: Workload provisioning runs as *four* atomic SSH-as-workload
sessions (step 11), each carrying a single cmdlet or native-binary
call per WSG WSp-105. The fourth session is `wsl.exe --shutdown`
alone, releasing the `NTUSER.DAT` hive before Windows unloads that
logon. Best-effort minimum-session-count: WSG-strict single-cmdlet
bodies and single-session-with-shutdown are mutually exclusive in
the transport stack, so multi-session demotion risk during sessions
1-3 is accepted; Layer 1 (caparison-installed absolute-path Match
block + admin-owned authorized_keys) is the auth defense if
demotion occurs. See Rationale.

NOTE: Cleanup (steps 4-8) is best-effort. Windows User Profile
Service performs asynchronous internal bookkeeping after profile
removal, so a single-pass 100% pristine state is not always
achievable; discipline is sufficient sweep that step 9's
`New-LocalUser` lands on a clean SID with the canonical profile
path.

NOTE: Garrison-w does not self-validate the locked-down round-trip.
The operator's first workload-SSH invocation post-garrison
(`buw-jwk`, `buw-jwc`, `buw-jws`) IS the round-trip validation; an
explicit invigilate fact for this assertion was retired as
overdesign once those operational tabtargets existed.

==== Rationale

*Four-session shape with final shutdown (step 11)*::
WSG WSp-105 (one cmdlet or native binary per body) forbids the
compound session shape (wsl --import + useradd + install-D + wsl
--shutdown all on one workload SSH connection). Each cmdlet
therefore opens its own session. Across sessions 1-3, helper
processes (wslhost.exe, wslservice) spawned by `wsl --import`
outlive their parent session and lock the NTUSER.DAT hive of the
workload's logon, exposing the demotion path (User Profile Service
relocating the canonical profile to `bujuw_user.<host>.NNN` on the
next logon). Session 4 is `wsl.exe --shutdown` as its only act,
releasing the WSL VM and the helper-process hive handles before
that session's logon unloads -- best-effort cleanup that limits but
does not eliminate the demotion risk across sessions 1-3. Layer 1
({busn_caparison}-installed absolute-path Match block + admin-owned
authorized_keys) is the auth defense if demotion still occurs:
sshd reads as SYSTEM from the absolute path and never resolves the
workload's `$HOME`. The lockdown rewrite (step 12) runs in admin
context after session 4 exits -- no in-session rewrite race.

*Two LIKE patterns in Phase 2*::
The query matches `LocalPath LIKE
'C:\Users\${BUJB_workload_user}%'` AND `LocalPath LIKE
'C:\Users\${BUJB_workload_user}.%'`. The first catches the
canonical path; the second catches numbered fallback dirs from
prior demotions (`bujuw_user.<host>.NNN`). One pattern alone leaks
state.

*Per-row Remove-CimInstance in Phase 2*::
WSG WSp-105 mandates one cmdlet per SSH body. The probe emits
rows; bash iterates and issues one Remove-CimInstance per row with
a LocalPath-specific `-Query`. A piped pattern (probe |
remove-CimInstance) would violate WSp-105 and is forbidden.

*Phase 5 is admin's rbtww-main, not workload's*::
Workload's distribution lives in workload's own `HKCU\Lxss`,
populated in step 11. Phase 5 cleans the workload's Linux user
vestige inside *admin's* `rbtww-main` -- vestige from prior
garrison shapes when workload shared admin's distribution. The
phase is retained because residual entries leak shell history and
HKCU pollution across garrisons.
