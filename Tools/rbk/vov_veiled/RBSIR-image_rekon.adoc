//axhob_operation
{rbtgo_image_rekon}

NOTE: This operation reports which canonical members are present under a
hallmark subtree in
{rbtgi_repository}
via the GAR REST API implemented by
{gar_service}.
Raw presence enumeration — no health classification (use {rbtgo_hallmark_tally} for that).

The
{rbtr_director}
rekons one domain:

* `rbfl_rekon_hallmark` — basenames present under a hallmark subtree

Rekon over Lodes is intentionally omitted: a reliquary cohort is a conclave {rbtga_lode} whose member tags are inspected via its Lode augur operation, and a bole Lode is a single image per anchor (1-deep), so neither warrants a presence table here — {rbtgo_image_audit} covers the catalog and {rbtgo_image_retrieve} covers retrieval.

== Hallmark Rekon

The
{rbtr_director}
rekons basenames under a hallmark:

//axhopt_typed_parameter
{rbst_hallmark}
. *Validate Input Parameters*:
.. {rbbc_require}
`«INPUT_HALLMARK»` provided as first argument
.. {rbbc_fatal}
if missing: "Usage: rbw-irh <hallmark>"

//axhos_step
. *Authenticate as
{rbtr_director}*:
.. Execute
{rbtoe_director_authenticate}
.. {rbbc_store}
`«DIRECTOR_TOKEN»` for subsequent API calls

//axhos_step
. *Enumerate Packages Under Hallmarks Root*:
.. {rbbc_store}
`«HALLMARKS_ROOT»` from `RBGL_HALLMARKS_ROOT` (kindled from `RBGC_GAR_CATEGORY_HALLMARKS`, the token `rbi_hm`)
.. {rbbc_store}
`«REPO_BASE»` as the GAR REST API repository base (`projects/«PROJECT»/locations/«REGION»/repositories/«REPOSITORY»`)
.. {rbbc_call}
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories.packages/list[GAR REST API: packages.list] with:
... method: `GET`
... endpoint: `«REPO_BASE»/packages?pageSize=1000`
... header: `Authorization: Bearer «DIRECTOR_TOKEN»`
.. {rbbc_require}
HTTP 200 response
.. Extract `.packages[].name`, strip the resource prefix, URL-decode `%2F` back to `/`, and retain only entries beginning with `«HALLMARKS_ROOT»/`
.. For each retained `<hallmark>/<basename>` package, {rbbc_call}
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories.packages.tags/list[GAR REST API: tags.list] for the package and {rbbc_require} at least one tag returned (HTTP 200 with non-empty `.tags[]`). Skip packages with zero tags — these are post-jettison walking-dead awaiting the depot's cleanup policy (see {rbtgo_depot_levy} step *Create Container Repository*). Filtering at this read site decouples the display from GAR's lazy reclamation cadence.
.. {rbbc_store}
`«HALLMARK_ROWS»` as `«HALLMARK» «BASENAME»` pairs derived from the live-tagged entries (a 2-deep enumeration)

//axhos_step
. *Filter to Input Hallmark*:
.. {rbbc_store}
`«PRESENT»` as the set of basenames in `«HALLMARK_ROWS»` whose hallmark equals `«INPUT_HALLMARK»`
.. {rbbc_fatal}
if `«PRESENT»` is empty: "Hallmark not found: «INPUT_HALLMARK»"

//axhoot_typed_output
{rbtga_ark}
. *Display Results*:
.. {rbbc_show}
table header `BASENAME | EXISTS | PACKAGE-PATH`
.. For each canonical basename in the order
{rbtga_ark_image},
{rbtga_ark_about},
{rbtga_ark_vouch},
`attest`,
{rbtga_ark_pouch},
{rbtga_ark_diags}:
... {rbbc_store}
`«MARK»` as `yes` if `«BASENAME»` ∈ `«PRESENT»`, else `no`
... {rbbc_store}
`«PATH»` as `«HALLMARKS_ROOT»/«INPUT_HALLMARK»/«BASENAME»` when `«MARK»` = `yes`, else the literal `(absent)`
... {rbbc_show}
row `«BASENAME» | «MARK» | «PATH»`

//axhoc_completion
The procedure completes successfully when the canonical row set has been displayed for the input identifier.
A subtree with no matching row across the canon is treated as the "not present" signal and reported as a fatal — empty subtrees do not yield empty tables.
The operation does not classify health; that is the role of {rbtgo_hallmark_tally}.
