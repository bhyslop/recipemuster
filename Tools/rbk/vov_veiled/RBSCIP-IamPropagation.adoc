==== Decision

IAM propagation delays for Cloud Build v2 connection creation are handled by
connection-creation retry with exponential backoff, not by an impersonation
readiness gate or fixed sleep delays.

Three propagation classes are recognized by the shared IAM grant/revoke harness in
`zrbgi_propagation_error_predicate`, all sharing the same exponential-backoff
budget defined by the `RBGC_PROPAGATION_*` constants:

* **Class A — forward member-visibility**: `setIamPolicy` and `getIamPolicy`
  calls fail with HTTP 400 "does not exist" when a newly-created service
  account is not yet visible to the IAM policy service as a valid member.
* **Class B — backward member-visibility**: `setIamPolicy` and `getIamPolicy`
  calls fail with HTTP 400 "is not deleted" when a recently-deleted service
  account is still referenced in existing policy bindings.
* **Class C — caller-recently-empowered**: `getIamPolicy` / `setIamPolicy`
  on a resource-scope target (AR repo, SA, bucket, secret) returns HTTP 403
  because the caller's recently-granted role (e.g. fresh `roles/owner` on a
  newly-levied depot) has not yet reached the resource-scope IAM cache,
  even though it has reached project- and SA-scope caches.

Class C is time-bounded only — no body fingerprinting. The bound itself is
the discriminator: real propagation succeeds within budget, real denial
waits the budget and fails cleanly. The same model is already proven in
{rbtgo_access_jwt_probe}. Resource-scope sites — both grant (`rbgi_add_*`)
and revoke (`rbgi_revoke_*`) — declare 403 tolerance with an empty body-glob;
project-scope sites do not. Class C attaches to how recently the *caller* was
empowered against the resource-scope IAM cache, not to the direction of the
policy edit: any resource-scope `getIamPolicy` / `setIamPolicy` issued by a
recently-empowered caller can draw the 403, whether it adds or removes a
member.

* **Secret access readiness**: The CB v2 connection creation LRO itself serves as the propagation test.
  If the Cloud Build Service Agent cannot yet access the secrets (IAM grant not propagated), the LRO fails.
  The caller retries with exponential backoff, deleting the zombie connection resource between attempts.
* **Secret resource propagation**: `getIamPolicy` on a newly created secret may return HTTP 404 while
  the Secret Manager resource propagates across Google's control plane.  The API documents that
  `getIamPolicy` returns an empty policy (HTTP 200) when no policy has been set — so 404 indicates
  the resource itself is not yet visible, not that the policy is absent.  After creating secrets,
  `rbuh_poll_until_ok` polls each secret's `getIamPolicy` endpoint until 200 before proceeding to
  grant IAM bindings.

* **SA keys subresource propagation**: After
  link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts/create[serviceAccounts.create]
  returns 200, the keys subresource read path
  (link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts.keys/list[serviceAccounts.keys.list])
  can lag the SA email path independently — empirically observed as a non-monotonic 200→200→404 flap where
  both the SA GET probe and a keys subresource `rbuh_poll_until_ok` probe return 200 while the real
  `keys.list` call immediately returns 404 on the same URL.  Mitigation: retry the actual `keys.list`
  call on HTTP 404 using the same backoff budget as step 7 keys.create (`RBGC_SA_KEY_CREATE_RETRY_*`) —
  same underlying physics (subresource visibility after parent creation).  A separate
  `rbuh_poll_until_ok` proxy probe before `keys.list` was found insufficient: it returned 200 in the
  same flap window that the real call observed 404, exemplifying why proxy probes are not retained as
  correctness gates (see "What Dies" below).  See evidence row below.

* **SA delete read-path flap**: The inverse of the create-side flap above.  After
  link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts/delete[serviceAccounts.delete]
  returns 200, the SA email GET read path flaps non-monotonically 404→200 across replicas while the
  deletion propagates — so a single 404 is not durable proof the account is gone.  A same-name recreate
  ({rbtgo_retriever_invest} / {rbtgo_director_invest}) racing a still-lagging replica then trips the
  invest existence preflight (200 → fail-loud "already exists").  Empirically observed where
  `rbuh_poll_until_gone` saw GET 200,200,404 and declared gone at 6s, then the recreate's preflight GET
  returned 200 two seconds later.  Mitigation: `rbuh_poll_until_gone` requires `RBGC_GONE_CONFIRM_STREAK`
  consecutive 404 observations before declaring the resource durably gone; any intervening 200 resets the
  streak, and the total wait stays bounded by `RBGC_MAX_CONSISTENCY_SEC`.  Same underlying physics as the
  create-side flap (multi-replica read convergence after a parent mutation).  See evidence row below.

* **Lifecycle split relocates both SA-churn races (keyfile tier)**: the
  idempotent establish / re-key split (see {rbtgo_retriever_invest} and
  {rbtgo_director_invest}) changes *when* the two races can fire.  The
  **create→grant race** (Class A forward member-visibility, plus the org's
  Domain-Restricted-Sharing lag on a brand-new SA member) now fires only at
  *first establish* — once per identity, off the re-key hot path — because a
  standing-depot re-invest rotates the key without minting a new SA.  The
  **delete→recreate race** (the SA delete read-path flap above) is
  *structurally eliminated* for re-key: the SA is no longer deleted and
  recreated, so there is no same-name recreate to race a lagging replica.  The
  `RBGC_GONE_CONFIRM_STREAK` mitigation survives only for the path that still
  deletes-then-reinvests — a genuine teardown ({rbtgo_retriever_divest} /
  {rbtgo_director_divest}) followed later by a fresh invest — now rare rather
  than per-cycle.  See
  `Memos/memo-20260604-credential-churn-leak-and-propagation-races.md`.

==== Rationale

The connection creation LRO tests the actual operation, not a proxy.
When the Cloud Build Service Agent accesses resources requiring recently-granted permissions,
a successful LRO proves end-to-end readiness: the IAM grant has propagated,
the service agent identity is active, and the resource content is valid.

Fixed `sleep` delays ("sleep and pray") are unreliable:

* Google's IAM propagation documentation states changes "typically take less than 2 minutes"
  but provides no guaranteed upper bound.
* A 15-second sleep is too short for worst-case propagation and wastefully long for the common case.
* Retry-on-error tests the actual operation: if the grant has propagated, the retry succeeds immediately;
  if not, the retry continues until the deadline.

The `testIamPermissions` API cannot serve as a readiness gate because it only tests the caller's own
permissions on a resource -- it cannot test whether a third-party service account (like the CB service agent)
has been granted access to a secret.

==== Superseded: Impersonation Readiness Gate

The prior approach used an impersonation readiness gate: mint a short-lived token as the CB service agent
via `generateAccessToken`, then call `accessSecretVersion` with that impersonated token, retrying on 403
until the grant propagates.  This approach was abandoned due to three problems discovered 2026-03-03:

. **roles/owner does NOT grant iam.serviceAccounts.getAccessToken** --
  The explicit `roles/iam.serviceAccountTokenCreator` grant (previously deleted as unnecessary per
  an earlier RBSCIP revision) was in fact required for impersonation.  Without it,
  `generateAccessToken` returns persistent 403.  roles/owner is a broad role but does not include
  the `iam.serviceAccounts.getAccessToken` permission needed for impersonation.

. **Wrong SA identity** -- The code impersonated the default build SA
  (`{PN}@cloudbuild.gserviceaccount.com`) but CB v2 connections use the Cloud Build Service Agent
  (`service-{PN}@gcp-sa-cloudbuild.iam.gserviceaccount.com`).  Even if impersonation worked,
  it would have tested the wrong principal.

. **No non-impersonation alternative exists** -- `testIamPermissions` and IAM Policy Analyzer
  reflect policy intent (what the policy document says), not enforcement state (whether the grant
  has actually propagated).  Only actual access as the principal proves propagation.

Pattern B (connection-creation retry) eliminates `generateAccessToken`, `accessSecretVersion`,
and any need for `roles/iam.serviceAccountTokenCreator` grants for the readiness gate.

==== What Dies

* The impersonation readiness gate (`generateAccessToken` + `accessSecretVersion` as a pre-check)
* Any `roles/iam.serviceAccountTokenCreator` grant solely for the readiness gate
  (build-time Mason impersonation grants are unrelated and preserved)
* Fixed `sleep 15` delays before CB v2 connection creation and after Governor SA creation
* Standalone `rbuh_poll_until_ok` as a correctness gate for IAM propagation
  (the SA GET poll remains as a fast-path existence check, not a grant propagation gate)
* Any assumption that SA existence implies SA usability in IAM policy operations

==== What Survives

* SA GET poll via `rbuh_poll_until_ok` -- retained as a fast-path check that the SA resource itself
  is readable, which typically resolves well before IAM policy propagation completes
* The existing 429/5xx transient retry loop in {rbtoe_iam_grant_project} step 3 --
  this handles a different failure class (rate limiting, server errors) and is preserved inside
  the new outer propagation retry
* All IAM grant functions ({rbtoe_iam_grant_project}, {rbtoe_iam_grant_repo},
  {rbtoe_iam_grant_sa}, {rbtoe_iam_grant_bucket}) -- enhanced with propagation retry,
  not replaced
* Cloud Build service agent `roles/iam.serviceAccountTokenCreator` grant on Mason SA --
  this is for build-time impersonation (Cloud Build executing as Mason), unrelated to the readiness gate

==== Backoff Profile

All IAM-grant propagation retries — every site in `rbgi_IAM.sh` plus the
inline GAR retry in `rbgg_invest_director` — use the same single profile
locked in `rbgc_Constants.sh`:

* Initial delay: `RBGC_PROPAGATION_INITIAL_DELAY_SEC` (3 seconds)
* Multiplier: 2x (per-site arithmetic, not a kindle constant)
* Cap: `RBGC_PROPAGATION_MAX_DELAY_SEC` (20 seconds)
* Deadline: `RBGC_PROPAGATION_DEADLINE_SEC` (420 seconds; 7 minutes)

This profile accommodates Google's documented "typically < 2 minutes" with margin for worst-case
propagation while avoiding indefinite waits. It is the single budget shared
across all three propagation classes (A: forward member-visibility, B:
backward member-visibility, C: caller-recently-empowered); operator value
changes propagate to every grant site by editing the constants once. The
JWT-bearer consumer-side retry uses its own budget
(`RBGC_SA_KEY_CONSUMER_RETRY_*`) — different cost-of-waiting profile.
