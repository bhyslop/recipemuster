{rbtgo_manor_affiance}

NOTE: This operation pledges the trust between a
{rbtgi_manor}
and an external OIDC identity provider — the
{rbtf_affiance}
act. It creates a provider under the manor's one standing workforce pool, with the
provider's attribute mapping. The pool itself is founded separately, at manor setup,
and affiance requires it already standing — it never founds the pool. One pool serves
every
{rbtgi_depot}
under the manor: the same federated `principal://` subject is grantable in each, so
the trust is a property of the manor, never of a depot.
The attribute mapping is mechanism-invariant; the provider's `oidc`
configuration is built per
{rbrf_mechanism}
— issuer-discovered web-sso under
{rbrf_mechanism_interactive},
an uploaded public JWKS under
{rbrf_mechanism_programmatic}.
Affiance stays vendor-agnostic — it consumes the regime's opaque values and never
learns which IdP product is behind them.

The
{rbtr_payor}
performs the affiance. The operation takes the target
{rbtf_foedus}
as a folio (param1) — the `rbef_`-sprued subdirectory name of a standing foedus in the
moorings foedera library, exactly as
{rbtgo_foedus_descry}
takes its subject — and acts on THAT named foedus's provider, resolved from the folio,
never the active or pinned foedus. Provider management is folio-addressed while the
{rbrr_active_foedus}
selector is reserved for the credential accessor (avow / don), completing the addressing
split
{rbtgo_foedus_descry}
already half-embodies. The vendor-agnostic core
`(RBRF_PROVIDER_ID, RBRF_IDP_ISSUER, RBRF_IDP_CLIENT_ID, RBRF_ATTRIBUTE_MAPPING)` is read
from the folio-resolved Federation regime unconditionally;
`RBRF_MECHANISM` then selects the provider body, and the pool to parent it under is
manor-level, read from the sibling Workforce regime (`RBRW_WORKFORCE_POOL_ID`) — the same
for every foedus. Under
{rbrf_mechanism_programmatic}
affiance additionally reads the uploaded public JWKS (`RBRF_IDP_JWKS_JSON`).
The interactive device-flow endpoints and scope (`RBRF_IDP_DEVICE_ENDPOINT`,
`RBRF_IDP_TOKEN_ENDPOINT`, `RBRF_IDP_SCOPE`) exist only under
{rbrf_mechanism_interactive}
and are the credential accessor's to read at
{rbtf_avow},
not affiance's.
An unresolvable folio — no such foedus in the library — is a precondition rejection
naming the absent foedus.

NOTE: Affiance is *idempotent-ensure-exists*, exercised once per foedus with
founding gravity. Re-running it against an already-seated provider neither duplicates
nor fails — the provider is ensured present (created when absent, left in place when
already there). The project's standing test trust (the federation-legs spike's
reference provider) is the type instance: against the seeded Federation regime this
operation is a no-op on the provider. A second foedus is a further affiance with a
different `RBRF_PROVIDER_ID` under the same standing pool; the pool persists across
providers, founded once at manor setup and never by affiance. This is the
{rbtf_affiance}
cinch — first or further provider alike — realized as one idempotent body.

NOTE: *Drift reconciliation is a deferred follow-up, not in the founding cut.* An
already-present provider is left as it stands; affiance does not yet patch a provider
whose live `attributeMapping` or `oidc` fields have drifted from the Federation
regime. The patch-to-converge behavior is a horizon item — until it lands, affiance
ensures existence, not field convergence. (The pool's own `sessionDuration`
reconciliation is the manor-setup finisher's, not affiance's.)

NOTE: Scope boundary. Affiance seats only the provider under the manor's standing
pool. The pool itself and the org-level workforce-pool admin grant (spike Finding F1)
are the manor-setup finisher's; the depot-scoped admission bindings (the
{rbtf_mantle}
service-account `tokenCreator` and the depot `serviceUsageConsumer` of spike
Finding F2) belong to the levy and admission ceremonies. Affiance sits between them —
neither the pool's founding nor a depot's admissions, only the provider.

//axhos_step
. *Ensure the pool provider and its attribute mapping*:
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/locations.workforcePools.providers/get[workforcePools.providers.get] for `RBRF_PROVIDER_ID` under `locations/global/workforcePools/RBRW_WORKFORCE_POOL_ID`
.. present (HTTP 200) → leave it in place, advance. Absent (HTTP 404) → create it:
.. {rbbc_submit}
link:https://cloud.google.com/iam/docs/reference/rest/v1/locations.workforcePools.providers/create[workforcePools.providers.create] with:
... parent: `locations/global/workforcePools/RBRW_WORKFORCE_POOL_ID`
... workforcePoolProviderId: `RBRF_PROVIDER_ID`
... body: `displayName` and `attributeMapping` are mechanism-invariant; the `oidc` block is built per `RBRF_MECHANISM`.
+
Under
{rbrf_mechanism_interactive}
(issuer-discovered keys, device-flow web-sso):
+
----
{
  "displayName": "RBRF_PROVIDER_ID",
  "attributeMapping": { <parsed from RBRF_ATTRIBUTE_MAPPING> },
  "oidc": {
    "issuerUri": "RBRF_IDP_ISSUER",
    "clientId":  "RBRF_IDP_CLIENT_ID",
    "webSsoConfig": {
      "responseType": "ID_TOKEN",
      "assertionClaimsBehavior": "ONLY_ID_TOKEN_CLAIMS"
    }
  }
}
----
+
Under
{rbrf_mechanism_programmatic}
(uploaded public JWKS, self-supplied JWT — no issuer discovery):
+
----
{
  "displayName": "RBRF_PROVIDER_ID",
  "attributeMapping": { <parsed from RBRF_ATTRIBUTE_MAPPING> },
  "oidc": {
    "issuerUri": "RBRF_IDP_ISSUER",
    "clientId":  "RBRF_IDP_CLIENT_ID",
    "jwksJson":  "RBRF_IDP_JWKS_JSON"
  }
}
----
.. {rbbc_await}
the returned operation to done
.. {rbbc_require}
the provider present. Ensure-exists only: an already-present provider is left as it stands — not recreated, not patched, not failed. Drift reconciliation of `attributeMapping` / `oidc` is the deferred follow-up noted above.
.. NOTE: `RBRF_ATTRIBUTE_MAPPING` is stored as a single `key=value` string in the regime and parsed into the API's `attributeMapping` object; the regime validator guarantees it maps `google.subject`. The mapped claim is vendor-specific (`assertion.oid` for Entra, `assertion.sub` elsewhere) and rides in the regime value, not in affiance.
.. NOTE: *Subject-namespacing is set here, provider-side.* Because one pool is shared across every foedus, the mapped `google.subject` is the co-tenancy seam: a production foedus maps to the citizen's IdP-stable canonical subject, while the synthetic test provider's regime maps `google.subject` to a namespace-DISJOINT value, so a test token can never satisfy a production `principal://…/subject/«S»` binding. Affiance writes the mapping verbatim from the regime (vendor-blind); the disjointness is the discipline the test foedus's `RBRF_ATTRIBUTE_MAPPING` carries — the load-bearing co-tenancy rule, enforced at this provider.
.. NOTE: `webSsoConfig` (`ID_TOKEN` / `ONLY_ID_TOKEN_CLAIMS`) is interactive-meaningful — the spike-proven shape that makes the provider serve the device flow. Under {rbrf_mechanism_programmatic} it is inert: GCP validates the self-supplied JWT against the uploaded `jwksJson`, never through web-sso. The proof observed `gcloud providers create-oidc` *demand* the web-sso flags even alongside `--jwk-json-path`; whether the REST `create` body must likewise carry an inert `webSsoConfig` or may omit it is an impl-confirm, not a hard contract.
.. NOTE: `jwksJson` is the uploaded public JWKS (REST field `oidc.jwksJson`, the `--jwk-json-path` target). GCP's parser is strict — only `kty/kid/use/alg/n/e` RSA members are accepted; an exporter's `x5c`/`x5t` certificate fields are rejected (misleadingly as `Only RSA, EC key types are supported`) and must be stripped before upload.
.. NOTE: The provider resource name `//iam.googleapis.com/locations/global/workforcePools/RBRW_WORKFORCE_POOL_ID/providers/RBRF_PROVIDER_ID` is the STS audience the accessor's second leg presents (spike Leg 2). Affiance seats the resource that name addresses; it does not itself exercise the legs.
