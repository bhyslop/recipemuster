NOTE: This operation banishes a whole
{rbtga_lode}
from
{gar_service}
via a GAR `packages delete` on the touchmark package — the whole-Lode delete.
The delete is **cloud-dispatched**: the workstation submits a
{rbtr_director}-run Cloud Build that deletes the package in-pool by convergence
(see the completion note) and verifies package absence — so the build's outcome IS
the delete outcome, closing the host trust-200 LRO gap by construction.
Kind-agnostic; the fetched-side parallel of made-side
{rbtgo_ark_abjure}, with which it shares the cloud-dispatch delete backbone.

NOTE: A
{rbtga_lode}
is exactly one GAR package, so `packages delete` removes the package and all its member versions and tags in one call.
("Atomic" = single operation, not transactional rollback.)
Banishment is whole-Lode; single-member cleanup — the fix-a-bad-member case — is a member/tag-grain operation in the image family ({rbtgo_image_delete}), not here.

+
A host-side presence probe drives the confirmation, so an absent Lode is reported as a clean "nothing to banish" rather than a delete error.
The in-pool delete is by **convergence**, not a single `packages delete`: GAR refuses to delete a child manifest while its parent index still exists (FAILED_PRECONDITION, "manifest is referenced by parent manifests"), so a whole-package delete of a multi-arch web removes nothing.
Each round the step fires a delete at every remaining version (force=true) and at the package shell — the parent indexes succeed and free their children, which a later round then deletes — and polls the package GET until it 404s.
Absence is the only success signal; the per-round "referenced by parent" skips are expected, not errors; a deadline is the sole failsafe.
This is GAR's documented parent-before-child behavior (and the shape of Google's own gcr-cleaner `--skip-errors`); rationale and posture live in RBSCB.
