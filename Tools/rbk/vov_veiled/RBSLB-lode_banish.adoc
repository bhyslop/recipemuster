//axhob_operation
{rbtgo_lode_banish}

NOTE: This operation banishes a whole
{rbtga_lode}
from
{gar_service}
via a single GAR `packages delete` on the touchmark package — the atomic whole-Lode delete.
Kind-agnostic; the fetched-side parallel of made-side
{rbtgo_ark_abjure}.

NOTE: A
{rbtga_lode}
is exactly one GAR package, so `packages delete` removes the package and all its member versions and tags in one call.
("Atomic" = single operation, not transactional rollback.)
Banishment is whole-Lode; single-member cleanup — the fix-a-bad-member case — is a member/tag-grain operation in the image family ({rbtgo_image_delete}), not here.

The
{rbtr_director}
banishes a Lode:

//axhoc_completion
Procedure completes when the touchmark package — the whole
{rbtga_lode}
with every member version and tag — has been removed from
{gar_service}
in one `packages delete`.
+
A presence probe drives the confirmation, so an absent Lode is reported as a clean "nothing to banish" rather than a delete error.
Because one Lode is one package, there is no per-member dance and no orphan-detection branch — the single delete is the whole operation.
If GAR ever refuses a single-call delete of a mixed-artifact package, the fallback is a pool-run delete loop over the package's tags, effectively atomic for these purposes.
