[[rbscig_iam_grant_contracts]]
==== IAM Grant API Contracts

Research memo documenting the actual behavioral contracts of the Google Cloud
IAM policy APIs used by the four grant patterns:
{rbtoe_iam_grant_project}, {rbtoe_iam_grant_repo},
{rbtoe_iam_grant_sa}, {rbtoe_iam_grant_bucket}.
Also covers Secret Manager ({rbtoe_iam_grant_secret}).

This document provides verified evidence for spec definition corrections and
future convergence decisions.  It should be re-reviewed when any grant function
behavior is modified, when new resource types require IAM grants, or when
Google API behavior changes are suspected.

Findings are verified against API reference documentation, upstream proto
definitions (`google.iam.v1.Policy`, per-service `service.proto` files), and
ecosystem evidence (Terraform provider issues, Red Hat bugzilla, Pulumi issues).

Research conducted: 2026-03-11.

===== HTTP Method and Body Format

[cols="2,2,2,2",options="header"]
|===
|API
|getIamPolicy
|setIamPolicy
|SET body format

|CRM Projects ({rbtoe_iam_grant_project})
|POST (body: `{"options":{"requestedPolicyVersion":3}}`)
|POST
|`{"policy":...}`

|IAM {giam_service_account_s} ({rbtoe_iam_grant_sa})
|POST (body: `{"options":{"requestedPolicyVersion":3}}`)
|POST
|`{"policy":...}`

|{gar_registry} ({rbtoe_iam_grant_repo})
|GET (query param)
|POST
|`{"policy":...}`

|Secret Manager ({rbtoe_iam_grant_secret})
|GET (query param)
|POST
|`{"policy":...}`

|{gcs_service} ({rbtoe_iam_grant_bucket})
|GET (query param)
|PUT
|raw policy (no envelope)
|===

The GET-vs-POST split reflects an API generation boundary.
{gcs_service} predates Google's gRPC transcoding convention (AIP-127/AIP-136)
and uses classical REST semantics (GET to read, PUT to replace).
{gar_registry} and Secret Manager define their own RPC methods in their
service protos with GET HTTP annotations, overriding the default IAM mixin POST.
CRM Projects and IAM {giam_service_account_s} use the standard IAM mixin POST
convention.

Note on gRPC transcoding tolerance: the {gar_registry} proto specifies GET
for `getIamPolicy` with no `additional_bindings` for POST.  AIP-127 documents
no automatic POST-to-GET fallback — if the proto annotation says `get:`, only
GET is accepted via the transcoded REST endpoint.  The previous implementation
sent POST to this GET-only endpoint, which worked intermittently via undocumented
transcoding tolerance but was observed failing with HTTP 404 on 2026-03-12
during director creation (the 404 was the transcoding layer rejecting the POST,
not a missing repository).  The inline director AR grant path also passed an
empty body `{}` that omitted `requestedPolicyVersion`, creating a latent
condition-mangling hazard.  Both issues are now corrected: the shared
{rbtoe_iam_grant_repo} function already used GET; the inline director path
has been fixed to match.

===== Etag Behavior

[cols="2,2,2,2",options="header"]
|===
|API
|Etag in GET response
|Mismatch error
|No-etag behavior

|CRM Projects ({rbtoe_iam_grant_project})
|Always present
|409 ABORTED
|Unconditional overwrite

|IAM {giam_service_account_s} ({rbtoe_iam_grant_sa})
|Always present
|409 ABORTED
|Unconditional overwrite

|{gar_registry} ({rbtoe_iam_grant_repo})
|Always present
|409 ABORTED
|Unconditional overwrite

|Secret Manager
|Always present
|409 ABORTED
|Unconditional overwrite

|{gcs_service} ({rbtoe_iam_grant_bucket})
|Always present
|412 Precondition Failed
|Unconditional overwrite
|===

**Universal finding: every API always returns an etag.**
The `etag` field is a required structural field of the `google.iam.v1.Policy`
proto message (`bytes etag = 3`).  There is no documented scenario where a
successful `getIamPolicy` response omits the etag.

**Unconditional overwrite is dangerous.**
All five APIs accept writes without etag and silently succeed.  If the existing
policy is version 3 (IAM Conditions) and the caller sends a version 1 policy
without etag, all conditions are silently destroyed.  The proto documentation
explicitly warns: "If you omit the etag field, then IAM lets you overwrite a
version 3 allow policy with a version 1 allow policy, and all of the conditions
in the version 3 policy are lost."

**{gcs_service} is the only 412 API.**
All standard IAM mixin APIs (CRM, IAM, {gar_registry}, Secret Manager) return
409 ABORTED on etag mismatch.  {gcs_service} returns 412 Precondition Failed.
This is a genuine API family difference, not arbitrary — {gcs_service} uses
metageneration-based preconditions and `If-Match` header semantics from its
legacy REST design.

**The single-writer invariant is necessary but not sufficient.**
The operational invariant at the top of `rbgi_iam.sh` ("Single admin actor.
There are no concurrent writers in the same project.") protects against
concurrent _Recipe Bottle_ operations.  However, Google's own internal
processes — API enablement auto-grants, service agent auto-provisioning,
organization policy inheritance changes — can modify project IAM concurrently
and without notice.  This is why every grant pattern's etag handling is
load-bearing even under single-writer: relying on the single-writer assumption
alone would leave the read-modify-write window unguarded against these
Google-internal modifications.

**Intentional etag extraction in current code.**
Every grant pattern now extracts the etag from the `getIamPolicy` response and
asserts it is non-empty (`buc_die "Empty etag"` and per-resource variants),
then threads it through the compose function
(`rbgi_jq_add_member_to_role_capture`), whose
`if $etag != "" then .etag = $etag else . end` filter writes the extracted etag
into the composed policy JSON.  The earlier behavior — passing empty string and
relying on the jq filter leaving the original response etag as accidental
passthrough — has been replaced by deliberate, asserted optimistic concurrency.
The write is now intentionally concurrency-safe rather than accidentally so.

===== Policy Version and requestedPolicyVersion

The `requestedPolicyVersion` parameter in `getIamPolicy` controls which policy
schema the caller declares it understands:

[cols="2,2,3",options="header"]
|===
|requestedPolicyVersion
|Policy has conditions?
|Result

|Omitted or 1
|No
|Returns version 1 policy

|Omitted or 1
|Yes
|Returns version 1; conditions appear as mangled role names with `_withcond_` suffix + hash (e.g., `roles/iam.securityReviewer_withcond_58e135cabb940ad9346c`). Actual condition objects are stripped.

|3
|No
|Returns version 1 policy (server won't inflate version unnecessarily)

|3
|Yes
|Returns version 3 policy with full condition objects
|===

All five grant patterns now request version 3.  {rbtoe_iam_grant_project}
and {rbtoe_iam_grant_sa} send `{"options":{"requestedPolicyVersion":3}}` in
the POST body; {rbtoe_iam_grant_repo}, {rbtoe_iam_grant_secret}, and
{rbtoe_iam_grant_bucket} append a `requestedPolicyVersion=3` query parameter
to their GET.  This protects against silent condition mangling: were any of
these resources to acquire conditioned IAM policies (via future work or
organization-level policy inheritance), a version-1 read would mangle the
conditions into `_withcond_` role suffixes and a subsequent `setIamPolicy`
would destroy them.  Requesting version 3 everywhere closes that hazard.

===== updateMask Default Behavior

The `setIamPolicy` request includes an optional `updateMask` field (FieldMask).
When not provided, it defaults to `bindings, etag`.  This means the `version`
and `auditConfigs` fields in the submitted policy JSON are _ignored_ by the
server unless the mask explicitly includes them.  The compose function should
not need to manage `auditConfigs` (Recipe Bottle does not configure audit
logging via IAM policy), but this default explains why submitting a policy with
version=1 does not downgrade a version=3 policy when the mask is defaulted —
the version field is simply ignored.

===== Empty-Policy / No-Policy Behavior

[cols="2,3,1",options="header"]
|===
|API
|getIamPolicy on new resource
|404 possible?

|CRM Projects ({rbtoe_iam_grant_project})
|200 + policy with bindings (project born with Owner grant)
|No (403 if project does not exist — deliberate enumeration prevention)

|IAM {giam_service_account_s} ({rbtoe_iam_grant_sa})
|200 + empty policy with etag
|No

|{gar_registry} ({rbtoe_iam_grant_repo})
|200 + empty policy with etag
|No

|Secret Manager
|200 + empty policy with etag
|No (404 = secret itself does not exist)

|{gcs_service} ({rbtoe_iam_grant_bucket})
|200 + policy with inherited bindings
|No
|===

**No API returns 404 for "no IAM policy."**
All resources that accept allow policies are created with default allow policies.
`getIamPolicy` returns 200 with an empty-ish policy (no bindings, but etag and
version present) for resources that have never had explicit IAM configuration.

===== Propagation Error Catalog

Three propagation classes feed into `zrbgi_propagation_error_predicate`'s
per-site tolerance list (see <<rbscip_trade_study>> for the shared backoff
profile):

[cols="1,1,2,2,2",options="header"]
|===
|Class
|HTTP code
|Body glob
|Scope applicability
|Discriminator

|A — forward member-visibility
|400
|`*does not exist*`
|All grant sites (project + resource scope)
|Body match; message content stable across ecosystem

|B — backward member-visibility
|400
|`*is not deleted*`
|All grant sites (project + resource scope)
|Body match; message content stable across ecosystem

|C — caller-recently-empowered
|403
|`""` (empty — match any body)
|Resource-scope sites only ({rbtoe_iam_grant_repo}, {rbtoe_iam_grant_sa}, {rbtoe_iam_grant_bucket}, {rbtoe_iam_grant_secret}, plus the inline GAR retry in {rbtgo_director_enrobe})
|Time-bound only; no body fingerprinting
|===

The HTTP 400 errors containing `"does not exist"` and `"is not deleted"` that
the grant functions retry on are **empirically stable but not contractually
guaranteed**.  Google does not document error message text as stable API
surface.  The HTTP status code (400) is the stable signal; the message
content is diagnostic, not machine-parseable contract.

The two 400 patterns represent different propagation directions:

* `"does not exist"` — forward propagation: a newly-created {giam_service_account}
  is not yet visible to the IAM policy service as a valid member.  The SA
  resource has been created (GET returns 200) but the policy service has not yet
  learned about it.

* `"is not deleted"` — backward propagation: a recently-deleted
  {giam_service_account} is still referenced in existing policy bindings.
  The SA resource has been deleted but the policy service still considers it
  a valid (or at least referenced) member.

Multiple independent ecosystem sources (Terraform provider, Red Hat OpenShift
installer, Pulumi) have depended on these strings for years without breakage.
The Terraform Google provider explicitly treats HTTP 400 as retryable in IAM
contexts (issue #8280), which constitutes strong ecosystem evidence that this
pattern is reliable enough for major infrastructure tools.
The practical risk of string instability is low but nonzero.

The HTTP 403 class C pattern represents **caller-empowerment propagation**
into resource-scope IAM caches: the caller (governor) has been freshly
granted `roles/owner` on the depot project, but the artifact-registry
resource-scope IAM cache has not yet learned about the grant, even though
project-scope and SA-scope caches already have (the asymmetry observed
2026-05-13 against `artifactregistry.repositories.getIamPolicy` on a
fresh-depot AR repo, with `roles/owner` granted ~30 seconds prior).
Cumulative additional propagation observed: ~30 seconds.

Class C uses **time-bound discrimination** rather than body matching: real
propagation succeeds within the `RBGC_PROPAGATION_DEADLINE_SEC` budget;
real denial waits the budget and fails cleanly.  This avoids the trap of
treating "denied because forbidden" as "denied because lagging" — the
distinction is observational, not parseable from the 403 response.  The
same time-bound discriminator is already proven in
{rbtgo_access_jwt_probe}'s `invalid_grant` retry harness.

Class C tolerance is **scope-discriminated**: project-scope grant sites
({rbtoe_iam_grant_project}) do not declare 403 tolerance because the
governor's `roles/owner` reaches the project-scope cache effectively
instantly (no observed lag).  Adding spurious 403 tolerance at project
scope would slow down real-denial diagnosis without absorbing any known
race.

**SA email vs numeric ID propagation.**
Google's documentation recommends using a service account's numeric ID rather
than its email address to reference it immediately after creation: "To identify
a service account just after it is created, use its numeric ID rather than its
email address."  Recipe Bottle uses email addresses throughout, which is why
propagation retry is necessary.  Switching to numeric ID could reduce (but
likely not eliminate) propagation delays.

**SA keys.create write-path propagation.**
Observed 2026-03-12 during fresh depot onboarding: `serviceAccounts.keys.create`
POST returned HTTP 404 "Service account ... does not exist" immediately after
both `serviceAccounts.get` (GET by email) and `serviceAccounts.keys.list` (GET)
succeeded on the same SA email path.  This demonstrates a third propagation
class: SA read operations (GET by email, keys.list) propagate independently
from SA write operations (keys.create) on the same endpoint family.
The 404 response confirms the POST was rejected without side effects — no key
was created — making retry safe.  `gcloud iam service-accounts keys list`
confirmed zero USER_MANAGED keys after the 404 failure.
The race is intermittent: a subsequent director creation (dir1011) with the same
code path succeeded on first attempt.
Mitigation: retry keys.create POST on 404 up to `RBGC_SA_KEY_CREATE_RETRY_MAX`
times with `RBGC_SA_KEY_CREATE_RETRY_DELAY_SEC` delay between attempts.

===== Member Removal (Revoke) Contract

The teardown path ({rbtgo_retriever_defrock}, {rbtgo_director_defrock}) removes a
service-account member from each binding it was granted — the inverse of the
grant trio above.  The contract:

* **Same envelope, inverse transform.**  Member removal is the same
  `getIamPolicy` (v3) -> modify -> `setIamPolicy` read-modify-write as the grant
  patterns, against the identical per-scope endpoints (CRM project, AR repo, IAM
  {giam_service_account}).  The only change is the jq transform: subtract the
  member from the named role's `members` array instead of adding it.  A binding
  left with an empty `members` array is pruned — `setIamPolicy` rejects a binding
  with no members, and only the named role can empty out (every other binding
  arrives non-empty from the read and is untouched).

* **Idempotent.**  Subtracting a member that is not present is a no-op, so the
  write re-asserts the same policy and returns HTTP 200.  A re-defrock, or a
  defrock of an identity already partially torn down, revokes cleanly.

* **Class C only — and why.**  The member-visibility classes (A: forward "does
  not exist", B: backward "is not deleted") cannot fire on a revoke — the member
  being removed is long-established, not freshly created or deleted — so the
  revokes drop them.  Class C (caller-recently-empowered, 403) is different: it is
  about the *governor*, not the member.  A governor freshly re-enrobed before a
  teardown has not yet propagated its `roles/owner` to the resource-scope IAM
  cache, so the **resource-scope** revokes (repo and Mason SA) retry the
  `getIamPolicy` on 403 against `RBGC_PROPAGATION_DEADLINE_SEC`, exactly as the
  grant resource-scope sites do.  **Project-scope** revoke carries none — project
  caches absorb governor empowerment instantly (the grant project sites declare no
  403 tolerance either).  `setIamPolicy` carries no Class C retry: it is gated by
  the same `roles/owner` permission as the read, so once `getIamPolicy` clears the
  write will not 403.

* **Etag conflict is fatal (single-writer invariant).**  As with the grant
  sites, RBGI is single-writer; an HTTP 409 on `setIamPolicy` (etag mismatch) is
  a real concurrency violation and aborts the operation rather than retrying.
  Transient `429/5xx` are retried within `RBGC_MAX_CONSISTENCY_SEC`.

* **Removal precedes deletion, by design.**  Teardown revokes the live member
  *before* deleting the SA so GCP never retags the binding as a
  `deleted:serviceAccount:...?uid=` ghost.  Deleting the SA first is precisely
  what accrues the unbounded tombstones the binding leak describes; see
  `Memos/memo-20260604-credential-churn-leak-and-propagation-races.md`.

===== Secret Manager Regional Endpoints

Secret Manager supports both global and regional secret endpoints:

* Global: `projects/*/secrets/*`
* Regional: `projects/*/locations/*/secrets/*`

The service proto defines `additional_bindings` for regional secrets at both
`getIamPolicy` and `setIamPolicy`.  The current {rbtoe_iam_grant_secret} grants use global endpoints.  If Recipe Bottle ever uses regional
secrets, the IAM grant patterns would need to construct regional resource paths.
Noting for completeness.

===== Code Convergence onto the API Contract

The implementation has converged onto the verified API contracts catalogued
above.  This section records, per pattern, how the current code in
`rbgi_iam.sh` realizes the contract — the gaps an earlier draft of this memo
flagged (wrong etag-mismatch code, vanilla-init on impossible responses,
unconditional etag-skipping overwrites, missing `requestedPolicyVersion`) are
now closed.

[cols="1,2,3",options="header"]
|===
|Pattern
|Contract point
|How the code realizes it

|{rbtoe_iam_grant_project}
|409 ABORTED on etag mismatch
|`setIamPolicy` is POST; the SET loop dies fatally on HTTP 409
 (`HTTP 409 ABORTED (etag mismatch — concurrent policy change)`).
 CRM uses the standard IAM mixin 409, not 412 — the code matches.

|{rbtoe_iam_grant_repo}
|GET getIamPolicy, no 404 vanilla-init
|`rbgi_add_repo_iam_role` reads via GET with the
 `options.requestedPolicyVersion=3` query parameter and has no 404
 vanilla-init branch.  A non-200 GET hits `rbuh_require_ok` and dies:
 404 means the repository does not exist (a real error), not an absent
 policy.  The inline director AR grant path in `rbgg_governor.sh` was
 the historical POST-on-GET-only site (observed failing with HTTP 404
 on 2026-03-12 as the transcoding layer rejected POST); it now uses GET.

|{rbtoe_iam_grant_repo}
|Non-empty etag, threaded to SET
|The GET response etag is extracted and asserted non-empty
 (`buc_die "Empty repo etag"`), then passed to the compose function and
 written into the SET body — intentional optimistic concurrency, not the

|Hard SA-existence preflight, no vanilla-init

|The getIamPolicy POST (with `{"options":{"requestedPolicyVersion":3}}`)

|{rbtoe_iam_grant_bucket}
|Non-empty etag; 412 primary, 409 defensive

|requestedPolicyVersion=3 on read
|All four resource-scope reads now request policy version 3 (query
 parameter for the GET sites, request body for the SA POST), matching
 {rbtoe_iam_grant_project}.  The earlier latent condition-mangling hazard
 on these resource types is closed.

===== Convergence Analysis

**Converged (zero new state space — now realized in code):**

* **Etag extracted and passed intentionally**: Every API always returns etag.
  Every grant pattern now extracts it explicitly, asserts it non-empty, and
  passes it to `setIamPolicy` through the compose function — intentionally
  concurrency-safe writes rather than accidental passthrough or unconditional
  overwrite.  This also protects against Google-internal concurrent policy
  modifications that the single-writer invariant does not cover.

* **Transient retry on SET (429/5xx)**: Every grant pattern's SET loop now
  treats 429 (rate limit) and 5xx (server error) as transient and retries
  within the eventual-consistency budget.

* **Uniform etag-mismatch handling**: All standard IAM mixin APIs (CRM, IAM,
  {gar_registry}, Secret Manager) die fatally on 409 ABORTED; {gcs_service}
  dies fatally on 412 Precondition Failed (with a defensive 409 branch).

* **Policy version 3 requested everywhere**: All five patterns request
  version 3 — a query parameter on the GET sites
  ({rbtoe_iam_grant_repo}, {rbtoe_iam_grant_secret}, {rbtoe_iam_grant_bucket})
  and a request-body field on the POST sites
  ({rbtoe_iam_grant_project}, {rbtoe_iam_grant_sa}) — closing the
  silent-condition-destruction hazard on the resource-scope types.

**Load-bearing differences (do not homogenize):**

* **GET vs POST for getIamPolicy**: API-mandated per service proto.
  Cannot be unified.

* **PUT vs POST for setIamPolicy**: {gcs_service} requires PUT.
  Cannot be unified.

* **Body wrapping** (`{"policy":...}` vs raw): {gcs_service} requires raw.
  Cannot be unified.

* **Project non-200 GET is a real error**: Projects always have IAM policies.
  Do not vanilla-init on non-200 for projects.

**Require investigation before converging** (tracked in <<rbshr_horizon_roadmap>>)**:**

* **SA preflight universalization** — {rbtoe_iam_grant_sa} does a hard
  SA-existence GET preflight; whether the other resource-scope patterns warrant
  an analogous preflight is open.
* **SA numeric ID for member references**

**Closed by design (not a convergence target):**

* **Read-back verification** — the post-set read-back verify loop was removed
  from {rbtoe_iam_grant_project} as non-load-bearing: the `setIamPolicy` 200 is
  the authoritative grant confirmation, and the email→uid read reconciliation
  could lag past any bounded poll and produce a false timeout on a grant that
  already succeeded.  Universalizing read-back is therefore explicitly not a
  direction; see the rationale comment in `rbgi_add_project_iam_role`.
