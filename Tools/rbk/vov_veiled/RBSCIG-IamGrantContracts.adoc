[[rbscig_iam_grant_contracts]]
==== IAM Grant API Contracts

Research memo documenting the actual behavioral contracts of the Google Cloud
IAM policy APIs used by the four grant patterns:
{rbtoe_iam_grant_project}, {rbtoe_iam_grant_repo},
{rbtoe_iam_grant_sa}, {rbtoe_iam_grant_bucket}.
Also covers Secret Manager ({rbtoe_iam_grant_secret}).

This document provides verified evidence for spec definition corrections and
future convergence decisions.  It should be re-reviewed when any grant function
behavior is modified, when new resource types require IAM grants, or when
Google API behavior changes are suspected.

Findings are verified against API reference documentation, upstream proto
definitions (`google.iam.v1.Policy`, per-service `service.proto` files), and
ecosystem evidence (Terraform provider issues, Red Hat bugzilla, Pulumi issues).

Research conducted: 2026-03-11.

===== HTTP Method and Body Format

[cols="2,2,2,2",options="header"]
|===
|API
|getIamPolicy
|setIamPolicy
|SET body format

|CRM Projects ({rbtoe_iam_grant_project})
|POST (body: `{"options":{"requestedPolicyVersion":3}}`)
|POST
|`{"policy":...}`

|IAM {giam_service_account_s} ({rbtoe_iam_grant_sa})
|POST (body: `{}`)
|POST
|`{"policy":...}`

|{gar_registry} ({rbtoe_iam_grant_repo})
|GET (query param)
|POST
|`{"policy":...}`

|Secret Manager ({rbtoe_iam_grant_secret})
|GET (query param)
|POST
|`{"policy":...}`

|{gcs_service} ({rbtoe_iam_grant_bucket})
|GET (query param)
|PUT
|raw policy (no envelope)
|===

The GET-vs-POST split reflects an API generation boundary.
{gcs_service} predates Google's gRPC transcoding convention (AIP-127/AIP-136)
and uses classical REST semantics (GET to read, PUT to replace).
{gar_registry} and Secret Manager define their own RPC methods in their
service protos with GET HTTP annotations, overriding the default IAM mixin POST.
CRM Projects and IAM {giam_service_account_s} use the standard IAM mixin POST
convention.

Note on gRPC transcoding tolerance: the {gar_registry} proto specifies GET
for `getIamPolicy` with no `additional_bindings` for POST.  AIP-127 documents
no automatic POST-to-GET fallback — if the proto annotation says `get:`, only
GET is accepted via the transcoded REST endpoint.  The previous implementation
sent POST to this GET-only endpoint, which worked intermittently via undocumented
transcoding tolerance but was observed failing with HTTP 404 on 2026-03-12
during director creation (the 404 was the transcoding layer rejecting the POST,
not a missing repository).  The inline director AR grant path also passed an
empty body `{}` that omitted `requestedPolicyVersion`, creating a latent
condition-mangling hazard.  Both issues are now corrected: the shared
{rbtoe_iam_grant_repo} function already used GET; the inline director path
has been fixed to match.

===== Etag Behavior

[cols="2,2,2,2",options="header"]
|===
|API
|Etag in GET response
|Mismatch error
|No-etag behavior

|CRM Projects ({rbtoe_iam_grant_project})
|Always present
|409 ABORTED
|Unconditional overwrite

|IAM {giam_service_account_s} ({rbtoe_iam_grant_sa})
|Always present
|409 ABORTED
|Unconditional overwrite

|{gar_registry} ({rbtoe_iam_grant_repo})
|Always present
|409 ABORTED
|Unconditional overwrite

|Secret Manager
|Always present
|409 ABORTED
|Unconditional overwrite

|{gcs_service} ({rbtoe_iam_grant_bucket})
|Always present
|412 Precondition Failed
|Unconditional overwrite
|===

**Universal finding: every API always returns an etag.**
The `etag` field is a required structural field of the `google.iam.v1.Policy`
proto message (`bytes etag = 3`).  There is no documented scenario where a
successful `getIamPolicy` response omits the etag.

**Unconditional overwrite is dangerous.**
All five APIs accept writes without etag and silently succeed.  If the existing
policy is version 3 (IAM Conditions) and the caller sends a version 1 policy
without etag, all conditions are silently destroyed.  The proto documentation
explicitly warns: "If you omit the etag field, then IAM lets you overwrite a
version 3 allow policy with a version 1 allow policy, and all of the conditions
in the version 3 policy are lost."

**{gcs_service} is the only 412 API.**
All standard IAM mixin APIs (CRM, IAM, {gar_registry}, Secret Manager) return
409 ABORTED on etag mismatch.  {gcs_service} returns 412 Precondition Failed.
This is a genuine API family difference, not arbitrary — {gcs_service} uses
metageneration-based preconditions and `If-Match` header semantics from its
legacy REST design.

**The single-writer invariant is necessary but not sufficient.**
The operational invariant at the top of `rbgi_IAM.sh` ("Single admin actor.
There are no concurrent writers in the same project.") protects against
concurrent _Recipe Bottle_ operations.  However, Google's own internal
processes — API enablement auto-grants, service agent auto-provisioning,
organization policy inheritance changes — can modify project IAM concurrently
and without notice.  This is why {rbtoe_iam_grant_project}'s etag handling is
load-bearing even under single-writer, and why {rbtoe_iam_grant_repo} and
{rbtoe_iam_grant_sa} should also use etag intentionally rather than relying on
the single-writer assumption alone.

**Accidental etag passthrough in current code.**
The compose function (`rbgu_jq_add_member_to_role_capture`) applies
`if $etag != "" then .etag = $etag else . end` — when the caller passes empty
string for etag, the jq filter does nothing, leaving the original `.etag` field
from the `getIamPolicy` response in the composed JSON.  This means
{rbtoe_iam_grant_repo} and {rbtoe_iam_grant_sa} may already be _accidentally_
sending etags via passthrough to `setIamPolicy`.  Whether those APIs then
enforce the passthrough etag is unverified.  The current behavior is neither
intentionally safe (explicit etag extraction and assertion) nor intentionally
unsafe (deliberate unconditional overwrite) — it is accidental, and should be
made intentional in either direction.

===== Policy Version and requestedPolicyVersion

The `requestedPolicyVersion` parameter in `getIamPolicy` controls which policy
schema the caller declares it understands:

[cols="2,2,3",options="header"]
|===
|requestedPolicyVersion
|Policy has conditions?
|Result

|Omitted or 1
|No
|Returns version 1 policy

|Omitted or 1
|Yes
|Returns version 1; conditions appear as mangled role names with `_withcond_` suffix + hash (e.g., `roles/iam.securityReviewer_withcond_58e135cabb940ad9346c`). Actual condition objects are stripped.

|3
|No
|Returns version 1 policy (server won't inflate version unnecessarily)

|3
|Yes
|Returns version 3 policy with full condition objects
|===

Currently only {rbtoe_iam_grant_project} requests version 3.
{rbtoe_iam_grant_repo}, {rbtoe_iam_grant_sa}, and {rbtoe_iam_grant_bucket}
do not request a specific version.  If any of those resources ever acquire
conditioned IAM policies, the `getIamPolicy` response would silently mangle
the conditions, and a subsequent `setIamPolicy` would destroy them.  This is
a latent hazard: Recipe Bottle does not currently use IAM Conditions on these
resource types, but future work or organization-level policy inheritance could
introduce them without Recipe Bottle's knowledge.

===== updateMask Default Behavior

The `setIamPolicy` request includes an optional `updateMask` field (FieldMask).
When not provided, it defaults to `bindings, etag`.  This means the `version`
and `auditConfigs` fields in the submitted policy JSON are _ignored_ by the
server unless the mask explicitly includes them.  The compose function should
not need to manage `auditConfigs` (Recipe Bottle does not configure audit
logging via IAM policy), but this default explains why submitting a policy with
version=1 does not downgrade a version=3 policy when the mask is defaulted —
the version field is simply ignored.

===== Empty-Policy / No-Policy Behavior

[cols="2,3,1",options="header"]
|===
|API
|getIamPolicy on new resource
|404 possible?

|CRM Projects ({rbtoe_iam_grant_project})
|200 + policy with bindings (project born with Owner grant)
|No (403 if project does not exist — deliberate enumeration prevention)

|IAM {giam_service_account_s} ({rbtoe_iam_grant_sa})
|200 + empty policy with etag
|No

|{gar_registry} ({rbtoe_iam_grant_repo})
|200 + empty policy with etag
|No

|Secret Manager
|200 + empty policy with etag
|No (404 = secret itself does not exist)

|{gcs_service} ({rbtoe_iam_grant_bucket})
|200 + policy with inherited bindings
|No
|===

**No API returns 404 for "no IAM policy."**
All resources that accept allow policies are created with default allow policies.
`getIamPolicy` returns 200 with an empty-ish policy (no bindings, but etag and
version present) for resources that have never had explicit IAM configuration.

===== Propagation Error Strings

The HTTP 400 errors containing `"does not exist"` and `"is not deleted"` that
the grant functions retry on (per <<rbscip_trade_study>> backoff profile) are
**empirically stable but not contractually guaranteed**.
Google does not document error message text as stable API surface.
The HTTP status code (400) is the stable signal; the message content is
diagnostic, not machine-parseable contract.

The two patterns represent different propagation directions:

* `"does not exist"` — forward propagation: a newly-created {giam_service_account}
  is not yet visible to the IAM policy service as a valid member.  The SA
  resource has been created (GET returns 200) but the policy service has not yet
  learned about it.

* `"is not deleted"` — backward propagation: a recently-deleted
  {giam_service_account} is still referenced in existing policy bindings.
  The SA resource has been deleted but the policy service still considers it
  a valid (or at least referenced) member.

Multiple independent ecosystem sources (Terraform provider, Red Hat OpenShift
installer, Pulumi) have depended on these strings for years without breakage.
The Terraform Google provider explicitly treats HTTP 400 as retryable in IAM
contexts (issue #8280), which constitutes strong ecosystem evidence that this
pattern is reliable enough for major infrastructure tools.
The practical risk of string instability is low but nonzero.

**SA email vs numeric ID propagation.**
Google's documentation recommends using a service account's numeric ID rather
than its email address to reference it immediately after creation: "To identify
a service account just after it is created, use its numeric ID rather than its
email address."  Recipe Bottle uses email addresses throughout, which is why
propagation retry is necessary.  Switching to numeric ID could reduce (but
likely not eliminate) propagation delays.

**SA keys.create write-path propagation.**
Observed 2026-03-12 during fresh depot onboarding: `serviceAccounts.keys.create`
POST returned HTTP 404 "Service account ... does not exist" immediately after
both `serviceAccounts.get` (GET by email) and `serviceAccounts.keys.list` (GET)
succeeded on the same SA email path.  This demonstrates a third propagation
class: SA read operations (GET by email, keys.list) propagate independently
from SA write operations (keys.create) on the same endpoint family.
The 404 response confirms the POST was rejected without side effects — no key
was created — making retry safe.  `gcloud iam service-accounts keys list`
confirmed zero USER_MANAGED keys after the 404 failure.
The race is intermittent: a subsequent director creation (dir1011) with the same
code path succeeded on first attempt.
Mitigation: retry keys.create POST on 404 up to `RBGC_SA_KEY_CREATE_RETRY_MAX`
times with `RBGC_SA_KEY_CREATE_RETRY_DELAY_SEC` delay between attempts.

===== Secret Manager Regional Endpoints

Secret Manager supports both global and regional secret endpoints:

* Global: `projects/*/secrets/*`
* Regional: `projects/*/locations/*/secrets/*`

The service proto defines `additional_bindings` for regional secrets at both
`getIamPolicy` and `setIamPolicy`.  The current {rbtoe_iam_grant_secret} grants use global endpoints.  If Recipe Bottle ever uses regional
secrets, the IAM grant patterns would need to construct regional resource paths.
Noting for completeness.

===== Current Code vs Actual API Contract

This section catalogs discrepancies between the current implementation and the
verified API contracts.  These are not necessarily bugs that cause failures —
several work by accident or are protected by the single-writer operational
invariant — but they represent incorrect assumptions in the code.

[cols="1,2,3",options="header"]
|===
|Pattern
|Discrepancy
|Detail

|{rbtoe_iam_grant_project}
|412 handler targets wrong error code
|Code handles 412 as "etag mismatch" (line 199 of rbgi_IAM.sh).
 CRM actually returns 409 ABORTED.
 The 412 case never fires; etag mismatches hit the 409 path
 (`buc_die "HTTP 409 Conflict (fatal by invariant)"`), which
 produces the correct outcome (fatal) but with the wrong diagnostic.

|{rbtoe_iam_grant_repo}
|404 handler for impossible response
|Code initializes vanilla policy on HTTP 404 (line 304).
 {gar_registry} `getIamPolicy` returns 200 with empty policy
 for existing repos, never 404.
 A 404 would mean the repository itself does not exist — vanilla init
 would mask this real error.

|{rbtoe_iam_grant_repo}
|Resolved — uses GET per proto
|The shared `rbgi_add_repo_iam_role` function already used GET.
 The inline director AR grant path in `rbgg_Governor.sh` was the
 remaining POST site — observed failing with HTTP 404 on 2026-03-12
 (transcoding layer rejecting POST on GET-only endpoint).  Fixed to
 GET with `requestedPolicyVersion=3` query parameter.  The 404
 vanilla-init handler was also removed (404 on GET means repo does
 not exist — a real error, not a policy absence).

|{rbtoe_iam_grant_repo}
|Skips available etag
|Passes empty string for etag (line 315), performing unconditional
 overwrite despite the API always returning etag.
 Safe under single-writer invariant but discards protection the API
 provides for free.  See "Accidental etag passthrough" above — the
 actual behavior may be passthrough rather than omission, but this
 is accidental, not intentional.

|{rbtoe_iam_grant_sa}
|Vanilla init for impossible response
|Code initializes vanilla policy on non-200 (line 411).
 IAM `getIamPolicy` returns 200 with empty policy for existing SAs.
 Non-200 means the SA does not exist or access denied — vanilla init
 masks a real error.

|{rbtoe_iam_grant_sa}
|Skips available etag
|Passes empty string for etag (line 419), performing unconditional
 overwrite.  Same single-writer safety, same accidental passthrough
 possibility, same missed intentional protection.

|{rbtoe_iam_grant_bucket}
|Etag described as conditional
|Code extracts etag with fallback to empty (line 507).
 {gcs_service} `getIamPolicy` always returns etag — the fallback
 should never trigger.  Minor: the RBS0 spec previously said "when
 present" but it is always present.

|Secret Manager ({rbtoe_iam_grant_secret})
|Resolved — now has propagation retry
|The three secret IAM grants (formerly inline in {rbtgo_depot_levy})

 retry per <<rbscip_trade_study>> backoff profile, explicit etag,
 transient retry, and `requestedPolicyVersion=3`.

|{rbtoe_iam_grant_repo}, {rbtoe_iam_grant_sa}, {rbtoe_iam_grant_bucket}
|No requestedPolicyVersion in getIamPolicy
|Only {rbtoe_iam_grant_project} requests policy version 3.
 The other three do not, creating a latent hazard if conditioned
 policies are ever introduced on those resource types (conditions
 would be silently mangled and destroyed on read-modify-write).
|===

===== Convergence Analysis

Based on these findings, the following convergence opportunities exist:

**Safe to converge (zero new state space):**

* **Always extract and pass etag intentionally**: Every API always returns etag.
  Extracting it explicitly and passing it to `setIamPolicy` converts accidental
  passthrough or unconditional overwrites to intentionally concurrency-safe
  writes.  The compose function already handles etag correctly when provided.
  This also protects against Google-internal concurrent policy modifications
  that the single-writer invariant does not cover.

* **Transient retry on SET (429/5xx)**: Currently only {rbtoe_iam_grant_project}
  retries transient errors.  Adding bounded retry for 429 (rate limit) and 5xx
  (server error) is universally correct and cannot cause new failures.

* **Uniform etag-mismatch handling**: Handle 409 ABORTED for all standard IAM
  mixin APIs, and 412 Precondition Failed for {gcs_service} only.  The current
  project code's 412 handler should become a 409 handler.

* **Request policy version 3 everywhere**: Protects against silent condition
  destruction if conditioned policies are ever introduced on repo, SA, or
  bucket resources.  For APIs that use GET (where version is a query param),
  this is a query parameter addition; for POST APIs, it is a request body
  field.

**Load-bearing differences (do not homogenize):**

* **GET vs POST for getIamPolicy**: API-mandated per service proto.
  Cannot be unified.

* **PUT vs POST for setIamPolicy**: {gcs_service} requires PUT.
  Cannot be unified.

* **Body wrapping** (`{"policy":...}` vs raw): {gcs_service} requires raw.
  Cannot be unified.

* **Project non-200 GET is a real error**: Projects always have IAM policies.
  Do not vanilla-init on non-200 for projects.

**Require investigation before converging** (tracked in <<rbshr_horizon_roadmap>>)**:**

* **Read-back verification universalization**
* **SA preflight universalization**
* **SA numeric ID for member references**
