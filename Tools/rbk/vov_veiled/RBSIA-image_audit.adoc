//axhob_operation
{rbtgo_image_audit}

NOTE: This operation produces a flat catalog of hallmark identifiers in
{rbtgi_repository}
via the GAR REST API implemented by
{gar_service}.
Identifiers only — no per-member presence detail (use {rbtgo_image_rekon} for that), and no health classification (use {rbtgo_hallmark_tally} for that).
Captured upstream artifacts are not audited here — base images and reliquary cohorts are now {rbtga_lode}s (bole and conclave kinds), enumerated by {rbtgo_lode_divine}.

The
{rbtr_director}
audits one domain:

* `rbfl_audit_hallmarks` — every hallmark identifier under the `{gar_hallmarks_namespace}/` root

Audit vs. {rbtgo_hallmark_tally}: audit is a {rbtr_director}-tier flat-list operation; tally is a {rbtr_retriever}-tier operation that classifies each entry by basename completeness and vouch status.
The two roles compose without overlap.

== Hallmarks Audit

The
{rbtr_director}
audits hallmark identifiers:

//axhos_step
. *Authenticate as
{rbtr_director}*:
.. Execute
{rbtoe_director_authenticate}
.. {rbbc_store}
`«DIRECTOR_TOKEN»` for subsequent API calls

//axhos_step
. *Enumerate Packages Under Hallmarks Root*:
.. {rbbc_store}
`«HALLMARKS_ROOT»` from `RBGL_HALLMARKS_ROOT` (kindled from `RBGC_GAR_CATEGORY_HALLMARKS`, the token `rbi_hm`)
.. {rbbc_store}
`«REPO_BASE»` as the GAR REST API repository base (`projects/«PROJECT»/locations/«REGION»/repositories/«REPOSITORY»`)
.. {rbbc_call}
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories.packages/list[GAR REST API: packages.list] with:
... method: `GET`
... endpoint: `«REPO_BASE»/packages?pageSize=1000`
... header: `Authorization: Bearer «DIRECTOR_TOKEN»`
.. {rbbc_require}
HTTP 200 response
.. Extract `.packages[].name`, strip the resource prefix, URL-decode `%2F` back to `/`, and retain only entries beginning with `«HALLMARKS_ROOT»/`
.. {rbbc_store}
`«HALLMARK_ROWS»` as `«HALLMARK» «BASENAME»` pairs derived from those entries (a 2-deep enumeration shared with {rbtgo_image_rekon})

//axhos_step
. *Project to Unique Identifiers*:
.. {rbbc_store}
`«HALLMARKS»` as the ordered set of distinct hallmark identifiers from the first column of `«HALLMARK_ROWS»`

//axhoot_typed_output
{rbtga_ark}
. *Display Results*:
.. If `«HALLMARKS»` is empty:
... {rbbc_show}
"No hallmarks found under «HALLMARKS_ROOT»/"
... return success with count `0`
.. Otherwise:
... {rbbc_show}
table header `HALLMARK`
... For each `«HALLMARK»` in `«HALLMARKS»`:
.... {rbbc_show}
`«HALLMARK»`
... {rbbc_show}
"Total hallmarks: «COUNT»"

//axhoc_completion
The procedure completes successfully when the ordered list of identifiers — possibly empty — has been displayed and the count line emitted.
The empty case is non-fatal: an empty domain reports zero rather than failing, since "registry holds no entries in this domain" is a valid director observation, not an error.
