{rbtgo_ark_about}

NOTE: This operation assembles the
{rbtga_ark_about}
metadata artifact for an existing
{rbtga_ark_image}
in
{rbtgi_repository}.
The about scripts (`rbgja01`–`rbgja04`) are uniform across all three vessel modes and always execute as Cloud Build steps.
This procedure describes those steps — not a standalone Cloud Build job.
Conjure and bind embed these steps in their primary Cloud Build job alongside image production; graft combines these steps with vouch steps in a single about+vouch Cloud Build job.
All jobs run as the
{rbtr_mason}
service account on the private pool (`RBRR_GCB_POOL_STEM`).

NOTE: The about steps receive their context via substitution variables (`_RBGA_*`) and workspace state, not via function parameters.
For conjure, the hallmark is read from `/workspace` (computed by the preceding build steps); for bind and graft, it comes from the `_RBGA_HALLMARK` substitution variable.
See <<rbtgo_about_substitutions>> below for the full variable reference.

//axhos_step
. *Cloud Build Step 1: Discover Platforms* (Python 3):
.. Image: gcloud (from reliquary)
.. Fetch OAuth token from GCE metadata server
.. {rbbc_call}
Docker Registry API: GET manifests for `«PREFIX»hallmarks/«HALLMARK»/image:«HALLMARK»` with OCI media type accept headers
.. If response is an OCI image index or Docker manifest list:
... Filter out attestation manifests before extracting platforms.
OCI image indexes may contain non-runnable attestation manifests (SLSA provenance, SBOMs) alongside real platform manifests.
Per the BuildKit attestation storage specification, these attestation manifests are deliberately assigned platform `unknown/unknown` (`"architecture": "unknown", "os": "unknown"`) to prevent container runtimes from pulling them as images.
They carry media types such as `application/vnd.in-toto+json` and are annotated with `"vnd.docker.reference.type": "attestation-manifest"`.
Exclude any manifest entry where _both_ `platform.os` and `platform.architecture` are `unknown`.
+
See: https://docs.docker.com/build/metadata/attestations/attestation-storage/[Docker Attestation Storage], https://github.com/moby/buildkit/blob/master/docs/attestations/attestation-storage.md[BuildKit Attestation Storage spec].
... {rbbc_store}
`«PLATFORMS»` as list of platform descriptors from the remaining (non-attestation) manifest entries
.. If response is a single OCI/Docker manifest (typical for bind and graft):
... {rbbc_call}
GET config blob via digest from manifest's `config.digest` field
... {rbbc_store}
`«PLATFORMS»` as single-element list from config's `os` and `architecture` fields
.. {rbbc_store}
platform count for subsequent step branching
.. After platform discovery, attempt diags artifact extraction (conjure-only):
... {rbbc_call}
Docker Registry API: HEAD manifests for `«PREFIX»hallmarks/«HALLMARK»/diags:«HALLMARK»`
... If HTTP 200: fetch manifest, iterate layer blobs, extract diagnostic files (`buildkit_metadata.json`, `cache_before.json`, `cache_after.json`, `recipe.txt`) to `/workspace`
... If HTTP 404 or error: skip gracefully (expected for bind/graft — no diags artifact exists)
.. NOTE: The diags artifact is pushed by conjure step 06 (`rbgjb06-push-diags.sh`) and contains build-time diagnostics that cannot be conveyed via Cloud Build substitution variables (Dockerfile content exceeding the ~4KB limit, BuildKit metadata, Docker daemon cache snapshots).

//axhos_step
. *Cloud Build Step 2: Syft SBOM Generation*:
.. Image: docker (from reliquary; runs syft via `docker run` with reliquary syft image)
.. Fetch OAuth token via metadata server (syft authenticates as {rbtr_mason})
.. For each platform in `«PLATFORMS»`:
... Conjure with per-platform tags (multi-platform): {rbbc_call}
`syft registry:«GAR_HOST»/«GAR_PATH»/hallmarks/«HALLMARK»/image:«HALLMARK»-«PLATFORM_SUFFIX»`
... Conjure single-platform (single manifest, not an index): {rbbc_call}
`syft registry:«GAR_HOST»/«GAR_PATH»/hallmarks/«HALLMARK»/image:«HALLMARK»`
... Bind/graft (always via per-platform digest): {rbbc_call}
`syft registry:«GAR_HOST»/«GAR_PATH»/hallmarks/«HALLMARK»/image@«PLATFORM_DIGEST»` using per-platform digest from step 1's manifest list
+
NOTE: Bind and graft images always use `@«PLATFORM_DIGEST»` pinning, even for single-platform images.
Grafted images may be stored as OCI image indexes containing attestation manifests (see step 1 filtering); scanning the main tag would cause syft to auto-select the Cloud Build worker's native platform (amd64) from the index, failing when the image contains only a different architecture.
Digest pinning bypasses index platform resolution entirely.
... {rbbc_store}
per-platform SBOM to `/workspace/sbom-«LABEL».json` (e.g., `sbom-amd64.json`; LABEL = architecture + variant without separators)
.. NOTE: Syft's `registry:` transport reads directly from the OCI registry — no docker CLI or local pull required.

//axhos_step
. *Cloud Build Step 3: Assemble build_info.json* (Python 3):
.. Image: gcloud (from reliquary)
.. For each platform in `«PLATFORMS»`:
... Construct `build_info.json` with mode-aware fields organized into nested JSON objects:
.... Root: `hallmark`, `vessel_mode`, `vessel_name`, `platform`, `image_digest`, `about_timestamp`
.... `git` object: `repo`, `branch`, `commit`
.... Conjure `build` object: `build_id` (from `_RBGA_BUILD_ID`), `inscribe_timestamp`, `qemu_used`
.... Conjure `slsa` object: `build_level`, `build_invocation_id`, `provenance_predicate_types`, `provenance_builder_id`
.... Bind `bind` object: `source` (from `_RBGA_BIND_SOURCE`)
.... Graft `graft` object: `source` (from `_RBGA_GRAFT_SOURCE`)
... {rbbc_store}
per-platform `build_info.json` to `/workspace/build_info-«LABEL».json` (e.g., `build_info-amd64.json`)
.. Write `recipe.txt` using dual-source logic:
... If `recipe.txt` already exists in `/workspace` (written by diags extraction in step 1): skip substitution variable write, log that `recipe.txt` came from the diags artifact
... Else if `_RBGA_DOCKERFILE_CONTENT` is non-empty: {rbbc_store}
`recipe.txt` to `/workspace/recipe.txt` from substitution variable content
... Else: omit `recipe.txt`
.. NOTE: The {rbtr_director} reads the Dockerfile locally and passes its content as `_RBGA_DOCKERFILE_CONTENT`.
If the file exceeds the Cloud Build substitution variable size limit (~4KB), the director omits it with a warning.
The diags path (conjure only) bypasses this limit by conveying the full Dockerfile via a registry artifact.
`_RBGA_DOCKERFILE_CONTENT` is retained for all modes as a fallback when diags extraction fails or is absent.

//axhos_step
. *Cloud Build Step 4: Build and Push About Artifact*:
.. Image: docker (from reliquary)
.. Create buildx builder instance (inspect-or-create pattern)
.. Build multi-platform `FROM scratch` container with per-platform SBOM, `build_info.json`, `recipe.txt` (when present), and conjure enrichment files when forwarded from the diags artifact: `buildkit_metadata.json`, `cache_before.json`, `cache_after.json` (conditional COPY — included only if present in `/workspace`)
.. For single-platform images: build single-platform about container
.. For multi-platform images: build with `buildx --push` using `TARGETARCH`/`TARGETVARIANT` to select per-platform files
.. Push to registry tagged `«PREFIX»hallmarks/«HALLMARK»/about:«HALLMARK»` via `docker buildx build --push`
.. {rbbc_require}
push succeeds

//axhoc_completion
Successful execution of these four Cloud Build steps produces a
{rbtga_ark_about}
artifact in the registry tagged as `«PREFIX»hallmarks/«HALLMARK»/about:«HALLMARK»`.
The about artifact is a multi-platform container matching the image artifact's platform list.
+
*Re-about idempotency*: Re-running about re-executes Syft and rebuilds `build_info.json`, overwriting the previous about artifact.
SBOM content may vary if Syft discovers different packages (e.g., after a Syft version update).
+
*Implementation note*: About and vouch share common Build resource JSON construction patterns (worker pool, mason SA, registry coordinates).
The implementation should extract a shared helper for `builds.create` request assembly.

[[rbtgo_about_substitutions]]
==== About Substitution Variables

All `_RBGA_*` substitution variables are passed in the `builds.create` request body.

*Registry and identity*::
[cols="1,2"]
|===
| `_RBGA_GAR_HOST` | GAR Docker host. Source: regime constants.
| `_RBGA_GAR_PATH` | GAR repository path. Source: regime constants.
| `_RBGA_VESSEL` | Vessel sigil. Source: loaded vessel configuration.
| `_RBGA_HALLMARK` | Full hallmark string. Source: input parameter.
| `_RBGA_VESSEL_MODE` | Vessel mode (`conjure`, `bind`, or `graft`). Source: loaded vessel configuration.
| `_RBGA_ARK_BASENAME_IMAGE` | Ark basename for image artifact. Source: `RBGC_ARK_BASENAME_IMAGE` constant.
| `_RBGA_ARK_BASENAME_ABOUT` | Ark basename for about artifact. Source: `RBGC_ARK_BASENAME_ABOUT` constant.
| `_RBGA_ARK_BASENAME_DIAGS` | Ark basename for diags artifact. Source: `RBGC_ARK_BASENAME_DIAGS` constant.
|===

*Git metadata*::
[cols="1,2"]
|===
| `_RBGA_GIT_COMMIT` | Main repo HEAD SHA at operation time.
| `_RBGA_GIT_BRANCH` | Main repo branch at operation time.
| `_RBGA_GIT_REPO` | Main repo identifier (owner/repo).
|===

*Mode-specific*::
[cols="1,2"]
|===
| `_RBGA_BUILD_ID` | Conjure Cloud Build job ID (conjure only). In the combined conjure job, populated from the built-in `$BUILD_ID`. In manual recovery standalone about, passed by the director from a prior conjure run. Used by step 3 to populate `build_info.json`. Empty for bind/graft.
| `_RBGA_INSCRIBE_TIMESTAMP` | Inscribe timestamp from hallmark (conjure only). Empty for bind/graft.
| `_RBGA_BIND_SOURCE` | Pinned upstream image reference (bind only). Source: {rbrv_bind_image}. Empty for conjure/graft.
| `_RBGA_GRAFT_SOURCE` | Local image reference at push time (graft only). Empty for conjure/bind.
| `_RBGA_DOCKERFILE_CONTENT` | Dockerfile content read locally by {rbtr_director} (all modes, optional). Source: {rbrv_conjure_dockerfile} (conjure, always populated), {rbrv_graft_optional_dockerfile} (graft, optional), {rbrv_bind_optional_dockerfile} (bind, optional). Empty if file exceeds ~4KB substitution limit or path not configured. Written to `recipe.txt` in step 3.
|===

NOTE: `«PLATFORMS»` is discovered in step 1 and passed to subsequent steps as a workspace variable (written to `/workspace/platforms.txt`), not as a substitution variable.
