//axhob_operation
{rbtgo_ark_about}

NOTE: This operation assembles the
{rbtga_ark_about}
metadata artifact for an existing
{rbtga_ark_image}
in
{rbtgi_repository}.
The
{rbtr_director}
submits a Cloud Build job via `builds.create` (direct submission), which runs as the
{rbtr_mason}
service account on the private pool (`RBRR_GCB_WORKER_POOL`).

NOTE: The about pipeline is uniform across all three vessel modes.
Once `-image` exists in GAR (however it got there — conjure, bind, or graft), the same about job runs Syft for SBOM and assembles mode-aware `build_info.json`.

NOTE: About is always submitted as a standalone `builds.create` Cloud Build job by the {rbtr_director}.
The about scripts (`rbgja01`–`rbgja04`) are uniform across all vessel modes.
`rbf_create` chains the about submission after the mode-specific image operation completes.
Future: for conjure and bind, the about steps may be embedded in the image-producing Cloud Build job for single-round-trip efficiency; in that case `rbf_create` would skip the standalone about submission.

NOTE: About uses `builds.create` rather than trigger dispatch because the
{rbtga_consecration}
is known only after the image arrives in GAR.

The
{rbtr_director}
assembles about for an ark:

{rbtga_ark}

vessel directory provided as first argument

.. {rbbc_store}
`«VESSEL_MODE»` from {rbrv_vessel_mode}

`POST «GCB_PROJECT_BUILDS_URL»` with Build resource JSON

Cloud Console URL for build monitoring

//axhos_step
. *Cloud Build Step 1: Discover Platforms*:
.. Image: {rbrg_gcloud_image_ref}
.. Fetch OAuth token via `gcloud auth print-access-token`
.. {rbbc_call}
Docker Registry API: GET manifests for `«CONSECRATION»-image` with OCI media type accept headers
.. If response is an OCI image index or Docker manifest list:
... {rbbc_store}
`«PLATFORMS»` as list of platform descriptors from the manifest list
.. If response is a single OCI/Docker manifest (typical for bind and graft):
... {rbbc_call}
GET config blob via digest from manifest's `config.digest` field
... {rbbc_store}
`«PLATFORMS»` as single-element list from config's `os` and `architecture` fields
.. {rbbc_store}
platform count for subsequent step branching
.. After platform discovery, attempt `-diags` artifact extraction (conjure-only):
... {rbbc_call}
Docker Registry API: HEAD manifests for `«CONSECRATION»-diags`
... If HTTP 200: fetch manifest, iterate layer blobs, extract diagnostic files (`buildkit_metadata.json`, `cache_before.json`, `cache_after.json`, `recipe.txt`) to `/workspace`
... If HTTP 404 or error: skip gracefully (expected for bind/graft — no `-diags` artifact exists)
.. NOTE: The `-diags` artifact is pushed by conjure step 07 (`rbgjb07-push-diags.sh`) and contains build-time diagnostics that cannot be conveyed via Cloud Build substitution variables (Dockerfile content exceeding the ~4KB limit, BuildKit metadata, Docker daemon cache snapshots).

//axhos_step
. *Cloud Build Step 2: Syft SBOM Generation*:
.. Image: {rbrg_docker_image_ref} (runs syft via `docker run` with pinned {rbrg_syft_image_ref})
.. Fetch OAuth token via metadata server (syft authenticates as {rbtr_mason})
.. For each platform in `«PLATFORMS»`:
... If single-platform (no manifest list): {rbbc_call}
`syft registry:«GAR_HOST»/«GAR_PATH»/«VESSEL»:«CONSECRATION»-image`
... If multi-platform with per-platform tags (conjure): {rbbc_call}
`syft registry:«GAR_HOST»/«GAR_PATH»/«VESSEL»:«CONSECRATION»-image-«PLATFORM_SUFFIX»`
... If multi-platform without per-platform tags (bind): {rbbc_call}
`syft registry:«GAR_HOST»/«GAR_PATH»/«VESSEL»:«CONSECRATION»-image@«PLATFORM_DIGEST»` using per-platform digest from step 1's manifest list
... {rbbc_store}
per-platform SBOM to `/workspace/sbom-«LABEL».json` (e.g., `sbom-amd64.json`; LABEL = architecture + variant without separators)
.. NOTE: Syft's `registry:` transport reads directly from the OCI registry — no docker CLI or local pull required.

//axhos_step
. *Cloud Build Step 3: Assemble build_info.json*:
.. Image: {rbrg_alpine_image_ref}
.. For each platform in `«PLATFORMS»`:
... Construct `build_info.json` with mode-aware fields:
.... Shared: `vessel_mode`, `vessel_name`, `platform`, `image_digest`, `about_timestamp`, `git_commit`, `git_branch`, `git_repo`
.... Conjure: `build_id` (from `_RBGA_BUILD_ID`), `inscribe_timestamp`, `qemu_used`, `slsa_build_level`, `build_invocation_id`, `provenance_predicate_types`, `provenance_builder_id`
.... Bind: `bind_source` (from `_RBGA_BIND_SOURCE`)
.... Graft: `graft_source` (from `_RBGA_GRAFT_SOURCE`)
... {rbbc_store}
per-platform `build_info.json` to `/workspace/build_info-«LABEL».json` (e.g., `build_info-amd64.json`)
.. Write `recipe.txt` using dual-source logic:
... If `recipe.txt` already exists in `/workspace` (written by `-diags` extraction in step 1): skip substitution variable write, log that `recipe.txt` came from `-diags`
... Else if `_RBGA_DOCKERFILE_CONTENT` is non-empty: {rbbc_store}
`recipe.txt` to `/workspace/recipe.txt` from substitution variable content
... Else: omit `recipe.txt`
.. NOTE: The {rbtr_director} reads the Dockerfile locally and passes its content as `_RBGA_DOCKERFILE_CONTENT`.
If the file exceeds the Cloud Build substitution variable size limit (~4KB), the director omits it with a warning.
The `-diags` path (conjure only) bypasses this limit by conveying the full Dockerfile via a registry artifact.
`_RBGA_DOCKERFILE_CONTENT` is retained for all modes as a fallback when `-diags` extraction fails or is absent.

//axhos_step
. *Cloud Build Step 4: Build and Push About Artifact*:
.. Image: {rbrg_docker_image_ref}
.. Create buildx builder instance (inspect-or-create pattern)
.. Build multi-platform `FROM scratch` container with per-platform SBOM, `build_info.json`, `recipe.txt` (when present), and conjure enrichment files when forwarded from `-diags`: `buildkit_metadata.json`, `cache_before.json`, `cache_after.json` (conditional COPY — included only if present in `/workspace`)
.. For single-platform images: build single-platform `-about` container
.. For multi-platform images: build with `buildx --push` using `TARGETARCH`/`TARGETVARIANT` to select per-platform files
.. Push to registry tagged `«CONSECRATION»-about` via `docker buildx build --push`
.. {rbbc_require}
push succeeds

//axhos_step
. *Wait for Build Completion*:
.. {rbbc_poll}
`GET «GCB_PROJECT_BUILDS_URL»/«BUILD_ID»` every 5 seconds
.. Continue while status in [`PENDING`, `QUEUED`, `WORKING`]
.. {rbbc_fatal}
if timeout or final status is not `SUCCESS`

{rbtga_ark_about}
. *Display Results*:
.. {rbbc_show}
about assembly complete with consecration and about artifact tag

//axhoc_completion
Successful execution creates a
{rbtga_ark_about}
artifact in the registry tagged as `«VESSEL»:«CONSECRATION»-about`.
The about artifact is a multi-platform container matching the `-image` artifact's platform list.
+
*Re-about idempotency*: Re-running about re-executes Syft and rebuilds `build_info.json`, overwriting the previous `-about` artifact.
SBOM content may vary if Syft discovers different packages (e.g., after a Syft version update).
+
*Proceed to vouch*: After about completes, run
{rbtgo_ark_vouch}
to produce the verification artifact that completes the ark.
+
*Implementation note*: About and vouch share common Build resource JSON construction patterns (worker pool, mason SA, registry coordinates).
The implementation should extract a shared helper for `builds.create` request assembly.

==== About Substitution Variables

All `_RBGA_*` substitution variables are passed in the `builds.create` request body.

*Registry and identity*::
[cols="1,2"]
|===
| `_RBGA_GAR_HOST` | GAR Docker host. Source: regime constants.
| `_RBGA_GAR_PATH` | GAR repository path. Source: regime constants.
| `_RBGA_VESSEL` | Vessel sigil. Source: loaded vessel configuration.
| `_RBGA_CONSECRATION` | Full consecration string. Source: input parameter.
| `_RBGA_VESSEL_MODE` | Vessel mode (`conjure`, `bind`, or `graft`). Source: loaded vessel configuration.
| `_RBGA_ARK_SUFFIX_IMAGE` | Ark suffix for image artifact. Source: `RBGC_ARK_SUFFIX_IMAGE` constant.
| `_RBGA_ARK_SUFFIX_ABOUT` | Ark suffix for about artifact. Source: `RBGC_ARK_SUFFIX_ABOUT` constant.
| `_RBGA_ARK_SUFFIX_DIAGS` | Ark suffix for diags artifact. Source: `RBGC_ARK_SUFFIX_DIAGS` constant.
|===

*Git metadata*::
[cols="1,2"]
|===
| `_RBGA_GIT_COMMIT` | Main repo HEAD SHA at operation time.
| `_RBGA_GIT_BRANCH` | Main repo branch at operation time.
| `_RBGA_GIT_REPO` | Main repo identifier (owner/repo).
|===

*Mode-specific*::
[cols="1,2"]
|===
| `_RBGA_BUILD_ID` | Conjure Cloud Build job ID (conjure only). Ephemeral director-side state retained between the conjure and about `builds.create` calls. Empty for bind/graft.
| `_RBGA_INSCRIBE_TIMESTAMP` | Inscribe timestamp from consecration (conjure only). Empty for bind/graft.
| `_RBGA_BIND_SOURCE` | Pinned upstream image reference (bind only). Source: {rbrv_bind_image}. Empty for conjure/graft.
| `_RBGA_GRAFT_SOURCE` | Local image reference at push time (graft only). Empty for conjure/bind.
| `_RBGA_DOCKERFILE_CONTENT` | Dockerfile content read locally by {rbtr_director} (all modes, optional). Source: {rbrv_conjure_dockerfile} (conjure, always populated), {rbrv_graft_optional_dockerfile} (graft, optional), {rbrv_bind_optional_dockerfile} (bind, optional). Empty if file exceeds ~4KB substitution limit or path not configured. Written to `recipe.txt` in step 3.
|===

NOTE: `«PLATFORMS»` is discovered in step 1 and passed to subsequent steps as a workspace variable (written to `/workspace/platforms.txt`), not as a substitution variable.
