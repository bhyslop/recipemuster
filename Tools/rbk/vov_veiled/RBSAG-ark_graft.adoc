//axhob_operation
{rbtgo_ark_graft}

NOTE: This operation grafts a locally-built container image into
{rbtgi_repository}
via native docker/podman push.
No Cloud Build involvement in the image push — the image is already built on the local workstation.
No crane dependency — docker tag + docker push is sufficient for locally-present images.
No dirty-tree guard — git state is irrelevant to an already-built container.

NOTE: The name "graft" comes from botanical grafting — foreign material spliced onto rootstock.
The graft may take or may not.
The metaphor fits the trust posture: grafted images have no provenance chain and receive a GRAFTED verdict from
{rbtgo_ark_vouch}.

NOTE: Graft uses whatever container runtime the `docker` command resolves to in PATH, matching other {rbtr_director}-side operations.
No separate regime variable governs runtime selection for graft.

NOTE: *Universal reliquary precondition*. Every ordain-path vessel — conjure, bind, and graft — must have `RBRV_RELIQUARY` non-empty before this operation runs. Graft's about+vouch combined Cloud Build job consumes reliquary tool images (gcloud, docker, alpine, syft) just like conjure and bind; the registry preflight enforces this before submission.

The
{rbtr_director}
grafts an image:

//axhoc_completion
Successful execution pushes the locally-built image to
{rbtgi_repository}
tagged as `{gar_hallmarks_namespace}/«HALLMARK»/image:«HALLMARK»` within the hallmark subtree.
The graft operation returns `«HALLMARK»` to {rbtgo_ark_ordain}, which chains the subsequent pipeline stages automatically:
{rbtgo_ark_about} and {rbtgo_ark_vouch} in a single combined `builds.create` job (about steps followed by vouch steps) to assemble metadata and record the GRAFTED verdict.
+
*No SLSA provenance*: Grafted images have no Cloud Build provenance.
The {rbtgo_ark_vouch} pipeline records a GRAFTED verdict, explicitly marking the absence of a provenance chain.
This is a deliberate trust boundary — grafted images are accepted into the system but carry less attestation than conjured images.
+
*Push atomicity*: `docker push` is atomic per manifest — a failed push (network timeout, authentication expiry) leaves no partial state in GAR.
The director can safely retry without cleanup.
+
*Single-platform only*: Graft is specified for single-platform images only.
`docker push` of a locally-present image pushes a single manifest.
The about pipeline's platform discovery step handles this correctly, producing a single-platform `about` basename under the hallmark subtree.
Multi-platform graft is not currently specified — `buildx --load` only loads one platform into the local daemon, so true multi-platform local images would require a different push mechanism.
