//axhob_operation
{rbtgo_ark_inspect}

NOTE: This operation plumbs an
{rbtga_ark}
by querying GAR directly via the Docker Registry API.
Artifacts are fetched from the
{rbtga_ark_about}
and
{rbtga_ark_vouch}
packages in the registry.
No local container runtime is required.
Authenticates as
{rbtr_retriever}.

NOTE: Two entry points provide different detail levels:
{rbtc_plumb_full} for complete diagnostic output and
{rbtc_plumb_compact} for a trust posture summary.

NOTE: Two invocation modes:
Vessel+hallmark (both arguments) fetches directly from the vessel package.
Hallmark-only (single argument) resolves the vessel via the vouches superdirectory.

The plumb operation:

//axhos_step
. *Display Mode-Aware Sections* (both detail levels):
.. *Vessel Type*: mode (conjure/bind/graft), platform, build strategy (native/QEMU for conjure), moniker
.. *Source*: git repository, branch, commit
.. Branch on vessel mode:
+
**Conjure sections**:
.. *Builder*: Cloud Build job ID, timestamps, image URI
.. *SLSA Provenance*: build level, invocation ID, builder ID, predicate types, explicit boundary statements (what SLSA does and does not attest)
.. *Base Image*: Dockerfile FROM line, syft-detected distro
+
**Bind sections**:
.. *Bind Source*: original upstream image reference, digest-pin status
.. *Trust Boundary*: digest-pin only, no SLSA provenance, no build transcript
+
**Graft sections**:
.. *Graft Source*: local image reference at push time
.. *Trust Boundary*: no provenance chain, no build transcript — GRAFTED acknowledgment
+
**Shared sections** (all modes):
.. *SBOM Summary*: package count and type breakdown from syft scan
.. *Vouch Results*: verdict type and details:
... Conjure: verifier identity, per-platform verification verdicts (`SLSA_VERIFIED`)
... Bind: digest-pin confirmation (`DIGEST_PIN_VERIFIED`)
... Graft: GRAFTED acknowledgment (`GRAFTED`)

//axhos_step
. *Display Full-Only Sections* ({rbtc_plumb_full} only):
.. *Package Inventory*: every syft-detected package (type, name, version) sorted by ecosystem
.. *Package Licensing*: per-package license and Package URL for compliance review
.. *Recipe*: the Dockerfile content from `recipe.txt` in the about artifact (available for any mode when the Dockerfile path was configured in the vessel regime). When `recipe.txt` is absent, display the Dockerfile path from `build_info.json` if available, or omit the section.

//axhoc_completion
Plumb is a read-only operation that queries GAR via the Docker Registry API.
It does not modify artifacts or require a local container runtime.
Authenticates as
{rbtr_retriever}
using OAuth credentials.
+
*GAR-direct extraction*: Artifacts are fetched via Registry API manifest and blob endpoints.
Multi-platform manifest lists are resolved to a single platform (first entry) since artifact content (JSON metadata) is architecture-independent.
+
*vouches hallmark-only mode*: When invoked with only a hallmark, plumb resolves the vessel by looking up the hallmark in the vouches superdirectory package.
The vouches package aggregates all vouch artifacts across all vessels, enabling reverse mapping from hallmark to vessel.
+
*Mode-aware display*: The plumb output adapts to vessel mode.
Bind and graft vessels display clear trust boundary statements rather than attempting to show provenance data that does not exist.
