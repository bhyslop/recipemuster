{rbtgo_ark_conjure}

NOTE: This operation conjures an
{rbtga_ark_image}
in
{rbtgi_repository}
by building a container image from a vessel specification. The
{rbtr_director}
performs the build via
{gcb_service}
using direct `builds.create` API submission.

NOTE: Conjure produces both the image and about artifacts in a single Cloud Build job submitted via `builds.create`.
The about steps (`rbgja01`-`rbgja04`) are embedded via stitch alongside the build steps.
The vouch artifact is created by the subsequent {rbtgo_ark_vouch} pipeline stage (always a separate Cloud Build job).

NOTE: The
{rbtga_hallmark}
is minted locally by the
{rbtr_director}
before submission.
The Director knows the complete artifact tag set before the build begins -- naming authority and verification authority are the same entity.

NOTE: The conjure build job runs on the worker pool selected by the vessel's `RBRV_EGRESS_MODE`: tether pool for vessels needing public internet (e.g., `apt-get` in Dockerfile), airgap pool for vessels with all dependencies held as Lodes in GAR.
The subsequent {rbtgo_ark_vouch} always runs on the airgap pool regardless of vessel egress mode.

NOTE: *Universal reliquary precondition*. Every ordain-path vessel — conjure, bind, and graft — must have `RBRV_RELIQUARY` non-empty before this operation runs. The about/vouch pipeline (gcloud, docker, alpine, syft) consumes reliquary tool images uniformly across all modes; the registry preflight below enforces this.

NOTE: The conjure Cloud Build job carries the {rbsk_in_pool_preflight} as its step 0, ahead of the extract-context step. Conjure populates `_RBGR_BASE_LOCATOR_n` substitutions from `RBRV_IMAGE_n_ANCHOR` slots, so the preflight HEAD-checks each anchored base in addition to the reliquary tools.

NOTE: *Dirty-tree guard*.
Conjure refuses on a working tree with unstaged or staged changes (`bug_require_clean_tree_creed`), before anything leaves the host.
The build stamps HEAD into the image (`git.commit` label, about build_info) and ships working-tree bytes (the pouch context, the composed step scripts), so the recorded commit must match the bytes built.
Conjure reads no chaining fact and performs no post-gate config write: it builds only from committed configuration — it is a pure chain head, output only.
The base anchor (`RBRV_IMAGE_n_ANCHOR`) is elected separately by {rbtgo_director_feoff} *before* conjure runs (ceremony: ensconce → feoff → commit → conjure), so the anchor conjure resolves is already committed when the build stamps HEAD.
This closes the provenance hole of the former in-build election, which wrote the anchor after the clean-tree gate and built from an uncommitted value, so the stamped commit named a tree whose {rbrv_regime} still held the prior anchor.
Graft is the deliberate exception to the guard — see {rbtgo_ark_graft}.

//axhos_step
. *Registry Preflight*:
.. *Reliquary check*: HEAD request per canonical tool to the conclave Lode package `{gar_lodes_namespace}/«RELIQUARY»`, manifest tag `:rbi_«tool»`, in
{gar_service}.
{rbbc_require}
HTTP 200 for every tool — on 404, fail with directive to run conclave first.
Misses accumulate across the whole tool set before the failure report, so the operator sees the full damage at once.
The reliquary is a co-versioned set of builder tool images captured from upstream into one conclave Lode in private GAR. Each vessel declares its reliquary touchmark; multiple reliquary Lodes can coexist.
.. *Anchor check*: For each indexed slot (n = 1, 2, 3) where both
{rbrv_image_origin}
and
{rbrv_image_anchor}
are set:
... Parse
{rbrv_image_anchor}
as a
{rbst_locator}
into package-path and tag
... HEAD request to `«LOCATOR-PATH»:«LOCATOR-TAG»` manifest in
{gar_service}
... {rbbc_require}
HTTP 200 — on 404, fail with directive to run {rbtgo_lode_ensconce} first.
Multiple vessels sharing the same base origin need only one ensconce.
.. *Airgap anchor requirement*: Under `rbnve_airgap` egress, every slot with a populated
{rbrv_image_origin}
must also carry a populated
{rbrv_image_anchor}
— the airgap pool has no upstream egress, so an empty anchor guarantees an in-pool build failure; preflight rejects it before submission with the population directive.
Two population paths, discriminated by whether the origin names a producer-vessel directory under the vessel tree: an upstream origin is captured into a bole locator by {rbtgo_lode_ensconce}; a producer-vessel origin is a hallmark-pin, written from the producer's ordain output (never ensconced — {rbtgo_lode_ensconce} rejects producer-vessel origins).
Under `rbnve_tether`, an anchorless origin passes through — buildx fetches upstream at build time.
.. Preflight prevents wasted
{gcb_service}
submission time on missing images, with copy-paste remediation commands

//axhos_step
. *Resolve Base Images*:
.. For each indexed slot (n = 1, 2, 3) where
{rbrv_image_origin}
is set:
... If
{rbrv_image_anchor}
is present: parse the
{rbst_locator}
and resolve to full GAR image reference (`«GAR-REPO-BASE»/«LOCATOR-PATH»:«LOCATOR-TAG»`)
... If
{rbrv_image_anchor}
is absent: pass
{rbrv_image_origin}
through as the upstream tag (development path)
.. The host resolves a *tag-level* reference; a cloud step (`rbgjb03-resolve-base-digests`) then resolves each populated slot's tag to its canonical `@sha256:` digest in GCP egress — `gcrane manifest «REF»` over the raw manifest bytes, the gcrane-fingerprint shape — reachable in-pool for an anchored GAR base and over egress for an upstream pass-through base
.. The pinned reference `«REF»@sha256:«DIGEST»` is used twice: as the `RBF_IMAGE_n` build-arg, so buildx provably builds `FROM` exactly the resolved digest rather than a tag that could move between resolution and build; and as a sprued image label recording what was actually resolved
.. The label key is `rbi_resolved_base_n` (one per populated slot — a `scratch` slot, having no base, gets none), sprued under the `rbi_` image domain where its neighbor build labels (`hallmark`, `git.commit`, `git.branch`) stay unsprued by deliberate divergence; the value is the same pinned `«REF»@sha256:«DIGEST»`, tamper-evident (rivet RBr_b4e)
