//axhob_operation
{rbtgo_ark_conjure}

NOTE: This operation conjures an
{rbtga_ark_image}
in
{rbtgi_repository}
by building a container image from a vessel specification. The
{rbtr_director}
performs the build via
{gcb_service}
using direct `builds.create` API submission.

NOTE: Conjure produces both the image and about artifacts in a single Cloud Build job submitted via `builds.create`.
The about steps (`rbgja01`-`rbgja04`) are embedded via stitch alongside the build steps.
The vouch artifact is created by the subsequent {rbtgo_ark_vouch} pipeline stage (always a separate Cloud Build job).

NOTE: The
{rbtga_hallmark}
is minted locally by the
{rbtr_director}
before submission.
The Director knows the complete artifact tag set before the build begins -- naming authority and verification authority are the same entity.

NOTE: The conjure build job runs on the worker pool selected by the vessel's `RBRV_EGRESS_MODE`: tether pool for vessels needing public internet (e.g., `apt-get` in Dockerfile), airgap pool for vessels with all dependencies enshrined in GAR.
The subsequent {rbtgo_ark_vouch} always runs on the airgap pool regardless of vessel egress mode.

NOTE: *Universal reliquary precondition*. Every ordain-path vessel — conjure, bind, and graft — must have `RBRV_RELIQUARY` non-empty before this operation runs. The about/vouch pipeline (gcloud, docker, alpine, syft) consumes reliquary tool images uniformly across all modes; the registry preflight below enforces this.

NOTE: The conjure Cloud Build job carries the {rbsk_in_pool_preflight} as its step 0, ahead of the extract-context step. Conjure populates `_RBGR_ENSHRINE_LOCATOR_n` substitutions from `RBRV_IMAGE_n_ANCHOR` slots, so the preflight HEAD-checks each anchored base in addition to the reliquary tools.

The conjure operation:

//axhopt_typed_parameter
{rbtga_ark}
. *Validate Input Parameters*:
.. {rbbc_require}
vessel provided as first argument
.. {rbbc_require}
vessel contains conjuring configuration (RBRV_CONJURE_DOCKERFILE, RBRV_CONJURE_BLDCONTEXT)

//axhos_step
. *Registry Preflight*:
.. *Reliquary check*: HEAD request to `«RELIQUARY»/docker:latest` manifest in
{gar_service}.
{rbbc_require}
HTTP 200 — on 404, fail with directive to run inscribe first.
The reliquary is a co-versioned set of builder tool images inscribed from upstream into private GAR. Each vessel declares its reliquary; multiple datestamped reliquaries can coexist.
.. *Enshrine check*: For each indexed slot (n = 1, 2, 3) where both
{rbrv_image_origin}
and
{rbrv_image_anchor}
are set:
... Parse
{rbrv_image_anchor}
as a
{rbst_locator}
into package-path and tag
... HEAD request to `«LOCATOR-PATH»:«LOCATOR-TAG»` manifest in
{gar_service}
... {rbbc_require}
HTTP 200 — on 404, fail with directive to run {rbtgo_ark_enshrine} first.
Multiple vessels sharing the same anchor need only one enshrine.
.. Preflight prevents wasted
{gcb_service}
submission time on missing images, with copy-paste remediation commands

//axhos_step
. *Resolve Base Images*:
.. For each indexed slot (n = 1, 2, 3) where
{rbrv_image_origin}
is set:
... If
{rbrv_image_anchor}
is present: parse the
{rbst_locator}
and resolve to full GAR image reference (`«GAR-REPO-BASE»/«LOCATOR-PATH»:«LOCATOR-TAG»`)
... If
{rbrv_image_anchor}
is absent: pass
{rbrv_image_origin}
through as the upstream tag (development path)
.. Resolved references become `RBF_IMAGE_n` build-args for the Dockerfile `ARG`/`FROM` pattern

//axhos_step
. *Assign Hallmark*:
.. Mint
{rbtga_hallmark}
locally on the Director's workstation
.. {rbbc_store}
`«HALLMARK»` -- complete artifact tag set is now known before build submission

//axhos_step
. *Push Pouch*:
.. Package build context (Dockerfile and context files) as a FROM SCRATCH OCI image
.. Push to GAR tagged `{gar_hallmarks_namespace}/«HALLMARK»/pouch:«HALLMARK»`
.. {rbbc_require}
push succeeds -- the pouch delivers build context to
{gcb_service}

//axhos_step
. *Execute Build*:
.. Stitch
{rbtgr_build_json}
with all step image references and substitution variables
.. {rbbc_submit}
`POST «GCB_PROJECT_BUILDS_URL»` with Build resource JSON via `builds.create`
.. All build parameters in stitched
{rbtgr_build_json}
-- zero custom substitution overrides
.. Waits for build completion
.. On success, the
{rbtga_ark_image}
artifact is created with the
{rbtga_hallmark}
.. {rbtgr_provenance}
is auto-generated by Google as a signed attestation on the built image

. *Display Results*:

vessel image built with hallmark tag

//axhoc_completion
Successful execution creates the
{rbtga_ark_image}
in
{rbtgi_repository}.
The {rbtr_director} retains `«BUILD_ID»` and `«HALLMARK»` from the conjure round-trip.
The `builds.create` job produces both the image and about artifacts under the hallmark subtree (about steps embedded via stitch).
{rbtgo_ark_ordain} chains only {rbtgo_ark_vouch} after the combined job completes, passing `«BUILD_ID»` for DSSE provenance verification.
