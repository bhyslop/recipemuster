//axhob_operation
{rbtgo_ark_enshrine}

NOTE: This operation enshrines upstream base images into
{gar_service}
as content-addressed, human-readable anchors.
The
{rbtr_director}
submits a Cloud Build job that copies each declared
{rbrv_image_origin}
from its upstream registry to GAR using
the reliquary skopeo image
(`skopeo copy --all`), preserving the manifest list and all per-platform manifests for multi-platform images.
The locator results are returned via `buildStepOutputs` and
{rbrv_image_anchor}
is then written back to the vessel regime locally as a
{rbst_locator}.

NOTE: Slots whose
{rbrv_image_anchor}
is already populated are processed under skip-if-set semantics.
The Director HEAD-validates that the locator resolves in
{gar_service}
and leaves the slot's
{rbrv_image_anchor}
untouched — no Cloud Build copy occurs for that slot.
This admits hallmark-base vessels: a vessel whose base image is an internal hallmark (e.g., the airgap-bottle building FROM another vessel's just-ordained hallmark) carries an
{rbrv_image_anchor}
of `{gar_hallmarks_namespace}/«HALLMARK»/«BASENAME»:«HALLMARK»` written by the orchestrating procedure (handbook step or onboarding-sequence fixture) before enshrine runs; enshrine then verifies the hallmark is reachable rather than re-mirroring it.
A vessel whose every slot is pre-populated requires no Cloud Build submission — enshrine reduces to local HEAD validation.

NOTE: Enshrine is a pre-build preparation step.
It must be run before
{rbtgo_ark_conjure}
when anchored base images are desired.
Conjure works without anchors (passing ORIGIN through), but anchored images provide reproducibility and air-gap readiness.

NOTE: Enshrine runs on Cloud Build, not the Director's workstation.
No local skopeo installation is required.
The GCB step uses Mason SA ambient credentials via the metadata server for GAR writes.

The enshrine operation:

//axhopt_typed_parameter
{rbrv_sigil}
. *Validate Input Parameters*:
.. {rbbc_require}
vessel provided as first argument
.. {rbbc_require}
vessel declares at least one base-image slot — that is, at least one of slots 1-3 has either
{rbrv_image_origin}
or
{rbrv_image_anchor}
populated
.. {rbbc_require}
vessel mode is `conjure` (`RBRV_VESSEL_MODE`)
.. {rbbc_require}
reliquary tool images exist in GAR (HEAD check on `docker:latest` canary)

//axhos_step
. *Classify Slots*:
.. For each indexed slot (n = 1, 2, 3):
... If
{rbrv_image_anchor}
is set: classify as *pre-populated*. HEAD-request the locator's `«PACKAGE-PATH»:«TAG»` manifest in
{gar_service}
. {rbbc_require}
HTTP 200 — on 404, fail with directive naming the missing locator (caller wrote an unreachable
{rbrv_image_anchor}
). The slot's
{rbrv_image_anchor}
is left intact; no Cloud Build copy is performed.
... Otherwise, if
{rbrv_image_origin}
is set: classify as *to-enshrine*. ORIGIN is passed to the Cloud Build step for inspect-and-copy.
... Otherwise the slot is empty and is omitted from the build.
.. If every declared slot is pre-populated, skip the Cloud Build submission entirely — the operation reduces to local HEAD validation.

//axhos_step
. *Submit Enshrine Cloud Build Job* (only if any slot is *to-enshrine*):
.. Compose `builds.create` JSON with:
... Single step using
the reliquary skopeo image
as the builder image (script: `rbgje01-enshrine-copy.sh`)
... `_RBGE_*` substitutions: `_RBGE_GAR_HOST`, `_RBGE_GAR_PATH`, `_RBGE_IMAGE_n_ORIGIN` (n = 1, 2, 3; empty for unused slots). Base images are stored in the shared `{gar_enshrines_namespace}/` namespace, with each anchor as a peer package (`{gar_enshrines_namespace}/«ANCHOR»:«ANCHOR»`), not per-vessel.
... Mason SA, private pool, timeout
.. Submit via `builds.create` and wait for completion
.. The GCB step, for each non-empty ORIGIN slot:
... Fetches the raw upstream manifest (`skopeo inspect --raw`) and computes its sha256 digest (`sha256sum` of the raw manifest bytes)
... {rbbc_store}
`<<DIGEST>>` -- the sha256 digest of the upstream manifest (manifest list digest for multi-platform images, single manifest digest for single-platform images)
... Constructs anchor: sanitize full origin string (`:` `/` replaced with `-`), append first 10 hex chars of sha256 digest
... Example: `python:3.11-slim` with `sha256:abc123def4...` yields anchor `python-3.11-slim-abc123def4`
... Composes the
{rbst_locator}
as `{gar_enshrines_namespace}/«ANCHOR»:«ANCHOR»` (e.g., `{gar_enshrines_namespace}/python-3.11-slim-abc123def4:python-3.11-slim-abc123def4`)
... Copies upstream image into the shared `{gar_enshrines_namespace}/` namespace in GAR as a peer package at `{gar_enshrines_namespace}/«ANCHOR»:«ANCHOR»` using `skopeo copy --all`, transferring the manifest list and all per-platform manifests atomically. Each anchor is its own package under the namespace — the path segment carries the anchor identity, and the tag redundantly restates it so divergence is a visible bug. The anchor is vessel-independent — multiple vessels sharing the same base image origin share one enshrined package.
... {rbbc_require}
copy succeeds -- the locator now resolves to a permanent, content-addressed reference in
{gar_service}
containing all upstream platforms
... Single-platform upstream images are handled identically -- `skopeo copy --all` is a no-op on the platform dimension when only one platform exists
.. Writes JSON locator results to `/builder/outputs/output` (returned as `buildStepOutputs`); each slot's `anchor` field carries the full locator string

//axhos_step
. *Extract Locators and Write Regime* (only for *to-enshrine* slots):
.. Read `buildStepOutputs[0]` from the completed build result (base64-decoded JSON)
.. For each slot present in the result:
... Write locator string to `RBRV_IMAGE_n_ANCHOR` in the vessel's `rbrv.env`
.. *Pre-populated* slots are not touched — their existing
{rbrv_image_anchor}
values stand

//axhoot_typed_output
{rbrv_image_anchor}
Per-slot locator string written to the vessel regime.

//axhoc_completion
Successful execution produces anchored base image references in
{gar_service}
(with all upstream platforms preserved) and updates the vessel regime with
{rbrv_image_anchor}
values.
The vessel is now ready for reproducible
{rbtgo_ark_conjure}
builds using GAR-mirrored base images.
Multi-platform conjure vessels pull the correct platform from the anchored manifest list automatically.
Re-running enshrine with the same
{rbrv_image_origin}
but a different upstream digest produces a new locator (upstream image was updated).
Re-running with the same digest is idempotent.
