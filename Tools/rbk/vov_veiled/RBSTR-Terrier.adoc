== Terrier access

The
{rbtf_terrier}
is a polity's durable, cloud-resident, authoritative record of which
{rbtf_citizen}
holds which
{rbtf_mantle}
— one
{rbtf_muniment}
per object, recording (principal subject, mantle held).
Exactly three sub-operations touch it, each a single conditioned REST call
against the bucket:

[[rbtf_engross]]
{rbtf_engross}::
Write a
{rbtf_muniment}
(to engross: to set a deed down in its formal, final copy).
Composed by
{rbtf_brevet},
which engrosses the muniment before it writes the IAM binding.
Carries the atomic write-semantics below.

[[rbtf_expunge]]
{rbtf_expunge}::
Withdraw a
{rbtf_muniment}
(to strike an entry from the record).
Composed by
{rbtf_unseat}.
Carries the atomic write-semantics below.

[[rbtf_peruse]]
{rbtf_peruse}::
Read
{rbtf_muniment_s}
(to examine the record through).
The pure list-and-fetch primitive — no precondition — that
{rbtf_rehearse}
composes, and the read side of the reconciliation diff.

=== Write semantics — atomic, server-adjudicated (settled)

Mutations of the Terrier —
{rbtf_engross}
and
{rbtf_expunge}
— are made atomic by Cloud Storage itself. No Cloud Build
invocation, and no external lock, is involved.

* *Create is conditional on absence.* The object write carries the
  `ifGenerationMatch=0` precondition, so Cloud Storage performs the write only if
  no live object exists at that key. Concurrent creators race cleanly: one
  succeeds; the others receive `HTTP 412 Precondition Failed` and treat an
  already-present identical muniment as idempotent success.
* *Update is conditional on generation.* Where a muniment is rewritten rather
  than created, the write carries `ifGenerationMatch=<generation>` read
  beforehand — optimistic concurrency control. The write lands only if the object
  has not changed since the read; otherwise the 412 directs a read-and-retry.
* Cloud Storage guarantees these preconditions are evaluated atomically and free
  of race conditions. There is no multi-object transaction across muniments, and
  none is needed: each muniment is an independent object, so distinct governors
  writing distinct entries never contend, and the precondition covers the rare
  same-entry case.

Each mutation is therefore a single conditioned REST call: the bucket adjudicates
atomicity, the caller adds one query parameter and checks the HTTP code. This is
within bash's good case under BCG — glue over a service, carrying no lock logic
of its own.

=== Settled — the terrier noun's internals (in RBS0)

The Terrier *noun* internals — object granularity, the muniment JSON shape and its
`rbgft_` sprue, the physical bucket name and its constant home, and the
managed-folder IAM grain — are settled in the RBS0 terrier quoin. They are
properties of the noun, not of this access subdoc, so they are not restated here
(ACG: reference the home).

One resolution bears on the write semantics above: granularity is per-entry — one
object per (principal subject,
{rbtf_mantle}
held) pair — so a muniment is immutable: a holding exists or it does not. Writes
are therefore create-and-delete only (
{rbtf_engross}
then
{rbtf_expunge});
the generation-conditional update clause stands as contract but is unexercised
under this granularity.
