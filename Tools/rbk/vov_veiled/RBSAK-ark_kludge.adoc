//axhob_operation
{rbtgo_ark_kludge}

NOTE: This operation builds a vessel image locally on the workstation using `docker build`, without authenticating to or fetching from Cloud Build or GAR. It produces a
{rbtga_hallmark}
with a `k` prefix (kludge) that composes cleanly with the nameplate and compose infrastructure but is obviously not a production hallmark. Kludge resolves the same {rbrv_image_anchor} declarations that {rbtgo_ark_conjure} does — so an enshrined vessel built via kludge consumes the same GAR-anchored base layers as the conjure path — but anchored layers must be pre-cached locally; the credentialed fetch happens out-of-band via {rbtgo_image_retrieve}. Origin-only slots (no anchor) require no credentials, so kludge pulls them inline if they're not already cached.

NOTE: Kludge is a deliberate development shortcut for fast edit-build-test cycles. It enables iterating on vessel Dockerfiles, entrypoint scripts, and package lists without waiting for a Cloud Build round-trip. The ugly name is intentional -- this is not part of the production hallmark pipeline.

NOTE: The kludge hallmark format is `k«YYMMDDHHMMSS»-«git-describe»`, paralleling conjure's `c«YYMMDDHHMMSS»` and bind's `b«YYMMDDHHMMSS»-r«YYMMDDHHMMSS»`. The timestamp provides chronological sorting; the `git describe --always` suffix ties the image to a specific commit. The dirty-tree guard ensures the describe always corresponds to a committed state. The `k` prefix makes kludge hallmarks visually distinct in nameplate files and `docker image ls` output.

The kludge operation:

//axhopt_typed_parameter
{rbtga_ark}
. *Validate Input Parameters*:
.. {rbbc_require}
vessel provided as first argument (sigil or path)
.. {rbbc_require}
vessel is conjure mode (bind and graft vessels have no local Dockerfile)
.. {rbbc_require}
vessel has conjuring configuration (RBRV_CONJURE_DOCKERFILE, RBRV_CONJURE_BLDCONTEXT)

//axhos_step
. *Resolve Base Images*:
.. For each indexed slot (n = 1, 2, 3) where
{rbrv_image_origin}
is set:
... If {rbrv_image_anchor} is also set, resolve the slot's `RBF_IMAGE_n` build-arg to the full GAR reference composed from the locator (`«REGISTRY_HOST»/«PROJECT»/«REPOSITORY»/«PACKAGE_PATH»:«TAG»`) — same shape conjure produces
... If {rbrv_image_anchor} is empty, pass {rbrv_image_origin} through verbatim as the build-arg
.. {rbbc_fatal}
if a slot has {rbrv_image_anchor} set but {rbrv_image_origin} empty: "Malformed regime — anchor without origin"

//axhos_step
. *Verify Local Image Presence*:
.. For each resolved slot ref, query the local container runtime (`docker image inspect`) to confirm the image is cached locally
.. On pass-through miss (origin without anchor): announce and run `docker pull «origin»` inline, then re-inspect to confirm the pull landed an image locally. No credentials are required for public upstream registries.
.. On anchored miss: accumulate the missed locator for post-loop refusal
.. {rbbc_fatal}
if any anchored miss accumulated, emit a remediation block pointing at {rbtgo_image_retrieve} for each missed locator and abort before invoking docker build
.. The credentialed fetch for anchored slots is deliberately out-of-band — kludge never authenticates against GAR. The inline origin pull reaches upstream without credentials and does not change this property.

//axhos_step
. *Assign Kludge Hallmark*:
.. Mint hallmark as `k«YYMMDDHHMMSS»-«git-describe»` from dispatch timestamp and `BURD_GIT_CONTEXT`
.. {rbbc_store}
`«HALLMARK»`

//axhos_step
. *Build Image Locally*:
.. Validate the vessel Dockerfile against {rbsfh_dockerfile_hygiene}: any column-0 `FROM` line must use `${RBF_IMAGE_1}`, `${RBF_IMAGE_2}`, `${RBF_IMAGE_3}`, or `scratch` as its image token. Abort before invoking `docker build` on violation.
.. Run `docker build` with resolved `RBF_IMAGE_n` build-args
.. The no-network property comes from the presence check above (every slot verified in the local cache — origin auto-pulled if absent, anchored refused if absent) combined with buildx's default behavior of not re-pulling base images. {rbsfh_dockerfile_hygiene} closes the residual hardcoded-`FROM` risk that the discontinued `--pull=never` flag used to backstop.
.. Build for host platform only (no multi-arch -- development builds run where they are built)
.. Tag as `«GAR_BASE»/{gar_hallmarks_namespace}/«HALLMARK»/image:«HALLMARK»` to match compose YAML interpolation format

//axhos_step
. *Create Fake Vouch Tag*:
.. `docker tag` the image ref to also carry the `vouch` basename under the same hallmark subtree (`«GAR_BASE»/{gar_hallmarks_namespace}/«HALLMARK»/vouch:«HALLMARK»`)
.. This is the same image aliased -- no provenance verification, no DSSE envelope
.. Satisfies the `rbob_charge` vouch gate (`docker image inspect` on vouch ref)

//axhos_step
. *Report Hallmark*:
.. Print the hallmark value for the operator to paste into the nameplate `.env` file
.. Persist hallmark to `BURD_OUTPUT_DIR` for downstream chaining

== Limitations

* Requires GCP depot configuration to construct GAR-formatted image tags (registry host, project, repository), but no credentials are consumed and no registry is contacted for hallmark publication — only origin-slot upstream pulls reach a registry, unauthenticated
* Anchored base images must be cached locally before kludge runs — fetched out-of-band via {rbtgo_image_retrieve}. Origin pass-through slots auto-pull inline if missing, so the cold-cache path for origin-only vessels is a single operator step; vessels with anchored slots remain two steps (wrest, then kludge).
* Host platform only -- a kludge-built `linux/arm64` image on Apple Silicon will not run on `linux/amd64` hosts
* No about artifact, no provenance chain -- the vouch alias carries no attestation content
* Not suitable for release or delivery -- use {rbtgo_ark_conjure} for production hallmarks
