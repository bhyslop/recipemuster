//axhob_operation
{rbtgo_ark_abjure}

NOTE: This operation formally renounces an
{rbtga_ark}
from
{rbtgi_repository}
by enumerating every package beneath the ark's hallmark subtree (host-side, GAR
REST `packages.list`) and then **cloud-dispatching** their deletion: the
workstation submits one
{rbtr_director}-run Cloud Build that loops the package list in-pool, deleting each
package by convergence (see the completion note) and verifying its absence — so
the build's outcome IS the delete outcome, closing the host trust-200 LRO gap.
Made-side abjure shares this cloud-dispatch delete backbone with fetched-side
{rbtgo_lode_banish}; the line cinched is tool-plane (GAR package deletes go cloud)
versus control-plane (SA / project / depot deletes stay host-side REST).

An
{rbtga_ark}
is the coherent unit of a Recipe Bottle build: a set of packages rooted at
`{gar_hallmarks_namespace}/«HALLMARK»/` and distinguished by basename.
The
{rbtga_ark_image}
(basename `image`) contains the container image itself.
The
{rbtga_ark_about}
(basename `about`) contains SBOM and build metadata.
The
{rbtga_ark_vouch}
(basename `vouch`) contains verification results.
The
{rbtga_ark_pouch}
(basename `pouch`) contains the build context (conjure/bind only, absent for graft).
A fifth basename, `diags`, is present for conjure arks only (build-time diagnostics forwarded to the about pipeline).
For conjure vessels, a single `attest` package holds per-platform tags (`attest:«HALLMARK»-«ARCH»`).
Abjuring an ark deletes every package discovered beneath the hallmark subtree as a single logical operation.

The
{rbtr_director}
abjures an ark:

{rbtga_ark}
. *Validate Input Parameters*:
.. {rbbc_require}
`«INPUT_VESSEL»` provided as first argument (retained for AAK signature compatibility; not used to construct the subtree path, drops in AAL)
.. {rbbc_require}
`«INPUT_HALLMARK»` provided as second argument
.. {rbbc_fatal}
if hallmark missing: "Hallmark parameter required"

//axhopt_typed_parameter
{rbtga_hallmark}
. *Validate Hallmark Format*:
.. {rbbc_require}
`«INPUT_HALLMARK»` matches pattern `[cbg]\d{12}-r\d{12}` (mode prefix + dual timestamp)
.. {rbbc_fatal}
if malformed: "Hallmark format invalid"

//axhos_step
. *Authenticate as
{rbtr_director}*:
.. Execute
{rbtoe_director_authenticate}
.. {rbbc_store}
`«DIRECTOR_TOKEN»` for subsequent API calls

//axhos_step
. *Construct Hallmark Subtree*:
.. {rbbc_store}
`«SUBTREE»` as `«HALLMARKS_ROOT»/«INPUT_HALLMARK»/` where `«HALLMARKS_ROOT»` derives from `RBGL_HALLMARKS_ROOT` (kindled to `{gar_hallmarks_namespace}`)
.. {rbbc_store}
`«REPO_BASE»` as the GAR REST API repository base (`projects/«PROJECT»/locations/«REGION»/repositories/«REPOSITORY»`)

//axhos_step
. *Enumerate Packages Under Subtree*:
.. {rbbc_call}
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories.packages/list[GAR REST API: packages.list] with:
... method: `GET`
... endpoint: `«REPO_BASE»/packages?pageSize=1000`
... header: `Authorization: Bearer «DIRECTOR_TOKEN»`
.. {rbbc_require}
HTTP 200 response
.. Extract `.packages[].name` from the response, strip the resource prefix, and URL-decode `%2F` back to `/`
.. {rbbc_store}
`«ARK_PACKAGES»` as the list of decoded package names whose prefix equals `«SUBTREE»`
.. {rbbc_fatal}
if `«ARK_PACKAGES»` is empty: "No packages found under «SUBTREE» — hallmark not present in registry"

//axhos_step
. *Confirm Abjuration*:
.. {rbbc_show}
"Will abjure «COUNT» packages under «SUBTREE»:" followed by each entry in `«ARK_PACKAGES»`
.. {rbbc_require}
user types `yes` to confirm
.. {rbbc_fatal}
if not confirmed: "prompt not confirmed."

//axhos_step
. *Dispatch Cloud Delete of the Enumerated Set*:
.. {rbbc_call}
the cloud-dispatched delete backbone with the whole `«ARK_PACKAGES»` list: a single {rbtr_director}-run Cloud Build whose in-pool step loops the list — for each `«PACKAGE»`, link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories.packages/delete[GAR REST API: packages.delete], poll the returned long-running operation to terminal, then verify package absence
.. {rbbc_require}
the build reaches terminal SUCCESS — one build per abjure, never one build per package; the build's outcome IS the delete outcome (no host-issued DELETE survives)
.. {rbbc_fatal}
if the build fails: the in-pool step has printed the failing package and verdict to the build log

//axhoot_typed_output
{rbtga_ark}
. *Display Results*:
.. {rbbc_show}
"Hallmark abjured: «INPUT_HALLMARK» («COUNT» packages)"

//axhoc_completion
Procedure completes successfully when the cloud-dispatched delete build reaches terminal SUCCESS — every package discovered beneath the hallmark subtree verified absent from the registry by the in-pool step — and the abjuration confirmation is displayed.
The ark is now formally renounced and no longer part of the repository corpus.
Because enumeration drives the delete set, there is no separate existence check and no orphan-detection branch: whatever lives in the subtree is removed, and an empty subtree is itself the "not present" signal.
Each enumerated package is deleted in-pool by **convergence**: GAR refuses to delete a child manifest while its parent index still exists (FAILED_PRECONDITION, "manifest is referenced by parent manifests"), so each round the step fires a delete at every remaining version (force=true) and at the package shell — parents free their children, which a later round deletes — and polls the package GET until it 404s, the per-round "referenced by parent" skips being expected rather than errors.
The delete build runs as {rbtr_director} (which already holds repoAdmin/delete); Mason stays writer-only.
Posture and the convergence rationale live in RBSCB.
