{rbtgo_foedus_descry}

NOTE: *Palpate-adjacent, but not a chaining consumer.* Descry reads a named
{rbtf_foedus}'s
provider presence under the manor pool for a transient inspection only and writes no durable config — the read-side
posture of a
{rbch_palpate}.
But it resolves no chained fact: its subject is an express foedus identity, never a value resolved
express-or-chain from a prior tabtarget's capture. So it is *not* a
{rbch_palpate}
member — it carries none of the depth-1 no-relay machinery, and it rejects into its *own* precision
band, never the chaining band that
{rbch_palpate}
and
{rbch_enchase}
share. The one discipline it does keep is the palpate guardrail: a read must never be extended to
write durable config.

NOTE: *Subject — any standing foedus, not only the active one.* The inspected foedus is named by its
`rbef_`-sprued subdirectory identity (the operation's folio) in the
{rbrf_regime}
library, which need not be the one
{rbrr_active_foedus}
currently selects. Where
{rbtf_canvass}
enumerates the foedera the Manor holds and
{rbrr_active_foedus}
selects the active one, descry inspects the presence of a single named foedus's provider under the manor pool.

NOTE: *Fixture role — the probe atom.* Descry is the precondition probe the foedus reuse-or-establish
fixture composes first; it stays a bare read, never folded into a fat reuse-or-create verb. The
composition contract is stated at
{rbtgo_foedus_instate}.

The ordered shape (do not read REST calls, function names, or a token actor into it
as fixed):

//axhos_step
. *Resolve the named foedus's provider*: read the named foedus subdirectory's
{rbrf_regime}
to obtain the
{rbrf_provider_id}
it configures, and the manor pool id ({rbrw_workforce_pool_id}) it hangs under. A name resolving to
no foedus subdirectory is fatal in descry's own band, naming the absent foedus.

//axhos_step
. *Read the provider's presence* from the Manor — whether the manor pool carries this foedus's provider
under it. The read is manor-scoped — a
{rbtf_canvass}-adjacent
interrogation of one provider rather than all — mutates nothing, and yields the presence verdict.

//axhos_step
. *Confirm the manor pool coordinates* (the folded drift guard): the same read confirms the live pool
under {rbrw_org_id} carries the id {rbrw_workforce_pool_id} — a mismatch (the RB-marked pool standing
under a different id) is a coordinate-drift deficit, surfaced like any other. This is descry's half of
the {rbrw_regime} sync guard; the manor-setup finisher's ensure-exists is the other.

//axhoc_completion
Successful execution reports the named foedus's provider presence — present, or a named deficit
(provider-absent, or a manor-pool coordinate mismatch) — for transient use by a caller or a fixture, having
written no durable config and mutated nothing in the Manor or the regime.
A broken read or an unresolvable foedus name rejects in descry's own precision band, distinct from
the chaining band and from
{rbtgo_foedus_instate}'s:
the two co-occur in the fixture's reuse-or-establish spawn path, so the band allocation rule forbids
them a shared code.
