[[rbshr_horizon_roadmap]]
==== Horizon Roadmap

Defined-but-unscoped future work across Recipe Bottle.
Each item has a rationale (why it matters), a trigger (when to revisit), and a reference (where the evidence lives).

Items here are not planned for any heat.
When an item becomes actionable, it moves into a heat as paces with proper dockets.
This is the single collection point for big changes — individual spec subdocuments should cross-reference here rather than maintaining local "Future Work" sections.

**SA numeric ID for member references.**
Google recommends using a
{giam_service_account}'s
numeric ID rather than its email address when referencing it immediately after creation, to reduce propagation delay.
Recipe Bottle uses email addresses throughout, which is why propagation retry
(<<rbscip_trade_study>>
backoff profile) is necessary.
Switching to numeric ID could reduce retry frequency but would require threading the numeric ID from SA creation responses through all call chains that construct `serviceAccount:` member strings.
Requires verification that `setIamPolicy` accepts numeric ID in member strings.
_Trigger_: if propagation retries become a significant wall-clock cost in depot creation or governor reset.
_Reference_:
<<rbscig_iam_grant_contracts>>
(SA propagation section, evidence table).

**Read-back verification universalization.**
Currently only
{rbtoe_iam_grant_project}
performs post-write verification (re-reading policy to confirm the granted role is observable).
{rbtoe_iam_grant_repo},
{rbtoe_iam_grant_sa},
and
{rbtoe_iam_grant_bucket}
trust the 200 from `setIamPolicy`.
Adding verification universally would provide stronger guarantees but adds an HTTP round-trip and a new timeout window per grant.
_Trigger_: after etag convergence (\u20a2AsAAF) has been validated in production and we have confidence in the corrected grant contracts.
_Reference_:
<<rbscig_iam_grant_contracts>>
(Convergence Analysis section).

**SA preflight universalization.**
Currently only
{rbtoe_iam_grant_sa}
performs a pre-flight existence check on the target
{giam_service_account}.
Applying it to all grant types when the member is a service account requires conditional logic and changes behavior for non-SA members.
_Trigger_: if grant failures due to non-existent members become a diagnostic problem.
_Reference_:
<<rbscig_iam_grant_contracts>>
(Convergence Analysis section).

**Podman as first-class deployment target.**
Recipe Bottle currently deploys to Podman on workstations via the bottle lifecycle (RBSBS, RBSBR, RBSBL).
Podman support is functional but not hardened — the sentry container, network isolation, and eBPF enforcement assume Linux Podman with rootful access.
Expanding to rootless Podman, Podman on macOS (via podman-machine), or alternative container runtimes would require revisiting the security model and network architecture.
_Trigger_: user demand for rootless deployment or macOS-native workflows beyond the current Podman VM approach.
_Reference_: RBSPV-PodmanVmSupplyChain.adoc (current VM supply chain);
RBSBS, RBSBR, RBSBL (bottle lifecycle specs).

**`podman-compose` lacks `--wait` for `compose up`.**
Docker Compose `--wait` (v2.1.1+) blocks until all container health checks pass, which `rbob_charge` relies on to guarantee crucible readiness before returning.
`podman-compose` does not implement `--wait` — the command returns immediately after scheduling containers, creating a race between charge completion and the active-check assertion in theurge.
Without `--wait`, Podman integration will need an alternative readiness gate (polling loop, explicit health-check query, or `podman wait --condition=healthy`).
_Trigger_: Podman first-class deployment (above).
_Reference_: containers/podman-compose#710 (June 2023, open);
containers/podman-compose#1329 (October 2025, open).

**Revised enshrine — universal capture verb under Lode recipe noun.**
Today's narrow `enshrine` mirrors only upstream OCI base images.
Three additional capture needs share the same gesture (pull from upstream, SLSA-wrap, store in GAR): build-time tool images (today's `mirror`/`reliquary` machinery), the WSL substrate distro that hosts the Windows workload, and podman VM disk images.
The revision generalizes `enshrine` into a universal verb operating on a single recipe noun — **Lode**, parallel to {rbtga_vessel} on the built side — with a kind enum that gates pipeline behavior: `base`, `tool`, `reliquary`, `wsl`, `podvm-wsl`, `podvm-native`.
Today's `reliquary` demotes from top-level noun to a kind enum slot — the date-cohort semantic survives, the cult-vocabulary weight drops.
Today's `mirror` verb retires entirely;
absorbed into enshrine on the `tool` and `reliquary` kinds.
Lode .env carries author intent only — no resolved FQIN field — mirroring RBRV's ORIGIN/ANCHOR pattern (kindle code computes upstream coordinates from kind primitives at module-startup).
The `wsl` kind delivers a project-controlled WSL rootfs tar that replaces the Microsoft-Store-appx-mutable distribution today's caparison-w depends on;
the `podvm-*` kinds collapse `rbv_PodmanVM.sh`'s existing GHCR mirror into GAR under the unified verb.
_Trigger_: when Windows host workload requires a project-controlled WSL distro, when podman VM supply chain needs project-owned bytes in GAR, or when the `mirror`/`reliquary` machinery accumulates enough friction to motivate consolidation.
_Reference_: RBSAE-ark_enshrine.adoc (current narrow-scope spec);
RBSRV-RegimeVessel.adoc (ORIGIN/ANCHOR precedent, kind-enum-gated variable groups pattern);
`Tools/rbk/vov_veiled/FUTURE/rbv_PodmanVM.sh` (existing podman-vm fan-out machinery).

**Windows host workstation support — shell transport hardening.**
Recipe Bottle commands invoked on a Windows workstation traverse a multi-leg shell transport stack — cmd.exe, PowerShell, Cygwin's bash.exe, and wsl.exe — each with its own escaping, heredoc, and quoting discipline.
The bash-console body authoring patterns that hold trivially on Linux/macOS need explicit per-leg discipline on Windows;
WSG-WindowsScriptingGuide.md catalogs the empirical rules.
Today's tabtargets and workbench bodies have not been hardened against this stack, and the test infrastructure lacks a Windows fundus signal.
Release-quality Windows host support requires both per-tabtarget audit for transport-stack safety and a green test suite from a Windows fundus, on top of the project-controlled WSL distro delivered by the revised-enshrine `wsl` kind.
_Trigger_: when Recipe Bottle moves toward a release claiming Windows host support;
naturally paired with the revised-enshrine entry above.
_Reference_: WSG-WindowsScriptingGuide.md (transport-stack discipline);
BUSJCW-CaparisonWindows.adoc, BUSJGW-GarrisonWsl.adoc, BUSJIW-InvigilateWindows.adoc (Windows workload-side specs).

**Hairpin peer access between crucibles.**
Bottled containers cannot currently reach each other's published entry ports.
Each bottle is isolated behind its own sentry on a separate enclave network.
The traffic path is feasible: Bottle A egresses through Sentry A's transit bridge, the host kernel routes to Sentry B's transit bridge, and Sentry B's existing entry path (DNAT + RBM-FORWARD) accepts the connection unchanged.
The security model is preserved — the sentry remains the sole enforcement point, with peer access as a new category of allowed destination alongside entry (inbound) and uplink (outbound internet).
Nameplate model: two new variables (`RBRN_PEER_ACCESS_MODE` disabled|allowlist, gated `RBRN_PEER_ALLOWED_MONIKERS` as comma-separated moniker list).
No global mode — peer access must always be explicit.
Implementation touches three areas: nameplate kindle/validate/render, a peer-access iptables step in security_config, and host-side peer IP discovery via container inspection.
Audit gains a cross-instance check: each named peer must exist and have entry enabled.
_Trigger_: when a use case requires bottle-to-bottle communication (e.g., microservice composition, service mesh patterns).
_Reference_: prior design conversation capture (paddock).

**Hoard: external artifact mirror in retention-locked GCS.**
A
{rbtgi_depot}
citizen alongside the
{rbtga_ark}
that mirrors external content-addressed corpora — canonical case: Hugging Face model checkpoints.
GCS archive is ~80× cheaper per byte than GAR and supports bucket-level retention locks for compliance-grade immutability that GAR does not offer.
Ingress reuses upstream SHA256 addressing (shallow git + LFS pointer resolution, SHA256-keyed GCS objects with natural dedup across revisions);
Cloud Build streams so egress stays Google-side;
auth path reuses the existing JWT/OAuth bearer curl stack — no new tooling dependency.
+
Intended consumption: just-in-time workstation mount from the retention-locked bucket into the
{at_bottle_container}
at charge time, rather than baking models into
{at_bottle_image_s}.
Keeps images lean, enables model swap without rebuild, keeps the retention property where it naturally lives.
Tradeoff: provenance becomes a
{at_nameplate}-declared
hoard digest recorded at charge time rather than a SLSA-attested image layer — weaker by one link.
_Trigger_: first concrete need to run LM inference inside a
{at_bottle_container}
against a specific HF checkpoint with retention guarantees.
_Reference_: this roadmap entry;
design conversation pending heat.

**CIDR allowlist granularity vs CDN-hosted domains.**
The sentry's iptables CIDR rules accept all protocols and all destination ports to allowed IP ranges.
This is intentional: the spec defines
{rbrn_uplink_access_mode_allowlist}
as "IP access restricted to allowed CIDR ranges" — not port-restricted.
The DNS allowlist
({rbrn_uplink_allowed_domains})
provides name-level precision;
the CIDR allowlist provides IP-level gating.
These two layers are independent and reinforce each other.
+
The weakness emerges when allowed domains are CDN-hosted.
IANA's example.com (RFC 2606) migrated to Cloudflare, requiring CIDR expansion from `93.184.216.0/24` to include `104.16.0.0/12` — a /12 covering ~1M IPs shared with a large fraction of Cloudflare-fronted sites.
The DNS allowlist remains precise (two domains), but the CIDR backstop is now porous: any destination within the Cloudflare range is reachable regardless of domain.
A compromised or attacker-controlled site sharing the CDN range would be reachable from the bottle.
+
Three potential mitigations, in order of increasing ambition:
+
. **Accept as-is with documentation.**
  The DNS allowlist is the primary gate;
  the CIDR is a coarse backstop.
  Document the known weakness for broad CIDRs.
  Suitable when the allowed domains are trusted and the CDN range is acceptable collateral exposure.
. **Dynamic CIDR from DNS resolution.**
  Have dnsmasq feed resolved IPs into iptables at resolution time, so the allowed set tracks actual DNS responses rather than static configuration.
  Eliminates CDN range sprawl.
  Requires new sentry infrastructure (dnsmasq hook scripts or ipset integration).
. **TLS SNI enforcement.**
  Inspect TLS ClientHello for the SNI field and enforce that it matches the DNS allowlist.
  Closes the CDN gap at the application layer without dynamic iptables.
  Requires a transparent TLS proxy or eBPF-based SNI extraction in the sentry — a different order of complexity.
+
_Trigger_: when allowed domains require CIDRs broader than /24, or when the threat model requires preventing lateral reachability within a CDN range.
_Reference_: rbjs_sentry.sh (Phase 3 CIDR rules);
the tadmor nameplate's {rbrn_regime} assignment file (current CIDR configuration).

**Encrypted tunnel conduit (WireGuard in sentry).**
Add WireGuard to the sentry image so the sentry terminates an encrypted tunnel directly to a VPC hosting PrivateLink endpoints for cloud services.
The bottle's traffic routes through a `wg0` interface on the sentry — one stable VPC CIDR replaces floating cloud-provider IP ranges.
WireGuard is ~400KB on Alpine, uses static keypair exchange (no PKI/CA), and fits naturally into the sentry's existing privileged-container, explicit-routing architecture.
New nameplate fields: tunnel mode, peer endpoint, public keys, allowed IPs.
VPC side requires a small WireGuard peer instance and PrivateLink endpoints for target services.
_Trigger_: when a bottle needs to reach a cloud service that supports PrivateLink (AWS Bedrock is the known first case; survey should include Vertex AI, Azure OpenAI, GitLab Dedicated, SageMaker, and others as they emerge).
_Reference_: `Memos/memo-20260415-crucible-conduit-architecture.md` (full design exploration, approach comparison, service compatibility survey).

**Cloud service conduit nameplate.**
A nameplate configuration pattern for bottles that access cloud services at runtime.
Near-term: allowlist-only nameplate targeting specific service CIDRs and domains (works today with existing sentry machinery, no development needed).
Longer-term: tunnel-mode nameplate using WireGuard conduit (above).
AWS Bedrock is the known requirement;
the pattern should be researched broadly across cloud AI inference services, managed Git hosting, and other PrivateLink-capable endpoints.
_Trigger_: immediate — Bedrock access is a current need.
Allowlist-only nameplate can be created now;
tunnel-mode nameplate follows WireGuard sentry work.
_Reference_: `Memos/memo-20260415-crucible-conduit-architecture.md` (approach A for allowlist-only, approach C for tunnel mode).

**Bottle credential custody.**
Move the service secrets a bottle workload uses — cloud API keys, IAM keys, SSH keys — off the operator's workstation and into the bottle itself, injected at charge time via nameplate/regime configuration rather than baked into the bottle image (a published image must never carry a live secret).
The workstation starts the crucible but never holds the service secret; its only credential becomes permission to charge.
This blocks credential theft from a compromised workstation — the one threat the sentry allowlist and the encrypted tunnel do not already close — by shrinking the secret's resting place to the contained bottle.
The existing RBRA file regime-path pattern provides the model.
Orthogonal to but naturally paired with the encrypted tunnel conduit: the tunnel grants the bottle network reach to a cloud service and the sentry custodies the tunnel key, while this item custodies the service secret inside the bottle — network access and the credential to use it configured together.
_Trigger_: alongside the conduit nameplate work.
_Reference_: `Memos/memo-20260415-crucible-conduit-architecture.md` (Credential Confinement section; the `Credential theft from workstation` row of the threat-model table).

NOTE: For SaaS services without PrivateLink (GitHub.com, GitLab.com), the sentry CIDR/domain allowlist + TLS remains the realistic ceiling.
The sentry still provides the core value: the bottle cannot reach anything the sentry doesn't allow, and the workstation is cut out of the data path.
The tunnel adds defense-in-depth only for PrivateLink-capable services.
Cross-reference: the existing "CIDR allowlist granularity vs CDN-hosted domains" item (Network Security Model) remains relevant for SaaS-only services where the tunnel cannot help.

**Cross-cloud architectural stance: asymmetric sovereignty.**
Recipe Bottle governs identities and infrastructure only in GCP.
AWS-side presence is shaped as delivered artifacts and policy templates, not owned principals — no AWS-Director, no AWS-Mason, no AWS-Retriever.
If Recipe Bottle ever needs to *build* in AWS rather than deliver to it, that would be a deliberate architectural decision triggering a full parallel role hierarchy, not an incremental drift.
Vocabulary used in related entries — embassy, envoy, bullion, entrust, accredit, recall, chantry — is tentative working language from a design conversation; concepts are firmer than the words.
_Trigger_: when AWS-side work moves into a heat with real scope.
_Reference_: this roadmap entry; design conversation pending heat.

**Embassy — AWS-side delivery premises.**
An S3 bucket (with Object Lock in Compliance mode for retention), an IAM role with a trust policy accepting the envoy's federated identity, and Recipe-Bottle-shipped IAM policy shapes that the AWS consumer attaches to whatever principal consumes delivered artifacts (EC2 role, ECS task role, Bedrock execution role, etc.).
Provisioned by the payor via operations congruent with
{rbtgi_depot}
creation — the AWS consumer's embassy is the sovereign-setup equivalent of the Manor's depot.
Recipe Bottle ships provisioning templates;
the AWS consumer owns the resulting resources.
_Trigger_: first concrete need to deliver artifacts to an AWS workload.
_Reference_: this roadmap entry;
design conversation pending heat.

**Envoy — GCP-side cloud-native cross-post actor.**
A
{giam_service_account}
in the depot, sibling to existing cloud-native SAs, whose sole purpose is executing cross-cloud pushes to an accredited embassy.
The envoy holds no long-lived AWS credentials;
it federates to AWS via OIDC Workload Identity Federation, exchanging short-lived GCP tokens for short-lived AWS STS credentials at operation time.
Preserves credential locality: Director holds only GCP credentials on the workstation, envoy holds only federation configuration in GCP, and AWS admin credentials never enter Recipe Bottle's custody.
_Trigger_: embassy entry above.
_Reference_: this roadmap entry.

**Bullion — AWS-side content-addressed artifact.**
Recipe Bottle's delivered artifact in an embassy bucket: a content-addressed filesystem image (squashfs/erofs, optionally with dm-verity) with Object Lock applied for compliance-grade retention.
A bullion may carry a vessel image payload (the "Director pushes a vessel to AWS" motivator), model weights, or any other content-addressed corpus.
Sibling-but-distinct from the hoard: hoards stay GCS-native and depot-resident, preserving the existing provenance story;
bullions are the new AWS-side citizen with S3-native retention semantics.
Provenance transports as naming-convention sibling objects (attestation JSON adjacent to the image blob), not as OCI referring artifacts, because the bullion is not an OCI artifact.
_Trigger_: embassy entry above.
_Reference_: this roadmap entry;
Artifact Preservation section (hoard entry) for the GCP sibling concept.

**Accredit / Recall — governor operations for embassy binding lifecycle.**
Accredit: governor configures the envoy's OIDC federation against an embassy's IAM role trust policy, and records the binding (embassy name, AWS account, region, role ARN, bucket name) in depot state.
Recall: governor removes the federation configuration and prunes the binding.
Follows the existing governor verb register (mantles, charters, knights, forfeits) with diplomatic-register verbs appropriate for foreign-court relationships.
A single depot may be accredited to multiple embassies.
_Trigger_: embassy entry above.
_Reference_: this roadmap entry.

**Entrust — director-triggered cross-post operation.**
Director invokes entrust on a specific hallmark (or other deliverable) with a target embassy name.
The envoy executes the cross-post from within Cloud Build: federates to the embassy's IAM role, streams bytes to S3, applies Object Lock retention.
Director holds no AWS credentials;
failure modes surface through the existing Cloud Build event stream.
_Trigger_: embassy entry above.
_Reference_: this roadmap entry.

**Chantry — Bedrock-hosted inference endpoint.**
Architecturally distinct from the embassy/bullion pattern.
A chantry is a Bedrock-registered custom model (or a native Bedrock foundation model) invoked by a
{at_bottle_container}
over the crucible conduit architecture — not an artifact Recipe Bottle delivers, but a foreign-sovereign service a bottle consumes.
Weights may originate as a bullion (entrusted to an embassy, then consumed by Bedrock Custom Model Import on the AWS side), but the chantry itself is the registered-and-invocable endpoint, not the weight storage.
Pairs with the "Cloud service conduit nameplate" entry for the invocation path.
_Trigger_: first concrete Bedrock inference requirement;
near-term per the Crucible Conduit Architecture section above.
_Reference_: this roadmap entry;
`Memos/memo-20260415-crucible-conduit-architecture.md`;
Crucible Conduit Architecture section above.

**Egress lockdown — IMPLEMENTED.**
Dual-pool architecture: tether pool (public egress) and airgap pool (`NO_PUBLIC_EGRESS`).
Operation-level routing: inscribe/enshrine use tether, about/vouch always airgap, conjure/bind per vessel `RBRV_EGRESS_MODE`.
Reliquaries (tool images) and anchors (base images) provide universal GAR-sourced dependencies.
_Reference_: RBSCB-CloudBuildPosture.adoc (Implemented: Egress Lockdown via Dual Pools).

**VPC Service Controls perimeter.**
Wrap depot project in VPC-SC perimeter restricting API access.
Requires egress lockdown (now implemented) as prerequisite, org-level admin for perimeter creation, and verification of every GCP API call in the pipeline.
_Trigger_: organizational security policy mandating VPC-SC, or data exfiltration risk assessment.
_Reference_: RBSCB-CloudBuildPosture.adoc (Potential Hardening: VPC Service Controls).
Cross-reference: if a VPC is stood up for the crucible conduit architecture (above), design the VPC-SC perimeter to serve both Cloud Build egress and bottle conduit consumers.

**Cosign image signing.**
Cryptographic signatures on images, verifiable by downstream consumers.
KMS-backed keyed signing preferred for airgap-pool compatibility.
Currently redundant given CB-native SLSA v1.0 Build Level 3 attestation.
_Trigger_: external image distribution, trust verification needed outside GCP, or builder strategy expanding beyond Google Cloud Build.
_Reference_: RBSCB-CloudBuildPosture.adoc (Deferred Items: Cosign).

**Binary Authorization.**
Deploy-time policy enforcement (only attested images can be deployed).
Technically achievable now with CB-native SLSA Level 3.
Not beneficial until deployment targets include GKE or Cloud Run.
_Trigger_: deployment targets change from Podman workstations to managed container platforms.
_Reference_: RBSCB-CloudBuildPosture.adoc (Deferred Items: Binary Authorization).

**Dockerfile base image digest validation.**
Builder step images are digest-pinned, but user-authored Dockerfiles in vessel build contexts are not validated.
A vessel with `FROM ubuntu:22.04` uses a mutable tag.
Constrains vessel authors (policy decision).
_Trigger_: vessel ecosystem matures beyond internal-only use.
_Reference_: RBSCB-CloudBuildPosture.adoc (Deferred Items: Dockerfile base image).

**Reproducibility verification.**
Periodic independent rebuilds compared by digest.
Requires reproducible toolchains — a vessel-author discipline, not a pipeline control.
_Trigger_: vessel ecosystem maturity and external distribution.
_Reference_: RBSCB-CloudBuildPosture.adoc (Deferred Items: Reproducibility).

**Operator federation — external-IdP identity for depot participants.**
Recipe Bottle grants
{rbtgi_depot}
capabilities to identities the depot's IAM recognizes.
The shipping model is the keyfile tier: each operator role holds a service-account key file (the
{rbtgi_rbra_file})
on disk, possession of the key is the identity, and no GCP organization is required.
Operator federation is the heavyweight successor tier for installations that cannot accept an operator secret on disk.
The depot owner configures the
{rbtgi_depot}
to trust an external OIDC/SAML identity provider (any compliant issuer — self-hosted Dex/Keycloak, or GCP Identity Platform as an in-house IdP), then grants capabilities to those federated `principal://` subjects: keyless, no Google accounts for participants, revocation central at the depot.
The
{rbtgi_depot}
remains the home of the participant roster (the role-to-identity mapping); the IdP owns only identity proofing.
Org-only — federation pools are organization-level (Workforce Identity Federation, the human-identity product), so a no-org
{rbtr_payor}
cannot use this tier; the organization requirement is the line between the free keyfile tier and the federation tier.
A qualifying organization is free via Cloud Identity once its owner verifies control of a DNS domain, so the true prerequisite is that the operator controls a domain — not that they pay Google.
The Google-side credential path is curl-native — an STS token exchange (RFC 8693), no gcloud and no added stack dependency; only the IdP-auth leg (device flow where the issuer supports it) varies by issuer.
Distinct from the envoy's egress federation (GCP-to-AWS delivery, the machine-identity Workload product): this is
{at_operator}
*access* identity arriving the other way.
_Trigger_: first concrete need from an organization that cannot accept on-disk operator keyfiles, or when the keyfile tier's secret-distribution and rotation burden materially blocks adoption rather than being theoretically frictionful.
