==== Decision

Cloud Build configuration files for trigger-based builds use `cloudbuild.json` format,
generated programmatically via `jq`.
YAML is not used.

==== Rationale

JSON is the Cloud Build REST API's native wire format.
The `projects.builds.create` and `projects.triggers.run` endpoints accept JSON request bodies
modeled on the API's `Build` resource.
YAML is a file-authoring convenience that gets parsed into the same `Build` structure before submission.

Programmatic generation via `jq` produces correct JSON directly — no YAML indentation
fragility, no multiline string quoting hazards, no dependency on `yq` or custom emitters.
The stitch function (`zrbf_stitch_build_json`) already generates JSON;
converting to YAML would add complexity with zero functional benefit.

==== What Dies

* Any assumption that `cloudbuild.yaml` is required for triggers or SLSA provenance
* The linked term `rbtgr_build_yaml` — replaced by
{rbtgr_build_json}
* YAML-specific concerns: multiline string quoting, indentation sensitivity, `$$` escaping conventions

==== What Survives

* `zrbf_stitch_build_json()` — continues to emit JSON via `jq`, adapted for trigger structure
* `rbgjb/*.sh` step scripts — inlined into JSON `args` arrays at inscribe time
* All `_RBGY_*` substitution variables — baked into JSON `substitutions` object
* OCI Layout Bridge pattern — unchanged

==== Provenance Note

SLSA v1.0 provenance records `buildConfigSource.path` which will contain `cloudbuild.json`.
This is a cosmetic difference from `cloudbuild.yaml` — the structured provenance fields
(build inputs, parameters, outputs) are identical regardless of config file format.

* Google docs consistently present YAML examples by convention, not by requirement
* The `gcloud builds submit` CLI internally converts any config format to JSON for API submission

==== Capture Composition Contract (rblds_ spine)

This section governs the host-composed `projects.builds.create` path for Lode
captures — distinct from the trigger-autodetect path the Decision above covers.
A Lode capture is composed on the host by the capture-assembly spine
(`rblds_Spine.sh`) from two data inputs handed by a per-kind body (e.g. bole's
`zrbld_ensconce_submit`): an ordered _recipe_ of step rows and an opaque
_substitutions_ blob. The spine carries no per-kind branch and no
kind-registration mechanism — a new Lode kind is a new body file the spine
consumes as data, never a conditional inside the spine.

===== Spine/body boundary

[cols="1,3",options="header"]
|===
|Owner |Carries

|Spine (`rblds_`)
|The step-composition loop (recipe rows -> `steps[]`); the Build-resource
envelope shape (`serviceAccount`, `options.automapSubstitutions`,
`options.logging`, `options.pool`, `timeout`); submit + build-id persist +
console link; the poll via `zrbfc_wait_build_completion`; and the
`buildStepOutputs[N]` base64-decode. The capture-domain constants — the mason
capture SA, the `TETHER` pool, the regime build timeout — are spine-owned,
because every Lode capture fetches upstream bytes as mason over the network.

|Body (`rbld«kind»_`)
|The recipe rows (pre-resolved); the substitutions blob; the poll ceiling (the
one kind-flavored build knob); and the interpretation of the decoded step
output (the per-member envelope fan-out).
|===

The only kind-flavored build knob the body passes is the poll ceiling (ensconce
borrows the enshrine ceiling). `serviceAccount`/`pool`/`timeout` are NOT body
parameters: they are capture-domain invariants the spine owns. A future kind
that needed a different pool or SA would change the spine's capture-domain
constants deliberately, not slip past as recipe data.

===== Recipe row format

A recipe is an ordered list of `|`-delimited step rows. The `|` delimiter is
load-bearing: builder image refs carry colons (tags and `sha256:` digests), so a
colon delimiter would be ambiguous.

----
script_path | builder_image | id | entrypoint
----

[cols="1,3",options="header"]
|===
|Field |Meaning

|`script_path`   |Absolute path to the step script, _pre-resolved by the body_. The spine owns no steps-directory knowledge.
|`builder_image` |The step's builder image ref (a reliquary tool image).
|`id`            |The Cloud Build step id.
|`entrypoint`    |`bash` \| `sh` \| `python3` — selects the composed shebang line. Any other value fails composition.
|===

The spine reads each script's body (shebang stripped), expands any
`#@rbgjs_include` markers against the shared snippet library (see _Composed-snippet
library_ below), prepends the shebang the entrypoint selects, and inlines the
result as the step's `script` field. This is the same shape the made-side
`rbfca_` assemblers use; the capture spine is the canonical single copy for the
fetched side.

===== Substitutions blob — opaque

The body composes the substitutions object (the `_RBGL_*` keys) into a file; the
spine slots it into the Build resource's `substitutions` field and reads no key
from it. The spine adds exactly one key of its own — `_RBGL_GIT_COMMIT`, the
dispatching repo's HEAD commit — dispatch provenance it owns the same way it
owns the pool and timeout; the shared vouch-push step splices it into every
envelope as `rblv_git_commit`, and recipes without that step carry the key
unread. `options.automapSubstitutions` maps the keys into each step's
environment. The spine's opacity to the body's keys is what keeps it
registration-agnostic — a new kind authors a different blob without touching the
spine.

Invariant (RBSCB): substitutions and the {rbtgr_workspace} carry non-secret data only;
secrets are step-local and re-minted.

===== Extract contract

The spine's extract decodes one step's `buildStepOutputs[N]` slot — base64 to a
destination file — where `N` is the only generic parameter. The decoded JSON's
shape (for bole: `slot_1`..`slot_N` member envelopes, each a stamp + vouch) is
the body's knowledge; the spine is member-count-agnostic. This keeps the
N-member envelope axis out of the spine.

===== Substitution-coverage check (enforced by zrbld_spine_validate)

A composed step reads two register channels:

* _Substitution registers_ — `_RBGL_*` keys present in the substitutions blob,
  automapped into every step's environment.
* _/workspace registers_ — non-secret files a step writes for a later step,
  named by path relative to `/workspace`.

The two channels are validated differently, because they differ in what can be
recovered safely from the step source. The substitution channel — where an
undefined reference is both silent (it expands to empty in-build) and expensive
to surface (it costs the whole cook) — is checked by derivation at dispatch; the
`/workspace` channel is left to the steps' own in-build guards.

*Substitution channel — derived, not declared.* A step's substitution _requires_
is exactly the set of `_RBGL_*` names its body references — the references ARE the
requires, so nothing separate is declared and nothing can drift from the code. At
dispatch `zrbld_spine_validate` scans each step's include-expanded body for
`_RBGL_*` tokens on non-comment lines and fails before submit at the first token
absent from the substitutions blob's keys, naming the offending register. Coverage
is _flat_: substitutions automap into every step, so a register is satisfied by
its presence in the blob alone — there is no recipe order and no cross-step
`provides`. The scan runs on the _include-expanded_ body; composed snippets carry
no `_RBGL_*` references (they speak the shell-var channel — see below), so only the
kind's own substitution reads are recovered.

*Workspace channel — guarded in-step, dispatch-validation deferred.* A step's
`/workspace` reads and writes are not safely recoverable by a static scan: the
paths are templated by runtime loop variables (e.g. `lode_${STAMP}_vouch.json`)
and read-vs-write classification in shell (redirections, `test -f`, `cp` of a
var-held path) is exactly what a scanner gets wrong. The live recipes already
guard their own `/workspace` preconditions at runtime — the vouch-push step opens
with `test -f /workspace/lode_stamps.txt || FATAL "step 01 must run first"` — so
the channel is crash-fast in-build. A dispatch-time `/workspace` declaration is
added only when a future kind needs one, not authored speculatively (generalize
against the instances we have, never stub the ones we don't).

The derivation is shaped by the capture domain's three axes:

* _N-member envelopes_ — the terminal step's single `buildStepOutputs` slot is
  body knowledge (member fan-out lives in the body), so it never enters the
  coverage check.
* _Tool variety_ — a `_RBGL_*` name appears as the same literal token whether a
  step is bash (`${_RBGL_X}`) or `python3` (`os.environ['_RBGL_X']`), so the scan
  is identical across builder image and entrypoint.
* _Opaque-blob payloads_ — the check tests key _presence_ by NAME, never value
  shape, preserving the spine's opacity to the blob.

===== Composed-snippet library (rbgjs)

Cloud Build offers no native step reuse. Where the same in-step logic recurs
across cloud steps, it is authored once as a _snippet_ in the shared cloud-step
home `Tools/rbk/rbgjs/` and stitched into each step at host-assembly time. The
home deliberately breaks the `rbgj«family»` letter scheme — a shared snippet
belongs to no single made-side family, so `s` is the no-family family.

*Mechanism.* A step marks an inclusion point with a directive line

----
#@rbgjs_include «name»
----

(any leading indentation tolerated). At assembly, `zrbfc_expand_includes`
(`rbfcb_BuildHost.sh`, sibling to `zrbfc_write_script_body`, sourced by both the
Lode spine and the made-side `rbfca_` vouch assembler) rewrites the step body,
splicing in `rbgjs-«name».sh` with its leading shebang stripped. A body with no
marker is rewritten unchanged, so an assembler may call the expander for every
step. An unknown `«name»` fails the expander, which fails dispatch before submit
(crash-fast — no silent skip). The snippet shebang exists only so shellcheck
lints the fragment as bash; it never reaches the build.

*Register channel — shell vars, not substitutions.* A snippet reads the plain
upper-case shell vars the kind sets immediately before the marker, and writes
plain shell vars the kind reads after — it is blind to `_RBGx_` substitution
names. This is load-bearing: it is precisely what lets one snippet serve callers
whose substitution namespaces are disjoint (e.g. Lode's `_RBGL_*` and
hallmark-verify's `_RBGV_*`). The substitution-register channel (automapped
`_RBGx_` keys) and the snippet shell-var channel are distinct; a snippet never
crosses into the substitution channel.

*The library (each snippet's requires/provides):*

[cols="1,3",options="header"]
|===
|Snippet |Contract

|`token-fetch`        |_requires_ none; _provides_ `TOKEN` (Mason SA OAuth2, from the GCB metadata server). Sole surviving caller is the immure residency guard (rbgjl09), whose curl blob-HEAD needs a bearer token — curl's floor role, not gcrane's. The registry steps themselves auth ambiently via gcrane (RBSCB); the skopeo bind mirror, formerly a consumer of inline token fetch, was converted to gcrane.
|`gcrane-fingerprint` |_requires_ `ORIGIN`, `RAW_FILE`; _provides_ `SHA`, `FINGERPRINT`. Read the raw manifest (`gcrane manifest`, registry bytes verbatim), take its canonical sha256, derive the sanitized-origin glance fingerprint.
|`buildx-bootstrap`   |_requires_ none; _provides_ the `rb-builder` buildx builder, created-if-absent and selected. Idempotent under retry; run once per step.
|`buildx-push`        |_requires_ `PUSH_URI`, `PUSH_PLATFORMS`, `PUSH_CTX` (and the bootstrap builder); _provides_ the image pushed to `PUSH_URI`. The irreducible FROM-scratch push.
|===

*Share the primitive, not the whole step.* The vouch-push step (Lode `rbgjl02`,
hallmark-verify `rbgjv03`) is NOT one shared step: the two diverge on five
load-bearing axes — push cardinality (loop-N members vs single), destination-URI
template and arity, platform source (literal `linux/amd64` vs a `/workspace`
platforms file), a conditional Dockerfile (conjure verify files), and disjoint
substitution sets. Only the buildx bootstrap and the push invocation are
identical, so only those are shared — as `buildx-bootstrap` + `buildx-push`.
Cardinality, URI construction, platform choice, and context assembly (which JSON,
which Dockerfile) stay in the per-kind step. The same boundary governs
`gcrane-fingerprint`: the read→digest→fingerprint core is identical across
captures, but the first GAR copy is excluded because its destination tag is
itself digest-derived and kind-specific, so it cannot be a clean shared
parameter. The rule the library follows: share the irreducible identical core,
leave kind-specific assembly in the kind.

*In-heat reach (partial dedup, forward payoff).* `buildx-bootstrap`/`buildx-push`
have two live callers (`rbgjl02`, `rbgjv03`). `gcrane-fingerprint` has one live
caller (bole's `rbgjl01`); the enshrine fork (`rbgje01`) keeps its own inline
read→digest→fingerprint copy until that cutover. `token-fetch` has one live
caller (the immure residency guard `rbgjl09`, whose curl blob-HEAD needs a
bearer token) — bole dropped it when it moved to gcrane's ambient auth,
and the bind mirror (skopeo) has been evicted and converted to gcrane.
The snippets are authored once: the library teaches "compose from the library,"
and the dedup completes when the doomed enshrine forks are removed.
