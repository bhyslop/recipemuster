//axhob_operation
{rbtgo_hallmark_tally}

NOTE: This operation enumerates every
{rbtga_ark}
in
{rbtgi_repository}
by listing packages under `hallmarks/` via the GAR REST API implemented by
{gar_service},
classifying each
{rbtga_hallmark}
by basename presence under its subtree.
A single `packages.list` call returns the full corpus; no per-vessel iteration is required because vessel identity is no longer encoded in the GAR path.

The
{rbtr_retriever}
tallies hallmarks:

//axhos_step
. *Authenticate as
{rbtr_retriever}*:
.. Execute
{rbtoe_retriever_authenticate}
.. {rbbc_store}
`«RETRIEVER_TOKEN»` for subsequent API calls

//axhos_step
. *Enumerate Hallmark Packages*:
.. {rbbc_store}
`«HALLMARKS_ROOT»` from `RBGL_HALLMARKS_ROOT` (kindled as `RBRR_CLOUD_PREFIX` + `hallmarks`)
.. {rbbc_store}
`«REPO_BASE»` as the GAR REST API repository base (`projects/«PROJECT»/locations/«REGION»/repositories/«REPOSITORY»`)
.. {rbbc_call}
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories.packages/list[GAR REST API: packages.list] with:
... method: `GET`
... endpoint: `«REPO_BASE»/packages?pageSize=1000`
... header: `Authorization: Bearer «RETRIEVER_TOKEN»`
.. {rbbc_require}
HTTP 200 response
.. Extract `.packages[].name` from the response, strip the resource prefix, and URL-decode `%2F` back to `/`
.. Filter to names whose path under `«HALLMARKS_ROOT»/` is exactly two segments (`<hallmark>/<basename>`); excludes both shallower and deeper entries, and excludes anything outside `hallmarks/` (reliquaries, enshrines)
.. {rbbc_store}
`«PAIRS»` as sorted `<hallmark> <basename>` pairs (one per line). An empty result is a valid outcome — no hallmarks in the repository.

//axhos_step
. *Classify Health Per Hallmark*:
.. Iterate `«PAIRS»` in sort order; each contiguous run of the same hallmark value constitutes one
{rbtga_hallmark}'s
ark inventory.
.. For each hallmark, record whether the basenames
{rbtga_ark_image},
{rbtga_ark_about},
and
{rbtga_ark_vouch}
each appear in its run. Other basenames (`pouch`, `diags`, `attest`) are retained for display but do not affect health classification.
.. Classify:
... `vouched`: `image` + `about` + `vouch` all present
... `pending`: `image` + `about` present, `vouch` absent
... `incomplete`: any other combination (including `image` without `about`, `about` without `image`, or only auxiliary basenames)

//axhoot_typed_output
{rbtga_hallmark}
. *Display Table*:
.. {rbbc_show}
three-column header: `HALLMARK`, `HEALTH`, `BASENAMES`
.. For each hallmark (in sort order):
... {rbbc_show}
hallmark, health state, space-separated list of basenames present under its subtree
.. {rbbc_show}
summary line: `Total hallmarks: N (vouched: V, pending: P, incomplete: I)`

//axhos_step
. *Recommend Next Actions*:
.. If any `pending` hallmarks:
... {rbbc_show}
tabtarget recommendation: {rbtgo_director_vouch} (`rbw-fV`)
.. If any `incomplete` hallmarks:
... {rbbc_show}
tabtarget recommendation: {rbtgo_ark_abjure} (`rbw-fA`) — incomplete typically means the build was interrupted or the about stitch failed

//axhoc_completion
Procedure completes successfully when the packages listing returns and the classified table displays.
No resources are modified.
Vessel identity is not encoded in the GAR path; restoration of a vessel column, if needed, reads `vessel_name` from `about` or `vouch` metadata and is AAL scope — not performed by tally.
