//axhob_operation
{rbtgo_hallmark_tally}

NOTE: This operation enumerates
{rbtga_ark}s
across all
{rbtgi_vessel}
packages in
{rbtgi_repository},
classifying each
{rbtga_hallmark}
by artifact completeness and vouch status.
Uses local regime vessel list for enumeration, queries
{gar_service}
Docker Registry HTTP API V2 for tag data.

The
{rbtr_director}
tallies hallmarks:

//axhos_step
. *Enumerate Vessels*:
.. {rbbc_call}
local regime vessel list via `rbrv_list_capture`
.. {rbbc_store}
`«VESSELS»` as list of vessel monikers

//axhos_step
. *Authenticate as
{rbtr_director}*:
.. Execute
{rbtoe_director_authenticate}
.. {rbbc_store}
`«DIRECTOR_TOKEN»` for subsequent API calls

//axhos_step
. *Fetch Tags Per Vessel*:
.. For each vessel in `«VESSELS»`:
... {rbbc_call}
Docker Registry HTTP API V2 tag list endpoint
... For each tag returned, match against
{rbtga_hallmark}
pattern (`[cbg]\d{12}-r\d{12}`):
.... Tags with `-image` suffix → classify by platform suffix (bare = consumer manifest list, `-amd64`/`-arm64` = per-platform)
.... Tags with `-about` suffix → about artifact present
.... Tags with `-vouch` suffix → vouch artifact present
.... Tags with `-diags` suffix → diags artifact present (conjure-only build diagnostics; parsed for completeness but does not affect health state)
.... Tags with unrecognized patterns → skip (not ark artifacts)
.... Other tags → skip (infrastructure aliases, not ark artifacts)
... {rbbc_store}
unique hallmarks with artifact type and platform data

//axhos_step
. *Evaluate Health Per Hallmark*:
.. For each unique vessel+hallmark pair:
... If `-about` present and `-vouch` present → `vouched`
... If `-about` present and `-vouch` absent → `pending`
... If `-image` present and `-about` absent → `incomplete`
.. {rbbc_store}
health state and platform list per hallmark

//axhoot_typed_output
{rbtga_hallmark}

//axhoot_typed_output axd_tally
{rbcc_fact_consec_infix}
One file per
{rbtga_hallmark}
per vessel.
Consumers check presence via `test -f` on composed filenames.

. *Present Per-Vessel Tables*:
.. For each vessel with hallmarks:
... {rbbc_show}
vessel header
... {rbbc_show}
column headers: `Hallmark`, `Platforms`, `Health`
... For each hallmark (newest first):
.... {rbbc_show}
hallmark timestamp, platform list (or `single`), health state

//axhos_step
. *Recommend Next Actions*:
.. If any `pending` hallmarks:
... {rbbc_show}
tabtarget recommendation for vouch
.. If any `incomplete` hallmarks:
... {rbbc_show}
tabtarget recommendation to abjure and rebuild (incomplete typically means the build was interrupted or about stitch failed)

//axhoc_completion
Procedure completes successfully when all vessels have been queried and the per-vessel tables display hallmark health.
The check reveals the registry's ark inventory and vouch coverage without modifying any resources.
