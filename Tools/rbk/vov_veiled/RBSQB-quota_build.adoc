//axhob_operation
{rbtgo_quota_build}

The
{at_operator}
reviews
{gcb_service}
private pool quota settings for a
{rbtgi_depot_project}
to verify that the *Concurrent Private Pool Build CPUs* quota supports the desired level of parallel build execution.

All depots use a private worker pool.
Machine type is set at pool creation time via
{rbrr_gcb_machine_type}
(a Compute Engine type such as `e2-standard-2`); individual build requests do not specify machine type.
Each
{rbtgo_trigger_build}
invocation consumes vCPUs according to the pool's machine type.
If the pool's vCPU count consumes most of the quota, builds serialize because a second invocation cannot acquire enough CPUs.

The primary control for concurrent build capacity is the pool's machine type (set via
{rbrr_gcb_machine_type}
at pool creation).
Common Compute Engine types and their vCPU counts:

* `e2-standard-2` (2 vCPUs): 5 concurrent builds in a 10-CPU quota
* `e2-highcpu-8` (8 vCPUs): 1 concurrent build in a 10-CPU quota
* `e2-highcpu-32` (32 vCPUs): requires 32+ CPU quota for even 1 build

//axhos_step
. *Navigate to Cloud Build Quotas*:
.. Open Google Cloud Console
.. Select the depot project (`«DEPOT_PROJECT_ID»`) from the project selector
.. Navigate to *IAM & Admin > Quotas & System Limits*
.. In the filter bar, enter `cloudbuild.googleapis.com` to show only
{gcb_service}
quotas
.. Locate the *Concurrent Private Pool Build CPUs* row for the depot region
.. {rbhg_store}
current CPU quota value as `«CPU_QUOTA»`
.. {rbhg_store}
current usage percentage — if near 100% with one build running, the machine type is consuming most of the quota

//axhos_step
. *Evaluate Machine Type vs CPU Quota*:
.. {rbhg_store}
current
{rbrr_gcb_machine_type}
from the regime configuration
.. Compute maximum concurrent builds: `«CPU_QUOTA»` ÷ vCPUs per pool machine type
.. {rbhg_require}
the result meets or exceeds
{rbrr_gcb_min_concurrent_builds}
(the regime's minimum concurrent build target)
.. Machine type is set at pool creation time.
To change it, the pool must be recreated with a different
{rbrr_gcb_machine_type}
(e.g., `e2-standard-2` for 2 vCPUs, `e2-highcpu-8` for 8 vCPUs).
.. If machine type adjustment alone cannot satisfy capacity requirements, a CPU quota increase can be requested via *Edit Quotas*.
This is a last resort and rarely necessary.

//axhos_step
. *Confirm Quota Headroom*:
.. After pool recreation or quota increase, return to *IAM & Admin > Quotas & System Limits*
.. Filter for `cloudbuild.googleapis.com` and verify the *Concurrent Private Pool Build CPUs* row
.. {rbhg_require}
quota headroom supports the required concurrent builds: `«CPU_QUOTA»` ÷ vCPUs per machine type ≥
{rbrr_gcb_min_concurrent_builds}

//axhoo_output_of_type
{rbtgi_depot_project}

The
{rbtgi_depot_project}
private pool quota is confirmed to support the desired level of concurrent build execution.

//axhoc_completion
NOTE: All depots use a private worker pool (created by
{rbtgo_depot_create}).
The relevant quota metric is *Concurrent Private Pool Build CPUs* on the pool's host project.
Machine type is set at pool creation time via
{rbrr_gcb_machine_type};
individual build requests do not specify machine type.
The
{rbtgo_trigger_build}
operation includes an automated quota preflight gate that queries this metric and verifies capacity against
{rbrr_gcb_min_concurrent_builds}
before dispatching.
The image listing operation performs the same check as a non-blocking advisory.
