//axhob_operation
{rbtgo_image_delete}

NOTE: This operation jettisons a single image tag from
{rbtgi_repository}
using the Docker Registry HTTP API V2 implemented by
{gar_service}.

`rbfl_jettison` is locator-generic: the same procedure backs
{rbtga_ark}
hallmark members
and Lode members.
The package path embedded in the
{rbst_locator}
selects the domain — the procedure itself does not branch.

NOTE: To delete an entire {rbtga_ark} (the full `{gar_hallmarks_namespace}/«HALLMARK»/` subtree — every basename including `image`, `about`, `vouch`, and any `pouch`/`diags`/`attest` tags present), use {rbtgo_ark_abjure} instead.

NOTE: Jettisoning an interior member of a reliquary Lode (e.g., a single tool image member tag) leaves a hole in an otherwise-coherent cohort. The operator accepts this piecemeal-by-image semantic; recovery is nuclear: re-run {rbtgo_lode_conclave} to restore the full reliquary corpus. The operation provides no integrity guard against such interior holes.

The
{rbtr_director}
jettisons an image by {rbst_locator}:

{rbst_locator}
. *Validate Input Parameters*:
.. {rbbc_require}
`«INPUT_LOCATOR»` provided as first argument in `package-path:tag` format
.. {rbbc_fatal}

`«TAG»` as portion after the final colon
.. {rbbc_fatal}

.. {rbbc_fatal}

//axhos_step
. *Authenticate as
{rbtr_director}*:
.. Execute
{rbtoe_director_authenticate}
.. {rbbc_store}
`«DIRECTOR_TOKEN»` for subsequent API calls

//axhos_step
. *Confirm Jettison*:
.. {rbbc_show}
`Will jettison: «INPUT_LOCATOR»`
.. {rbbc_require}
user types `yes` to confirm
.. {rbbc_fatal}
if not confirmed: "prompt not confirmed."

//axhos_step
. *Jettison Tag*:
.. {rbbc_call}
link:https://github.com/opencontainers/distribution-spec/blob/main/spec.md#deleting-tags[Docker Registry API: DELETE manifests] with:
... method: `DELETE`
... endpoint: `«REGISTRY_BASE»/«PACKAGE_PATH»/manifests/«TAG»`
... header: `Authorization: Bearer «DIRECTOR_TOKEN»`
.. {rbbc_require}
HTTP 202 (Accepted) or 204 (No Content) response
.. {rbbc_fatal}
if jettison fails: "Jettison failed with HTTP «STATUS_CODE»"
.. NOTE: V2 DELETE-by-tag has producer-side asymmetric semantics. For single-platform pushes (`docker push`), the tag points to a single image manifest; DELETE removes it, the package becomes empty, and GAR drops the package container from `ListPackages`. For multi-platform pushes (`docker buildx --push`, `skopeo copy --all`), the tag points to an OCI image index referencing N per-platform child manifests; DELETE removes only the index, leaving the N children as untagged orphan content. The package container persists in `ListPackages` because it still holds those untagged child manifests.
.. NOTE: The orphan-children residue is reclaimed autonomously by the depot's GAR cleanup policy — see {rbtgo_depot_levy} step *Create Container Repository* for the policy install, schema-verification sources, and cadence semantics. User-facing display surfaces ({rbtgo_hallmark_tally} and {rbtgo_image_rekon}) filter out packages with zero live tags, so tooling output reflects the correct post-jettison state immediately even though GAR-side reclamation is lazy. External observers querying GAR REST `ListPackages` directly will see the walking-dead package container until the cleanup policy's next run.

//axhoc_completion
Procedure completes successfully when the tag has been jettisoned from the registry and the jettisoned message is displayed.
The locator-generic shape means a hallmark member or a Lode member is deleted by the same call — domain identity lives in the locator's package path, not in branched logic.
