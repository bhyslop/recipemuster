{rbrf_regime}

This regime specifies one foedus's provider-side IdP trust within the Recipe Bottle system.
Each
{rbrf_regime}
instance represents a single foedus — a provider under the manor's one workforce pool, plus the provider configuration a federated assertion is acquired against.
A foedus IS that provider: {rbtf_affiance} creates it, {rbtf_jilt} deletes it. The pool it hangs under is manor-level and carried by the sibling {rbrw_regime}, not here — one pool serves every depot under the manor, the same federated subject grantable in each.

The trust divides into a vendor-agnostic core and a single acquisition-mechanism gate.
The Google-side provider configuration is the same structure for every OIDC IdP — only the values differ — so *vendor identity is not a regime field*; a real vendor is a setup guide plus values, never a schema or code change.
The one axis that changes the required-field shape is the *acquisition mechanism* — how a citizen acquires a federated assertion — held in
{rbrf_mechanism}
and gating the two arms below.
The programmatic arm's secret material never enters this regime: the uploaded JWKS is *public*, itself a public identifier, and the asserter key and client secret arrive only as path references — sourced regime variables land in process environment, so secret VALUES must never ride them; the ships-committed / no-secrets premise stands un-amended.
RBRF assignment files form a library — one rbrf.env per standing foedus, each in its own subdirectory of the moorings foedera library, the subdirectory name the foedus's stable identity, the configuration stored once with no copied active file.
A foedus subdirectory name bears the `rbef_` foedus-instance sprue — the standing members are `rbef_entrada` (the interactive Entra trust) and `rbef_keycloak` (the programmatic test trust) — and is the operator-facing folio passed to the switch toothings ({rbtgo_foedus_instate} and {rbtgo_foedus_descry}).
Every value is a public identifier and carries no secrets, so each file ships committed; affiance rewrites a foedus's values when founding a fresh trust against a real org and IdP tenant.

NOTE: *One sanctioned ships-committed deviation — the `rbef_keycloak` test foedus.* The ships-committed rule holds for every foedus but one. `rbef_keycloak` (the programmatic test trust) commits only an `rbrf.env.template` carrying the vendor-agnostic core and the namespaced attribute mapping; its LIVE `rbrf.env` is git-IGNORED and rendered per charge by the Keycloak test-facility orchestrator (`rbxk_keycloak`) from that template plus TWO rendered fields — the realm's fresh public JWKS ({rbrf_idp_jwks_json}) and the reachable grant endpoint ({rbrf_grant_endpoint}) — both composed from the orchestrator's own knowledge (nameplate port, facility constants), never from realm contents. The deviation is confined to this one foedus and is forced by the twiddle: the realm signing key is ephemeral — minted fresh per charge ({rbtf_realm}) — so its public JWKS cannot be a committed canonical value; the grant endpoint rides the same render because the reachable POST target is charge-time knowledge (the crucible's port), not a committed IdP fact. `rbef_entrada` and every real-IdP foedus keep ships-committed untouched: the live file IS the committed file.

[[RBr_e4a]]
//axvc_camber
RBr_e4a::
Deliberate deviation from the ships-committed rule: `rbef_keycloak` alone commits only an `rbrf.env.template`, its LIVE `rbrf.env` git-ignored and rendered per charge by `rbxk_keycloak`.
A foedus subdirectory standing without its live regime file is this one sanctioned exception, not a fault — so a reader must not "repair" the missing-file tolerance into a hard presence check.

NOTE: *Active foedus and selection.* A foedus's stable identity is its subdirectory name in this library; the manor pool it hangs under is stable and manor-lifetime, so nothing rotates beneath the foedus. The single *active*
{rbtf_foedus}
— the configured federation a run authenticates against — is named not here but by the {rbrr_active_foedus} selector in the sibling {rbrr_regime}; this regime carries one foedus's configured shape, never which is active. The manor may stand up several foedera, each its own provider under the one pool and its own subdirectory; the authoritative registry of what exists is the Manor itself, enumerated by
{rbtf_canvass}
(a `providers.list` under the one pool), not a local list. Production stays single-foedus-clean — one foedus in the library, named by the selector, rarely re-pointed. The switch toothings live outside this regime:
{rbtgo_foedus_instate}
re-points the selector and
{rbtgo_foedus_descry}
reads a named foedus's provider presence.

NOTE: *Per-vendor guide contract.* A real vendor's IdP-console setup (app registration, claims, scopes) is foreign-console human work, not a regime field and not a tabtarget. Each vendor gets a setup guide whose contract is to yield the vendor-agnostic core values this regime reads (issuer, client-id, subject claim via the attribute mapping, and the mechanism-appropriate gated values). Entra is the first such guide; further vendors are further guides under the same contract.

//axhrv_variable
{rbrf_provider_id}

Workforce pool provider ID — the external IdP trust root within the manor's one pool, and the per-foedus discriminator that distinguishes one foedus from another under that shared pool.
Core (both mechanisms); required; lowercase alphanumeric and hyphen, must not start with the reserved `gcp-` prefix.

//axhrv_variable
{rbrf_mechanism}

The acquisition-mechanism discriminator — how a citizen acquires a federated assertion against this foedus.
Must be
{rbrf_mechanism_interactive}
(human device-flow sign-in) or
{rbrf_mechanism_programmatic}
(self-supplied JWT against an uploaded public JWKS, the test-bed's no-human path).
Gates which of the two mechanism groups below applies; modeled on
{rbrv_vessel_mode}.
Core (always present); required.

//axhrv_variable
{rbrf_idp_issuer}

OIDC issuer URI of the external IdP.
Core (both mechanisms); required.
Under
{rbrf_mechanism_interactive}
this is a resolvable `https://` issuer whose discovery document backs the device flow.
Under
{rbrf_mechanism_programmatic}
it is a self-declared `https://` identifier string-matched against the JWT `iss`; GCP still requires the `https://` scheme but never resolves it (the uploaded JWKS is the key source), so a fictional https value is valid and a local IdP's `http://localhost` issuer is not.

//axhrv_variable
{rbrf_idp_client_id}

Public client (application) ID registered at the IdP.
Core (both mechanisms); required.
Under
{rbrf_mechanism_interactive}
it is the device-authorization-flow client.
Under
{rbrf_mechanism_programmatic}
it must equal the JWT `aud`.

//axhrv_variable
{rbrf_attribute_mapping}

Workforce provider attribute mapping written into the provider by affiance.
Must map `google.subject`; the mapped IdP claim becomes the grantable federated subject.
Core (both mechanisms); required.

//axhrgb_group
{rbrf_group_interactive}

//axhrgc_gate
{rbrf_mechanism}
{rbrf_mechanism_interactive}

Variables in this group apply only when
{rbrf_mechanism}
is
{rbrf_mechanism_interactive}.
The interactive mechanism is the human device-authorization flow (RFC 8628): a live human avows each sitting, and these fields drive that flow's Leg 1.

//axhrgv_variable
{rbrf_idp_scope}

Device-flow OAuth scope requested at Leg 1.
Required; must request `openid`, and must NOT request `offline_access` — a refresh token would let a run begin outside a live sitting, contradicting the human-present premise.

//axhrgv_variable
{rbrf_idp_device_endpoint}

IdP device authorization endpoint (RFC 8628) — the Leg 1 device-code request target.
Explicit field rather than a tenant-derived template, so a non-Entra IdP needs only new values, no code change — the vendor-invariance of the core, in miniature.
Required; https:// URI.

//axhrgv_variable
{rbrf_idp_token_endpoint}

IdP token endpoint — the Leg 1 device-code polling target.
Required; https:// URI.

//axhrgb_group
{rbrf_group_programmatic}

//axhrgc_gate
{rbrf_mechanism}
{rbrf_mechanism_programmatic}

Variables in this group apply only when
{rbrf_mechanism}
is
{rbrf_mechanism_programmatic}.
The programmatic mechanism deletes the human by construction: there is no device flow and no sitting to open — a caller self-supplies a JWT and GCP validates it against an uploaded public JWKS.
It is the test-bed's unattended path; the human-present premise is deliberately voided under it.
The group carries both sides of that self-supply: what GCP validates against (the uploaded public JWKS) and what the caller needs to mint and present the grant (the reachable endpoint, the two secret-path references, and the three committed assertion facts).

//axhrgv_variable
{rbrf_idp_jwks_json}

The active foedus's uploaded *public* JWKS — the key snapshot GCP validates self-supplied JWTs against (REST `oidc.jwksJson`, the `--jwk-json-path` target).
The realm signing key is ephemeral — minted fresh per charge, see {rbtf_realm} — so this public snapshot is re-synced to the provider each charge (the twiddle) rather than held as a committed canonical reference, and the matching *private* key is persisted nowhere and never enters this regime.
Used because the test IdP runs as a local crucible unreachable from Google, so issuer-discovery cannot serve it — the key snapshot must be uploaded regardless.
Required under the programmatic mechanism.

//axhrgv_variable
{rbrf_grant_endpoint}

The reachable token-endpoint URL the RFC 7523 grant is POSTed to.
Live-rendered into the git-ignored live rbrf.env per charge by the establishment orchestrator, beside the JWKS — composed from the orchestrator's own knowledge (nameplate port, facility constants), never from realm contents.
Deliberately a distinct programmatic-gated field, never a promotion of the interactive
{rbrf_idp_token_endpoint}
to core: that field is a committed IdP fact validated https-only, while this one is a charge-time render admitting loopback cleartext — commit-vs-render provenance and https-vs-loopback validation are load-bearing differences.
Required under the programmatic mechanism; an `https://` URI or a loopback `http://` URI, cleartext tolerated on loopback only — the test IdP is a local crucible, and
{rbtf_realm}
pins the POST target as local, distinct from the fictional https frontend issuer in
{rbrf_idp_issuer}.

//axhrgv_variable
{rbrf_asserter_key_file}

Repo-root-relative path to the asserter's private signing key — the RFC 7523 caller signs its assertion with it.
A path reference, never the material: for the test foedus the committed caged home keeps the keypair (the caged test asserter of
{rbtf_realm}'s
two-keys doctrine), and a production programmatic foedus points the same field at a station-fenced file instead — the field's shape does not change between test and production.
Required under the programmatic mechanism.

//axhrgv_variable
{rbrf_client_secret_file}

Repo-root-relative path to the client secret the grant authenticates with at the token endpoint.
The grant requires a confidential client — the public-client path is a surveyed dead end (see
{rbtf_realm}'s
first-contact trips) — so the secret is load-bearing; it enters this regime only as a path reference, the caged committed home keeping the value beside the asserter key.
Required under the programmatic mechanism.

//axhrgv_variable
{rbrf_asserter_kid}

The asserter key id the assertion's JWT header `kid` carries, matched by the IdP against its registered asserting key.
A public assertion fact — a committed template value, dual-homed by contract with the realm DATA exactly as
{rbrf_idp_client_id}
already is.
Required under the programmatic mechanism.

//axhrgv_variable
{rbrf_asserter_issuer}

The asserting party's issuer — the assertion's `iss`, matched by the IdP against its registered asserting trust.
Distinct from
{rbrf_idp_issuer},
the IdP's own issuer, which the assertion's `aud` targets: the assertion aud is cinched to that existing field, deliberately no separate field.
A public assertion fact, committed and dual-homed with the realm DATA.
Required under the programmatic mechanism.

//axhrgv_variable
{rbrf_asserter_subject}

The external subject the assertion's `sub` carries — the identity the IdP resolves to its federated-linked user.
NOT the GCP-side federated subject: the IdP issues its own internal user id as the token `sub` (a first-contact trip of
{rbtf_realm}),
so this value never appears in a `principal://` binding.
A public assertion fact, committed and dual-homed with the realm DATA.
Required under the programmatic mechanism.

//axhro_kindle
{rbkro_kindle}

Enrolls all RBRF_ variables via buv enrollment — the vendor-agnostic core unconditionally, the interactive and programmatic groups gated on
{rbrf_mechanism}.
Guards against unexpected RBRF_ variables via scope sentinel.
Locks all enrolled variables against mutation.

//axhro_validate
{rbkro_validate}

Validates fields via buv_vet type checkers, then custom format checks in a separate enforce step.
Core, both mechanisms: the provider ID as lowercase alphanumeric/hyphen with the `gcp-` prefix reserved; the attribute mapping as mapping google.subject; the mechanism as one of the two enum values. The manor pool's org, pool id, and session duration validate in the sibling {rbrw_regime}, not here.
Interactive only: both device-flow endpoints as resolvable https:// URIs, the scope as requesting openid but never offline_access (the human-present premise), and the issuer as a resolvable https:// URI.
Programmatic only: the issuer as an https://-scheme iss-matching string (scheme required, resolution not); the uploaded JWKS as present and well-formed; the grant endpoint as an `https://` URI or a loopback `http://` URI (the loopback-http escape from the https gate — cleartext tolerated on loopback only); the two secret-path fields as present, non-empty repo-root-relative paths; and the three assertion facts (kid, issuer, subject) as present non-empty strings.
