==== Multi-Architecture Strategy Caveat

All sections below assume Google Cloud Build (GCB) remains the sole builder.
GCB provides only x86 workers — default and private pools alike.
Multi-platform builds use QEMU emulation via the binfmt setup step (rbgjb02).
The binfmt step is conditionally included: vessels whose `RBRV_CONJURE_PLATFORMS` matches the build runner platform (`RBGC_BUILD_RUNNER_PLATFORM`) skip it, reducing attack surface for native-only builds.

If a vessel requires native ARM builds, the builder strategy must expand beyond GCB, and further GCB-specific hardening should be re-evaluated.

If native ARM builds become necessary, builder-agnostic controls (cosign signing, portable SLSA provenance) become more valuable than GCB-specific infrastructure (private pools, VPC Service Controls).
See cosign deferral rationale in Deferred Items.

==== Current Posture

Properties of the build pipeline as of 2026-03.
Authoritative procedures live in the operational specs (RBSAC, RBSAV, RBSDC, RBSDN);
this section states what holds and why it matters.

**Build definitions are fully static.**
Each vessel's `cloudbuild.json` is stitched from step scripts (`rbgjb*.sh`) and current regime pins -- zero custom substitution overrides at submission time.
The build is fully determined by the stitched JSON and the pouch (build context as OCI image in GAR).
See RBSAC.

**SLSA v1.0 Build Level 3 provenance — verified.**
All `builds.create` submissions produce SLSA v1.0 Build Level 3 provenance via `requestedVerifyOption: VERIFIED` and the `images:` field declaring per-platform image URIs.
Google Container Analysis generates both v0.1 and v1 predicate attestations per image.
Provenance chain: stitched JSON + pouch -> per-platform image digest.
Verification via DSSE envelope signature verification (jq + openssl against embedded GCB attestor keys).

Production-verified build IDs:

* Single-arch: `16d3b60f` (busybox-amd64), `fc36b970` (busybox-arm64), `9180c42a` (trbim-macos)
* Multi-platform: `2e172ce0` (rbev-busybox, 3 platforms — amd64, arm64, armv7)

**Security boundary: pouch and stitched JSON.**
Cloud Build receives build context via the pouch (FROM SCRATCH OCI image in GAR) -- per-vessel Dockerfile and context files.
The main repo (tooling, specs, regime files) is never exposed to Google.
The Director controls what enters the pouch; the cloud is just labor.
See RBSAC.

**Builder step images are reliquary-sourced.**
All GCB step images are pulled from a datestamped reliquary in GAR — co-versioned tool images frozen by the inscribe ceremony.
`RBRV_RELIQUARY` in each vessel regime identifies the active reliquary.
Base images are separately enshrined via `RBRV_IMAGE_n_ANCHOR`.

**Supply-chain pinning boundary — staged on the reliquary.**
The governing rule: reliquary _generation_ (the inscribe/conclave ceremony that mints the sealed tool cohort) may pull unpinned upstream images — bootstrapping the first tool must start somewhere, and generation is the only phase permitted to.
Every build that _consumes_ a sealed reliquary carries zero unpinned aspects: each tool image resolves, pinned, from the cohort, whose durable home is our own Artifact Registry (GAR), never an upstream registry.
The made side already conforms — all conjure/vouch step images are reliquary-sourced (above).
The capture family is brought into conformance by the crane-embrace cutover: the capture image tool (gcrane) becomes a reliquary cohort member, so a sealed-reliquary capture (bole/wsl/podvm) resolves a _pinned_ gcrane, leaving the floating Google-hosted bootstrap gcrane (`ZRBLD_GCRANE_BUILDER`) only in conclave — which builds the reliquary and therefore cannot resolve from it.
Until that lands, capture builders ride floating bootstrap tags (see Deferred Items).

**Dual private pool workers (tether + airgap).**
Every depot gets two private pools: a tether pool (public egress) and an airgap pool (`NO_PUBLIC_EGRESS`).
Pool routing is operation-level: inscribe and enshrine use tether (upstream access required), about and vouch always use airgap (verification isolated from public registries), conjure and bind use the vessel's `RBRV_EGRESS_MODE`.
Pools are fully managed by Google (scale to zero, no VMs to maintain).

**Artifact Registry storage.**
Built images are stored in Google Artifact Registry (GAR), not Docker Hub or other external registries.
Regime variables configure project, region, and repository.

**Dedicated Mason service account.**
Scoped to depot-project permissions, specified in the `serviceAccount` field of each build JSON.

**SBOM generation.**
Per-platform Syft scans via `registry:` transport (reads directly from GAR using an OAuth2 token obtained from the GCB metadata server — no Docker daemon coupling).
Each platform image gets its own JSON and table-format SBOM, packaged into the per-platform metadata container.
Aligned with CISA 2025 minimum elements and EU CRA requirements.
The upgrade path is runtime-aware analysis, not removal.

**Cloud Build logging.**
Build requests set `options.logging: CLOUD_LOGGING_ONLY`.
Log hygiene (no credential echo) was audited during trigger migration script inlining.

**Registry authentication is ambient and per-step, with no credential-helper dependency.**
Docker-based steps authenticate to GAR automatically — Cloud Build pre-populates Docker credentials, so no `docker login` step is needed.
The capture steps use **`gcrane`** (crane's Google-auth sibling — same `cp`/`manifest`/`tag` engine), whose `google.Keychain` matches `*.pkg.dev` and draws credentials from Application Default Credentials → the GCE metadata server.
On a Cloud Build worker the Mason SA *is* that metadata-server identity, so gcrane reads and writes GAR ambiently with **no token fetch, no `crane auth login`, no credential-helper image, and no new IAM grant**.
This is the same ambient model the docker-based steps already use, extended from docker to crane — one auth story across the capture family rather than two.
The bind mirror (made-side) previously used **skopeo**, which required a per-step OAuth2 token fetch from the GCB metadata server (the Mason SA's ambient identity) via `--src-creds`/`--dest-creds`.
That token was held in step-local memory, never written to `/workspace`.
_Invariant: the {rbtgr_workspace} carries non-secret build data only — secrets stay step-local._
The bind mirror now uses **gcrane** (`gcrane cp`), evicting skopeo entirely.
Ambient gcrane auth is now universal for registry-tool steps — no registry tool fetches a token.
The one surviving per-step token fetch is the immure residency guard (`rbgjl09`), whose raw curl blob-HEAD needs a bearer token — curl's floor role, not a registry tool's.

**Provenance on demand.**
`rbfl_tally()` lists all hallmarks and queries Container Analysis inline for each per-platform image, displaying SLSA build level, build invocation ID, and single-origin verification per hallmark.
Tabtarget:
{rbtgo_hallmark_tally}.

===== Pipeline Architecture: Single-Platform Vessels

Five build steps plus CB-native `images:` push (binfmt step excluded at stitch time).
Applies to vessels whose `RBRV_CONJURE_PLATFORMS` matches `RBGC_BUILD_RUNNER_PLATFORM`.

1. **derive-tag-base** — compute timestamps and tag components from CB substitutions; log build strategy
2. **build-and-load** — `docker buildx build --load` (single platform into local daemon)
3. **sbom** — Syft scan via `registry:` transport
4. **assemble-metadata** — per-platform `build_info.json` with SLSA summary fields
5. **build-and-push-metadata** — multi-platform `about` container pushed to `{gar_hallmarks_namespace}/<hallmark>/about:<hallmark>` via `buildx --push`

`images:` field declares the single per-platform URI → CB pushes and generates SLSA Level 3.

===== Pipeline Architecture: Multi-Platform Vessels

Seven build steps plus CB-native `images:` push (binfmt step included at stitch time).
Applies to vessels whose `RBRV_CONJURE_PLATFORMS` differs from `RBGC_BUILD_RUNNER_PLATFORM`.
About steps (`rbgja/`) are embedded in the build job via stitch (combined image+about).
Vouch (`rbgjv/`) runs as a separate Cloud Build job.

1. **derive-tag-base** — compute timestamps and tag components; log build strategy
2. **qemu-binfmt** — register QEMU user-static for cross-arch emulation
3. **buildx-push-multi** — `docker buildx build --push` all platforms to intermediate `-multi` tag; capture `cache_before.json` (Docker daemon state pre-build) and `buildkit_metadata.json` (via `--metadata-file`)
4. **per-platform-pullback** — `docker pull --platform` each arch, tag individually
5. **push-per-platform** — `docker push` each per-platform tag (pre-push for imagetools); capture `cache_after.json` (Docker daemon state post-push)
6. **imagetools-create** — reassemble consumer-facing manifest list from per-platform tags
7. **push-diags** — build `FROM scratch` container with build-time diagnostic files (`buildkit_metadata.json`, `cache_before.json`, `cache_after.json`, `recipe.txt`), push as `{gar_hallmarks_namespace}/«HALLMARK»/diags:«HALLMARK»`

`images:` field declares all per-platform URIs → CB re-pushes (idempotent) and generates SLSA Level 3.
The reassembled manifest list provides transparent platform resolution to attested per-platform images.

==== Implemented: Egress Lockdown via Dual Pools

Egress lockdown is implemented through the dual-pool architecture.
The airgap pool (`NO_PUBLIC_EGRESS`) blocks all public internet access.
Operations that require upstream access (inscribe, enshrine) run on the tether pool.
Operations that must be isolated (about, vouch) always run on the airgap pool.
Conjure and bind are routed per vessel's `RBRV_EGRESS_MODE`.

**Dependency mirroring is universal.**
All GCB step images come from reliquaries (inscribed tool images in GAR).
All base images come from anchors (enshrined base images in GAR).
This is not just an airgap feature — it is a universal robustness measure against upstream registry churn.

**Operation pool routing:**

[cols="1,1,2",options="header"]
|===
|Operation
|Pool
|Rationale

|Inscribe
|tether
|Pulls tool images from upstream registries

|Enshrine
|tether
|Pulls base images from upstream via gcrane

|Conjure/Bind
|vessel's RBRV_EGRESS_MODE
|Build-time deps vary per vessel

|About
|airgap (always)
|Only needs GAR + Google APIs (Private Google Access)

|Vouch
|airgap (always)
|Verification isolated from public registries
|===

**About/vouch always-airgap is a security property**: verification can never be influenced by a compromised public registry, even for tether vessels.

==== Implemented: Cloud-Dispatched Tool-Plane Deletes

GAR image-package deletes — `banish` (whole Lode) and `abjure` (every package under a hallmark subtree) — are **cloud-dispatched**, not host-issued.
The workstation submits a Cloud Build and blocks until terminal (conjure-shaped); the in-pool step deletes each package by convergence (see *Delete-by-convergence* below) and verifies package absence.
The build's outcome IS the delete outcome, which closes a real correctness gap: a host-issued `packages.delete` returns HTTP 200 immediately and the prior code trusted that 200, but the actual deletion runs asynchronously as an LRO whose terminal verdict the workstation never saw (the trust-200 LRO gap).

**The cinched line is tool-plane versus control-plane.**
Only GAR *package* deletes move cloud-side: `banish` and `abjure`.
Control-plane deletes — service accounts, projects, the depot, liens, GCS buckets — stay host-issued REST, as do the member/tag-grain maintenance verbs (the raw image backdoor), which must not depend on the pipeline they exist to clean up after.
The recorded canon is the line itself, not a trajectory toward all-deletes-go-cloud.

**The delete build runs as Director, not Mason.**
Director already holds `repoAdmin` (delete); Mason is writer-only and stays that way, so the identity that executes untrusted upstream capture bytes is never widened with delete authority.
Running a build *as* the Director SA, submitted by the Director-authenticated workstation, requires one investiture addition: Director holds `serviceAccountUser` (actAs) on *itself* — the self-actAs binding that lets a SA run a build as itself — alongside the `workerPoolUser` and Cloud Build Editor grants it already carries.
No new GAR grant is introduced.

**The delete builder is digest-pinned (2026-06-10).**
Every other reliquary-less bootstrap (the underpin/wsl gcrane floaters) runs as Mason, writer-only — that asymmetry is the design: the identity that executes unpinned bytes never holds delete authority.
The delete build breaks that pairing by running as Director, so its builder may not float: `ZRBFC_DELETE_BUILDER` rides `gcr.io/cloud-builders/gcloud` pinned by digest, not `:latest`.
The pin preserves the reliquary-less property the delete build requires (a digest needs no reliquary to resolve from; the delete of the last reliquary still works), and since the step's needs are frozen (python3 + urllib + json), staleness cost is near zero — the digest refreshes deliberately, as a reviewed one-line change.

**Delete-by-convergence at the vendor Palisade.**
GAR refuses to delete a child manifest while its parent index still exists: a `versions.delete` (or a whole-`packages.delete`) on a referenced child returns FAILED_PRECONDITION, "manifest is referenced by parent manifests", and a single `packages.delete` of a multi-arch web removes nothing — the package survives intact.
This is GAR's documented behavior — its cleanup policies "don't delete images referenced by a parent manifest; if the parent manifest is deleted, then any related images are deleted" on the next run — and the reason Google's own gcr-cleaner ships a `--skip-errors` flag rather than ordering the topology.
The in-pool step does the same: each round it fires a delete at every remaining version (force=true to drop tags) and at the package shell, skipping the per-round "referenced by parent" preconditions, and polls the package GET until it 404s.
The parent indexes delete first (they have no parent), freeing their children for a later round; absence is the only success signal, the deadline the only failsafe — the same eventual-consistency shape as the host `rbuh_poll_until_gone`.
This is conduct at the vendor Palisade: an empirical probe (2026-06-09) recorded the FAILED_PRECONDITION signature directly — the earlier paddock guess of an LRO terminating NOT_FOUND (code 5) "even when the delete effectively completed" was wrong; the package does not delete at all under a single call, and the step converges on the observable end state rather than trusting any LRO verdict.
Multi-arch capture (bole/wsl keep full-fidelity index Lodes) keeps index-web packages a live shape, so the convergence loop is load-bearing.

**Accepted cost: every fixture banish/abjure becomes a build.**
Cloud dispatch moves these deletes from a sub-second host REST call to a full Cloud Build (minutes, serialized with other fixture builds).
This is accepted as the price of delete correctness; the depot is disposable, so debris is test material, not cleanup duty.

==== Potential Hardening: VPC Service Controls Perimeter

Wrap the depot project in a VPC Service Controls (VPC-SC) perimeter.
This restricts which GCP APIs can be called and from where, reducing data exfiltration risk.

**Required infrastructure:**

* VPC-SC perimeter around the depot project (organizational policy — may require org-level admin)
* Access levels for operator workstations (so builds can still be submitted from outside the perimeter)
* Ingress/egress rules for the perimeter (which services may cross the boundary)
* Testing: every GCP API call in the pipeline must be verified inside the perimeter — Cloud Logging writes, GCS uploads, Artifact Registry pushes, Cloud Build API calls can break subtly

**Prerequisite:** Egress Lockdown is in place (dual-pool architecture).

==== Deferred Items

Evaluated and consciously deferred; each has a specific revisit trigger.

**Dockerfile base image digest validation.**
Builder step images are mostly pinned (reliquary tools ride a content-frozen cohort stamp); the always-pullable Google-hosted bootstrap builders (`gcr.io/cloud-builders/docker`, `gcr.io/go-containerregistry/gcrane:debug`) ride floating tags by design, and user-authored Dockerfiles in vessel build contexts are not validated.
A vessel with `FROM ubuntu:22.04` passes today — mutable tag, supply-chain risk.
This constrains vessel authors (policy decision with RBSRV implications), orthogonal to pipeline infrastructure.
Separate future heat.

**Bootstrap builder pinning + gcr.io provenance — recorded decision (2026-06).**
The reliquary's own generators cannot resolve their tools from the reliquary (a cycle), so a small set of bootstrap builders is pulled unpinned: `gcr.io/cloud-builders/docker` and `gcr.io/go-containerregistry/gcrane:debug`.
This is acceptable because it is bounded to reliquary _generation_ (see _Supply-chain pinning boundary_, Current Posture) — sealed-reliquary builds never touch these.
On gcr.io durability: the Container Registry _storage product_ shut down (writes 2025-03-18, reads 2025-06-03), but Google-_owned_ images served via gcr.io URLs (cloud-builders, go-containerregistry, distroless) were migrated onto Artifact Registry under the preserved gcr.io URL and are explicitly exempt, with no announced end-of-life.
RB stores nothing at gcr.io — every RB artifact lives in Artifact Registry — so the Container Registry shutdown does not touch RB's foundation; the only gcr.io exposure is these bootstrap-generation pulls, which are non-load-bearing (rare, supervised, re-runnable; a one-line source change if an alias is ever retired).
Hardening (sibling itch, not yet scoped): digest-pin the bootstrap builders, or mirror them into our Artifact Registry for full control — the durable answer the reliquary already embodies for consumed tools.
See _Registry transition_ references below.

**BuildKit provenance flag.**
`--provenance=true` on `docker buildx build` records builder internals — useful for deeper forensics.
GCB-native SLSA v1.0 provenance covers the primary attestation use case.
Redundant for the current threat model.
Revisit if build reproducibility verification becomes a priority.

**Cosign image signing.**
Cryptographic signatures attached to images, verifiable by downstream consumers.
Two modes: keyless (Sigstore/Fulcio — requires public internet, conflicts with Egress Lockdown) and keyed (Cloud KMS — no external dependency).
With CB-native SLSA v1.0 Build Level 3 verified on all vessel types, cosign is redundant for the current threat model (building and consuming within own GCP project).
Becomes valuable if: images are distributed externally, trust verification needed outside GCP, or builder strategy expands beyond GCB (losing Google's provenance).
If adopted, prefer KMS-backed keyed signing (`cosign sign --key gcpkms://...`) for Egress Lockdown compatibility.

**Binary Authorization.**
Deploy-time policy enforcement: only attested images can be deployed.
Relevant for GKE and Cloud Run.
Now technically achievable — CB-native SLSA Level 3 satisfies BinAuth attestation requirements.
Recipe Bottle deploys to Podman on workstations — no benefit until deployment targets change.

**Reproducibility verification.**
Periodic independent rebuilds compared by digest.
Requires reproducible toolchains — a vessel-author discipline, not a pipeline control.
Not actionable until vessel ecosystem matures.

**skopeo credential helper — rejected, not merely deferred (evaluated 2026-06). Moot: skopeo fully evicted.**
The standalone `docker-credential-gcr` helper could have let skopeo authenticate to GAR via a `credHelpers` config entry instead of an inline token fetch.
Rejected: on GCB the helper mints the _same_ metadata-server token we already obtained (no new capability), but required injecting a new binary into the skopeo container — a fresh vouched supply-chain dependency, the opposite of our minimize-the-dependency-manifold posture — and skopeo→GAR via credential helper carried documented 400-error friction (skopeo issue #2573, locked/unresolved), with the community fallback being the explicit `oauth2accesstoken` token, i.e. exactly the approach we used.
The question is now closed: skopeo has been evicted from the cohort entirely (bind mirror converted to gcrane), so neither the inline-token approach nor the credential-helper approach remains in use.

**Capture-machinery duplication — vouch-push.**
The cloud capture steps presently duplicate logic by copy: the `FROM scratch` vouch-push step (Lode `rbgjl02` and hallmark-verify `rbgjv03`).
(The registry-tool metadata-token fetch is retired with skopeo's eviction — the `token-fetch` snippet survives solely for the immure residency guard's curl blob-HEAD; the read→digest→fingerprint→copy block is unified under the shared `gcrane-fingerprint` snippet and gcrane's ambient auth.)
This is acknowledged, not accidental — Cloud Build offers no native step reuse (no YAML anchors or includes), and its blessed reuse mechanism, custom builder images, conflicts with the _cloud is just labor; the Director controls what enters_ posture above (a custom builder holding our logic would be a new vouched artifact rather than inline-from-repo).
Remedy (built in the capture-machinery-unification refactor): a host assembly spine (aligned with the `stitch` vocabulary), a shared cloud step-library (`rbgjs`) of compose-once snippets, and a dispatch-time recipe validator that derives each step's substitution coverage — composed once and referenced, replacing copy-by-example with compose-from-library. The `/workspace` inter-step channel stays guarded in-step (the steps' own `test -f` preconditions) rather than by a dispatch contract.
Dedup is deliberately partial: only the survivor (bole) rides the spine; the doomed enshrine and inscribe forks are left in place for later removal, so the duplicated copies above persist until that cutover.
Behavior-preservation is confirmed against the live system at the heat's milestone skirmish gate (full skirmish + service).
