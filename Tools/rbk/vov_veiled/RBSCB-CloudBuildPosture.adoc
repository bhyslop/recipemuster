==== Multi-Architecture Strategy Caveat

All sections below assume Google Cloud Build (GCB) remains the sole builder.
GCB provides only x86 workers — default and private pools alike.
Multi-platform builds use QEMU emulation via the binfmt setup step (rbgjb02).
The binfmt step is conditionally included: vessels whose `RBRV_CONJURE_PLATFORMS`
matches the build runner platform (`RBGC_BUILD_RUNNER_PLATFORM`) skip it,
reducing attack surface for native-only builds.

If a vessel requires native ARM builds, the builder strategy must expand beyond GCB,
and further GCB-specific hardening should be re-evaluated.

If native ARM builds become necessary, builder-agnostic controls (cosign signing, portable SLSA provenance)
become more valuable than GCB-specific infrastructure (private pools, VPC Service Controls).
See cosign deferral rationale in Deferred Items.

==== Current Posture

Properties of the build pipeline as of 2026-03.
Authoritative procedures live in the operational specs (RBSRI, RBSTB, RBSDC, RBSDN);
this section states what holds and why it matters.

**Build definitions are fully static.**
Each vessel's `cloudbuild.json` is generated by the inscribe command directly into the
{rbtgr_rubric_repo}
from step scripts (`rbgjb*.sh`) and current regime pins — zero custom substitution overrides
at dispatch time.
The build is fully determined by the generated JSON at a specific rubric repo commit.
See RBSRI, RBSTB.

**SLSA v1.0 Build Level 3 provenance — verified.**
All trigger-dispatched builds produce SLSA v1.0 Build Level 3 provenance via
`requestedVerifyOption: VERIFIED` and the `images:` field declaring per-platform image URIs.
Google Container Analysis generates both v0.1 and v1 predicate attestations per image.
Provenance chain: rubric repo commit → committed JSON → per-platform image digest.
No DIY provenance tooling — CB-native attestation only.

Production-verified build IDs:

* Single-arch: `16d3b60f` (busybox-amd64), `fc36b970` (busybox-arm64), `9180c42a` (trbim-macos)
* Multi-platform: `2e172ce0` (rbev-busybox, 3 platforms — amd64, arm64, armv7)

**Security boundary: rubric repo.**
Cloud Build sees only the rubric repo via a CB v2 connection (GitLab `gitlabConfig`
with project access token — 3 secrets in Secret Manager, fully programmatic setup) —
a single shared repo with per-vessel directories (cloudbuild.json + Dockerfile + build context).
The main repo (tooling, specs, regime files) is never exposed to Google.
SLSA provenance records the rubric repo commit, which maps directly to the build material.
See RBSRI.

**Builder step images are digest-pinned.**
All `RBRR_GCB_*_IMAGE_REF` regime variables enforce `@sha256:<64hex>` digests,
validated by `buv_val_odref()`.
Inscribe verifies pin freshness (< 1 day) as a blocking gate;
stale pins require explicit refresh, commit, and re-inscribe.

**Private pool workers.**
Every depot gets a private pool — no default-pool path exists.
Expanded machine type options (64 vs default pool's 5), higher concurrent build limits.
Public egress is retained — no dependency mirroring cascade.
Pools are fully managed by Google (scale to zero, no VMs to maintain).
Foundation for a future egress lockdown if warranted.

**Artifact Registry storage.**
Built images are stored in Google Artifact Registry (GAR), not Docker Hub or other external registries.
Regime variables configure project, region, and repository.

**Dedicated Mason service account.**
Scoped to depot-project permissions, specified in the `serviceAccount` field of each build JSON.

**SBOM generation.**
Per-platform Syft scans via `registry:` transport (reads directly from GAR using an OAuth2 token obtained from the GCB metadata server — no Docker daemon coupling).
Each platform image gets its own JSON and table-format SBOM,
packaged into the per-platform metadata container.
Aligned with CISA 2025 minimum elements and EU CRA requirements.
The upgrade path is runtime-aware analysis, not removal.

**Cloud Build logging.**
Build requests set `options.logging: CLOUD_LOGGING_ONLY`.
Log hygiene (no credential echo) was audited during trigger migration script inlining.

**Provenance on demand.**
`rbf_check_consecrations()` lists all consecrations and queries Container Analysis inline for each per-platform image,
displaying SLSA build level, build invocation ID, and single-origin verification per consecration.
Tabtarget: {rbtgo_consecration_check}.

===== Pipeline Architecture: Single-Platform Vessels

Five build steps plus CB-native `images:` push (binfmt step excluded at stitch time).
Applies to vessels whose `RBRV_CONJURE_PLATFORMS` matches `RBGC_BUILD_RUNNER_PLATFORM`.

1. **derive-tag-base** — compute timestamps and tag components from CB substitutions; log build strategy
2. **build-and-load** — `docker buildx build --load` (single platform into local daemon)
3. **sbom** — Syft scan via `registry:` transport
4. **assemble-metadata** — per-platform `build_info.json` with SLSA summary fields
5. **build-and-push-metadata** — multi-platform `-about` container via `buildx --push`

`images:` field declares the single per-platform URI → CB pushes and generates SLSA Level 3.

===== Pipeline Architecture: Multi-Platform Vessels

Seven build steps plus CB-native `images:` push (binfmt step included at stitch time).
Applies to vessels whose `RBRV_CONJURE_PLATFORMS` differs from `RBGC_BUILD_RUNNER_PLATFORM`.
About steps (`rbgja/`) are embedded in the build job via stitch (combined image+about). Vouch (`rbgjv/`) runs as a separate Cloud Build job.

1. **derive-tag-base** — compute timestamps and tag components; log build strategy
2. **qemu-binfmt** — register QEMU user-static for cross-arch emulation
3. **buildx-push-multi** — `docker buildx build --push` all platforms to intermediate `-multi` tag; capture `cache_before.json` (Docker daemon state pre-build) and `buildkit_metadata.json` (via `--metadata-file`)
4. **per-platform-pullback** — `docker pull --platform` each arch, tag individually
5. **push-per-platform** — `docker push` each per-platform tag (pre-push for imagetools); capture `cache_after.json` (Docker daemon state post-push)
6. **imagetools-create** — reassemble consumer-facing manifest list from per-platform tags
7. **push-diags** — build `FROM scratch` container with build-time diagnostic files (`buildkit_metadata.json`, `cache_before.json`, `cache_after.json`, `recipe.txt`), push as `«CONSECRATION»-diags`

`images:` field declares all per-platform URIs → CB re-pushes (idempotent) and generates SLSA Level 3.
The reassembled manifest list provides transparent platform resolution to attested per-platform images.

===== Tag Scheme

[cols="1,2",options="header"]
|===
|Tag Pattern
|Purpose

|`{INSCRIBE_TS}-multi`
|Intermediate `buildx --push` target (multi-platform only)

|`{CONSECRATION}-image-{SUFFIX}`
|Per-platform image (persistent, consecration-tagged)

|`{INSCRIBE_TS}-image-{SUFFIX}`
|SLSA alias tag (`images:` field, provenance generation trigger — ephemeral, overwritten on re-conjure)

|`{CONSECRATION}-image`
|Consumer-facing manifest list (reassembled by `imagetools create`, multi-platform only)

|`{CONSECRATION}-about`
|Metadata container (build provenance — multi-platform, see About Artifact Contents below)

|`{CONSECRATION}-diags`
|Build-time diagnostics container (conjure only — `buildkit_metadata.json`, `cache_before.json`, `cache_after.json`, `recipe.txt`). Pushed by conjure step 07, pulled by about step 01.
|===

Platform suffix derivation: `linux/amd64` → `amd64`, `linux/arm64` → `arm64`, `linux/arm/v7` → `armv7`.
Computed at inscribe time from `RBRV_CONJURE_PLATFORMS`.

===== About Artifact Contents

The `-about` artifact is a `FROM scratch` multi-platform container carrying build provenance files at the root filesystem.
Contents are extracted by {rbtgo_ark_inspect} for trust posture display.

[cols="1,2",options="header"]
|===
|File
|Contents

|`build_info.json`
|Per-platform build metadata: consecration, image URI, moniker, platform, QEMU usage, git coordinates (repo/branch/commit), build timestamps, and SLSA summary (build level, invocation ID, builder ID, predicate types).

|`sbom.json`
|Syft SPDX JSON output: package inventory with type, name, version, license, and Package URL for every detected package.

|`recipe.txt`
|The exact Dockerfile used to build the image.

|`buildkit_metadata.json`
|BuildKit resolved base image references and build parameters (conjure only, forwarded from `-diags`).

|`cache_before.json`
|Docker daemon image state before the build (conjure only, forwarded from `-diags`).

|`cache_after.json`
|Docker daemon image state after all pushes (conjure only, forwarded from `-diags`).

|===

NOTE: The three enrichment files (`buildkit_metadata.json`, `cache_before.json`, `cache_after.json`) are present only for conjure consecrations.
They are forwarded from the `-diags` registry artifact by the about pipeline's step 1.
For bind and graft, these files are absent and the about artifact contains only `build_info.json`, `sbom.json`, and optionally `recipe.txt`.

===== OCI Layout Bridge — Superseded

The original OCI Layout Bridge (RBSOB) — export to OCI tarball, push via crane,
split with skopeo — is fully superseded for all vessel types.
Single-platform vessels use `buildx --load` into the local daemon.
Multi-platform vessels use `buildx --push` with per-platform pullback.
CB pre-populates GAR credentials; no explicit docker login step is needed.
See RBSOB superseded notice.

==== Potential Hardening: Egress Lockdown

Set `egressOption: NO_PUBLIC_EGRESS` on the Private Pool.
Workers lose all public internet access.
This would be the largest cascading change in the pipeline's history.

**Dependency mirroring required.**
Every external resource fetched during build must be pre-staged in GAR or GCS:

[cols="1,2,1",options="header"]
|===
|Build Step
|External Dependency
|Mitigation

|binfmt step
|`tonistiigi/binfmt` from Docker Hub
|Mirror to GAR

|build step
|Base images from Docker Hub (per vessel Dockerfile)
|Mirror to GAR; enforce GAR-only `FROM` refs

|sbom step
|`anchore/syft` from Docker Hub
|Mirror to GAR
|===

**Ongoing maintenance cost.**
Mirror refresh becomes a recurring operation.
The existing pin-refresh workflow expands to include mirroring.
Every tool image security update requires a mirror-refresh-and-commit cycle.

**Additional infrastructure:**

* Private Cloud DNS zone for internal hostname resolution
* NAT/proxy if any exception egress paths are needed (treat as security event)
* All `RBRR_GCB_*_IMAGE_REF` pins must point to GAR mirrors, not upstream registries

**Note:** The rubric repo architecture (Current Posture) already narrows the build context to
per-vessel directories — Cloud Build never fetches from the main repo.
This limits the egress surface to builder images and base images only.

==== Potential Hardening: VPC Service Controls Perimeter

Wrap the depot project in a VPC Service Controls (VPC-SC) perimeter.
This restricts which GCP APIs can be called and from where, reducing data exfiltration risk.

**Required infrastructure:**

* VPC-SC perimeter around the depot project (organizational policy — may require org-level admin)
* Access levels for operator workstations (so builds can still be submitted from outside the perimeter)
* Ingress/egress rules for the perimeter (which services may cross the boundary)
* Testing: every GCP API call in the pipeline must be verified inside the perimeter — Cloud Logging writes, GCS uploads, Artifact Registry pushes, Cloud Build API calls can break subtly

**Prerequisite:** Egress Lockdown should be in place for the perimeter to be meaningful.

==== Deferred Items

Evaluated and consciously deferred; each has a specific revisit trigger.

**Dockerfile base image digest validation.**
Builder step images are digest-pinned (Current Posture), but user-authored Dockerfiles
in vessel build contexts are not validated.
A vessel with `FROM ubuntu:22.04` passes today — mutable tag, supply-chain risk.
This constrains vessel authors (policy decision with RBSRV implications),
orthogonal to pipeline infrastructure.
Separate future heat.

**BuildKit provenance flag.**
`--provenance=true` on `docker buildx build` records builder internals —
useful for deeper forensics.
Google's trigger-level SLSA v1.0 provenance covers the primary attestation use case.
Redundant for the current threat model.
Revisit if build reproducibility verification becomes a priority.

**Cosign image signing.**
Cryptographic signatures attached to images, verifiable by downstream consumers.
Two modes: keyless (Sigstore/Fulcio — requires public internet, conflicts with Egress Lockdown)
and keyed (Cloud KMS — no external dependency).
With CB-native SLSA v1.0 Build Level 3 verified on all vessel types,
cosign is redundant for the current threat model (building and consuming within own GCP project).
Becomes valuable if: images are distributed externally, trust verification needed outside GCP,
or builder strategy expands beyond GCB (losing Google's provenance).
If adopted, prefer KMS-backed keyed signing (`cosign sign --key gcpkms://...`)
for Egress Lockdown compatibility.

**Binary Authorization.**
Deploy-time policy enforcement: only attested images can be deployed.
Relevant for GKE and Cloud Run.
Now technically achievable — CB-native SLSA Level 3 satisfies BinAuth attestation requirements.
Recipe Bottle deploys to Podman on workstations — no benefit until deployment targets change.

**Reproducibility verification.**
Periodic independent rebuilds compared by digest.
Requires reproducible toolchains — a vessel-author discipline, not a pipeline control.
Not actionable until vessel ecosystem matures.
