{rbtgo_manor_jilt}

NOTE: This operation dissolves one foedus's trust between a
{rbtgi_manor}
and an external OIDC identity provider — the
{rbtf_jilt}
act, the structural inverse of
{rbtf_affiance}.
It deletes that foedus's provider from the manor's one workforce pool; the pool itself
stands, untouched. Deleting a provider ends federated access for citizens asserted
through it, while a
{rbtgi_depot}
drawing on another foedus is unaffected — a deliberately confirmed act, but no longer
the manor-wide break that tearing down the shared pool would be (that is the separate
manor-teardown, the finisher's inverse).

The
{rbtr_payor}
performs the jilt. The operation takes the target
{rbtf_foedus}
as a folio (param1) — the `rbef_`-sprued library subdirectory name, exactly as
{rbtf_affiance}
and
{rbtgo_foedus_descry}
take theirs — and deletes THAT named foedus's provider (`RBRF_PROVIDER_ID`, read from
the folio-resolved Federation regime) under the manor pool `RBRW_WORKFORCE_POOL_ID` — the
same provider
{rbtf_affiance}
seated for that folio. Provider management is folio-addressed, never the active or pinned
foedus; the
{rbrr_active_foedus}
selector is reserved for the credential accessor. An unresolvable folio is a precondition
rejection naming the absent foedus.

NOTE: *Mechanism-blind.* Jilt carries no
{rbrf_mechanism}
arm, the structural asymmetry with
{rbtf_affiance}.
Deleting the provider ends the trust regardless of how citizens acquired tokens
against it — the interactive device flow and the programmatic self-supplied JWT alike
validate against a provider that no longer exists, so an
{rbrf_mechanism_programmatic}
foedus is torn down by exactly the same gesture as an interactive one. Jilt's one
interactive touch — the operator confirmation below — is governed by the orthogonal
confirm-skip seam, not the mechanism gate.

NOTE: Jilt is *idempotent*. A provider that never existed or was already deleted (the
probe returns `404`) is reported already-dissolved and the operation exits clean
without issuing a delete. Only a present provider is deleted.

NOTE: *Provider-only deletion.* Jilt deletes exactly the provider
{rbtf_affiance}
created
(`//iam.googleapis.com/locations/global/workforcePools/RBRW_WORKFORCE_POOL_ID/providers/RBRF_PROVIDER_ID`)
and nothing else — the shared pool stands for the next foedus and for every other
already standing. This is the
{rbtf_jilt}
cinch — dissolve one betrothal, never the pool the manor keeps.

NOTE: *Re-affiance recreates.* A jilted provider is gone from the pool; a subsequent
{rbtf_affiance}
of the same foedus recreates it under the standing pool (ensure-exists,
created-when-absent), no graveyard to clear. The pool's own soft-delete graveyard and
the refuse-and-rotate ruling belong to the manor-teardown act that deletes the pool
({rbtgo_depot_unmake} being the depot-project analogue), not to jilt — jilt never
reaches the pool.

NOTE: Scope boundary. Jilt unmakes only one provider — one foedus's IdP trust. The
shared pool and the org-level admin grant (spike Finding F1) are the manor-teardown's,
never jilt's; the depot-scoped admission bindings (the
{rbtf_mantle}
service-account `tokenCreator` and the depot `serviceUsageConsumer` of spike Finding
F2) and the mantle service accounts belong to the levy and admission ceremonies and
their own teardowns. Jilt is affiance's exact inverse — the provider, and only the
provider.

//axhos_step
. *Safety confirmation* (zero traffic — the deliberate pause precedes authentication):
.. {rbbc_require}
operator confirmation matching the provider id `RBRF_PROVIDER_ID`.
Deleting the provider ends federated access for citizens asserted through this foedus;
the confirmation is the gate. The standard confirm-skip seam suppresses the interactive
prompt for non-interactive test runs.

//axhos_step
. *Authenticate as
{rbtr_payor}*:
.. Execute
{rbtoe_payor_authenticate}
.. {rbbc_require}
authentication successful

//axhos_step
. *Probe the provider*:
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/locations.workforcePools.providers/get[workforcePools.providers.get] for `RBRF_PROVIDER_ID` under `locations/global/workforcePools/RBRW_WORKFORCE_POOL_ID`
.. {rbbc_store}
`«PROVIDER_CODE»` (HTTP status)
.. *Already-dissolved short-circuit* — exit clean (idempotent no-op), no delete issued, when `«PROVIDER_CODE»` is `404` (no provider of this id under the pool).
.. Any other non-`200` is
{rbbc_fatal}.
.. A present provider (`200`) advances to deletion.

//axhos_step
. *Delete the provider*:
.. {rbbc_submit}
link:https://cloud.google.com/iam/docs/reference/rest/v1/locations.workforcePools.providers/delete[workforcePools.providers.delete] for the provider resource name
.. {rbbc_require}
the delete accepted (HTTP `200`, returning a long-running operation)
.. NOTE: only the provider is deleted; the manor pool is never touched by jilt.

//axhos_step
. *Verify dissolution*:
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/locations.workforcePools.providers/get[workforcePools.providers.get] for the provider resource name
.. {rbbc_poll}
the provider at `RBGC_EVENTUAL_CONSISTENCY_SEC` intervals up to `RBGC_MAX_CONSISTENCY_SEC`
.. {rbbc_require}
a dissolved terminal: `state: DELETED` *or* a `404` (gone). Either is success — the
verify tolerates whether the GET surfaces a soft-deleted provider, mirroring the
depot-unmake resource-state poll
({rbtgo_depot_unmake}).

//axhoot_typed_output
{rbtgi_manor}
. *Confirm Jilt*:
.. {rbbc_show}
single-line banner naming the
{rbtgi_manor},
the dissolved provider resource name, the dissolved terminal reached, and the
re-affiance recovery hint (a fresh affiance recreates the provider under the standing pool).

//axhoc_completion
Successful execution leaves the foedus jilted: its provider is deleted from the manor
pool, and no
{rbtf_citizen}
can
{rbtf_avow}
against that dissolved trust until a fresh
{rbtf_affiance}
re-seats the provider under the standing pool. The pool itself, every other foedus's
provider, and the depot-scoped
{rbtf_mantle}
bindings and mantle service accounts are untouched: the pool is the manor-teardown's
concern and the bindings the levy and admission concern, never jilt's.
