//axhob_operation
{rbtgo_director_vouch}

NOTE: This operation vouches for all pending
{rbtga_ark}
hallmarks across all
{rbtgi_vessel}
repositories.
The
{rbtr_director}
enumerates vessels, identifies hallmarks whose subtree contains an `about` basename but no `vouch` basename, pre-checks SLSA provenance availability for conjure-mode hallmarks, and submits individual vouch builds for each vouchable hallmark.
Conjure hallmarks lacking provenance are skipped with a warning; bind and graft hallmarks are always vouchable.

The
{rbtr_director}
batch-vouches arks:

//axhos_step
. *Enumerate Vessels*:
.. Invoke `rbrv_list_capture` to discover all vessel sigils
.. {rbbc_require}
at least one vessel exists

//axhos_step
. *Iterate Vessels*:
.. For each vessel sigil:
... Load vessel configuration via `zrbf_load_vessel`
... {rbbc_call}
Docker Registry API: enumerate packages under `hallmarks/<hallmark>/` for the vessel
... For each hallmark subtree, record which basenames are present (`image`, `about`, `vouch`, `pouch`, `diags`, `attest`)

//axhos_step
. *Classify Hallmark Health*:
.. For each unique hallmark across all vessels, determine health state by basename presence under its `hallmarks/<hallmark>/` subtree:
... *vouched*: `about` and `vouch` basenames both present — healthy, skip
... *vouchable*: `about` present, `vouch` absent, and either (a) mode is bind/graft, or (b) mode is conjure and SLSA provenance exists — queue for vouch
... *unvouchable*: `about` present, `vouch` absent, mode is conjure, no SLSA provenance — skip with warning
... *incomplete*: `image` present but `about` absent — about pipeline has not run, skip with warning
.. Vessel mode is derived from the hallmark mode prefix (`c`=conjure, `b`=bind, `g`=graft)
.. {rbbc_warn}
if hallmark mode prefix does not match loaded {rbrv_vessel_mode} — may indicate vessel reconfiguration or corruption

//axhos_step
. *Pre-check Provenance* (conjure only):
.. For each conjure-mode hallmark with `about` present and `vouch` absent:
... {rbbc_call}
`gcloud artifacts docker images describe --show-provenance` to test provenance availability
... If provenance unavailable: classify as *unvouchable*, continue to next
.. Bind and graft hallmarks skip this step — they are always vouchable

//axhos_step
. *Submit Vouch Builds*:
.. For each *vouchable* hallmark:
... Invoke `rbf_vouch` with vessel directory and hallmark parameters
... {rbbc_require}
vouch completes successfully

//axhoot_typed_output
{rbtga_ark_vouch}
. *Display Summary*:
.. {rbbc_show}
per-vessel table: hallmark, health state, action taken
.. {rbbc_show}
batch totals: N vouched, M skipped (no provenance), P already vouched, Q incomplete

//axhoc_completion
Successful execution creates
{rbtga_ark_vouch}
artifacts for all vouchable hallmarks across all vessels.
Conjure hallmarks without SLSA provenance are skipped gracefully — the batch never aborts on these.
Bind and graft hallmarks are always vouchable (no provenance pre-check needed).
+
*No-provenance resilience* (conjure only): Older conjure builds may lack SLSA provenance records.
Rather than submitting a Cloud Build that fails with an opaque `invalid DSSE envelope payload` error, the batch pre-checks provenance availability and skips with a clear message.
+
*Mode-aware vouch*: The batch delegates to {rbtgo_ark_vouch}, which branches on vessel mode internally.
Bind gets digest-pin verification; graft gets GRAFTED stamp; conjure gets full SLSA verification.
