//axhob_operation
{rbtgo_director_vouch}

NOTE: This operation vouches for every *pending*
{rbtga_ark}
in
{rbtgi_repository},
where *pending* means a hallmark whose subtree contains both
{rbtga_ark_image}
and
{rbtga_ark_about}
basenames but no
{rbtga_ark_vouch}.
The
{rbtr_director}
enumerates the full hallmark corpus via a single GAR REST `packages.list` call (the same shared helper used by
{rbtgo_hallmark_tally}),
identifies pending hallmarks, resolves the vessel name from each hallmark's
{rbtga_ark_about}
metadata (since vessel identity no longer lives in the GAR path), and dispatches
{rbtgo_ark_vouch}
per hallmark.
Mode-awareness (conjure, bind, graft) lives inside
{rbtgo_ark_vouch};
batch vouch does not branch on mode.

NOTE: *Fail-fast semantics.* If
{rbtgo_ark_vouch}
fails for any hallmark, the batch terminates via fatal error at that hallmark.
There is no skip-with-warning path and no summary-count-masking of failures: a broken hallmark demands immediate attention, not quiet continuation.

The
{rbtr_director}
batch-vouches arks:

//axhos_step
. *Authenticate as
{rbtr_director}*:
.. Execute
{rbtoe_director_authenticate}
.. {rbbc_store}
`«DIRECTOR_TOKEN»` for subsequent API calls

//axhos_step
. *Enumerate Hallmark Packages*:
.. {rbbc_call}
the shared hallmark enumeration helper — a single
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories.packages/list[GAR REST API: packages.list]
call under `«HALLMARKS_ROOT»/` (`RBGL_HALLMARKS_ROOT`), filtered to `<hallmark>/<basename>` shape
.. {rbbc_store}
`«PAIRS»` as sorted `<hallmark> <basename>` pairs (same shape produced for
{rbtgo_hallmark_tally})

//axhos_step
. *Identify Pending Hallmarks (Pass 1)*:
.. Iterate `«PAIRS»` in sort order; each contiguous run of the same hallmark is one
{rbtga_hallmark}'s
ark inventory.
.. For each hallmark, record whether
{rbtga_ark_image},
{rbtga_ark_about},
and
{rbtga_ark_vouch}
each appear in its run.
.. A hallmark is *pending* when `image` and `about` are present and `vouch` is absent.
Collect pending hallmarks into `«PENDING»` (in-memory sequence).
.. If `«PENDING»` is empty, the procedure completes successfully with no vouches submitted.

//axhos_step
. *Resolve Vessel and Vouch Per Pending Hallmark (Pass 2)*:
.. For each `«HALLMARK»` in `«PENDING»`:
... Extract the hallmark's
{rbtga_ark_about}
artifact (package `«HALLMARKS_ROOT»/«HALLMARK»/about`, tag `«HALLMARK»`) to a scratch directory via the shared GAR artifact extraction helper
... Read `.vessel_name` from the extracted `build_info.json` via `jq`
... {rbbc_store}
`«VESSEL_DIR»` as `«RBRR_VESSEL_DIR»/«VESSEL_NAME»`
... {rbbc_require}
`«VESSEL_DIR»` exists on disk — the vessel named in `about` metadata must correspond to a configured vessel in the current regime
... {rbbc_fatal}
if the `about` artifact cannot be extracted, `build_info.json` is missing, `.vessel_name` is empty, or `«VESSEL_DIR»` does not exist — the `about` artifact is the sole source of vessel identity for this batch; a gap there is a fail-fast stop
... Invoke
{rbtgo_ark_vouch}
with `«VESSEL_DIR»` and `«HALLMARK»`.
{rbtgo_ark_vouch}
loads the vessel's
{rbrv_vessel_mode}
internally and dispatches the appropriate verification (conjure: SLSA DSSE verification; bind: digest-pin; graft: GRAFTED stamp).
... {rbbc_fatal}
if
{rbtgo_ark_vouch}
fails — the batch terminates at the broken hallmark.

//axhoot_typed_output
{rbtga_ark_vouch}
. *Display Summary*:
.. {rbbc_show}
`Batch vouch complete — N/N hallmark(s) vouched` where N is the initial pending count

//axhoc_completion
Successful execution creates one
{rbtga_ark_vouch}
artifact per pending hallmark (N total, where N = initial pending count).
The batch runs sequentially; a failure at any hallmark halts the remainder.
+
*Scope of vessel resolution*: Reading `vessel_name` from `build_info.json` is a narrow anticipation of the broader vessel-from-metadata pattern (AAL territory) — applied locally inside batch vouch only, because
{rbtgo_ark_vouch}
's signature still takes a vessel directory.
Other operations (tally, plumb, abjure) continue to not surface vessel.
+
*Two-pass shape*: Pass 1 collects pending hallmarks into memory before Pass 2 begins processing.
Each
{rbtgo_ark_vouch}
invocation sources credentials and issues Cloud Build calls that would silently consume loop stdin if the pending set were streamed; the in-memory collection decouples enumeration from dispatch.
+
*Idempotency*: Re-running batch vouch after a successful batch finds an empty pending set (every prior pending hallmark now has `vouch` present) and exits with no action.
+
*No mode pre-check*: Earlier shapes of this operation pre-probed SLSA provenance for conjure-mode hallmarks to skip-with-warning those lacking provenance.
That pattern is gone — all mode-specific verification happens inside
{rbtgo_ark_vouch},
and any failure there surfaces as a fail-fast stop rather than being absorbed into a summary count.
