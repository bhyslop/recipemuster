//axhob_operation
{rbtgo_director_yoke}

NOTE: This operation writes `RBRV_RELIQUARY=«STAMP»` into every vessel's `rbrv.env` under `${RBRR_VESSEL_DIR}` so that subsequent Cloud Build operations (conjure, bind, graft, enshrine) consume the named tool image set as GCB step images.
The about/vouch pipeline (gcloud, docker, alpine, syft) consumes reliquary tool images uniformly across all ordain modes; bind additionally uses skopeo for image-copy.
The stamp is validated once against GAR before any vessel file is touched: all six tool packages must be present under the reliquary subtree.
On stamp validation failure the operation errors diagnostically without writing any vessel.
On per-vessel write failure the diagnostic identifies the failing vessel and the set of vessels already written, so the operator can resume from a known point.
This primitive does not commit; the caller (orchestrator or operator) commits the rbrv.env changes.

The
{rbtr_director}
yokes a reliquary stamp across every vessel:

//axhopt_typed_parameter
{rbtr_director}
. *Validate Input Parameters*:
.. {rbbc_require}
`«STAMP»` provided as first positional argument (e.g., `r260327172456`)

//axhos_step
. *Authenticate as
{rbtr_director}*:
.. Execute
{rbtoe_director_authenticate}
.. {rbbc_store}
`«DIRECTOR_TOKEN»` for subsequent API calls

//axhos_step
. *List Reliquary Packages in GAR*:
.. {rbbc_store}
`«DEPOT_PROJECT_ID»` as `${RBRR_CLOUD_PREFIX}d-${RBRR_DEPOT_MONIKER}` (resolved by `RBDC_DEPOT_PROJECT_ID`)
.. {rbbc_store}
`«GAR_REPOSITORY»` as `${RBRR_CLOUD_PREFIX}${RBRR_DEPOT_MONIKER}-gar` (resolved by `RBDC_GAR_REPOSITORY`)
.. {rbbc_call}
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories.packages/list[packages.list] for
`«DEPOT_PROJECT_ID»`
/
{rbrr_gcp_region}
/
`«GAR_REPOSITORY»`
with `pageSize=1000`
.. {rbbc_require}
HTTP 200 response
.. Filter response package names to those starting with `{gar_reliquaries_namespace}/«STAMP»/`
.. Strip the subtree prefix to yield the bare tool-name set present in GAR

//axhos_step
. *Verify Tool Image Completeness*:
.. Expected tool name set: `gcloud`, `docker`, `alpine`, `syft`, `binfmt`, `skopeo` (per
{rbtgo_depot_inscribe})
.. {rbbc_require}
all six names present in the filtered list
.. {rbbc_fatal}
if any expected tool is missing — the error message names `«STAMP»`, the reliquary subtree path, and the specific missing tool(s); remediation points at
{rbtgo_depot_inscribe}

//axhos_step
. *Iterate Vessels under `${RBRR_VESSEL_DIR}`*:
.. Enumerate every immediate subdirectory of `${RBRR_VESSEL_DIR}` containing an `rbrv.env` file
.. {rbbc_require}
at least one vessel found — empty enumeration is a fatal error (nothing to yoke)
.. For each enumerated vessel, perform *Rewrite rbrv.env* below in directory order
.. Maintain a running list of vessels successfully written; emit per-vessel progress

//axhos_step
. *Rewrite rbrv.env* (per vessel):
.. Load the vessel's `rbrv.env` into memory, line by line
.. For each line matching `^RBRV_RELIQUARY=` — substitute with `RBRV_RELIQUARY=«STAMP»`, preserving position
.. All other lines pass through unchanged
.. If no `RBRV_RELIQUARY=` line was found — append one under a `# Tool Image Reliquary` comment header
.. Write the composed content to a per-vessel temp file, then atomically replace `rbrv.env` via rename
.. {rbbc_fatal}
on any per-vessel I/O failure — the error message names the failing vessel and the list of vessels already written, so the operator can resume from a known point

//axhoc_completion
Successful execution writes `RBRV_RELIQUARY=«STAMP»` into the `rbrv.env` of every vessel discovered under `${RBRR_VESSEL_DIR}`; no GAR state is modified.
The primitive does not commit — the caller (orchestrator or operator) commits the modified `rbrv.env` files with their usual git workflow.
The reliquary tool images are now linked, but vessel images must be rebuilt
({rbtgo_ark_ordain})
to pick up the new tool versions.
