//axhob_operation
{rbtgo_depot_levy}

NOTE: This operation creates an
{rbtgi_depot}
- a cohesive unit of Recipe Bottle infrastructure including the depot project, build bucket, container repository, Mason service account, and private worker pool
that share their productive lifecycle.

The
{rbtr_payor}
creates a new depot infrastructure from the
{rbtgi_payor_project}.
The operation accepts no CLI parameters; depot identity is read from the kindled regime as `(RBRD_CLOUD_PREFIX, RBRD_DEPOT_MONIKER)`.

. *Create Private Worker Pools*:
.. For each worker pool variant in [`tether`, `airgap`]:
... {rbbc_submit}
link:https://cloud.google.com/build/docs/api/reference/rest/v1/projects.locations.workerPools/create[workerPools.create] with:
.... parent: `projects/«DEPOT_PROJECT_ID»/locations/{rbrd_gcp_region}`
.... workerPoolId: `«WORKER_POOL_STEM»-«VARIANT»`
.... body for `tether`:
+
----
{
  "privatePoolV1Config": {
    "workerConfig": {
      "machineType": "«RBRD_GCB_MACHINE_TYPE»"
    }
  }
}
----
.... body for `airgap`:
+
----
{
  "privatePoolV1Config": {
    "workerConfig": {
      "machineType": "«RBRD_GCB_MACHINE_TYPE»"
    },
    "networkConfig": {
      "egressOption": "NO_PUBLIC_EGRESS"
    }
  }
}
----
... {rbbc_await}
returned operation
.. NOTE:
{rbrd_gcb_machine_type}
is a Compute Engine machine type (e.g., `e2-standard-2`), consumed at pool creation time.
Pool scales to zero workers when idle — no idle cost accrues.
.. NOTE: The two variants differ only in egress posture. The `airgap` body declares `networkConfig.egressOption: NO_PUBLIC_EGRESS`, which Cloud Build enforces at pool level — builds routed through the airgap pool cannot reach the public internet regardless of build configuration. Awaiting the LRO terminal state at create time surfaces machine-type quota denial here rather than as a downstream "workerpool not found" at the subsequent per-pool probe submission step.

//axhos_step

.. {rbbc_poll}
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories/list[artifactregistry.repositories.list] for `«DEPOT_PROJECT_ID»` in `{rbrd_gcp_region}`
.. Poll at `RBGC_EVENTUAL_CONSISTENCY_SEC` intervals until HTTP 200 or `RBGC_MAX_CONSISTENCY_SEC` timeout
.. {rbbc_require}
HTTP 200 response confirming IAM propagation complete
.. {rbbc_fatal}

//axhos_step
. *Create Build Bucket*:
.. {rbbc_call}
link:https://cloud.google.com/storage/docs/json_api/v1/buckets/insert[storage.buckets.insert] with:
... name: `«BUILD_BUCKET»`
... location: `{rbrd_gcp_region}`
... project: `«DEPOT_PROJECT_ID»`
... lifecycle rule: delete objects after 1 day

//axhos_step
. *Create Container Repository*:
.. {rbbc_submit}
link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories/create[artifactregistry.repositories.create] with:
... repositoryId: `«GAR_REPOSITORY»`
... format: `DOCKER`
... location: `{rbrd_gcp_region}`
... parent: `projects/«DEPOT_PROJECT_ID»/locations/{rbrd_gcp_region}`
... cleanupPolicyDryRun: `false` (active mode — eligible artifacts are deleted, not just logged)
... cleanupPolicies: a map keyed by policy ID with a single entry:
+
----
{
  "rb-delete-untagged": {
    "id": "rb-delete-untagged",
    "action": "DELETE",
    "condition": {
      "tagState": "UNTAGGED",
      "olderThan": "86400s"
    }
  }
}
----
.. {rbbc_await}
returned operation
.. NOTE: The cleanup policy underwrites the multi-platform contract documented in {rbtgo_image_delete}. V2 DELETE-by-tag against an OCI image index removes only the parent manifest, leaving per-platform children as untagged orphan content. GAR's cleanup background job (executes ~daily on Google's schedule, exact time unspecified) deletes manifests matching this policy on its next run; layer storage is reclaimed by GAR's separate daily layer-GC job. The 86400s (1-day) `olderThan` is measured from manifest creation time, not from when the manifest became untagged — orphans become eligible once they have existed in GAR for at least one day and are reaped on the next subsequent daily cleanup execution.
.. NOTE: REST schema verified against canonical sources. `cleanupPolicies` is a *map* keyed by policy ID, not an array — the gcloud CLI policy-file format used with `gcloud artifacts repositories set-cleanup-policies --policy=FILE` is a different (array, nested `{type: "Delete"}` action) schema that does not apply to this REST call. `action` is a *string enum* with values `ACTION_UNSPECIFIED | DELETE | KEEP`. `condition.tagState` enum values: `TAGGED | UNTAGGED | ANY`. `condition.olderThan` is a Duration-format string (seconds + `s` suffix). Maximum 10 cleanup policies per repository (this clause uses one). Schema sources for future re-verification:
... link:https://cloud.google.com/artifact-registry/docs/reference/rest/v1/projects.locations.repositories[Repository resource reference] — `cleanupPolicies` and `cleanupPolicyDryRun` field shapes
... link:https://artifactregistry.googleapis.com/$discovery/rest?version=v1[Discovery document v1] — canonical CleanupPolicy / CleanupPolicyCondition schemas (authoritative for action enum values, condition field types)
... link:https://cloud.google.com/artifact-registry/docs/repositories/cleanup-policy[Configure cleanup policies] — concept guide, daily-cadence statement, dry-run semantics

//axhos_step
. *Submit Per-Pool Probe Builds*:
.. For each worker pool variant in [`tether`, `airgap`]:
... {rbbc_submit}
link:https://cloud.google.com/build/docs/api/reference/rest/v1/projects.locations.builds/create[builds.create] with:
.... `serviceAccount`: `projects/«DEPOT_PROJECT_ID»/serviceAccounts/«MASON_EMAIL»`
.... `options.pool.name`: full pool resource path for the variant
.... `options.logging`: `CLOUD_LOGGING_ONLY`
.... `steps[0]`: posture-check assertion (builder: `gcr.io/google.com/cloudsdktool/cloud-sdk:alpine`)
..... tether asserts `https://example.com` reachable AND `https://storage.googleapis.com` reachable (curl without `-f`; any HTTP response counts as reached)
..... airgap asserts `https://example.com` connection blocked AND `https://storage.googleapis.com` reachable
.... `steps[1]`: GAR write probe (builder: `gcr.io/cloud-builders/docker`) — `FROM scratch` image with a single marker file, pushed to `«REGION»-docker.pkg.dev/«DEPOT_PROJECT_ID»/«GAR_REPOSITORY»/rbi_df/probe-«VARIANT»:probe`
... {rbbc_require}
submission returns HTTP 200/201 (build accepted). All other codes are fatal.
... {rbbc_await}
build terminal state via `builds.get` polled every `ZRBGP_BUILD_POLL_INTERVAL` seconds, with a `ZRBGP_BUILD_POLL_CEILING` ceiling (10 minutes total). SUCCESS is the expected terminal state on correct posture; FAILURE means the in-build assertion tripped (egress-posture mismatch) and operators investigate via the console link. Any other terminal state (CANCELLED / TIMEOUT / EXPIRED / INTERNAL_ERROR) indicates Cloud Build infrastructure trouble and is similarly diagnostic, not silently accepted.
... {rbbc_show}
per-pool probe block carrying the pool ID, GAR artifact reference, build ID + Cloud Build console link, and terminal status.
.. NOTE: Builder images are Google-hosted only (`gcr.io/...`). The airgap pool cannot reach `docker.io`; any non-Google builder would hang the build.
.. NOTE: Submission itself populates the per-pool concurrent-build quota row in Google's Cloud Build console UI — what the preflight diagnostic calls "after some build activity." A SUCCESS terminal probe confirms three things end-to-end: the pool can dispatch builds, the worker exhibits the egress posture stamped into its network config, and the worker can authenticate to GAR and push to `rbi_df`. Operators bump per-pool quota afterward via the now-populated console row to enable subsequent parallel gauntlet activity.
.. NOTE: Submission awaits terminal state inline. The two iterations of the loop run serially because the await blocks the loop body before the next submission; this is load-bearing. An earlier non-awaiting variant caused the late-clearing-probe-collision wedge — the airgap probe queued for ~19 minutes behind subsequent gauntlet dispatches, and when it finally cleared, the next tether conjure (e.g., `ordain_conjure_jupyter`) wedged in `QUEUED` until `queueTtl=3600s` expired the build. See `Memos/memo-20260517-cloudbuild-default-quota-wedge/`. The serialized await also makes the airgap probe's slot acquisition fast in the no-contention case, since no other gauntlet builds are competing during the levy window.
.. NOTE: In-build posture-assertion failure (e.g., airgap pool reaching the public internet, tether pool unable to resolve `storage.googleapis.com`) surfaces in the build's terminal status as FAILURE — a hard infra-drift signal. The posture-check script uses `curl` without `-f` so that HTTP 4xx/5xx responses from a reached server still count as "reached"; only honest network failures (DNS, connection refused, timeout) trip the assertion.
.. NOTE: The `rbi_df` namespace is a flat sibling of `rbi_hm` / `rbi_rq` / `rbi_es` under the GAR repository. {rbtgo_image_audit} / {rbtgo_image_rekon} enumerators ignore it by design — probe artifacts are operational, not part of the hallmark / reliquary / enshrine image catalogue. Filenames under `rbi_df` form a flat sea (no subdirs); each filename names one depot-scoped OCI artifact.

. *Inscribe Depot Regime Tripwire*:
.. Build a `FROM scratch` image carrying the {rbrd_regime} assignment file and push it host-side to `«REGION»-docker.pkg.dev/«DEPOT_PROJECT_ID»/«GAR_REPOSITORY»/rbi_df/rbrd:tripwire`, authenticating to GAR with the Payor OAuth access token already in scope.

{rbrd_regime}
(CLOUD_PREFIX, DEPOT_MONIKER, GCP_REGION, GCB_MACHINE_TYPE) into GAR. Every cloud-submitting command thereafter pulls and exact-byte-diffs the inscribed copy against the local {rbrd_regime} assignment file before submitting, so a post-levy hand-edit fails loud rather than silently diverging from the depot's actual worker-pool quotas, region, and identity. Implemented host-side (Cloud Build is not involved) by `rbrd_inscribe` in `Tools/rbk/rbndb_base.sh`. The tripwire push runs after the depot-tracking refresh; a failure here leaves the depot live but un-inscribed, recoverable via `tt/rbw-rdi.sh` once the cause is resolved.
