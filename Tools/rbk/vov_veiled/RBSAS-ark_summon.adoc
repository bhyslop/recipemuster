//axvd_sheaf rbs0_imprimatur axd_normative
//axhob_operation
{rbtgo_ark_summon}

NOTE: This operation summons an
{rbtga_ark}
from
{rbtgi_repository},
pulling
{rbtga_ark_image},
{rbtga_ark_about},
and
{rbtga_ark_vouch}
artifacts to the local container runtime. Uses
{rbtr_retriever}
credentials when available, falling back to
{rbtr_director}.

NOTE: This is the ark-level view of image retrieval. For single-artifact retrieval, see
{rbtgo_image_retrieve}.

NOTE: *Read-side chain consumer.*
The hallmark is resolved express-or-chain for a transient retrieval only — summon writes no durable config — so a broken chain dies loud as an ordinary {rbbc_fatal}, categorically lower severity than the durable-leak links
{rbtgo_director_feoff}
and
{rbtgo_director_yoke},
whose rejections ride the chaining precision band.
Unlike those kind-gated links the resolve carries no kind decode: a
{rbst_hallmark}
bears no sub-kind these verbs discriminate, so the express-or-chain resolution is the whole typecheck.
Summon must never be extended to write config without joining that durable-leak surface and its depth-1 no-relay plus named-band-reject discipline.

The summon operation:

//axhopt_typed_parameter
{rbst_hallmark}
. *Resolve the Hallmark* (express-or-chain):
.. `«HALLMARK»` resolves express-or-chain: an express folio argument (e.g., `c260305133650-r260305160530`) wins; absent, it falls back to the hallmark a prior build ({rbtgo_ark_ordain} or {rbtgo_ark_kludge}) handed forward through the depth-1 cross-tabtarget chain — so a no-arg summon immediately after a build pulls the just-built hallmark. The chain is read terminally — never relayed.
.. {rbbc_fatal}
if neither an express argument nor a chained hallmark is present — a broken chain

//axhos_step
. *Authenticate as
{rbtr_retriever}*:
.. Execute
{rbtoe_retriever_authenticate}
.. Fall back to
{rbtoe_director_authenticate}
if retriever credentials unavailable
.. {rbbc_store}
authentication token for subsequent API calls

//axhos_step
. *Construct Ark Tags*:
.. {rbbc_store}
image tag as `{gar_hallmarks_namespace}/«INPUT_HALLMARK»/image:«INPUT_HALLMARK»`
.. {rbbc_store}
about tag as `{gar_hallmarks_namespace}/«INPUT_HALLMARK»/about:«INPUT_HALLMARK»`
.. {rbbc_store}
vouch tag as `{gar_hallmarks_namespace}/«INPUT_HALLMARK»/vouch:«INPUT_HALLMARK»`

//axhos_step
. *Verify Ark Existence*:
.. {rbbc_call}
Docker Registry API HEAD requests for all three artifacts
.. {rbbc_store}
image existence status
.. {rbbc_store}
about existence status
.. {rbbc_store}
vouch existence status

//axhos_step
. *Evaluate Ark State*:
.. If both artifacts missing:
... {rbbc_fatal}
"Ark not found"
.. If exactly one missing:
... {rbbc_warn}
"Orphaned artifact detected"

//axhos_step
. *Docker Login and Pull*:
.. {rbbc_call}
docker login to registry
.. For each existing artifact:
... {rbbc_call}
docker pull to retrieve artifact to local container runtime

//axhoot_typed_output
{rbtga_ark}
. *Display Results*:
.. {rbbc_show}
which artifacts were retrieved

//axhoc_completion
Successful execution retrieves the {rbtga_ark} artifacts to the local container runtime, making them available for use.
Once locally present, {rbtgo_ark_inspect} can display the trust posture from the about and vouch artifacts.
