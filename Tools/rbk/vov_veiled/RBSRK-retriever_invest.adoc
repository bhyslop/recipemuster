//axhob_operation
{rbtgo_retriever_invest}

NOTE: This operation invests a
{rbtr_retriever}
{giam_service_account}
into the
{rbtgi_depot_project}
with read-only
{rbtgi_repository}
access.
The
{rbtr_governor}
mints the SA, generates its credential key, and writes the RBRA bytes to the assay output slot.
The operation is identity-parameterized: a required identity folio (param1) is supplied by the operator; the SA account name is composed as `${RBCC_account_retriever}-<identity>`.
Idempotent establish: a pre-existing SA is not drift — its identity (email, `uniqueId`) and IAM bindings are preserved and only the credential key is rotated; an absent SA is created first. Either path ends with exactly one fresh USER_MANAGED key and the role's bindings in place.

//axhos_step
. *Authenticate as
{rbtr_governor}*:
.. Execute
{rbtoe_governor_authenticate}
.. {rbbc_store}
`«GOVERNOR_TOKEN»` for subsequent API calls

//axhos_step
. *Compose
{rbtr_retriever}
Service Account Identity*:
.. {rbbc_require}
identity folio (param1) is non-empty
.. {rbbc_store}
`«RETRIEVER_ACCOUNT_NAME»` as `${RBCC_account_retriever}-<identity>` (operator-supplied identity composing the account name)
.. {rbbc_store}
`«RETRIEVER_EMAIL»` as `«RETRIEVER_ACCOUNT_NAME»@${RBDC_DEPOT_PROJECT_ID}.iam.gserviceaccount.com`
.. NOTE: `RBDC_DEPOT_PROJECT_ID` is derived from `(RBRD_CLOUD_PREFIX, RBRD_DEPOT_MONIKER)` at regime kindle time; no operator-supplied project ID field is referenced.

//axhos_step
. *Preflight: Detect Existing Service Account*:
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts/get[iam.serviceAccounts.get] for `projects/-/serviceAccounts/«RETRIEVER_EMAIL»`
.. {rbbc_store}
`«SA_EXISTS»` true on HTTP 200 (the SA and its bindings are kept; only the key is rotated below), false on HTTP 404 (the SA is created below)
.. {rbbc_fatal}
on any other HTTP code — an indeterminate preflight is not a re-key signal

//axhos_step
. *Create
{rbtr_retriever}
Service Account* (when `«SA_EXISTS»` is false):
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts/create[iam.serviceAccounts.create] with:
... accountId: `«RETRIEVER_ACCOUNT_NAME»`
... displayName: `Recipe Bottle Retriever (<identity>)`
... description: `Read-only access to Google Artifact Registry - identity: <identity>`
... project:
{rbtgi_depot_project}
.. {rbbc_require}
HTTP 200 response
.. {rbbc_store}
`«RETRIEVER_UNIQUE_ID»` from response `.uniqueId` field
.. NOTE: Skipped when `«SA_EXISTS»` is true — the identity is preserved and `«RETRIEVER_UNIQUE_ID»` is read from the preflight response instead.

//axhos_step
. *Verify Service Account Propagation* (when `«SA_EXISTS»` is false):
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts/get[iam.serviceAccounts.get] for `projects/-/serviceAccounts/«RETRIEVER_EMAIL»`
.. {rbbc_poll}
with backoff until HTTP 200 (newly-created SA may not be immediately resolvable by email)
.. NOTE: Poll must use the email address, not the numeric ID. The email-based lookup path has independent propagation from the UID-based path, and subsequent steps (keys.list, keys.create) all use the email address. Polling by UID can succeed while the email path still returns 404.
.. NOTE: Skipped when `«SA_EXISTS»` is true — a pre-existing SA is already propagated.
.. {rbbc_require}
service account accessible by email

//axhos_step
. *Blunt-Kill Existing Keys*:
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts.keys/list[serviceAccounts.keys.list] for `«RETRIEVER_EMAIL»`
.. Retry up to `RBGC_SA_KEY_CREATE_RETRY_MAX` times on HTTP 404 with `RBGC_SA_KEY_CREATE_RETRY_DELAY_SEC` delay between attempts
.. NOTE: Keys subresource read-path may lag SA email-path readiness — the SA GET probe (previous step) can return 200 while `keys.list` on the same email immediately returns 404, observed as a non-monotonic 200→200→404 flap on freshly-minted depots. The 404 confirms `keys.list` returned no state, making retry safe. Shares budget with step 7 keys.create — same underlying physics (subresource visibility after parent creation). See `Memos/memo-20260514-pristine-gauntlet-iam-flap.md` for transcript evidence and <<rbscip_trade_study>> for the doctrinal pattern.
.. {rbbc_store}
`«EXISTING_KEYS»` as the `.name` of every `.keys[]` entry with `keyType: USER_MANAGED`
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts.keys/delete[serviceAccounts.keys.delete] for each `«EXISTING_KEYS»` entry
.. {rbbc_require}
HTTP 200 OR HTTP 404 (key already gone — idempotent) for each delete
.. NOTE: Re-key rotates the Key lifecycle only — the SA and its IAM bindings are untouched. Deleting every user-managed key before minting guarantees exactly one fresh key when this operation completes (the blunt kill). This replaces the former fail-loud "manual cleanup via Console" abort.

//axhos_step
. *Generate Service Account Key*:
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts.keys/create[serviceAccounts.keys.create] for `«RETRIEVER_EMAIL»` with:
... privateKeyType: `TYPE_GOOGLE_CREDENTIALS_FILE`
.. Retry up to `RBGC_SA_KEY_CREATE_RETRY_MAX` times on HTTP 404 with `RBGC_SA_KEY_CREATE_RETRY_DELAY_SEC` delay between attempts
.. NOTE: SA read operations (GET by email, keys.list) may succeed while keys.create POST returns 404 "does not exist" due to write-path propagation delay. Google link:https://cloud.google.com/iam/docs/service-accounts-create[documents]: "you might need to wait 60 seconds or more before you use the service account." The 404 confirms no key was created, making retry safe. See <<rbscig_iam_grant_contracts>>.
.. {rbbc_store}
`«RETRIEVER_PRIVATE_KEY_DATA»` from response `.privateKeyData` field
.. {rbtoe_rbra_generate}
converts `«RETRIEVER_PRIVATE_KEY_DATA»` to `«RETRIEVER_RBRA_FILE»`

//axhos_step
. *Grant
{rbtgi_repository}
Read Access*:
.. {rbtoe_iam_grant_project} for
{rbtgi_depot_project}
.. Add `serviceAccount:«RETRIEVER_EMAIL»` with role:
... `roles/artifactregistry.reader` (read-only access to all Artifact Registry repositories)
.. Policy version: 3
.. {rbbc_require}
IAM policy update successful
.. NOTE: This grant is project-scope (against the depot project's IAM policy), not resource-scope.  Project-scope IAM caches absorb the governor's `roles/owner` propagation effectively instantly, so this step does not need class C (HTTP 403) propagation tolerance — only class A/B (HTTP 400 member-visibility) coverage from the grant harness.  See <<rbscig_iam_grant_contracts>> propagation catalog.
