//axhob_operation
{rbtgo_director_divest}

NOTE: This operation tears down a
{rbtr_director}
{giam_service_account}
from the
{rbtgi_depot_project}.
The
{rbtr_governor}
synthesizes the SA email from the supplied identity folio, *revokes the SA's own IAM bindings across every scope it was granted*, then issues a REST DELETE against the IAM API, and opportunistically removes the production RBRA file if its `client_email` matches the synthesized email.
Revoking the live member before the delete is what keeps the binding leak closed: a member removed while still live leaves no `deleted:...?uid=` ghost, whereas deleting the SA first would have GCP retag its bindings as unbounded tombstones.
A required identity folio (param1) is supplied by the operator; the SA email is composed as `${RBCC_account_director}-<identity>@${RBGD_SA_EMAIL_FULL}`.

//axhos_step
. *Authenticate as
{rbtr_governor}*:
.. Execute
{rbtoe_governor_authenticate}
.. {rbbc_store}
`«GOVERNOR_TOKEN»` for subsequent API calls

//axhos_step
. *Synthesize Service Account Email*:
.. {rbbc_require}
identity folio (param1) is non-empty
.. {rbbc_store}
`«SA_EMAIL»` as `${RBCC_account_director}-<identity>@${RBGD_SA_EMAIL_FULL}` (composed from the role prefix and the operator-supplied identity)

//axhos_step
. *Revoke
{rbtr_director}
IAM Bindings*:
.. NOTE: Run before the DELETE so each live member is removed while the SA still exists — no `deleted:...?uid=` tombstone is minted on any resource. Mirrors exactly the bindings granted by {rbtgo_director_invest}; the {rbtr_mason} `roles/artifactregistry.writer` binding is Mason's, not the director's, and is left intact. See the member-removal contract in <<rbscig_iam_grant_contracts>>.
.. Remove `serviceAccount:«SA_EMAIL»` from the
{rbtgi_depot_project}
IAM policy, roles:
... `roles/cloudbuild.builds.editor`
... `roles/viewer`
... `roles/cloudbuild.workerPoolUser`
.. Remove `serviceAccount:«SA_EMAIL»` with `roles/iam.serviceAccountUser` from the
{rbtr_mason}
service account IAM policy
.. Remove `serviceAccount:«SA_EMAIL»` with `roles/artifactregistry.repoAdmin` from the
{rbtgi_repository}
IAM policy
.. Policy version: 3; a binding left with no members is pruned
.. {rbbc_fatal}
on revoke failure — teardown is fail-loud: a non-OK policy status or an HTTP 409 etag conflict (single-writer invariant) aborts the divest before the SA is deleted; the operator reruns. The {rbtgi_repository} and {rbtr_mason} revokes presume those standing resources exist (they outlive director SAs); a missing resource there surfaces only during a full depot teardown.
.. NOTE: Idempotent — removing an absent member is a no-op, so a re-divest revokes cleanly without error.

//axhos_step
. *Divest Director Service Account*:
.. {rbbc_call}
link:https://cloud.google.com/iam/docs/reference/rest/v1/projects.serviceAccounts/delete[iam.projects.serviceAccounts.delete] with:
... method: `DELETE`
... endpoint: `https://iam.googleapis.com/v1/projects/{rbtgi_depot_project}/serviceAccounts/«SA_EMAIL»`
... header: `Authorization: Bearer «GOVERNOR_TOKEN»`
.. {rbbc_require}
HTTP 200 response OR HTTP 404 (already divested)
.. {rbbc_warn}
if HTTP 404: "Service account not found (already divested)"

//axhos_step
. *Confirm Deletion Propagated*:
.. {rbbc_poll}
the SA GET endpoint until it returns HTTP 404 for `RBGC_GONE_CONFIRM_STREAK` consecutive observations, confirming the
{giam_service_account}
is durably gone. A delete-accepted response precedes propagation, and the post-delete read path flaps 404→200 across replicas (see <<rbscip_trade_study>>), so a single 404 is not durable proof — an immediately following same-name
{rbtgo_director_invest}
would otherwise race a still-visible account. Any intervening non-404 resets the streak.
.. NOTE: skipped when the DELETE returned HTTP 404 — the account was already absent, so there is nothing to wait for
.. {rbbc_require}
`RBGC_GONE_CONFIRM_STREAK` consecutive HTTP 404s within the eventual-consistency deadline

//axhos_step
. *Opportunistic RBRA Cleanup*:
.. If the role's RBRA file exists at
{rbrr_director_rbra_file},
source it and read `RBRA_CLIENT_EMAIL`
.. If `RBRA_CLIENT_EMAIL` equals `«SA_EMAIL»`, remove the file
.. NOTE: The removal is conditional — if the installed RBRA's `client_email` does not match the divested `«SA_EMAIL»` (e.g., a different identity is currently installed), the file is left in place. No fatal error is raised when the file is absent or non-matching.

//axhos_step
. *Display Result*:
.. {rbbc_show}
single-line success naming `«SA_EMAIL»` and confirming the
{rbtr_director}
{giam_service_account}
was divested, or a notice that it was already absent

//axhoc_completion
Successful execution revokes the
{rbtr_director}
{giam_service_account}'s IAM bindings across every scope it held (project,
{rbtr_mason},
{rbtgi_repository}) and then removes it from
{rbtgi_depot_project}, leaving no `deleted:` ghost on any policy.
If the installed RBRA at
{rbrr_director_rbra_file}
matches the divested identity, it is also removed; otherwise it is left intact.
A 404 response from the cloud DELETE is accepted as success — the
{giam_service_account}
is absent either way.
Proceed to
{rbtgo_director_invest}
to re-issue a fresh
{rbtr_director}
{giam_service_account}
when needed.
