//axhob_operation
{rbtgo_lode_ensconce}

NOTE: This operation captures a single upstream base image into a
{rbtga_lode}
— one
{gar_service}
package under the
{gar_lodes_namespace}
namespace, identified by a host-minted
{rbtga_touchmark}.
It is the bole kind: base images consumed as a Dockerfile FROM line.
Ensconce is the canonical capture skeleton — the remaining Lode kinds (tool, reliquary, wsl, podvm) reuse its shape, varying only member cardinality and trust grade.

NOTE: Acquisition runs cloud-side.
The
{rbtr_director}
submits a
{gcb_service}
job on the tethered (egress-permitted) pool, executed by the
{rbtr_mason}
service account via the metadata server.
The workstation never fetches the upstream bytes — it trusts the GCP environment over a potentially-compromised station.

NOTE: Ensconce is capture-pure.
It mints the Lode, emits the
{rbtga_lode_vouch}
provenance envelope, and reports the touchmark — but writes NO consumer config.
Election (where a consumer pins a touchmark) is a separate consumption-side layer that reads the
{rbf_fact_lode}
this operation writes.

NOTE: Ensconce captures one base per Lode per invocation.
A vessel declaring multiple base slots is dispatched once per base; a vessel
{rbrv_image_origin}
that names a local producer-vessel directory is a made-side hallmark-pin, not an upstream base, and is rejected.

The
{rbtr_director}
ensconces a base Lode:

//axhos_step
. *Capture Step (in-pool — rbgjl01)*:
.. Fetch the raw upstream manifest (`skopeo inspect --raw`) and compute its sha256 (`sha256sum` of the raw bytes) — the canonical OCI digest
.. {rbbc_store}
`«DIGEST»` — the manifest-list digest for multi-platform images, the single-manifest digest otherwise
.. Compose the unsprued glance fingerprint: sanitize the
{rbrv_image_origin}
(`:` and `/` to `-`) and append the first 10 hex chars of `«DIGEST»`
.. {rbbc_require}
the touchmark carries no conflicting capture (collision guard, before the copy): inspect the existing `:rbi_bole` handle on the Lode package — absent means a fresh touchmark and capture proceeds; present means its canonical digest must equal `«DIGEST»` (a Cloud Build retry re-copying the same bytes, which proceeds idempotently)
.. {rbbc_fatal}
if the existing `:rbi_bole` digest differs from `«DIGEST»` — a touchmark collision; the step fails loud rather than clobber the prior capture (banish the Lode first, or ensconce under a fresh touchmark). The guard is digest-aware, not naive fail-if-exists, so retries pass; it is co-located with the copy in this same step — atomic with the GAR write in that it admits no host pre-submit window
.. Copy the upstream image into the Lode package with `skopeo copy --all` (manifest list and all per-platform manifests) under the `:rbi_sha256-«DIGEST»` tag
.. Apply the remaining member tags by GAR→GAR retag (blobs deduplicate): `:rbi_bole` (uniform handle) and `:«SANITIZED-ORIGIN»-«SHA10»` (fingerprint)
.. Author the
{rbtga_lode_vouch}
envelope JSON (schema `rbld-vouch-1`, trust grade `verified-against-published`, single-element `members[]`) and stage it on the {rbtgr_workspace}
.. Write the per-slot result JSON to `buildStepOutputs`

//axhos_step
. *Vouch Step (in-pool — rbgjl02)*:
.. For the staged envelope, build a `FROM scratch` image copying `vouch.json` and push it to the Lode package under the `:rbi_vouch` tag — a distinct manifest sharing the one package

//axhos_step
. *Wait for Build Completion*:
.. {rbbc_poll}
link:https://cloud.google.com/build/docs/api/reference/rest/v1/projects.locations.builds/get[cloudbuild.builds.get] for `«BUILD_ID»`
.. {rbbc_require}
final status is `SUCCESS`

//axhoot_typed_output
{rbf_fact_lode}
. *Write Capture-Facts*:
.. Read `buildStepOutputs[0]` (base64-decoded JSON) from the completed build to recover the host-minted `«TOUCHMARK»`
.. {rbbc_store}
the touchmark value and the bole kind-brand enum as the two bare single-form facts of the
{rbf_fact_lode}
— carrying no provenance content; the envelope is the in-GAR `:rbi_vouch` tag, written cloud-side by the vouch step

//axhoc_completion
Successful execution captures the upstream base into a single
{gar_service}
package at `{gar_lodes_namespace}/«TOUCHMARK»`, carrying the base manifest under its digest, handle, and fingerprint tags plus the `:rbi_vouch`
{rbtga_lode_vouch}
envelope, and hands the touchmark forward host-side as the
{rbf_fact_lode}.
+
The operation is capture-pure: no vessel or nameplate config is written.
A consumer elects the touchmark separately by reading the capture-file — so election retries independently and survives, and a one-to-many fan-out iterates by reading the file rather than by a verb taking N consumer arguments.
+
Capture-fidelity (`recorded == upstream`) is established by construction — the same `skopeo inspect` that records the digest is the truth it would be checked against — so fidelity is a live/service-tier property, not an offline-checkable one.
Re-running against the same upstream digest reproduces the same canonical digest tag; a rotated upstream yields a different digest, and that is the capture surfacing the change, not a fault.
