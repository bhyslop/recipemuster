== System Overview

=== Core Trust Challenge

Modern development increasingly requires running foreign code with access to local work.
{st_container}-based development tools processing source code, local AI/ML tools analyzing proprietary data, and third-party services needing controlled network access create fundamental security challenges.

This creates a fundamental trust dilemma:

*
{st_container_s}
provide powerful tool isolation
* But they need controlled access to do useful work
* Standard
{st_container}
networking grants unfettered outbound access or else network services completely unavailable.

=== The Bottle Pattern

{at_rbm_system}
solves this by creating two simple, robust boundaries around a certain
{st_container}
type here referred to as a
{at_bottle_container}:

. File Access Boundary
*
{at_bottle_container}
sees only explicitly controlled set of
{st_volume_mount_s}
* No other access to
{at_workstation}
filesystem

. Network Boundary
*
{at_bottle_container}
has no direct network access
* All traffic flows through dedicated
{at_sentry_container},
a
{st_container}
configured with very tight network access and visibility controls.
* Only declared
{st_port_map_s}
and destinations allowed
* Full logging of allowed and denied traffic

Each
{rbsc_crucible}
maintains these boundaries through a three-container architecture:

* One
{at_sentry_container}
enforcing network policy on dual networks (bridge and enclave)
* One
{at_pentacle_container}
establishing and configuring the network namespace
* One
{at_bottle_container}
running the actual service, sharing the pentacle's network namespace via compose `network_mode: "service:pentacle"`
* Clear configuration in a single
{at_nameplate}
file, extended by per-nameplate compose fragments for volume mounts

Docker Compose orchestrates the container lifecycle declaratively.
A base compose file (`rbob_compose.yml`) defines the three-container topology with health check chains: the
{at_sentry_container}
must be healthy before the
{at_pentacle_container}
starts, and the
{at_pentacle_container}
must be healthy before the
{at_bottle_container}
starts.
This prevents firewall bypass — the bottle cannot send traffic before security policies are enforced.
Per-nameplate compose fragments (e.g., `«moniker».compose.yml`) add volume mount declarations.
Env-files passed via `--env-file` serve a dual purpose: YAML interpolation (`${VAR}` references in compose files resolve image refs, subnet, container names) and container environment forwarding (bare variable names in `environment:` sections inject values into containers at runtime).

The
{at_pentacle_container}
is a privileged container that configures routing to ensure all traffic flows through the
{at_sentry_container},
then shares this pre-configured namespace with the
{at_bottle_container}
via compose `network_mode: "service:pentacle"`.
This ensures security policies are enforced from the first packet, and the
{at_bottle_container}
experiences only a functional path to its
{at_sentry_container}
gateway

=== Security Properties

The
{at_rbm_system}
maintains following essential guarantees:

. File Access Control
*
{at_bottle_container_s}
can access only explicitly configured
{st_volume_mount_s}
* No access to broader system resources

. Network Isolation
* All traffic flows through dedicated
{at_sentry_container}
* Each permitted traffic flow maintains exactly one controlled pathway
* Precise ingress, forwarding, and NAT rules enforce strict boundaries
* Only explicitly configured
{st_port_map_s}
allow traffic between
{at_transit_network}
and
{at_enclave_network}
*
{at_bottle_container_s}
have no direct network access; all connectivity flows through
{at_sentry_container_s}
* Comprehensive traffic logging

. Service Separation
* Each
{rbsc_crucible}
fully isolated
* No cross-service communication
* Independent security policies
* Clean state separation

. Service Access Control
* External access to services only permitted when
{rbrn_entry_mode}
is enabled
* No inter-container communication allowed within pods
* All service access must originate from outside the pod
* Entry traffic flows: external -> pod ->
{at_sentry_container}
->
{at_bottle_container}

These properties hold across all operations and failure modes.

== Google API Engagement

{at_recipe_bottle}
invokes
{gcp_service}
APIs through
{giam_service_account}
JWT authentication and direct HTTPS REST calls.

* *Authentication*:
{giam_service_account_s}
use private key credentials to sign JWTs with RS256.
These JWTs are exchanged for
{oauth_token_s}
via Google's token endpoint.
All API calls include the bearer token in the Authorization header.

* *Invocation Pattern*:
Operations execute through curl with JSON payloads and responses.
Each call captures both HTTP status codes and response bodies to files for processing.
Success is determined by HTTP 200/201/204 codes.

* *Long-Running Operations*:
{gcp_service}
{gcp_lro_s}
return operation IDs that must be polled until completion.
The system polls at configurable intervals until completion.
The system handles both success and error states.
Operations utilize
{rbbc_submit}
and
{rbbc_await}
to handle all asynchronous operations.

* *IAM Policies*:
All IAM policy modifications enforce version 3 to ensure conditional role binding support across all
{gcp_service}
resources.

* *IAM Member Format*:
All IAM member bindings use `serviceAccount:«EMAIL»` format.
+
GCP IAM policy `members` fields only accept email-format identifiers (`serviceAccount:name@project.iam.gserviceaccount.com`).
The numeric `uniqueId` returned by `serviceAccounts.create` is valid for *API URL paths* (e.g., `projects/-/serviceAccounts/«UNIQUE_ID»` for GET/verification) but *not* for policy binding members.
`serviceAccount:«NUMERIC_ID»` returns HTTP 400 "Invalid service account" on all setIamPolicy endpoints — project, repository, and bucket alike.
+
Google's documentation recommends numeric IDs "to identify a service account just after it is created" (link:https://docs.google.com/iam/docs/manage-access-service-accounts[Manage access to service accounts]), but this applies to API resource addressing, not to IAM policy member strings.
The internal IAM system resolves email→numeric transparently; callers must use email.
+
Google-provisioned service agents (Cloud Build) are not affected — they are managed by Google, not newly created.

== Operational Invariants

=== Concurrent Session Safety

{at_recipe_bottle}
commands are designed for concurrent execution from multiple independent sessions against the same filesystem.
This enables parallel workflows where different developers or automation systems invoke commands simultaneously.

Concurrency safety is achieved through:

* *Ephemeral Isolation*:
Each command invocation creates a unique temporary directory using `BURD_NOW_STAMP` (timestamp + process ID + random component), ensuring intermediate files never collide.

* *Unique Artifact Naming*:
Cloud resources (build IDs, inscribe timestamps) incorporate unique identifiers, preventing overwrite conflicts in
{rbtgi_repository}.

* *Read-Only Git Queries*:
Git operations are non-mutating (status checks, metadata reads).
The `git fetch` operation uses git's internal locking for safe concurrent execution.

* *Semantic Preconditions*:
Operations like
{rbtgo_ark_conjure}
require clean git state.
This is a workflow constraint ensuring reproducible builds, not a collision guard.

NOTE: Concurrent builds of the same vessel produce independent images with different timestamps.
Both builds succeed; neither overwrites the other.

NOTE: The
{gcb_service}
private pool enforces a *Concurrent Build CPUs (Private Pool)* quota per project per region.
Machine type is set at pool creation time via
{rbrr_gcb_machine_type}
(a Compute Engine type such as `e2-standard-2`); individual build requests do not specify machine type.
If the pool's vCPU allocation is nearly exhausted, a second
build invocation will remain in `QUEUED` status while the first is `WORKING`.
The
{rbrr_gcb_min_concurrent_builds}
regime variable declares the operator's minimum concurrency requirement.
See
{rbtgo_quota_build}
for the manual Console procedure to review build capacity settings.

== Key Premises

[[rbsk_no_gcloud]]
//axl_voices axk_premise
{rbsk_no_gcloud}::
{at_recipe_bottle}
scripts MUST NOT invoke the `gcloud` CLI in local bash scripts or transitively through helper scripts.
All local
{gcp_service}
interaction uses direct REST API calls via `curl` with JWT-authenticated bearer tokens.
+
**Exception — vouch provenance fetch**: The {rbtgo_ark_vouch} verification step (conjure mode only) invokes `gcloud artifacts docker images describe --show-provenance` for SLSA provenance envelope retrieval.
This wraps complex Container Analysis API interaction that would be burdensome to replicate via raw REST.
All other GCB step operations use the GCE metadata server for OAuth token acquisition and direct registry REST API calls — no gcloud.
The provenance fetch runs as Python 3 on the gcloud builder image after the primary image is built; the reproducibility concern that motivates this premise is weaker for post-build verification.
Works under NO_PUBLIC_EGRESS (Private Google Access reaches Google APIs).
+
In GCB build step contexts (conjure, bind, graft, enshrine), registry authentication uses the GCE metadata server to obtain an access token and writes a static Docker config JSON.
+
Rationale: `gcloud` is an opaque, version-sensitive binary that obscures API semantics, introduces implicit authentication state, and breaks reproducibility.
The `gcloud auth configure-docker` approach also fails mechanically in GCB steps — the gcloud builder image lacks `docker` in PATH, so the credential helper it installs cannot function.
+
NOTE: The gcloud reliquary image exists because every ordain-path build step (conjure, bind, graft) uses it for the about/vouch pipeline (`bash`, `date`, and `gcloud` for provenance retrieval per the exception above).
The reliquary toolchain is mode-uniform: all ordain modes consume gcloud, docker, alpine, and syft from the reliquary; bind additionally uses skopeo for image-copy.

[[rbsk_in_pool_preflight]]
//axl_voices axk_premise
{rbsk_in_pool_preflight}::
Every ordain-path Cloud Build job — conjure's combined image+about, bind's combined mirror+about, graft's combined about+vouch, and enshrine — prepends a step 0 (`rbgjr01-reliquary-preflight.sh`) that HEAD-checks reliquary tool images from the worker pool's vantage before expensive work runs.
Conjure additionally HEAD-checks the vessel's enshrined base anchors (slots 1-3 from `RBRV_IMAGE_n_ANCHOR`); other modes pass empty locator substitutions.
+
Coverage is 5 of 6 reliquary tools — gcloud, docker, syft, binfmt, skopeo.
Alpine, the preflight step's own image, is validated implicitly by Cloud Build pulling it before the script runs.
Failure produces a structured "this pool can't see «TOOL»" message naming each missed item, with remediation directive (re-inscribe + re-yoke + re-ordain).
+
Rationale: defense-in-depth on top of the host-side `zrbfd_registry_preflight` (which validates the Director's host-side view of GAR).
The in-pool check catches divergence the host-side check cannot — airgap private-VPC routing, Director↔Mason IAM split, and submit↔execute time skew.
+
Scope boundary: standalone vouch (`zrbfv_vouch_submit`) and standalone about (`zrbfv_about_submit`) are operator-recovery paths and are not currently covered.

[[rbsk_invest_serialized]]
//axl_voices axk_premise
{rbsk_invest_serialized}::
{rbtgo_retriever_invest}, {rbtgo_director_invest}, and {rbtgo_governor_mantle} write credential bytes to a singleton assay slot (`RBDC_ASSAY_RBRA_FILE`), not a per-identity slot.
The operator MUST move `RBDC_ASSAY_RBRA_FILE` to a safe place before initiating the next operation that writes to the assay slot — typically the role's production slot (`RBDC_RETRIEVER_RBRA_FILE`, `RBDC_DIRECTOR_RBRA_FILE`, or `RBDC_GOVERNOR_RBRA_FILE`) when the invester and the consumer are the same operator, otherwise a delivery channel to the consumer.
Two consecutive assay-slot writers without an intervening move lose the first writer's bytes.
Batched invest is not supported — by design.
+
Rationale: a singleton assay slot keeps the producer-output discipline uniform across roles and frees the implementation from path-space per-identity routing.
The operator-driven move step is the deliberate promotion gate from producer output to role-production consumer; it also acts as the operator's explicit acknowledgement of which identity owns the role at runtime, since `RBRA` files encode exactly one identity at a time.
Move (not copy) prevents the assay slot from accumulating stale credential bytes after promotion.

[[rbtga_ark]]
//axl_voices axig_artifact_group
{rbtga_ark}::
The coherent unit of a
{at_recipe_bottle}
build output: a set of packages in the
{rbtgi_repository}
sharing a common
{rbtga_hallmark}
and rooted at the `{gar_hallmarks_namespace}/«HALLMARK»/` subtree.
An ark comprises the
{rbtga_ark_image}
(basename `image`) containing the container image, the
{rbtga_ark_about}
(basename `about`) containing build metadata and SBOM, the
{rbtga_ark_vouch}
(basename `vouch`) containing verification results, the
{rbtga_ark_pouch}
(basename `pouch`) containing the build context, and the
{rbtga_ark_diags}
(basename `diags`) containing build diagnostics.
+
An ark reaches the repository through one of three arrival paths:
+
* *Conjure*: A single `builds.create` {gcb_service} job builds the image (context delivered via {rbtga_ark_pouch}) and embeds about steps (via stitch), producing both the `image` and `about` packages. Then {rbtgo_ark_vouch} verifies SLSA provenance via DSSE envelope signature verification in a separate Cloud Build job.
* *Bind*: A single `builds.create` {gcb_service} job copies the pinned upstream image and runs about steps, producing both the `image` and `about` packages. Then {rbtgo_ark_vouch} records a digest-pin verdict in a separate Cloud Build job.
* *Graft*: Local docker/podman push places the `image` package. Then a single `builds.create` {gcb_service} job runs about steps followed by vouch steps, producing both `about` and `vouch` packages in one submission.
+
Conjure and bind each use two Cloud Build round-trips (primary job + vouch job). Graft uses one (combined about+vouch job), since the image arrives via local push rather than a Cloud Build step.
Vouch is mandatory: an ark without a vouch artifact is incomplete.
Arks are abjured atomically by
{rbtgo_ark_abjure}.

[[rbtga_hallmark]]
//axl_voices axtg_timestamp
{rbtga_hallmark}::
The identifier marking when a vessel was hallmarked into an
{rbtga_ark}.
Minted on the host before submission — the host passes the hallmark to Cloud Build via the `_RBGY_HALLMARK` substitution variable.
+
Mode prefixes and formats:
+
* `c` — conjure (Cloud Build build from Dockerfile): `c«YYMMDDHHMMSS»-r«YYMMDDHHMMSS»`
* `b` — bind (mirror pinned upstream image): `b«YYMMDDHHMMSS»-r«YYMMDDHHMMSS»`
* `g` — graft (local build pushed to GAR): `g«YYMMDDHHMMSS»-r«YYMMDDHHMMSS»`
* `k` — kludge (local build for development): `k«YYMMDDHHMMSS»-«GIT_CONTEXT»`
+
For conjure, bind, and graft: `«MODE»YYMMDDHHMMSS-rYYMMDDHHMMSS` where the `r` prefix means "realized" — the moment the image materialized in GAR.
Regex: `[cbg]\d{12}-r\d{12}`.
Example: `c260224153022-r260224160530` (27 characters).
+
T1 (mode-prefixed): when the operation was initiated — inscribe for conjure, mirror for bind, OCI image creation time for graft.
T2 (r-prefixed): when the image landed in GAR.
+
For graft, T1 is an intrinsic fact about the image (its `created` metadata), not an operational timestamp.
Reproducible builds may produce `T1=700101000000` — this is a true statement, not an error.
+
For kludge, the hallmark uses a git-context suffix instead of a realized timestamp, since no GAR push occurs.
Kludge hallmarks are local-only and never appear in the {rbtgi_repository}.
+
Keyed into complete package references:
+
[cols="3,1,5"]
|===
|Reference |Lifecycle |Purpose

|`{gar_hallmarks_namespace}/«HALLMARK»/image:«HALLMARK»`
|Durable
|Consumer multiplatform manifest list (from `buildx --push`), or single-platform image

|`{gar_hallmarks_namespace}/«HALLMARK»/about:«HALLMARK»`
|Durable
|SBOM + build info

|`{gar_hallmarks_namespace}/«HALLMARK»/vouch:«HALLMARK»`
|Durable
|SLSA verification record

|`{gar_hallmarks_namespace}/«HALLMARK»/diags:«HALLMARK»`
|Durable
|Build diagnostics

|`{gar_hallmarks_namespace}/«HALLMARK»/pouch:«HALLMARK»`
|Durable
|Build context (FROM SCRATCH OCI image in GAR)

|`{gar_hallmarks_namespace}/«HALLMARK»/attest:«HALLMARK»-«ARCH»`
|Durable
|Per-platform provenance-carrying images pushed via CB `images:` field (one package, per-platform tags)
|===
+
The `attest` tags satisfy the {rbtgr_build_json} `images:` field for SLSA provenance generation.
Provenance is keyed by digest in the Container Analysis API, but the per-platform `attest` tags are the only objects carrying those digests — the classic Docker image store re-serializes manifests on pull/push, producing different digests than the buildx-native manifests in the `image` package.
The tags are durable ark artifacts, persisting alongside `image`, `about`, `vouch`, `pouch`, and `diags` under the hallmark subtree.
+
Multiple conjures produce distinct hallmarks (unique realized timestamps) with independently-attested images.
+
Hallmark-only lookup falls out of the layout: because every ark's packages live under `{gar_hallmarks_namespace}/«HALLMARK»/`, {rbtgo_ark_inspect} resolves an ark from a bare hallmark by navigating the subtree directly — no cross-vessel aggregator tag is needed.
Look up `{gar_hallmarks_namespace}/«HALLMARK»/vouch:«HALLMARK»`, read `vouch_summary.json` to discover the originating vessel, then fetch `about` from the same subtree.

[[rbtga_ark_vouch]]
//axl_voices axig_verification_artifact
{rbtga_ark_vouch}::
The mandatory verification artifact within an
{rbtga_ark},
the `vouch` package under the hallmark subtree (`{gar_hallmarks_namespace}/«HALLMARK»/vouch:«HALLMARK»`).
Created by
{rbtgo_ark_vouch}
as a
{gcb_service}
job submitted after
{rbtgo_ark_about}
completes.
The vouch verdict depends on vessel mode:
+
* *Conjure*: DSSE envelope signature verification of GCB-signed provenance against each per-platform digest. Records per-platform verification verdicts.
* *Bind*: Digest-pin comparison — verifies the image digest in GAR matches the pinned upstream reference. No SLSA provenance exists for upstream images.
* *Graft*: GRAFTED stamp — records that the image was locally built and pushed without provenance chain. An explicit trust boundary marker.
+
The vouch artifact is a multi-platform `FROM scratch` container holding `vouch_summary.json` and per-platform `verify-«SUFFIX».json` files (conjure) or a single `vouch_summary.json` (bind/graft).
Content is architecture-independent metadata; multi-platform is for pull ergonomics (no platform mismatch warning on arm64 hosts).
+
**Vouch depends on about**: {rbtgo_ark_vouch} gates on `about` package existence before submission.
The about pipeline must complete before vouch can run.
+
**DSSE envelope verification** (conjure mode only, ₣Av):
Provenance signatures verified against embedded GCB attestor public keys from `projects/verified-builder` KMS.
Step 1 (Alpine) writes the attestor public key to `/workspace/keys/`.
Step 2 (gcloud/Python 3) fetches provenance via `gcloud artifacts docker images describe --show-provenance`, extracts the v1.0 DSSE envelope, decodes base64url payload + signature, reconstructs PAE, and verifies via `openssl dgst -sha256 -verify` subprocess against the embedded attestor key.
No slsa-verifier, no jq — Python `json` + `base64` modules + openssl subprocess only.
+
**Vouch Cloud Build job architecture**:
Vouch uses the RunBuild API (`builds.create`) rather than trigger dispatch — the hallmark is known only after the image arrives in GAR, so a pre-registered trigger cannot provide it.
The build runs as the {rbtr_mason} service account.
The vouch step branches on `_RBGV_VESSEL_MODE`: conjure runs DSSE provenance verification, bind runs digest-pin comparison, graft stamps GRAFTED.
+
Discovered constraints from implementation:
+
  * **Private worker pool**: The RunBuild API caller ({rbtr_director}) requires `cloudbuild.workerpools.use` to specify a private pool, a permission not included in `roles/cloudbuild.builds.editor`.
    {rbtgo_director_invest} grants `roles/cloudbuild.workerPoolUser` at project scope.
    Vouch runs on the same private pool as conjure (the depot's worker pool, derived from `(RBRR_CLOUD_PREFIX, RBRR_DEPOT_MONIKER)`), eliminating an untested default-pool-with-buildx permutation.
  * **gcloud exception**: Vouch verification (conjure mode) invokes `gcloud artifacts docker images describe` for provenance retrieval (see {rbsk_no_gcloud} exception).
    All other steps use the GCE metadata server for OAuth tokens and direct registry REST API calls.
  * **Builder image selection**: Step 1 (Alpine) writes attestor keys (conjure) or is a no-op (bind/graft).
    Step 2 (gcloud/Python 3) runs mode-aware verification.
    Step 3 (Docker) assembles and pushes the vouch container (`docker buildx build --push` for multi-platform).
  * **GCB dollar escaping**: Step scripts embedded in Build JSON require `$$` for literal `$` and un-escaped `${_RBGV_*}` for GCB substitution variables, following the same `sed` pattern as conjure's `${_RBGY_*}`.
  * **DSSE verification**: Provenance envelope signatures verified against embedded GCB attestor public keys (`projects/verified-builder` KMS). PAE reconstruction + `openssl dgst -sha256 -verify` subprocess. No slsa-verifier, no jq — Python 3 standard library only.
    builds.create provenance achieves SLSA Build Level 3 (₢AvAAB research confirmed).
+
**Idempotent re-vouch**:
Re-running {rbtgo_ark_vouch} on a hallmark that already has a `vouch` package re-executes verification.
For conjure and bind, the new vouch content should match the existing vouch (same digests + same provenance/pin + same verifier = identical output).
For graft, the GRAFTED stamp is always identical.
A mismatch for conjure/bind indicates provenance change or tampering and must produce a loud alarm.
If determinism proves unreliable in practice, narrow the comparison to verification verdict, digests, and builder identity.
+
**Vouch provenance resolution**: The vouch verifier resolves per-platform digests from `attest:«HALLMARK»-«ARCH»` tags (HEAD request per tag) rather than from the `image` manifest index.
The per-arch `attest` digests are the GCB-attested digests; the `image` manifest's per-platform digests are buildx-native and carry no GCB provenance.
+
**Abjure**: {rbtgo_ark_abjure} enumerates every package under `{gar_hallmarks_namespace}/«HALLMARK»/` and deletes each — `image`, `about`, `vouch`, `pouch`, `diags`, and (for conjure vessels) `attest`. Deleting the single `attest` package removes all its per-arch tags.
+
**Display**: {rbtgo_hallmark_tally} classifies each hallmark as `vouched`, `pending`, or `incomplete` by basename presence under the hallmark subtree (`vouched` = image+about+vouch all present; `pending` = image+about without vouch; `incomplete` = any other combination).

[[rbtgr_provenance]]
//axl_voices axig_slsa_provenance
{rbtgr_provenance}::
SLSA v1.0 build provenance attestation auto-generated by Google for `builds.create` submissions.
Records build inputs, environment, and outputs as a cryptographically signed attestation on the built image.
+
Requires two conditions in the
{rbtgr_build_json}:
(1) `options.requestedVerifyOption: VERIFIED` — opts private pool builds into provenance generation (private pools default to no provenance);
(2) the top-level `images:` field must list the image URI — Cloud Build generates provenance only for images it pushes via this field.
Cloud Build pre-populates Docker credentials for all GAR regions, so explicit docker-login steps are unnecessary.
+
**Single-arch vessels**: `buildx --load` places the built image into the local Docker daemon's store.
The `images:` field in {rbtgr_build_json} causes Cloud Build to push the image and generate SLSA v1.0 Level 3 provenance.
Verified on builds.create submissions — rbev-busybox on depot10030 (3 platforms: amd64, arm64, armv7) with DSSE envelope signature verification (₣Av ₢AvAAD).
+
**Single-arch pipeline** (3 steps + `images:` push, single build invocation):
+
  1. `derive-tag-base` — receive host-minted {rbtga_hallmark} via `_RBGY_HALLMARK`, write to `.hallmark` + `/builder/outputs/output`
  2. `qemu-binfmt` — register cross-architecture emulation (skipped if native platform)
  3. `build-and-load` — `buildx --load` into local Docker daemon (single platform)
+
About assembly (syft, build_info, metadata push) uses reusable {rbtgo_ark_about} scripts (`rbgja/`) embedded in the stitched JSON alongside the build steps.
+
**Multi-platform vessels**: production-validated architecture.
Docker's local store (classic image store on CB workers, Docker 20.10.24) is single-platform only — `buildx --load` cannot load multi-platform images.
The validated approach uses per-platform pullback after `buildx --push`:
`buildx --push` builds all platforms and pushes the consumer-facing `image` manifest list directly to GAR;
`docker pull --platform «ARCH»` pulls each platform variant back to the local daemon (requires Experimental: true, confirmed on CB workers);
each pulled image is tagged as `{gar_hallmarks_namespace}/«HALLMARK»/attest:«HALLMARK»-«ARCH»`;
the `images:` field declares all per-arch `attest` tags — Cloud Build pushes each and generates independent SLSA v1.0 Level 3 provenance with the same `buildInvocationId`, proving single-build origin.
+
The per-arch `attest` tags are **ephemeral attestation scaffolding**: they exist solely to bridge `buildx --push` output to the `images:` field for SLSA provenance generation.
Cloud Build has no awareness of the `buildx --push` to GAR — the `images:` field is CB's only mechanism for pushing and attesting images.
Provenance is keyed by digest in the Container Analysis API, not by tag.
The host sweeps per-arch `attest` tags after vouch completes using {rbtr_director} credentials (repoAdmin); the {rbtr_mason} lacks delete permission.
Deleting these tags does not delete provenance — the digests referenced by the `image` manifest list retain their attestations.
+
Validated in experiments and production on depot `demo1025`:
per-platform provenance (`b3fd60c7`) — all 3 platforms (amd64, arm64, armv7) achieved SLSA Level 3 with shared `buildInvocationId`;
production e2e (`2e172ce0`) — build of `rbev-busybox` (3 platforms) achieved SLSA Level 3 on all per-platform images with shared `buildInvocationId`, consumer-facing manifest list resolved correctly.
The manifest list itself has `slsa_build_level: unknown` (expected — Cloud Build did not build the manifest list artifact), but consumers pulling the `image` reference get transparent platform resolution to attested per-platform digests with full SLSA Level 3.
+
Implementation notes:
`docker pull --platform` works on CB workers (Docker 20.10.24, Experimental: true);
`$${VAR}` escaping required for shell variables in {rbtgr_build_json} (Cloud Build interprets `${VAR}` as substitution variables);
`jq` is not available in `gcr.io/cloud-builders/docker` — the `--platform` flag approach avoids JSON manifest parsing entirely.
See `Memos/memo-20260305-provenance-architecture-gap.md` for full experiment details.
+
**Multi-platform pipeline** (6 steps + `images:` push, single build invocation):
+
  1. `derive-tag-base` — receive host-minted {rbtga_hallmark} via `_RBGY_HALLMARK`, write to `.hallmark` + `/builder/outputs/output`
  2. `qemu-binfmt` — register cross-architecture emulation (if needed)
  3. `buildx --push` — build all platforms, push consumer-facing `image` manifest list to GAR under `{gar_hallmarks_namespace}/«HALLMARK»/image:«HALLMARK»`
  4. `per-platform-pullback` — `docker pull --platform «ARCH»` from the `image` package, `docker tag` to `{gar_hallmarks_namespace}/«HALLMARK»/attest:«HALLMARK»-«ARCH»`
  5. `push-per-platform` — push each per-arch `attest` tag to GAR
  6. `push-diags` — diagnostics artifact
+
About assembly (syft, build_info, metadata push) uses reusable {rbtgo_ark_about} scripts (`rbgja/`) embedded in the stitched JSON alongside the build steps.
+
`images:` lists the per-arch `attest` tags — Cloud Build pushes each (idempotent if already present from step 5) and generates SLSA provenance.
`options.requestedVerifyOption: VERIFIED` ensures build fails if provenance cannot be generated.
The consumer-facing `image` package (produced by `buildx --push` in step 3) is platform-transparent, identical to single-arch.

Peer to {rbsc_crucible}, which handles local runtime containment.

{at_operator_s}

=== Local Architecture Definitions

[[at_sessile_service]]
{at_sessile_service}::
A long-running
{rbsc_crucible}
that charges via
{rbsc_charge}
and persists until explicitly quenched.
The sessile pattern is used for services that maintain state and provide ongoing functionality.

[[at_agile_service]]
{at_agile_service}::
A transient
{rbsc_crucible}
that executes via
{rbsc_enjoin}
and terminates when its command completes.
The agile pattern is used for one-shot operations that produce output and exit.

[[rbsc_crucible]]
{rbsc_crucible}::
A complete service instance consisting of one
{at_sentry_container},
one
{at_pentacle_container},
and one
{at_bottle_container}
working together to provide secure functionality.
Orchestrated by Docker Compose with a base compose file (`rbob_compose.yml`) defining the three-container topology and health check chains, plus optional per-nameplate compose fragments for volume mounts.
Each service maintains strict file system and network boundaries while enabling controlled access to resources.

[[rbsc_pentacle]]
{rbsc_pentacle}::
The privileged container within a
{rbsc_crucible}
that establishes authority over the contained network space.
Shares the
{at_sentry_image}
and configures routing so all traffic flows through the
{at_sentry_container}.
The
{at_bottle_container}
shares its network namespace via compose `network_mode: "service:pentacle"`.

[[rbsc_sessile]]
{rbsc_sessile}::
A
{rbsc_crucible}
service mode where the bottle container runs persistently.
Charged via
{rbsc_charge}
and persists until explicitly quenched.
Used for services that maintain state and provide ongoing functionality.

[[rbsc_agile]]
{rbsc_agile}::
A
{rbsc_crucible}
service mode where transient bottles are dispatched into an already-running sentry and pentacle.
Each invocation via
{rbsc_enjoin}
creates an ephemeral bottle that terminates when its command completes.
Used for one-shot operations that produce output and exit.

[[rbsc_quench]]
{rbsc_quench}::
Tear down a
{rbsc_crucible},
removing all containers and networks for the nameplate's compose project.

[[rbsc_hail]]
{rbsc_hail}::
Diagnostic operation: open an interactive shell into the
{at_sentry_container}
of a running
{rbsc_crucible}.
Call out to the guard to inspect security enforcement state.

[[rbsc_rack]]
{rbsc_rack}::
Diagnostic operation: open an interactive shell into the
{at_bottle_container}
of a running
{rbsc_crucible}.
Compel the demon to reveal its state.

[[rbsc_scry]]
{rbsc_scry}::
Diagnostic operation: observe network traffic on a running
{rbsc_crucible}
via tcpdump.
Solomonic divination — see through the veil to the hidden topology.

[[at_sentry_container]]
{at_sentry_container}::
A privileged container that enforces network security policies for a
{rbsc_crucible}.
It connects to both
{at_transit_network}
and
{at_enclave_network},
controlling all network traffic and providing DNS services.
The sentry discovers its network interfaces by IP at startup using `ip -o addr show to «enclave_ip»`, not by hardcoded interface name, because Docker Compose does not guarantee network-to-interface ordering.
Initialization and iptables configuration scripts are baked into the image at build time (zero volume mounts for security containers).

[[at_pentacle_container]]
{at_pentacle_container}::
A privileged container sharing the
{at_sentry_image}
that configures routing to ensure all traffic flows through the
{at_sentry_container}.
The
{at_bottle_container}
shares its network namespace via compose `network_mode: "service:pentacle"`, ensuring security policies are enforced from the first packet.
Initialization scripts are baked into the sentry image at build time (zero volume mounts for security containers).
The pentacle discovers its network interface by IP at startup using `ip -o addr show`, not by hardcoded interface name, because Docker Compose does not guarantee network-to-interface ordering.

[[at_bottle_container]]
{at_bottle_container}::
A container that runs the actual service functionality, inheriting its network namespace from the
{at_pentacle_container}.
It is connected only to the
{at_enclave_network}
and configured with specific
{st_volume_mount_s}.
All external communication flows through its associated
{at_sentry_container}.

[[at_enclave_network]]
{at_enclave_network}::
An isolated network connecting a
{at_bottle_container}
to its
{at_sentry_container},
using
{rbrn_enclave_base_ip}/{rbrn_enclave_netmask}
for addressing.
This network enforces strict security policies and provides the only network communication path for the
{at_bottle_container}.

[[at_transit_network]]
{at_transit_network}::
The network interface that connects a
{at_sentry_container}
to the
{at_workstation_p}
network resources.
This network enables controlled external communication while maintaining security boundaries.

[[at_nameplate]]
{at_nameplate}::
A configuration file that defines all aspects of a
{rbsc_crucible},
including its images, network settings, and security policies.
Each nameplate provides a complete service specification that the
{at_rbm_system}
can instantiate.

{rbrr_gcb_machine_type}::
Compute Engine machine type for the private worker pool, consumed at pool creation time by
{rbtgo_depot_levy}.
Default: `e2-standard-2`.
The pool resource holds this value; individual build requests do not specify machine type.

NOTE: *Private pool is always required.*
Private pools are a prerequisite for VPC Service Controls and `NO_PUBLIC_EGRESS` hardening (see `RBSCB-CloudBuildPosture.adoc`).
Pricing is region-dependent: in us-central1/us-east4, private pool e2-standard-2 matches the default pool rate ($0.006/build-minute); in other regions the private pool rate is up to 1.4× higher.
This cost is accepted because the security prerequisite is non-negotiable and maintaining two code paths (conditional pool vs. default) doubles configuration surface.
+
Cloud Build supports two distinct machine type naming systems:
the default shared pool uses Cloud Build enum values (`E2_HIGHCPU_8`, `N1_HIGHCPU_32`) in `options.machineType` of the build JSON;
private pools use Compute Engine instance type names (`e2-standard-2`, `n2d-standard-4`, `c3-standard-8`) in `workerConfig.machineType` at pool creation.
Valid private pool families: `e2`, `n2d`, `c3` (availability varies by region).
Since Recipe Bottle uses only private pools, only the Compute Engine format is valid for
{rbrr_gcb_machine_type}.

== Orchestration Patterns

Recipe Bottle orchestration patterns define reusable implementation sequences for system operations.

[[rbtoe_rbra_generate]]
//axvo_procedure axd_transient axd_internal
{rbtoe_rbra_generate}::
A pattern that converts Google
{giam_service_account}
credentials into
{rbtgi_rbra_file}
format:
extracts `client_email`, `private_key`, and `project_id` from base64-encoded key data or JSON structure,
converts to RBRA environment variables (`RBRA_CLIENT_EMAIL`, `RBRA_PRIVATE_KEY`, `RBRA_PROJECT_ID`),
and writes to specified file path with 600 permissions (owner read/write only).

[[rbtoe_rbra_load]]
//axvo_procedure axd_transient axd_internal
{rbtoe_rbra_load}::
A pattern that sources an
{rbtgi_rbra_file}
and validates required fields:
sources the specified RBRA file into the current environment,
validates that `RBRA_CLIENT_EMAIL`, `RBRA_PRIVATE_KEY`, and `RBRA_PROJECT_ID` are all present and non-empty,
and triggers
{rbbc_fatal}
with standardized error message if any field is missing or invalid.

[[rbtoe_jwt_oauth_exchange]]
//axvo_procedure axd_transient axd_internal
{rbtoe_jwt_oauth_exchange}::
A pattern that exchanges
{giam_service_account}
RBRA credentials for an
{oauth_token}:
constructs JWT with RS256 signature using RBRA_PRIVATE_KEY,
sets JWT claims (iss=RBRA_CLIENT_EMAIL, aud=https://oauth2.googleapis.com/token, scope=https://www.googleapis.com/auth/cloud-platform),
POSTs to OAuth2 token endpoint,
returns access_token from response,
triggers
{rbbc_fatal}
on non-200 HTTP response.

[[rbtoe_api_enable]]
//axvo_procedure axd_transient axd_internal
{rbtoe_api_enable}::
A pattern that ensures a specified Google Cloud API is enabled in a project with idempotent behavior:

. attempts to enable the API via serviceusage.services.enable,
. tolerates HTTP 200 (success) or HTTP 400 with "already enabled" error,
. executes
{rbbc_await}
if Long Running Operation returned,
. verifies API state is ENABLED via serviceusage.services.get,
. triggers
{rbbc_fatal}
on unexpected errors (409 conflicts, permission denied, etc.),
. returns success when API is confirmed enabled.


[[rbtoe_oauth_refresh]]
//axvo_procedure axd_transient axd_internal
{rbtoe_oauth_refresh}::
A pattern that exchanges OAuth refresh token for access token:

. loads RBRO_REFRESH_TOKEN and RBRO_CLIENT_SECRET from
{rbro_regime},
. loads RBRP_OAUTH_CLIENT_ID from
{rbrp_payor_project_id}
configuration,

[[rbtoe_depot_list_update]]

A pattern that synchronizes
{rbtgi_depot}

. executes

. {rbbc_call} Cloud Billing v1 `billingAccounts/${RBRP_BILLING_ACCOUNT_ID}/projects` to enumerate projects with `billingEnabled=true`,

. classifies state: `ACTIVE` → `COMPLETE`, `DELETE_REQUESTED` → `DELETE_REQUESTED`; other states are logged and skipped,
. emits per-moniker fact files: `«MONIKER».depot` (state) and `«MONIKER».depot-project` (canonical project_id),
. invoked by
{rbtgo_payor_install},
{rbtgo_depot_levy},
