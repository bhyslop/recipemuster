# Paddock: gcb-trigger-migration-tier2

## Context

Migrate the Recipe Bottle build pipeline from ad-hoc `builds.create` API calls
to Cloud Build trigger-based builds. This is Tier 2 of the RBSCB Cloud Build
Roadmap, selected as MVP-required for SLSA v1.0 provenance.

### The Rubric Concept

A **Rubric** (`rbtgr_rubric`) is the composite per-vessel build definition comprising:
- **Rubric YAML** (`rbtgr_build_yaml`) — committed `cloudbuild.yaml` per vessel,
  generated from step scripts + regime pins
- **Rubric Trigger** (`rbtgr_vessel_trigger`) — per-vessel Cloud Build trigger resource in GCP
- **Rubric Provenance** (`rbtgr_provenance`) — SLSA v1.0 attestation auto-generated by Google
  for trigger-invoked builds

The rubric lifecycle operations:
- **Inscribe** (`rbtgo_rubric_inscribe`) — generate the build yaml from step scripts + pins
- **Assay** (`rbtgo_rubric_assay`) — validate freshness (diff committed vs regenerated)
- **Enshrine** (`rbtgo_rubric_enshrine`) — create the Cloud Build trigger resource (one-time)
- **Dispatch** (`rbtgo_trigger_build`, revised) — execute triggers.run with substitution overrides

Developer Connect provides the bridge from GitHub to GCB:
- **Source Connection** (`rbtgi_source_connection`) — the Developer Connect connection
- **Source Link** (`rbtgi_source_link`) — the git repository link binding

### Key Design Decisions

- **Main repo as trigger source** — triggers connect to the existing GitHub repo via
  {rbtgi_source_connection}, not per-vessel repos. Eliminates Courier PAT, vessel repo
  management, and publish ceremony. {rbtgr_provenance} records the main repo commit hash —
  acceptable for alpha.
- **GitHub as host** (Developer Connect + OAuth GitHub App, no CLI dependencies, no PAT)
- **Per-vessel {rbtgr_build_yaml}** — committed to the main repo, generated-artifact-must-be-committed
- **Single-phase build**: Foundry runs {rbtgo_rubric_assay}, then dispatches via triggers.run
- **Dockerfiles stay in parent repo** — alongside their {rbtgr_build_yaml} in vessel directories
- **Phased rollout** — rbev-busybox first, verify {rbtgr_provenance}, then migrate remaining vessels
- **Tier 1 log hygiene** folded into YAML authoring (audit during script inlining)
- **All API calls via curl/REST** — consistent with existing Foundry pattern, no gcloud/gh dependency
- **One {rbtgi_source_connection}** to the main repo serves all vessel triggers
- **rb*-prefixed linked terms** with AXLA voicing annotations — not google-domain prefixes
  (legacy gcb_*/gar_*/gcs_*/giam_*/gcp_* cleanup deferred to ₣Aj)
- **OCI Layout Bridge pattern** survives unchanged (build/push/SBOM phase split)

### Decision Rationale

**Why GitHub (not GitLab, CSR, or SSM)?**
Evaluated four options. Cloud Source Repositories is End of Sale (June 2024) — unavailable
to new customers. Secure Source Manager requires 100-user minimum at $10/user = $1,000/month —
enterprise pricing, non-starter. GitLab was more automatable (PAT-only, no OAuth browser step)
but GitHub was chosen because the project already lives there. GitLab remains a viable
future migration if GitHub relationship deteriorates.

**Why main repo triggers (not per-vessel repos)?**
The original design used per-vessel GitHub repos for provenance isolation: one commit = one
vessel's build definition. But this requires a Courier PAT regime, N repo creation/management
scripts, a publish ceremony, and pin-refresh integration per vessel — substantial infrastructure
for alpha. With main repo triggers: Developer Connect uses OAuth GitHub App (one-time browser
auth, no PAT lifecycle), triggers.run dispatches against the main repo commit, and Google
still generates {rbtgr_provenance} recording the exact commit hash. The provenance commit
may include non-vessel changes, but for a single-author alpha project this is acceptable.
Per-vessel repos remain a future option if provenance isolation becomes a real requirement.

**Why GDC prefix for Developer Connect regime variables?**
Developer Connect is a Google service, not a GitHub concept. RBRR regime variables use
the GDC prefix (RBRR_GDC_*). The linked terms for the resources themselves use rbtgi_*
(RB-specific instances) with AXLA voicing annotations back to axig_* motifs.

**Why "Rubric"?**
The build definition per vessel needed a name. The metaphor chain is Vessel → Foundry → Ark.
A rubric is "the written instructions that tell the Foundry how to transform a vessel into
an ark." The lifecycle verbs (inscribe, assay, enshrine) write themselves from the metaphor.

### Constraints

**Build context scope**: With main repo triggers, Cloud Build checks out the entire repo,
not a minimal tarball. The builder sees all files. Acceptable since the repo contains no
secrets and Dockerfile/context paths are parameterized via substitution variables.

**Developer Connect setup**: Requires one-time interactive browser OAuth to authorize the
Google Developer Connect GitHub App on the main repo. Not scriptable end-to-end — the
human must complete the GitHub authorization flow once. Subsequent trigger operations are
fully automated via REST API.

### What Dies

- `zrbf_stitch_build_json()` and the rbgjb/*.sh script files
- GCS tarball upload chain (package, compose name, upload)
- `zrbf_compose_build_request_json()` and `builds.create` API path
- The `$$` dollar-escaping dance for Cloud Build substitutions
- Per-vessel GitHub repos concept (no vessel repos needed)
- Courier credential regime concept (no GitHub PAT needed)

### What's Born

- {rbtgr_rubric} — per-vessel build definition (yaml + trigger + provenance)
- {rbtgr_build_yaml} — per-vessel cloudbuild.yaml with inline step scripts (committed to main repo)
- {rbtgi_source_connection} — Developer Connect connection to main GitHub repo (OAuth GitHub App)
- {rbtgr_vessel_trigger} — per-vessel Cloud Build trigger pointing to main repo
- {rbtgo_trigger_build} revised — dispatch via triggers.run API
- {rbtgr_provenance} — SLSA v1.0 attestation on all trigger-built images
- rbtgo_rubric_inscribe, rbtgo_rubric_assay, rbtgo_rubric_enshrine — lifecycle operations
- AXLA motifs: axig_developer_connect, axig_build_trigger, axig_repo_link, axig_build_config, axig_slsa_provenance

### Vessel Modes

Currently implementing conjure mode only. Repo structure designed to accommodate:
- **Conjure**: {rbtgr_build_yaml} + Dockerfile + context (build from scratch)
- **Mirror**: {rbtgr_build_yaml} + manifest (pull public, push to GAR) — future
- **Local push**: {rbtgr_build_yaml} + OCI tarball reference — future

## Build Requirements

This heat is primarily bash/GCP tooling (RBW). No Rust builds required for most paces.
BCG compliance is mandatory for all new bash code — enterprise-grade orchestration.
