1014-
1015-The prohibition above is specific to error-handling blocks where `exit`/`return` must reach the calling shell. Subshells are legitimate in these contexts:
1016-
1017-- **Test execution** — principled isolation between suites and cases; see **Test Execution Patterns**

1020-
1021-#### Isolation Subshells (environment containment)
1022-
1023-When a sequence of commands mutates the shell environment (sourcing files, `cd`, venv activation, `export`), a subshell prevents those mutations from leaking into subsequent commands. The exit status propagates across the `)` boundary, so the caller can detect and handle failures.
--
1027-**Two variants** — same principle, different caller handling:
1028-
1029-```bash
1030-# Variant 1: Error propagation — die on any internal failure
1031-(
1032:  source "${z_config_file}" || buc_die "Failed to source: ${z_config_file}"
1033:  «prefix»_operation || buc_die "Operation failed"
1034:) || buc_die "Isolation subshell failed"
1035-
1036-# Variant 2: Status capture — caller inspects and decides
1037-local z_status=0
1038-(
1039:  source "${z_config_file}" || buc_die "Failed to source: ${z_config_file}"
1040:  «prefix»_operation || buc_die "Operation failed"
1041-) || z_status=$?
1042-```
1043-
1044:Inside the subshell, `buc_die` calls `exit` which terminates only the subshell — producing a non-zero exit status that the outer `|| buc_die` or `|| z_status=$?` catches. Failure propagates through exit status, not through `exit` reaching the parent.
1045-
1046-**Constraints:**
1047:- Every command inside must have explicit `|| buc_die` — `set -e` is suppressed inside `( ... ) ||` (as with all BCG code, never rely on it)
1048:- The outer boundary must have `|| buc_die` or `|| z_status=$?` — never leave the subshell exit status unchecked
1049-- Sourcing inside the subshell is permitted (this is a primary use case — isolation prevents sourced variables from leaking)
1050-- Keep subshell bodies short and focused — if the body grows complex, delegate to a CLI script via `exec`
